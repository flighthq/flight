# Package Map

Full per-package descriptions for the Flight SDK. Consult this when you need to understand a specific package's scope, API surface, or relationship to neighbors. For quick navigation, the compact list is in [index.md](../index.md#package-map).

Run `npm run api <name>` to query exported function signatures directly.

## Core and Primitives

- `@flighthq/types`: shared interfaces, kind symbols, and cross-package type contracts. This is the codebase's header layer — all public API shapes live here.
- `@flighthq/entity`: entity/runtime primitives used by higher-level packages.
- `@flighthq/geometry`: rectangles, vectors, matrices, typed-array capacity helpers, and pools.
- `@flighthq/math`: scalar math utilities — constants (EPSILON, TAU, DEG_TO_RAD), clamping/saturation, interpolation (lerp, inverseLerp, remap, smoothStep, smootherStep, damp, moveTowards, pingPong, repeat, lerpAngle), angles (degToRad, radToDeg, normalizeAngle, deltaAngle), rounding/quantization (roundTo, floorTo, ceilTo, fract, euclideanMod), comparison (approxEqual, approxEqualRelative, approxZero), power-of-two family (nextPowerOfTwo, previousPowerOfTwo, isPowerOfTwo, ceilPowerOfTwo, floorPowerOfTwo, nextMultipleOf), number theory (gcd, lcm, factorial, isEven, isOdd, hypot2, sign, quantize), seeded random and convenience helpers (randomRange, randomInt, randomBool, randomSign), random distributions (randomGaussian, randomGaussianPair, randomOnUnitCircle, randomInsideUnitDisc, randomOnUnitSphere, pick, shuffle, shuffleInPlace, randomWeighted), deterministic hashing (hashUint32, hashCombine, hash2D, hash3D, createRandomSourceFromHash), and statistics (mean, median, variance, standardDeviation, weightedAverage). Pure free functions, no allocation, tree-shakable.
- `@flighthq/node`: graph hierarchy, transforms, bounds, appearance, and invalidation.
- `@flighthq/signals`: strictly-typed signals and slots for event dispatching. Signals support multiple listeners, priority, and cancellation. The package is effectively always present in the SDK; specific signal groups are opt-in via `enable*` functions defined in the owning package. Signals is fundamental infrastructure and should have few dependencies.

## Scene Graph and Display

- `@flighthq/displayobject`: Flash/OpenFL-style display objects such as bitmaps, shapes, containers, masks, stages, and videos.
- `@flighthq/text`: text display objects — single-format `TextLabel` and multi-format `RichText` (built on the text-layout spine, with a lazily-ensured layout cache), plus `NativeText`, the platform-rendered text field measured outside the spine by the host engine. ("native-rendered text" is a property of the display object, not a package boundary — the `native` namespace is reserved for the platform/OS-integration suite.)
- `@flighthq/sprite`: sprite/tilemap/quad-batch graph for atlas-based batch rendering.
- `@flighthq/scene`: 3D world graph for spatial scene management. A doorway for future development; the road is mostly untaken and the package is not yet built out.
- `@flighthq/clip`: hard geometric clip regions as plain data — constructors (`createClipRegionFrom*` for rectangles, paths, rounded rectangles, ellipses, circles, and raw contours), composition (`intersectClipRegions`, `unionClipRegions`), queries (`clipRegionContainsPoint`, `clipRegionIntersectsRectangle`, `clipRegionContainsRectangle`), transform (`transformClipRegion`; axis-aligned matrices preserve scissor-eligibility, rotation/skew promotes to a quad contour), utilities (`cloneClipRegion`, `copyClipRegion`, `getClipRegionBounds`, `isClipRegionEmpty`, `isClipRegionRectangular`, `clipRegionsEqual`, `normalizeClipRegion`, `invalidateClipRegion`), and a pool bracket (`acquireClipRegion`/`releaseClipRegion`). No rendering; rendering is provided by the `displayobject-<backend>` clip modules.
- `@flighthq/path-formats`: tree-shakable serialization codecs for `@flighthq/path`. First-build scope is SVG path data (the `d` attribute): `parseSvgPathData(d): Path | null` parses the full SVG grammar (`M L H V C S Q T A Z`, relative lowercase, implicit repeated commands, the `S`/`T` smooth shorthands, comma/space/scientific-notation/packed-arc-flag tokenizing) into a `Path` via the `appendPath*` builders, and `formatSvgPathData(path, options?)` serializes a `Path` to absolute-command `d` text by walking `forEachPathSegment` (optional coordinate `precision`); `appendSvgPathData(path, d)` parses into an existing path. Codec only — owns no geometry math. Arcs are stored by `@flighthq/path` as cubic beziers, so a round-trip preserves geometry but re-emits `A` as `C`.
- `@flighthq/path-boolean`: 2D constructive-solid-geometry boolean operations for `@flighthq/path` — a tree-shakable neighbor package. Per-operation API `unionPaths(a, b, out?, options?)` / `intersectPaths` / `differencePaths` / `xorPaths` plus the general `booleanPaths(subject, clip, operation, out?, options?)`; each flattens both paths via `flattenPath` at the option tolerance, combines the contours through the active kernel, and rebuilds the result into `out` (a fresh `nonZero` path when omitted, holes traced counter-wound) — `out` may safely alias either input. Operations dispatch through a swappable `PathBooleanBackend` seam (`getPathBooleanBackend`/`setPathBooleanBackend`/`createDefaultPathBooleanBackend`) so a heavier/native/wasm kernel drops in without touching the operation API. The default kernel (`createMartinezPathBooleanBackend`) is a from-scratch floating-point Martinez–Rueda–Feito sweep-line: stage one sweeps to build the segment arrangement (crossing + collinear-overlap subdivision), stage two merges coincident sub-segments by winding, classifies each unique segment by the winding number on each perpendicular side under the fill rule (`PathBooleanOptions.fillRule`, `nonZero` default / `evenOdd`), and traces fill-oriented directed rings. Correct on coincident/overlapping edges, shared vertices, holes, and self-intersecting input; degenerate contours (empty, sub-triangle, zero-area, single point) yield sane empty/passthrough results rather than throwing. Polygon offsetting is `offsetPath(path, delta, options?)` — flatten, offset each contour (miter/bevel/round/square joins with miter-limit fallback; butt/round/square end caps for open paths), then self-union the raw rings through the active kernel to dissolve concave-corner self-overlap and emit a clean hole-correct outline; positive `delta` inflates against a canonical orientation regardless of input winding, over-deflation past self-collapse drops the ring (empty path). `PathOffsetJoin`/`PathOffsetEnd`/`PathOffsetOptions` live in `@flighthq/types`. Region simplification is `simplifyPath(path, options?)` — flatten and self-union a single (possibly self-intersecting/multi-contour) path through the kernel under `options.fillRule` to a clean outline (the Clipper `SimplifyPaths`/Skia `Simplify` op); it shares the module-private `resolvePathRegions` "raw rings → clean region" primitive with `offsetPath`. Kernel-free vertex cleanup (`cleanPath`, dedup coincident/near-collinear/spike vertices) lives in `@flighthq/path`, not here, so dedup never pulls the clipper into a bundle. `unionAllPaths(paths, out?, options?)` folds a path list through one kernel union pass. Positive/negative fill is a path-boolean-local `PathBooleanFillRule = PathWinding | 'positive' | 'negative'` superset (core `PathWinding` stays lean; positive/negative reachable only through the backend seam) — `offsetPath` self-unions under `positive` (Clipper's InflatePaths cleanup fill), correctly dissolving concave-corner self-overlap that non-zero could not. Kernel snap tolerance is magnitude-relative (`extent · 1e-9`); a seeded (xorshift32) fuzz suite asserts union commutativity, `A ∖ A == ∅`, `A ∪ A == simplify(A)`, simplify idempotence, offset identity, convex double-offset stability, and concave-offset self-intersection-freeness. **Full AAA set complete** (all four build phases). Known weak spots (documented in status.md): deflating dense round-join output, and open-polyline-with-coincident-endpoints closedness (unrecoverable from flattened geometry).
- `@flighthq/shape-formats`: tree-shakable serialization codecs for `@flighthq/shape`. First-build scope is native command-stream JSON: `formatShapeJson(shape, options?): string` walks the shape's `shape.data.commands` buffer and emits `{ shapeFormat: 1, commands: [{ key, args }] }`, serializing each arg by type (numbers/strings/booleans/numeric arrays verbatim, `Matrix` as `{a,b,c,d,tx,ty}`, `null` verbatim); `parseShapeJson(text, options?): Shape | null` replays each entry through the matching `appendShape*` builder, returning `null` on malformed JSON, a missing/mismatched `shapeFormat` tag, a non-array `commands`, a malformed entry/arg, or an unknown command key. Lossless for every non-bitmap command. `beginBitmapFill`/`lineBitmapStyle` resources have no serializable id, so they serialize as an ordinal `ShapeBitmapReference` (`{ index }`) and are rehydrated on parse via `options.resolveBitmap`; without a resolver (or when it returns `null`) the bitmap-bearing command is dropped and the rest parses intact. Codec only — owns no drawing, tessellation, or hit-testing. `options.space` pretty-prints. SVG export is a deferred follow-on format.
- `@flighthq/interaction`: hit testing, pointer dispatch, and object overlap detection.

## Rendering

- `@flighthq/render`: renderer registration, render state/queue, render node data, update pipeline, transform/color propagation. Image render caching lives in the renderer packages (`imageRenderCache`, `canvasRenderCache`, `webglRenderCache`, `domRenderCache`), not in a standalone package.
- `@flighthq/render-canvas`, `@flighthq/render-dom`, `@flighthq/render-webgl`: concrete renderers.
- `@flighthq/filters`: blur, glow, bevel, drop-shadow, color-matrix, and convolution filters as plain data descriptors with explicit Canvas/CSS and multi-pass WebGL backends. Not OpenFL-style filter objects.
- `@flighthq/filters-math`: shared, backend-agnostic filter math — shadow/bevel light-angle offsets, box-blur radius/sigma conversions, Gaussian kernel weights, the linear-sampling (bilinear-tap) blur optimization, and large-sigma downsample-level selection. Pure closed-form functions with `out` params, depended on by the filters intents package and every filter backend.
- `@flighthq/filters-gl`: GPU leaf-shader set for WebGL 2 — one `apply*FilterToGl` per filter descriptor, shared `applyGlBlitPass`/`applyGlTintPass` compositing primitives, and `clearGlFilterProgramCache` for deterministic GPU-resource release. This package is a collection of leaf shaders, not a chain applier; orchestration and scratch-target allocation belong to the caller (see `get*FilterGlScratchCount` helpers). Kernels are bounded: convolution ≤ `GL_CONVOLUTION_MAX_KERNEL_SIZE × GL_CONVOLUTION_MAX_KERNEL_SIZE` (7×7), median ≤ `GL_MEDIAN_MAX_RADIUS` (2). A chain dispatcher (`applyFiltersToGl`) is out of scope here by the tree-shaking rule; if needed, it belongs in `render-gl` or a `filters-gl-chain` neighbor.
- `@flighthq/materials`: the 3D material system — a full PBR material taxonomy (unlit, Blinn-Phong, metallic-roughness PBR with clearcoat/anisotropy/emissive/transmission, depth) plus color-transform. Built (the 20-material taxonomy, 922 tests); canonical design in [3d-materials-architecture.md](../3d-materials-architecture.md).
- `@flighthq/surface`: pixel-level manipulation of `ImageSource` values — read from or generate image data. Not used internally by renderers; user-facing.
- `@flighthq/capture`: the deterministic render-verification policy and format layer — the SDK-side vocabulary the tools, harnesses, and the Rust/wasm `flighthq-capture` parity crate share. Owns the tolerant fingerprint comparison policy (`compareCaptureFingerprints`, returning a `Number.POSITIVE_INFINITY` sentinel when a fingerprint is unparseable or grid sizes differ so any tolerance check fails), the `evaluateCaptureRegression`/`evaluateCaptureParity` tolerance checks, the default tolerances (`CAPTURE_REGRESSION_TOLERANCE` 5, `CAPTURE_PARITY_TOLERANCE` 15), the `regression`/`parity`/`smoke` tier vocabulary (`CaptureCheckTier`), and the committed baseline-store record shape (`CaptureBaseline` = `Record<column, { fingerprint?, sha256? }>`) with pure `createCaptureBaseline`/`formatCaptureBaseline`/`parseCaptureBaseline` and field get/set. Depends on `@flighthq/surface` for the pixel/fingerprint math (`parseSurfaceFingerprint`, `compareSurfaceFingerprints`) and does not re-export it. This is the standalone first build; browser driving (Playwright), dev servers, the gallery, CI wiring, and baseline file I/O stay in the tools (`scripts/compare-render.ts`, `scripts/baseline-store.ts`, `tools/harness/verify.ts`) — tool adoption of this package is a blessed later step, deferred.

## Resources

- `@flighthq/image`: image resources — pixel sources and constructors (from canvas/ImageBitmap/element, load from URL/ArrayBuffer/Base64/Blob). MIME detection now lives in `@flighthq/image-codec`; `image` re-exports `detectImageMimeType` for consumers and uses it to type Blobs when loading from bytes.
- `@flighthq/image-codec`: DOM-free byte↔pixel seam. A global MIME-keyed decoder/encoder registry (`registerImageDecoder`/`getImageDecoder`/`hasImageDecoder`/`unregisterImageDecoder`/`clearImageDecoders` and encoder twins), dispatchers `decodeImage(bytes, mimeType?)` / `decodeImagePremultiplied(bytes, mimeType?)` / `encodeImage(image, mimeType, options?)` returning `null` when no codec is registered (decode auto-detects MIME via `detectImageMimeType` when omitted), the migrated `detectImageMimeType` magic-byte sniffer, and opt-in web registrars (`registerWebImageDecoders` over `createImageBitmap`+`OffscreenCanvas`; `registerWebImageEncoders` over `OffscreenCanvas.convertToBlob`). `DecodedImage` is straight RGBA8; the web decoder premultiplies in a JS pass since `getImageData` is always straight. Registry is empty at import (no side effects); depends only on `types`.
- `@flighthq/font`: font and font-resource lifecycle — loaders (name/url/bytes), `document.fonts` load-status queries, magic-byte `detectFontFormat`, and `inferFontFormatFromUrl`.
- `@flighthq/video`: `VideoResource` lifecycle at image parity — load options (crossOrigin/muted/playsInline/preload/readiness), url/blob/mediastream sources, dispose, inspection getters, MIME infer/detect, and codec negotiation.
- `@flighthq/audio`: `AudioResource` lifecycle — the full loader matrix (url/bytes/blob/base64), sample-tier constructors, clone/dispose/predicates, buffer inspection getters, MIME infer/detect, and codec negotiation. (Playback channels + mixer/bus graph live in `@flighthq/media`.)
- `@flighthq/textureatlas`: texture atlases — regions, UVs, and constructors over image resources (depends on `@flighthq/image`).
- `@flighthq/tileset`: tilesets — uniform-grid texture atlases and constructors from images (depends on `@flighthq/textureatlas`).
- `@flighthq/loader`: type-agnostic batch load orchestrator over `() => Promise<T>` factories (or `ResourceLoadItem<T>` descriptors). `createResourceLoader`/`queueResourceLoad`/`startResourceLoad` drive a bounded-concurrency worker pool (`setResourceLoaderConcurrency`) with per-item priority (`setResourceLoadPriority`), key dedupe, weight-aware progress, retries with none/linear/exponential backoff, per-item timeouts, and optional token-bucket bandwidth throttling. Runtime control via `pauseResourceLoad`/`resumeResourceLoad`/`cancelResourceLoad`/`resetResourceLoader`; status/progress queries via `getResourceLoadItemStatus`/`getResourceLoadProgress`. Batch signals (`onProgress`/`onComplete`/`onError`/`onCancel`/`onPause`/`onResume`) are always present; per-item signals opt in via `enableResourceLoaderItemSignals`. `disposeResourceLoader` detaches listeners. Depends only on `@flighthq/signals` + `@flighthq/types`.
- `@flighthq/assets`: id-keyed asset library layered over `@flighthq/loader` (scheduling) and the per-resource decoders. All state lives on an `AssetLibrary` entity from `createAssetLibrary()` (no module globals); every function takes the library first. Type-agnostic: `registerAssetLoader(library, type, { load, dispose })` binds an `AssetType` to how it loads and frees through an open, last-write-wins registry, so the package depends on `@flighthq/loader` + `@flighthq/signals` + `@flighthq/types` only — never a resource package. `loadAssetManifest(library, manifest)` records the plain-data `AssetDescriptor[]` (id → url/type/group) and group membership without loading. `acquireAsset(library, id)` reference-counts and resolves the loaded value: resident → immediate, in-flight → the shared promise (dedup, one `adapter.load` per id across concurrent acquires), else it resolves the descriptor → its type's adapter and loads at refcount 1; it rejects (an async sentinel, not a throw) for an unknown id or a type with no adapter. `releaseAsset` decrements and, at zero, immediately calls `adapter.dispose(value)` and drops the entry (deterministic free; releasing below zero is a no-op; a release mid-load frees the orphaned value on settle). `getAsset(library, id)` returns the resident value synchronously or `null`; `getAssetRefCount` exposes the holder count. `loadAssetGroup(library, name, options?)` batch-preloads a group's not-yet-resident members through the loader (bounded concurrency, aggregate `AssetLoadProgress` via `options.progress`), acquiring each member so the group stays resident; `releaseAssetGroup` releases each. `disposeAssetLibrary` disposes every resident asset and clears the maps. Open directions (charter): LRU size-budget cache at refcount-zero, asset dependency graph, hot reload.

## Animation and Simulation

- `@flighthq/spritesheet`: animation layer built on raw resources — a logical package providing sprite-based animation, analogous in structure to `particles`.
- `@flighthq/particles`: headless particle emitter simulation, a pure sim leaf — `ParticleEmitterConfig`/`createParticleEmitterConfig`, `ParticleEmitterState`/`createParticleEmitterState`/`ensureParticleEmitterStateCapacity`, signals (`enableParticleEmitterSignals`/`getParticleEmitterSignals`), force fields (`applyParticleForces`), colliders (`applyParticleCollisions`), the object-pool path (`updateParticleObjects`/`stepParticleObjects`), config validation (`validateParticleEmitterConfig`/`normalizeParticleEmitterConfig`), and curve helpers (`particleColorCurveFromKeyframes`, `particleCurveFromKeyframes`, `sampleParticleCurve`). The `ParticleEmitter` display node and its sim→node glue live in `@flighthq/particleemitter`; particles depends only on `geometry` + `signals` + `types` (no display/node coupling).
- `@flighthq/particleemitter`: display-object composition layer over `@flighthq/particles`, the same node/sim split as `movieclip`/`timeline`. Owns the `ParticleEmitter` display node (`createParticleEmitter`, `createParticleEmitterData`, `createParticleEmitterRuntime`, `reserveParticleEmitter`, append/remove/compact particle ops, bounds) and the sim→node glue: `updateParticleEmitter`/`isParticleEmitterComplete`, `emitParticleBurst`, `stepParticleEmitter` (SoA), and `prewarmParticleEmitter`. Consumes the `@flighthq/particles` sim (curve sampling, emitter state/signals); depends on `particles` + `displayobject` + `geometry` + `node` + `types`.
- `@flighthq/particles-formats`: import/export of particle emitter configs to and from industry-standard authoring-tool formats — Particle Designer plist, Spine 4.x particle JSON, Unity Shuriken JSON. Full round-trip via `*Document` models, honest `warnings[]` channel, curve baking, and unified auto-detection via `detectParticleFormat`/`parseParticleConfig`.
- `@flighthq/timeline`: the pure timeline frame engine — keyframes, labels, and frame scripts driving a `TimelineSource`. Depends only on `signals` + `types`.
- `@flighthq/movieclip`: display-object composition layer that wraps a `Timeline` and drives a `DisplayObject`. Owns the `MovieClip` node (`createMovieClip`, `createMovieClipData`, `createMovieClipRuntime`, `getMovieClipRuntime`), `setMovieClipSource`, `updateMovieClip` (non-recursive — mirrors `updateTimeline`), goto/play/stop and frame-script mirrors delegating to the underlying timeline, `enableMovieClipSignals`/`disposeMovieClipSignals`/`getMovieClipSignals` over the separate `MovieClipSignals` interface, and `createSpritesheetTimelineSource` (exposes a spritesheet animation as a `TimelineSource`, swapping the atlas region on a lazily-created child bitmap per target). Depends on `timeline` + `displayobject` + `node` + `types`.
- `@flighthq/tween`: tween managers, tweens, and timers.
- `@flighthq/clock`: hierarchical, pausable, scalable time primitive. A `Clock` tree driven by the app loop via `advanceClock(root, dtSeconds)`; child rate = product of ancestor scales, and a paused ancestor freezes its subtree. Consumers read scaled `deltaTime`/`elapsed` fields; opt-in `onTick` signal (via `enableClockSignals`) composes with the signal rate helpers. Depends only on `types` + `signals`.
- `@flighthq/easing`: easing functions for use with tween or any animation system.
- `@flighthq/spring`: spring-physics animation — a value that chases a moving target with second-order damped-harmonic motion (accelerate, overshoot, settle), driven by a per-frame `deltaTime` rather than a fixed duration. Plain-data `Spring = { value, velocity }` and mass-independent `SpringConfig = { frequency, dampingRatio }` (frequency in Hz; dampingRatio 0 undamped / <1 underdamped-overshoots / 1 critical-fastest-no-overshoot / >1 overdamped), with `createSpring`, `createSpringConfig`, and `createSpringConfigFromPhysical(stiffness, damping, mass)` (converts k/c/m → `frequency = sqrt(k/m)/(2*PI)`, `dampingRatio = c/(2*sqrt(k*m))`). The core step `updateSpring(spring, target, config, deltaTime)` is the **closed-form analytic** solution of `c'' + 2*zeta*omega*c' + omega^2*c = 0` (`omega = 2*PI*frequency`, `c = value - target`) evaluated exactly over `deltaTime` — three branches (over/critical/under-damped) computing four transition coefficients, then `value/velocity` written from locals. Because it is exact rather than explicit-Euler, it is unconditionally stable (a frequency-1000 spring at `dt = 1` lands on target instead of exploding) and, for a constant target, exactly frame-rate independent (`M(dt)^n == M(n*dt)`); `deltaTime <= 0` and non-positive frequency are no-ops. The `target` is per-step, not stored, so it can change mid-flight. `isSpringSettled` (near target AND near-zero velocity), `resetSpring`, and componentwise `Spring2D`/`Spring3D` (`createSpring2D/3D`, `updateSpring2D/3D`, `isSpring2D/3DSettled`) that apply the one scalar solver per axis. `out`-param, allocation-free hot path. The duration-less, overshoot-capable, second-order complement to `easing`/`tween` (fixed-shape/fixed-duration) and distinct from `math`'s first-order `damp` (no overshoot, no velocity). Depends on `math` + `types` only.
- `@flighthq/motionpath`: path-following animation — advances a marker along a `@flighthq/path` by **arc length** (constant real speed along the curve, not the speed-distorting raw bezier parameter) and reports its world position, tangent, and heading angle, so a caller can move and orient an object along a curve. Plain-data `MotionPath = { path, length, distance, speed, loopMode, direction }` (`distance` = current arc-length position in `[0, length]`; `length` cached at create via `getPathLength`; `speed` a units/sec magnitude; `direction` the `1|-1` travel sign, stored so `pingpong` remembers its heading across frames). `createMotionPath(path, speed?, loopMode?, tolerance?)` measures and caches the length; `updateMotionPath(mp, deltaTime)` advances `distance` by `speed·direction·deltaTime` and resolves the `MotionPathLoopMode` end behavior — `clamp` (stop at the ends), `loop` (wrap modulo length, both directions), or `pingpong` (reflect off each end via a period-`2·length` triangle-wave phase and flip `direction`, correct even when one large `deltaTime` crosses the path several times). `getMotionPathPosition(mp, pointOut, tangentOut)` samples via path's `getPathPositionAtDistance` (returns `false` for an empty/degenerate path); `getMotionPathHeading(mp)` is `atan2` of the tangent (module-scratch, allocation-free); `getMotionPathProgress(mp)` the normalized `0..1`; `setMotionPathDistance`/`setMotionPathProgress` seek with clamping. `deltaTime <= 0` and a zero-length path are no-ops; inputs read to locals before writes (alias-safe on `mp`). The animation-family driver over path's arc-length sampling: path owns geometry and "point at distance D", motionpath owns "where is the marker now and where is it heading" over frames. Depends on `path` + `geometry` + `types` only.

## Game

- `@flighthq/camera2d`: the 2D game-camera cell — a plain-data camera that maps a 2D world to the screen and back, follows a target, and reports what is visible. Types (in `@flighthq/types` › `Camera2D.ts`): `Camera2D {x, y, zoom, rotation, viewportWidth, viewportHeight}` where `(x, y)` is the world point rendered at the **viewport center** (center-anchored), `zoom` > 1 magnifies, `rotation` is radians CCW about the center; `Camera2DFollowOptions {deadzoneHalfWidth?, deadzoneHalfHeight?, smoothTime?, worldBounds?}`; and `Camera2DOptions {x?, y?, zoom?, rotation?}` for `createCamera2D`. API: `createCamera2D(viewportWidth, viewportHeight, options?)` (the only allocator; identity defaults x=y=0, zoom=1, rotation=0), `getCamera2DViewMatrix(camera, out)` composing the world→screen affine as `translate(viewportCenter) · rotate(-rotation) · scale(zoom) · translate(-x, -y)` (built from `setTransformMatrix` + `translateMatrixByVectorXY`), `projectCamera2DPoint`/`unprojectCamera2DPoint` (world↔screen, the unproject using the inverse view matrix and exactly inverting project), `updateCamera2DFollow(camera, targetX, targetY, deltaTime, options?)` (deadzone box → minimum goal move so the target sits on the deadzone edge, then `@flighthq/math` `damp` toward the goal with `smoothTime` as the time constant — snap when `smoothTime<=0` or `deltaTime<=0` — then a world-bounds clamp that centers on any axis smaller than the view), `getCamera2DVisibleBounds(camera, out)` (the axis-aligned world rectangle the viewport covers — exact at rotation 0, the enclosing AABB of the rotated view otherwise, always centered on `(x, y)`; feeds `@flighthq/spatial`/renderer cull), `zoomCamera2DAtScreenPoint(camera, screenX, screenY, zoom)` (zoom-to-cursor: keeps the world point under a screen point fixed by re-centering the camera by the unproject delta), and `getCamera2DParallaxPoint(camera, factor, out)` (screen-space scroll offset for a depth-`factor` layer: 0 = screen-locked, 1 = the camera's full screen translation / world-locked, 0.5 = half; computed as `(view.tx − centerX, view.ty − centerY) · factor`). Small side-effect-free `out`-parameter functions, allocation-free in the hot path (module scratch matrices/vectors); every function carries the full `Camera2D` type name. Depends on `geometry` + `math` + `types` only — no display object, renderer, or scene graph. Distinct from the 3D `@flighthq/camera` (perspective/orthographic frustums). Open directions: camera shake/impulse, multi-camera/split-screen viewport offsets, and `damp`-smoothed zoom/rotation follow.
- `@flighthq/collision`: the 2D narrow-phase collision cell — discrete shape-vs-shape overlap between plain-data colliders returning a **contact manifold**, the detection layer queried after a broadphase. Types (in `@flighthq/types` › `Collision.ts`): `CollisionShapeKind` (open string union `circle`/`aabb`/`obb`/`polygon`/`segment`/`point`), the six shape interfaces (`CollisionCircle {x,y,radius}`, `CollisionAabb {minX,minY,maxX,maxY}`, `CollisionObb {x,y,halfW,halfH,rotation}`, `CollisionPolygon {points}` flat convex CCW list, `CollisionSegment {x0,y0,x1,y1}`, `CollisionPoint {x,y}`), the discriminated `CollisionShape` union, and `CollisionManifold {overlapping, normalX, normalY, depth}` (unit minimum-translation axis oriented to push shape **A out of B**, `depth` the penetration; on a miss all zero). API: the ten manifold pair tests `testCircleCircleCollision`/`testCircleAabbCollision`/`testCircleObbCollision`/`testCirclePolygonCollision`/`testAabbAabbCollision`/`testAabbObbCollision`/`testAabbPolygonCollision`/`testObbObbCollision`/`testObbPolygonCollision`/`testPolygonPolygonCollision`, the generic `testCollision(a, b, out)` dispatcher (kind-ranked, negates the normal on reversed argument order), `getCollisionShapeContainsPoint(shape, x, y)`, the boolean segment queries `testSegmentSegmentCollision`/`testSegmentCircleCollision`/`testSegmentAabbCollision`/`testSegmentObbCollision`/`testSegmentPolygonCollision`, and `createCollisionManifold`/`clearCollisionManifold`. Convex pairs (AABB/OBB/polygon) run through one shared separating-axis (SAT) core over materialized vertex lists using the min-of-both-directions **separation** penetration (correct for containment); circle pairs are special-cased with closest-point / radial math and a center-to-nearest-vertex axis for corner cases. `out`-parameter and allocation-free in the hot path (box vertices go to module scratch buffers). **Touching is exclusive** (zero penetration → not overlapping) for the manifold tests; the containment/segment queries are boundary-inclusive. Distinct from `@flighthq/geometry` (3D bounding volumes) and `@flighthq/interaction` (display-object hit-testing). Depends on `geometry` + `types`. Phase 1 = discrete overlap + manifolds; swept/time-of-impact (phase 2) and full contact-point sets + capsule/rounded shapes (phase 3) are chartered follow-ons.

- `@flighthq/gamestate`: the game-flow state-stack cell — the top-level *screen/mode* stack (boot, menu, play, pause, game-over) with a paired enter/exit/pause/resume/update lifecycle, the small orchestration primitive every engine has (Phaser scenes, a Unity/MonoGame state stack). It is flow control, not save data — serializable game state is a separate concern. Types (in `@flighthq/types` › `GameState.ts`): `GameState {name?, onEnter?, onExit?, onPause?, onResume?, onUpdate?, updateBelow?, renderBelow?}` — all callbacks optional (`onEnter`/`onExit`/`onPause`/`onResume: () => void`, `onUpdate: (deltaTime) => void`), `name?` is a debug label only, and the two boolean flags mark a transparent overlay whose state beneath keeps updating (`updateBelow`) / stays render-visible (`renderBelow`); `GameStateStack {states}` is the stack ordered bottom-to-top (last element = active top), a plain `{states}` with no entity/runtime split and **no module-global stack** (a game holds its own). Lifecycle is **direct callbacks**, not signals — a state's transitions have a single owner (the stack) firing in a defined, deterministic order, exactly when the SDK prefers direct wiring over loose signal dispatch. API (each carries the full `GameState`/`GameStateStack` name): `createGameStateStack()` (the only allocator, an empty stack); `pushGameState(stack, state)` → the current top's `onPause`, then push, then the new state's `onEnter` (pause-then-enter); `popGameState(stack)` → the top's `onExit`, then the revealed state's `onResume`, returns the popped state or `null` if empty (sentinel, not a throw); `replaceGameState(stack, state)` → the top's `onExit` + pop then the new state's `onEnter`, **without** pausing/resuming the state below (it stays covered), and a bare push-and-enter on an empty stack; `clearGameStateStack(stack)` → `onExit` every state top-to-bottom then empties (no pause/resume — a teardown, not a layering); `updateGameStateStack(stack, deltaTime)` → ticks the top's `onUpdate(deltaTime)`, then walks downward ticking the state beneath each visited state that sets `updateBelow`, stopping at the first opaque overlay (a no-op on an empty stack); `getActiveGameState(stack)` (top or `null`), `getGameStateStackDepth(stack)`, and `getGameStateStackVisibleStates(stack, out)` which clears then fills `out` with the render-visible states **bottom-to-top** — the top plus its contiguous `renderBelow` run — so the caller draws `out` front-to-back (index 0 first). Small side-effect-free functions over a plain-data stack; depends on `@flighthq/types` only — no display, renderer, or scene graph. Open directions: opt-in `enableGameStateStackSignals` transition events, async `onEnter` loading gates over `@flighthq/loader`/`@flighthq/assets`, and cross-fade/slide transition effects.

- `@flighthq/spatial`: the 2D broadphase cell — a spatial index over many objects' AABBs that answers "which pairs are close enough to be worth a narrow-phase test?" and "which objects overlap this region/point/ray?" without the O(n²) all-pairs scan. It is the locality filter in front of `@flighthq/collision`'s narrow-phase (and the culling structure a camera/renderer uses to skip off-screen objects). Types (in `@flighthq/types` › `Spatial.ts`): `SpatialObjectId` (a `number` handle — the caller maps its objects to ids; cache-friendly), `SpatialAabb {minX,minY,maxX,maxY}` (spatial's own 2D bounds — structurally like `CollisionAabb` but defined here so spatial never depends on `collision`; distinct from geometry's 3D `Aabb`), `SpatialPair {a,b}` (unordered candidate pair), the `SpatialIndexBackend` seam (`insert`/`update`/`remove`/`clearSpatialIndex`/`querySpatialPairs`/`querySpatialRegion`/`querySpatialPoint`/`querySpatialRay`), and the `SpatialIndex` entity (`{runtime:{backend}}`, entity/runtime split). API: `createSpatialIndex(backend?)` (defaults lazily to a uniform grid, no import side effect), `createUniformGridSpatialBackend(cellSize)`, and the operations dispatching to the backend — `insertSpatialObject`/`updateSpatialObject`/`removeSpatialObject`, `querySpatialPairs(index, out)`, `querySpatialRegion`/`querySpatialPoint`/`querySpatialRay`, `clearSpatialIndex`. All queries fill a caller-provided `out` array (cleared then filled) — no per-call structural allocation. The uniform grid is a spatial hash: an object's AABB maps to the block of `cellSize` cells it covers, each cell holds its co-occupant ids. `querySpatialPairs` emits each co-occupant candidate pair **exactly once** by emitting only from the pair's canonical (top-left of their overlapping cell-ranges) cell — dedup with zero per-pair allocation — and never (a,a); the pair is a broadphase candidate the caller confirms. `querySpatialRegion`/`Point`/`Ray` gather ids from the covered cells (region/ray dedup'd via a reused scratch set) and **confirm each candidate against its real bounds** (rectangle overlap / point-containment via geometry, ray via a slab test), so a false cell-mate is dropped; the ray walks cells with an amanatides-woo DDA bounded to the occupied cell range. Confirmed pairs are invariant to `cellSize` (resolution changes candidates, not confirmed results). Depends on `geometry` (2D AABB/rectangle math) + `types` — **not** `collision`. Phase 1 = the seam + uniform-grid default; quadtree (P2) and sort-and-sweep (P3) are chartered drop-in backends behind the same operations.

## Input and Text

- `@flighthq/input`: unified input manager for keyboard, pointer, wheel, and gamepad — normalized signal dispatch (`onKeyDown`/`onKeyUp`/`onPointerDown`/`onPointerMove`/`onPointerUp`/`onWheel`/`onGamepadButtonDown`/`onGamepadButtonUp`/`onGamepadAxisMove`/`onGamepadConnect`/`onGamepadDisconnect`/`onTextInput`), held-state snapshots (`InputState` — per-key, per-button, per-axis queries), frame-edge queries (`wasInputKeyPressed`/`wasInputKeyReleased`), analog dead zones (`applyGamepadAxisDeadZone`/`applyGamepadStickDeadZone`), gamepad button/axis name mappings, key-repeat timers, pointer lock/capture. Attach via `attachKeyboardInput`/`attachPointerInput`/`attachGamepadInput`/`attachWheelInput`/`attachTextInput`.
- `@flighthq/textinput`: editable-text-field behavior layered onto a `RichText` via the `enableTextInput` opt-in (idempotent; `disableTextInput`/`hasTextInput`/`getTextInputState`). Caret/selection model with word/line/all selection (`selectWordAtTextInputIndex`/`selectLineAtTextInputIndex`/`selectAllTextInput`/`setTextInputSelection`) and caret navigation — horizontal, word-boundary (`moveTextInputCaretByWord`), vertical (`moveTextInputCaretUp`/`Down`, desired-x column preserved), and line-relative (`moveTextInputCaretToLineStart`/`moveTextInputCaretToLineEnd`). Text mutation via `insertTextInput`/`appendTextInput`/`replaceTextInput`/`replaceSelectedTextInput` and selection-aware delete/backspace (`deleteTextInputBackward`/`Forward`, `deleteTextInputWordBackward`/`Forward`). Undo/redo with mergeable edit records (`undoTextInput`/`redoTextInput`/`canUndoTextInput`/`canRedoTextInput`/`clearTextInputHistory`, `historyLimit`). Input restrictions (`applyTextInputRestriction`, OpenFL `restrict` accept/decline ranges), password masking (`getTextInputDisplayText`, `displayAsPassword`/`passwordCharacter`), and caret-visibility scrolling (`scrollTextInputCaretIntoView`). Keyboard commands are resolved and applied by `handleTextInputKeyboard`: Home/End are line-relative, Ctrl/Cmd+Home/End are document-level, plus clipboard cut/copy/paste, word motion (Ctrl/Alt+arrows), and word delete. Focus managers dispatch pointer/keyboard/wheel events to the focused field: `createTextInputManager` (editable) and `createSelectableRichTextManager` (read-only, selection + copy only), wired to input signals via `connectInputToTextInput`.
- `@flighthq/textlayout`: renderer-agnostic glyph layout engine — line breaking (word-wrap, character-wrap, explicit newlines), horizontal alignment (left/center/right/justify), interWord and interCharacter justification, leading/line-spacing, max-width constraint. Produces positioned `TextLayoutGroup` runs with per-character advance widths. Consumes text-shaper output for measurement.
- `@flighthq/textshaper`: text-shaping seam over a swappable `TextShaperBackend` (`setTextShaperBackend`/`getTextShaperBackend`). Two tiers: a measure-only default (`measureText` — string→width, sufficient for layout + Canvas text) and a full-glyph shaper (`shapeTextRun`/`shapeTextRunInto`/`shapeTextRuns` — glyphs with ids, advances, offsets, clusters, required for GPU text). Also: glyph extents (`getGlyphExtents`/`getGlyphExtentsBatch`/`getGlyphExtentsInto`), font metrics (`getFontMetrics`/`getFontMetricsInto`), caret positions (`getCaretPositionsForRun`), text itemization (`itemizeText`), shaped-run caching (`shapeTextRunCached`), and backend-change signals. Correct international text (Arabic/Indic/kerning/ligatures) needs a full-glyph backend (HarfBuzz). The Rust port mirrors this with `flighthq-textshaper` (rustybuzz backend). See [rust/text](../rust/text.md).
- `@flighthq/textshaper-canvas`: Canvas 2D text-shaper backend — advances-only shaping via the Canvas `measureText` API. Provides `createCanvasTextShaperBackend`/`clearCanvasTextShaperBackendCache`. Sufficient for Canvas-rendered text and layout measurement; does not produce glyph ids or cluster mappings.

## Application

- `@flighthq/intl`: locale-aware formatting over the ECMAScript `Intl` API — a **utility** (like `math`/`easing`), no `*Backend` seam, since `Intl` is universal to every JS engine Flight targets. Small typed free functions, each fetching-or-building a **cached** `Intl.*` instance keyed by `(kind, locale, serialized-options)`: numbers (`formatNumber`, `formatCurrency`, `formatPercent`, `formatUnit`, `formatCompactNumber`), date/time (`formatDate`/`formatTime`/`formatDateTime` over `Date | number`, documented numeric default field sets), `formatList`, `formatRelativeTime`, plural/ordinal category selection (`selectPluralCategory`/`selectOrdinalCategory` → `Intl.LDMLPluralRule`), and collation (`compareStrings` + a non-mutating `sortStrings`). `locale` is `string | readonly string[]` (`LocaleInput`, in `@flighthq/types`); options mirror the native `Intl.*Options`. The cache is a bounded (256-entry, FIFO-evicting) module-level `Map` of immutable formatter instances — a legitimate memoization cache, not mutable app state — behind a shared `getCached` helper. Invalid input degrades to `Intl`'s own output (`NaN` → `'NaN'`, invalid `Date` → `'Invalid Date'`); a malformed locale/currency that `Intl` throws on stays a throw. Depends on `@flighthq/types` only. Open directions (charter): ICU message formatting (`intl-message`), `Intl.Segmenter`, `Intl.DurationFormat`/`DisplayNames`.
- `@flighthq/application`: optional package providing a main loop, application lifecycle events, and the **windowing API** — `ApplicationWindow` (size/position/state + signals), web event wiring (`attach*`/`detach*`), and window-control commands (title, position, size, minimize/maximize/restore, fullscreen, always-on-top, constraints, `openWindow`, close-with-veto via `onCloseRequest`) over a swappable `WindowBackend` (web default; native hosts register their own), matching the platform suite's backend-seam pattern.
- `@flighthq/log`: leveled, structured, capture-aware logging. Emit side (`log`, `logError/Warn/Info/Debug/Verbose`, `logWith` context variants, `logAssert`, `logOnce`) plus a full listener side: multi-sink fan-out (`addLogSink`/`removeLogSink`/`clearLogSinks`), global and per-channel level gates, `LogContext`-bound contextual loggers, pluggable formatters (`createJsonLogFormatter`/`createTextLogFormatter`), a console-capture sink, an in-memory ring-buffer sink, a file sink over a swappable `LogTransportBackend` (`setLogTransportBackend`/`createWebLogTransportBackend`; web default is a no-op), and sink combinators (buffered/filter/fanout/rate-limited/sampled). Diagnostics extras: timing (`startLogTimer`/`endLogTimer`), tracing spans (`createLogSpan`/`enterLogSpan`/`exitLogSpan`), nesting groups (`beginLogGroup`/`endLogGroup`), field redaction (`setLogRedactionPaths`), custom kind serializers (`registerLogSerializer`), `serializeLogError`, and an opt-in `LogSignals` event group (`enableLogSignals`). Not Canvas/DOM-coupled. Tree-shakes: the emit-only import carries only the gate check and the `LogLevel` enum.
- `@flighthq/media`: audio and video playback channels.
- `@flighthq/sdk`: convenience barrel for applications and examples.

## Platform Integration Suite

Host/OS integration so applications need no escape hatch out of the SDK. Each capability is a self-contained cell: flat free functions over a swappable `*Backend` (defined in `@flighthq/types`). A web/DOM backend is always lazily available, so every function works on the web; a native host (Electron, Tauri, Capacitor, a C/C++ shell) replaces it via the capability's `set*Backend`. "Electron support" is one backend, not a coupling. Two shapes: **command** capabilities expose flat functions + `get*Backend`/`set*Backend`/`createWeb*Backend`; **event** capabilities expose an entity of signals with `create*`/`attach*`/`detach*`/`dispose*` (mirroring `@flighthq/application`'s window wiring). Web backends guard every API and return sentinels (`null`/`false`/`-1`/`''`/`[]`/no-op) when unavailable rather than throwing.

- `@flighthq/platform`: root identification seam — OS name, desktop/mobile/web kind, arch, locale, touch.
- `@flighthq/accessibility`: assistive-technology (ARIA) bridge — publishes a canvas/game UI's semantics to screen readers, which otherwise see only an opaque `<canvas>`. A **command capability whose backend holds the mirrored tree** (like `storage`): `setAccessibilityNode(node)`/`removeAccessibilityNode(id)` (drops the descendant subtree)/`clearAccessibilityTree()` register plain-data ARIA-aligned nodes (`AccessibilityNode` = `id`/`role`/`label?`/`description?`/`value?`/`parentId?`/`bounds?`/`states?`; `AccessibilityRole` open ARIA-role union; `AccessibilityState` booleans + range/level numerics); `setAccessibilityFocus(id)` moves platform focus (sentinel `false` on miss); `announceAccessibility(message, liveness?)` speaks a `'polite'`/`'assertive'` live message. `get/set/createWebAccessibilityBackend`; the web backend is a visually-hidden ARIA DOM overlay (`Map<id, HTMLElement>` in an `sr-only` container, node→element `role`+`aria-*` mapping, `parentId` nesting, `aria-live` regions), lazy and import-side-effect-free, a no-op with no DOM. Depends on `@flighthq/types` (+ DOM in the web backend); does not read the scene graph — the caller maps its UI onto nodes.
- `@flighthq/clipboard`: system clipboard read/write (text, HTML).
- `@flighthq/dialog`: file open/save and message/confirm/prompt dialogs.
- `@flighthq/filesystem`: file read/write/list/stat and standard directory paths (web backend over OPFS).
- `@flighthq/notification`: OS notifications and permission.
- `@flighthq/shell`: open external URLs/paths, reveal in folder, move to trash, beep.
- `@flighthq/menu`: native application-menu and context-menu descriptors (native host required to realize).
- `@flighthq/tray`: system tray / menu-bar icon (icon, tooltip, title, context menu, click events). The application/dock badge lives in `@flighthq/app`, not here.
- `@flighthq/shortcut`: global OS hotkeys (native host required).
- `@flighthq/screen`: display enumeration, work area, scale factor.
- `@flighthq/storage`: synchronous persistent key/value (web backend over localStorage).
- `@flighthq/device`: static device/OS identity — model, manufacturer, OS, memory, safe-area insets. Battery is _not_ here; it is a live concern owned by `@flighthq/power`.
- `@flighthq/share`: native share sheet.
- `@flighthq/haptics`: vibration and impact/notification/selection feedback.
- `@flighthq/geolocation`: current position and position watches.
- `@flighthq/webcam`: take photo / pick image.
- `@flighthq/statusbar`: mobile status-bar style, visibility, color.
- `@flighthq/net` (command): HTTP(S) transport — the URLLoader/URLRequest home. `sendNetRequest(request, options?)` over a swappable `NetBackend` (`getNetBackend`/`setNetBackend`/`createWebNetBackend`); plain-data `NetRequest`/`NetResponse`, `text`/`json`/`arraybuffer`/`blob` decode, timeout + `AbortSignal` cancellation, opt-in download-progress signal. Expected transport failures (network error, DNS, timeout) resolve to a sentinel `NetResponse` (status 0, ok false); a non-2xx is a normal response with the real status. Distinct from `@flighthq/network` connectivity status.
- `@flighthq/socket` (event): bidirectional persistent-connection transport — the Socket/WebSocket/XMLSocket home, the long-lived sibling of `@flighthq/net` (one-shot request/response). A plain `Socket` entity plus an opaque `SocketRuntime` (live connection + opt-in signals + readyState) over a swappable `SocketBackend` (`getSocketBackend`/`setSocketBackend`/`createWebSocketBackend`, DOM `WebSocket` by default, lazy and import-side-effect-free). `createSocket(options)` opens; `attachSocket`/`detachSocket` gate event delivery; `disposeSocket` closes-if-open then detaches and drops the signal group (dispose is entity teardown, distinct from the `closeSocket` connection command). Commands: `sendSocketMessage` (text/binary; sentinel `false` when not open, never throws), `closeSocket` (code/reason), `getSocketReadyState` (`connecting`/`open`/`closing`/`closed`). Typed events opt-in via `enableSocketSignals` → `onSocketOpen`/`onSocketMessage` (text vs binary `SocketMessage`)/`onSocketClose` (`SocketCloseInfo` code/reason/wasClean)/`onSocketError`. Web backend returns a null connection for unsupported transports (raw TCP/UDP); native hosts add them via `setSocketBackend`. Distinct from `@flighthq/network` connectivity status.
- `@flighthq/permission` (command): OS-permission query/request — the shared "may I?" gate the capability packages (webcam/geolocation/notification/clipboard) sit behind. `getPermissionState(name)` (query, non-prompting) and `requestPermission(name)` (may trigger the OS prompt) over a swappable `PermissionBackend` (`getPermissionBackend`/`setPermissionBackend`/`createWebPermissionBackend`). `PermissionName` = open string union (`'camera'|'microphone'|'geolocation'|'notifications'|'clipboard-read'|'clipboard-write'|'persistent-storage'|'push'|'midi'|'screen-wake-lock' | (string & {})`); `PermissionState` = `'granted'|'denied'|'prompt'` (the Permissions-API vocabulary; `'prompt'` = not yet decided). Web backend queries via `navigator.permissions.query({ name })` (name→descriptor map; identity for the built-ins), falling back to `Notification.permission` (`'default'`→`'prompt'`) for notifications and the `'prompt'` sentinel otherwise when the API is absent or the name is unqueryable. `request` routes each name to its real web trigger — notifications → `Notification.requestPermission`, camera/microphone → `getUserMedia({video/audio})` then stop the granted tracks immediately, geolocation → a one-shot `getCurrentPosition`, persistent-storage → `navigator.storage.persist` — and a name with no request path falls back to a plain state query. A missing API resolves to a `'prompt'`/`'denied'` sentinel; nothing throws. State + request only: it never opens the stream, reads the location, or shows the notification — the capability packages do that. Depends on `@flighthq/types` + the DOM.
- `@flighthq/network` (event): connectivity status and online/offline signals.
- `@flighthq/power` (event): battery/charging status, low-power and keep-awake.
- `@flighthq/lifecycle` (event): app active/inactive/background, resume/pause, back button.
- `@flighthq/keyboard` (event): on-screen keyboard visibility/height (type `SoftKeyboard`, avoiding the DOM `Keyboard`).
- `@flighthq/sensors` (event): accelerometer, gyroscope, device orientation.

Application/process layer (host shell integration beyond a single window):

- `@flighthq/app`: application identity (name/version/locale), control (quit/relaunch/focus), single-instance lock + `onSecondInstance`, the canonical app badge (`setAppBadgeCount`) + dock badge/menu/bounce, and app events (`onActivate`, `onOpenFile`).
- `@flighthq/protocol`: custom URI-scheme / deep-link registration plus an `onOpenURL` handler entity.
- `@flighthq/updater` (event): auto-update lifecycle — checking/available/progress/downloaded/error signals plus check/download/quit-and-install commands.
- `@flighthq/ipc`: inter-process messaging — `sendIpcMessage`, `invokeIpc`, `onIpcMessage` over a host channel backend (for split-process hosts like Electron main↔renderer).

Inbound host events are delivered through the same seam: command-style capabilities that also receive events expose a flat `on*(listener): () => void` over a backend `subscribe*` method — `onMenuSelect`, `onTrayEvent` (+ `setTrayContextMenu`), `onNotificationClick`/`onNotificationAction`, `onScreenChange`, power `onSuspend`/`onResume`. The window backend delivers OS-originated changes by mutating the `ApplicationWindow` and emitting its signals directly (it owns the `win`↔OS-window mapping from `openWindow`); native window backends additionally implement icon/opacity/progress/attention/skip-taskbar/menu-bar/parent controls.

Host backends (the concrete adapters that fill the seams) are a distinct package class — they carry a host dependency, are not tree-shakable, and are named `host-<runtime>`:

- `@flighthq/host-electron`: Electron main-process implementation of the window/app/dialog/clipboard/menu/tray/shortcut/screen/power/notification/shell/protocol/updater/ipc seams. The consumer passes the `electron` module explicitly — `registerElectronBackends(electron)` — typed against a local `ElectronApi` interface (the exact Electron surface Flight depends on), so the package needs no `electron` dependency and is unit-testable with a fake. Each `createElectron*Backend(electron)` is also exported for granular use. **Not** re-exported from `@flighthq/sdk` (it is an adapter you install in the host process, not app-facing API). Mobile seams and `filesystem` (node `fs`) are out of scope here — a future `host-capacitor` / a node-fs injection covers those. Future siblings: `host-tauri`, `host-capacitor`.
