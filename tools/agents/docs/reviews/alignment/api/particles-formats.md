# API Alignment: @flighthq/particles-formats

**Verdict:** Largely clean and well-symmetric with its sibling `spritesheet-formats`; the only real convention breaches are throwing on expected failure (corrupt asset JSON) and an entity-vs-format naming inconsistency in the Particle Designer functions.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `parseSpineParticle`, `parseSpineParticleDocument`, `parseUnityParticle`, `parseUnityParticleDocument` | These `throw new Error(...)` (via internal `parseSpineJson` / `parseUnityJson`) on malformed/empty/non-object JSON. Per the map's sentinel rule, parsing user-supplied asset files is an **expected** failure (the code's own comment says these files are "frequently hand-edited or produced by external tools"), so it should return a sentinel (`null`), not throw — `throw` is reserved for programmer error. | Return `ParticleEmitterConfig \| null` / `SpineParsed \| null` (etc.) and produce `null` on parse failure, or move the failure detail into a `warnings`/`error` field on the `*Parsed` result. |
| Medium | `parseParticleDesignerPlist` vs `parseSpineParticle` / `parseUnityParticle` | Behavioral inconsistency for the same operation across the package: the PD parser (regex-based) silently tolerates garbage and falls back to defaults and never throws, while the Spine/Unity parsers throw on bad input. Same verb, same job, opposite failure contract. | Pick one contract for "parse a possibly-corrupt asset string" and apply it to all three formats (recommended: all return a sentinel / surface a warning; none throw). |
| Low | `parseParticleDesignerPlist`, `parseParticleDesignerPlistDocument`, `serializeParticleDesignerPlist` | Naming asymmetry vs the other two formats. The entity being parsed in all three cases is a _particle_ emitter config; Spine/Unity name after the entity (`parseSpineParticle`), but Particle Designer names after the file format (`...Plist`). This also makes the document variant `...PlistDocument` instead of `...ParticleDocument`. | Rename to `parseParticleDesignerParticle` / `parseParticleDesignerParticleDocument` / `serializeParticleDesignerParticle` for entity-word symmetry, OR (since `spritesheet-formats` names after the subject `Spritesheet`) standardize all three on the subject word `Particle` — `parseSpineParticle` already does this; align PD to it. |
| Low | `ParticleDesignerParsed`, `SpineParsed`, `UnityParsed` (the `warnings: string[]` field) | Warnings are free-form English strings, so a consumer cannot programmatically branch on "which feature was dropped" — only display them. Matches `spritesheet-formats`, so not a divergence, but worth flagging as a shared design limit if these are meant to drive an asset pipeline. | Consider a structured warning (`{ code, message }`) shared via `@flighthq/types` if pipelines need to act on specific dropped features; otherwise leave as-is for parity with the sibling. |

## Clean

- **Full, unabbreviated type words** in every exported name (`ParticleDesigner`, `Spine`, `Unity`, `Particle`, `Document`); no abbreviations. Names are globally unique and self-identifying from the barrel.
- **Verb consistency with the sibling package.** `parse*` / `parse*Document` / `serialize*` and the `*Parsed` result shape mirror `spritesheet-formats` exactly (`parseAsepriteSpritesheet` / `parseAsepriteSpritesheetDocument` / `serializeAsepriteSpritesheet` / `AsepriteParsed`). The single-pass-vs-document split is applied uniformly across all three formats.
- **Parameter-order symmetry.** Every `serialize*` is `(config: Readonly<ParticleEmitterConfig>, existing?, options?)` and every `parse*` is `(text: string, options?)` — consistent across formats, with the `existing` round-trip document always second.
- **`Readonly<T>` discipline.** All three `serialize*` functions take `Readonly<ParticleEmitterConfig>`; internal `configToDocument` helpers do too. Mutation is never implied where not intended.
- **No bad teardown / accessor verbs.** No `dispose`/`destroy`/`acquire`/`release` misuse (none needed — pure value transforms). No `get*` returning a boolean; no boolean function that should be `is*`/`has*`.
- **`import type {}` on its own lines** in every file; no inline `import { type Foo, bar }` mixing.
- **Type placement is correct.** The `*Document` / `*Schema` / `*Parsed` / `*Options` types are package-internal (confirmed: not imported by any other package), so defining them inline rather than in `@flighthq/types` is appropriate — they are not cross-package contracts. Cross-package types (`ParticleEmitterConfig`, `ParticleBlendMode`, `ParticleCurve`, `ColorKeyframe`, `CurveKeyframe`) are correctly imported from `@flighthq/types`.
- **Exports alphabetized** within each file and across the barrel; nothing for `npm run order` to flag.
- **`parse*` / `serialize*` are the right allocation verbs** for format I/O (allocating transforms, no hot-path `out` parameter to misuse).
