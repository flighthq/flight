# API Alignment: @flighthq/scene-gl

**Verdict:** Strongly aligned — full unabbreviated type words, consistent `Gl` prefix-first naming, disciplined `Readonly<>` and out-param/scratch usage, correct sentinel-vs-throw split; the only findings are minor naming-symmetry and verb-consistency nits, no structural violations.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- | --- |
| Low | `register*GlMaterial` (e.g. `registerStandardPbrGlMaterial`, `registerLambertGlMaterial`, `registerWireframeGlMaterial`) vs `registerGlMeshMaterialRenderer` / `getGlMeshMaterialRenderer` / `resolveGlMeshMaterialRenderer` | Verb-target inconsistency for one concept. The ~20 per-family public convenience registrars use the type word **`Material`** (`register<Family>GlMaterial`), while the generic registry trio uses **`GlMeshMaterialRenderer`**. A reader scanning the barrel sees `registerStandardPbrGlMaterial` register a _material_ but the underlying function registers a _renderer_; the family name (`StandardPbr`) is also infixed rather than the type-word-last form used elsewhere. | Acceptable as an intentional split (registrar binds a material _kind_; registry stores _renderers_), but consider unifying on one type word, e.g. `registerStandardPbrGlMeshMaterial` / `register*GlMeshMaterial`, so the family registrars and the generic registry share the `GlMeshMaterial` token. At minimum document the split as deliberate. |
| Low | `getGlPbrVertexSource()` / `getGlClassicVertexSource()` / `getGlWireframeVertexSource()` (bare) vs `getGl<Family>VertexSourceForKey(key)` | Inconsistent source-getter surface across families. PBR and Classic export **both** a bare `getGl*VertexSource()`/`getGl*FragmentSource()` and a `...SourceForKey(key)`; Debug, Toon, Unlit, Matcap export **only** `...ForKey`; Wireframe exports **only** the bare form (it has no define key). The asymmetry is partly justified (Wireframe has no variants) but the bare PBR/Classic getters duplicate `...ForKey` with a default key. | Decide one canonical shape per family: keep `...ForKey` as the single getter for define-keyed families and drop the bare PBR/Classic variants (or document them as the "default-key" convenience). Keep Wireframe's bare-only form since it is variant-free. |
| Info | `linkGlProgram`, `compileGl<Family>Program` (via internal `compileGlShader`) | `throw new Error(...)` on shader compile/link failure. This is correctly classified as programmer error (malformed prelude) per the comment, and conforms to the throw-only-on-misuse rule. Flagged only to confirm the call sites are unreachable in correct usage — a corrupt/unsupported-driver runtime compile failure would surface as a throw rather than a sentinel. | No change; behavior matches convention. If driver-level (non-programmer) compile failure becomes a real runtime concern, revisit whether `compileGl*Program` should return `GlProgram | null`. |
| Info | `drawGlScene` doc comments reference `drawScene`, `registerGlMaterialRenderer`, `getGlMaterialRenderer`, `getGlMaterialRenderer` | Comments name functions without the package's `Gl`/`Mesh` tokens (`drawScene` for `drawGlScene`, `getGlMaterialRenderer` for the 2D analog). Not exported names, so no API violation, but the drift can mislead. | Update comments to the exact exported names (`drawGlScene`, and clarify the 2D analog is `getGlMaterialRenderer` in another package if that is the intent). |

## Clean

- **Full, unabbreviated type words everywhere.** Every export spells out its type: `getGlMeshMaterialRenderer`, `bindGlPbrStandardBlock`, `setGlMeshViewProjection`, `buildGlPbrStandardDefineKey`, `ensureGlWireframeUpload`. No `Mtl`/`Prog`/`DObj`-style abbreviations.
- **Globally unique, `Gl`-prefixed names.** Backend token is prefix-first and present on every function and type (`Gl`/`gl*`), so root-barrel names won't collide with the wgpu/canvas siblings. Filenames match (`glPbrStandardBlock.ts`, `glMeshMaterialRegistry.ts`).
- **Allocation discipline by verb.** `create*` is only used in the geometry imports; hot-path math writes to module scratch (`scratchViewProjection`, `scratchInverseView`, `scratchNormalMatrix`, `scratchRgba`) or to GL directly — no per-call allocation in `setGlMeshViewProjection`, `setGlMeshCameraPosition`, `bindGlPbrStandardBlock`, `drawGlMeshSubset`. `ensure*` (cache-or-create) and `compile*`/`build*` (allocate) are used correctly.
- **`Readonly<T>` applied consistently.** All entity/struct params that aren't mutated are `Readonly<>`: `Readonly<GlMeshProgram>`, `Readonly<Camera>`, `Readonly<Material>`, `Readonly<StandardPbrMaterialProperties>`, `Readonly<SceneRenderProxy>`, `Readonly<MeshGeometry>`, `Readonly<Texture>`. The deliberately-mutable runtime (`GlSceneRuntime`, `GlMeshUpload`) and `GlRenderState` are correctly left writable, with the mutability documented.
- **Sentinels for expected failure.** `getGlMeshMaterialRenderer` and `resolveGlMeshMaterialRenderer` return `null` for an unregistered/unresolvable kind; `drawGlScene` skips `null`-resolving subsets rather than throwing. Throwing is reserved for shader programmer errors only.
- **Accessor/predicate prefixes correct.** `get*` returns values/renderers/sources; the lone boolean is `hasTexturePixels` (`has*`), correctly named — no `get*` returning a boolean.
- **No out-param alias hazards.** Out-style writes go to dedicated module scratch objects, never to a caller object aliased with an input, so the read-before-write rule is not at risk here.
- **Type hygiene.** Every cross-package type comes from `@flighthq/types` / `@flighthq/materials` / `@flighthq/camera` (none defined inline); `import type {}` is always on its own line (no `import { type Foo, bar }` mixing). Package is `"sideEffects": false` with registration exposed only through opt-in `register*` functions — no top-level renderer registration.
- **No teardown-verb misuse.** The package exposes no `dispose*`/`destroy*`/`release*`; teardown is deferred to the render-state owner (noted in `glSceneRuntime` comments), so there is no synonym drift to flag.
