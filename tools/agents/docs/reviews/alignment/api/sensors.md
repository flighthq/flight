# API Alignment: @flighthq/sensors

**Verdict:** Strong. The exported surface follows the event-capability seam pattern (create / attach / detach / dispose / `get*Backend` / `set*Backend` / `createWeb*Backend`) consistently with sibling event packages, uses full unabbreviated type words, and throws nowhere; the only gaps are a missing out-param snapshot accessor (symmetry with `power`/`network`) and a shared-mutable-buffer hazard in the signal payloads.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Low | (package) | No out-param snapshot accessor for the latest reading. Sibling event packages expose `getNetworkStatus(out: NetworkStatus)` and `getPowerStatus(out: PowerStatus)` for hot-path polling, but `sensors` only ships allocating `createMotionReading()` / `createOrientationReading()` constructors and pure signal delivery. A consumer that wants the current value without subscribing has no zero-alloc path. | If polling is in scope, add `getMotionReading(out: MotionReading): MotionReading` / `getOrientationReading(out: OrientationReading): OrientationReading` (and magnetometer) mirroring `getPowerStatus(out)`. If signal-only delivery is the deliberate design, no change — but the asymmetry with `power`/`network` is worth a one-line rationale. |
| Low | `createWebSensorsBackend` (signal payloads) | The web backend reuses module-level scratch buffers (`_motionAcceleration`, `_motionRotationRate`, `_orientation`, `_magnetometer`) and passes them straight into the listener / signal emit. A listener that stores the `Readonly<MotionReading>` it receives will see it mutate on the next event. The `Readonly<>` annotation prevents writes but not aliasing surprise. | Document on `Sensors` signals / `SensorsBackend.subscribe*` that the reading is a transient view valid only during the callback and must be copied to retain — or have `attachSensors` copy into per-`Sensors` buffers before emit. Matches the "comment when a name cannot carry aliasing" rule. |
| Info | `requestSensorsPermission` | Returns `Promise<boolean>` and resolves `true` when ungated/granted, `false` on denial/error — sentinel-on-expected-failure is correct and no throw leaks. Noted only to confirm the async permission path conforms. | None. |

## Clean

- **Full type words, no abbreviation.** Every export carries the complete type word it operates on: `attachSensors`, `disposeSensors`, `createMotionReading`, `createOrientationReading`, `createWebSensorsBackend`, `getSensorsBackend`, `setSensorsBackend`, `requestSensorsPermission`. No `getSensorRdg`-style elisions.
- **Verb consistency with sibling event packages.** `attachSensors` / `detachSensors` / `disposeSensors` / `createSensors` / `createWebSensorsBackend` / `getSensorsBackend` / `setSensorsBackend` line up 1:1 with `@flighthq/network` (`attachNetwork` …) and `@flighthq/power` (`attachPower` …). Same seam, same names.
- **Allocation discipline by verb.** `create*` functions are the only allocators (`createMotionReading`, `createOrientationReading`, `createSensors`, `createWebSensorsBackend`); none of the non-`create*` exports allocate on the public path.
- **Teardown verbs used correctly.** `disposeSensors` detaches backend subscriptions and releases the entity to GC (no non-GC resource to free), and its doc comment states exactly that — correct `dispose*` vs `destroy*` choice. `detachSensors` is the inverse of `attachSensors` and is idempotent / safe-when-unattached.
- **Sentinels over throws.** No `throw` anywhere; the web backend degrades to no-op unsubscribers when `window`/`Magnetometer`/`DeviceMotionEvent` are absent, returns `true` for ungated permission, `false` on denial, and `-1` for unknown compass heading.
- **Accessor / boolean prefixes.** `getSensorsBackend()` is a true getter returning `SensorsBackend`; no `get*` returns a boolean and there are no mis-prefixed boolean exports.
- **Idempotent attach.** `attachSensors` calls `detachSensors` first, so re-attaching tears down the prior subscription — no listener leak on double-attach.
- **Type hygiene.** `import type { … }` is on its own line; all cross-package types (`MotionReading`, `OrientationReading`, `Sensors`, `SensorsBackend`) come from `@flighthq/types`, defined in `packages/types/src/Sensors.ts`, not inline. Web-only DOM shapes (`WebDeviceMotionEvent`, `WebMagnetometer`, …) are kept local and unexported, correctly not leaked to the header layer.
- **Side-effect-free module.** `"sideEffects": false`; the backend is lazily created in `getSensorsBackend` rather than registered at module top level, and no listeners/timers start on import.
