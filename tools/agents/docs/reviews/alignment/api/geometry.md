# API Alignment: @flighthq/geometry

**Verdict:** Strong and internally disciplined overall (alias-safe out-params, full type words, `Readonly<>` everywhere, clean `import type` lines, correct `create`/`clone`/`copy`/`set`/`acquire`/`release` allocation verbs) — but four real consistency defects: the singular-matrix failure contract differs across `inverse*`, `normalizeVector2` is a different operation wearing the `normalize` name, boolean predicates use three competing prefixes, and the `setRectangle*` mutator family splits between `out` and `target`.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `inverseMatrix` / `inverseMatrix3` / `inverseMatrix4` | Three different contracts for the **same** expected-failure case (singular / det == 0). `inverseMatrix` (2D) silently writes a degenerate matrix and returns `void`; `inverseMatrix3` `throw new Error('Matrix is not invertable')` (matrix3.ts:190); `inverseMatrix4` returns `boolean` (`false`). A non-invertible matrix is data-dependent, not programmer error, so throwing violates the sentinel rule, and the trio disagrees with itself. | Pick one contract for all three. Per convention, return `boolean` (the `inverseMatrix4` shape) — `false`/no-write on singular input. Remove the `throw`. |
| High | `normalizeVector2` | Not a normalize. Signature is `(out, source, newLength): void` — it scales to a caller-supplied length and returns nothing, while `normalizeVector3`/`normalizeVector4` are `(out, source): number` (unit-normalize, return prior length). Same verb, different operation, different arity, different return. Misleading and breaks 2/3/4 symmetry. | Make `normalizeVector2(out, source): number` match v3/v4 (unit length, return length). Move the scale-to-length behavior to a distinctly named function (e.g. `setVector2Length` / `scaleVector2ToLength`); add the v3/v4 siblings if wanted. |
| Medium | `getAabbContainsPoint`, `getBoundingSphereContainsPoint` | `get*` prefix on a `boolean`-returning predicate. `get*` is reserved for accessors; booleans take `has*`/`is*`/a bare predicate verb. | Rename to a non-`get` predicate consistent with the chosen package style (see next row), e.g. `containsAabbPoint` / `containsBoundingSpherePoint` or `isAabbContainingPoint`. |
| Medium | containment/intersection predicates (cross-file) | Three competing schemes for the same concept: bare verb (`containsRectanglePoint`, `enclosesRectangle`, `intersectsRectangle`), `is*Verb-ing` (`isFrustumContainingPoint`, `isFrustumIntersectingAabb`), and `get*Contains` (the two above). | Settle on one (the bare-verb `containsXPoint` / `intersectsX` form is already the majority and reads best), and bring frustum + aabb + sphere onto it. |
| Medium | `setRectangleBottom/Left/Right/Top/BottomRight` vs `setRectangleSize/TopLeft/Identity-style` | Mutable-target parameter named `target` in five setters but `out` in `setRectangleSize` and `setRectangleTopLeft` — one family, two names for the same role. | Standardize on `out` (the package-wide mutation-output name) across the whole `setRectangle*` group; reserve `target` only if a deliberate distinction is intended (there is none here). |
| Low | `addVector2`/`addVector3`/`addVector4`, `multiplyMatrix*` use `(out, a, b)` while `subtractVector2/3/4` and `crossVector3` use `(out, source, other)` | Binary vector/matrix ops disagree on operand parameter names for structurally identical signatures. | Pick one convention for binary operands (`a, b` for symmetric ops; `source, other` is fine but apply it uniformly) and align `add`/`subtract`/`cross` to it. |
| Low | `expandRectangleToPoint(out, sourceRect, sourceVec2, ...)` | Outlier param names `sourceRect` / `sourceVec2` where the rest of the file uses `source` + a role name (`point`, `other`). The `Vec2`/`Rect` abbreviations also clip the type word. | Rename to `source` (the rectangle) and `point` (the Vector2), matching `expandAabbByPoint(out, aabb, point)` and `offsetRectangleByPoint`. |
| Low | `setMatrixFromFloat32Array` / `writeMatrixToFloat32Array` exist for `Matrix` and `Vector2` only | `Vector3`/`Vector4` (and `Matrix3`/`Matrix4`) have no Float32Array read/write siblings — a symmetry/completeness gap rather than a naming defect. Flagged for the depth pass, not blocking here. | Add the missing `set*FromFloat32Array` / `write*ToFloat32Array` for v3/v4 (and the matrix variants) if buffer interop is intended to be uniform. |

## Clean

- **Full, unabbreviated type words** in every exported name — `getRectangleBottomRight`, `transformBoundingSphereByMatrix4`, `setMatrix3NormalFromMatrix4`. No `Vec`/`Mat`/`Rect` abbreviations in function names (only the one param-name slip noted above).
- **Allocation discipline by verb is textbook.** `create*`/`clone*` allocate and return entity types; `copy*`/`set*` mutate via `out`; pool `acquire*`/`release*` are matched brackets (`acquireVector2`/`releaseVector2`, plus `acquireEmpty*`/`acquireIdentity*` variants and `clear*Pool`). Math/transform/bounds helpers all write into `out` and do not allocate.
- **Out-params are alias-safe.** Spot-checked `multiplyMatrix`, `crossVector3`, `inverseMatrix` — all read every input field into locals before writing any output field, so `out === source` is safe.
- **`Readonly<T>` is applied consistently** to all non-mutated object params and `*Like` inputs; mutable outputs are `out`/`target` (the latter is the only inconsistency, noted above).
- **`equals*` family returns `boolean` and accepts `null | undefined`** (sentinel-friendly comparison) rather than throwing — correct expected-failure handling.
- **Throws are confined to genuine programmer error** in the row/column accessors (`getMatrix3Element`, `setMatrix4Element`, etc.) — index-out-of-bounds is API misuse, so `RangeError` is correct there. The only misplaced throws are the `inverse*` singular cases above.
- **`import type {}` is always on its own line**; no inline `import { type Foo, bar }`. `*Like` types are sourced from `@flighthq/types`, not redefined inline.
- **`create*` returns entity types while inputs accept `*Like`** — matches the entity-vs-structural-literal rule.
