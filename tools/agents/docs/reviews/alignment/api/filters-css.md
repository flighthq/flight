# API Alignment: @flighthq/filters-css

**Verdict:** Mostly clean and consistent internally, but two real issues stand out — a cross-package geometry helper (`getShadowFilterOffset`) is trapped in this CSS backend and allocates instead of using an `out` param — plus a package-naming asymmetry (`filters-css` vs the `*-canvas`/`*-gl`/`*-wgpu`/`*-surface` siblings).

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `getShadowFilterOffset` | This is a **backend-agnostic geometry helper** (angle + distance → dx/dy for `DropShadowFilter \| InnerShadowFilter \| BevelFilter`), not a CSS concern. It lives only in `filters-css`, yet `filters-surface/src/surfaceDropShadowFilter.ts` already references it by name in its doc comment — meaning the shared offset math is owned by the CSS backend, forcing any other backend that wants the same offset to import from `@flighthq/filters-css` or silently re-derive it. Cross-backend math should not be reachable only through one backend's barrel. | Move the function to a backend-neutral home — `@flighthq/filters` (data + shared math) — and have `filters-css` (and surface/canvas/gl) import it. Keep one definition of the shadow-offset rule. |
| High | `getShadowFilterOffset` | **Allocates per call.** Returns a fresh `{ dx, dy }` object every invocation. This is hot-path transform/offset math (run per filter per frame), which by the allocation-discipline rule must write into an `out`/`target` parameter, not allocate. The `get*` prefix is also wrong for an allocating compute — `get*` is for accessors. | Reshape to `computeShadowFilterOffset(filter, out: { dx; dy })` (or write into a `Vector`-like out), no allocation. Rename off `get*` since it computes rather than reads a stored value. |
| Medium | package name `filters-css` | Naming asymmetry with every sibling backend. The other filter backends are `filters-canvas`, `filters-gl`, `filters-wgpu`, `filters-surface`; the CSS one is `filters-css`. That is internally fine (CSS is the backend token), but the **function naming diverges too**: siblings expose `applyBlurFilterToCanvas(...)` (verb `apply*`, backend as a `To<Backend>` suffix on the full type word), while this package exposes `computeBlurFilterCss(...)` (verb `compute*`, backend `Css` suffix). Different verb for what is conceptually the same per-backend filter realization. | This is a defensible split — CSS produces a _string_ (pure compute, no destination) while canvas _applies_ to a context — so `compute*` is arguably correct here and `apply*` would be misleading. Worth a one-line note in the package/docs that `compute*Css` returning a string is intentionally distinct from `apply*To<Backend>`. No rename required, but keep the `compute*` verb reserved for "produce a descriptor/string" across backends so the distinction stays principled. |
| Low | `rgbaFromInt` (internal, in both files) | Duplicated verbatim in `cssDropShadowFilter.ts` and `cssOuterGlowFilter.ts`. Not exported, so not an API issue, but it is the same color-unpack logic copied. If `getShadowFilterOffset` moves to `@flighthq/filters`, a shared `formatRgba`/color helper could ride along. | De-duplicate into one shared internal (or a `@flighthq/filters` color util). Minor. |
| Low | color unpacking (`>> 16`/`>> 8`/`& 0xff` + separate `alpha`) | `rgbaFromInt` treats `filter.color` as packed **RGB** with alpha supplied separately. This matches the `DropShadowFilter.color` type (`color?: number` + `alpha?: number`), so the export is correct against its input contract. It does, however, sit against the SDK map's stated "colors are packed RGBA `0xRRGGBBAA`" convention. That is a `@flighthq/types` decision (the filter types model RGB+alpha), not a `filters-css` bug — flagging only so the divergence is visible at audit time. | No change in this package. Track the RGB-vs-RGBA filter-color convention at the `@flighthq/types` layer. |

## Clean

- **Full, unabbreviated type words** in every export: `computeBlurFilterCss`, `computeDropShadowFilterCss`, `computeOuterGlowFilterCss`, `getShadowFilterOffset` — no `DO`/`Obj`/`Blur`-shortening.
- **Globally unique** exported names; the `Css` suffix keeps them distinct from the canvas/gl/wgpu/surface siblings.
- **Sentinels for expected failure, no throws.** Every `compute*Css` returns `string | null`, returning `null` for the genuinely-unrepresentable cases (anisotropic blur, knockout, zero/negative blur) rather than throwing. Exactly the expected-failure-→-sentinel rule.
- **`Readonly` intent honored at the type layer.** Inputs are the `*Filter` interfaces whose fields are all `readonly`; functions are pure and mutate nothing.
- **`import type {}` on its own line** in all three source files (`import type { BlurFilter } from '@flighthq/types'`, etc.); no inline `import { type X, y }` mixing.
- **Cross-package types come from `@flighthq/types`** (`BlurFilter`, `DropShadowFilter`, `OuterGlowFilter`, `InnerShadowFilter`, `BevelFilter`) — none defined inline; sole runtime dependency is `@flighthq/types`.
- **Exports alphabetized** within `index.ts` and within files; `compute*Css` verb is consistent across the three filter computers.
- **`sideEffects: false`**, single `.` export, thin barrel re-export — no per-file subpaths. Packaging shape matches the map.
