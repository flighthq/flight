# API Alignment: @flighthq/tray

**Verdict:** Mostly conformant — correct command/event seam shape and full type words, but one real inconsistency: the setters split between the `TrayIcon` and `Tray` type words (`setTrayIconTitle`/`setTrayIconTooltip` vs. `setTrayContextMenu`), and the entity-vs-raw-id seam is asymmetric (mutators take `TrayIcon`, `onTrayEvent` delivers a raw `id: number`).

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `setTrayContextMenu` vs. `setTrayIconTitle` / `setTrayIconTooltip` | Three mutators operate on the same `TrayIcon` entity but disagree on the type word: two spell it out (`setTrayIcon*`) while `setTrayContextMenu` drops `Icon`. The convention requires the full, unabbreviated type word and verb/word consistency for the same concept across the package. As written, a reader cannot predict whether a tray mutator is `setTray*` or `setTrayIcon*`. | Pick one form for all three. `setTrayIconContextMenu` aligns them on the entity (`TrayIcon`), but note the codebase map (index.md L247) blesses the name `setTrayContextMenu` specifically — so the more likely-correct resolution is to rename the other two to `setTrayTitle` / `setTrayTooltip` and standardize the package on `setTray*`. Either way, make all three consistent. |
| Low | `onTrayEvent` listener `(id: number, event: TrayEventType)` | The mutators are entity-facing (`tray: TrayIcon`), but the event listener delivers a raw `id: number` instead of a `TrayIcon`. A consumer holding a `TrayIcon` from `createTrayIcon` must compare `tray.id === id` by hand to route events back. Sibling event seams key on the same token they hand the caller (`onNotificationClick` → `tag`, `onMenuSelect` → `id: string`), but here the create/mutate side speaks `TrayIcon` and the event side speaks `number`. | Consider delivering `(tray: Readonly<TrayIcon>, event: TrayEventType)` for symmetry with the mutators, or document that `TrayIcon.id` is the correlation key. Backend stays id-based; the public `onTrayEvent` wrapper can wrap the id into `{ id }`. Surfaced as a suggestion since the payload shape also lives in `@flighthq/types` (`TrayBackend.subscribe`). |
| Low | `destroyTrayIcon(tray: TrayIcon)` | `destroy*` is the correct verb only if a non-GC host resource is freed now. The `TrayIcon` entity is plain GC memory (`{ id }`), but the native tray icon behind the id is a real OS handle, so `destroy*` is defensible. The borderline point: on the web backend `destroy` is a pure no-op, so for the default backend nothing is actually freed. | Keep `destroy*` — the verb is chosen for what the native backend does (frees an OS tray handle), which is the right basis. No change; flagged only to confirm the verb choice was deliberate, not `dispose*`-vs-`destroy*` drift. |
| Low | `destroyTrayIcon` parameter type | `destroyTrayIcon(tray: TrayIcon)` takes a mutable `TrayIcon`; the function only reads `tray.id`. Per the `Readonly<>`-by-default rule, parameters that are not mutated should be `Readonly<>`. Same applies to `setTrayIconTitle` / `setTrayIconTooltip` / `setTrayContextMenu`, which all only read `tray.id`. (`createTrayIcon` correctly uses `Readonly<TrayIconOptions>`.) | Mark the `tray` parameter `Readonly<TrayIcon>` on all four functions that consume but do not mutate the entity. |
| Info | No `isTraySupported` | Sibling capabilities expose a boolean support query (`isNotificationSupported`). Tray instead signals "unsupported" by returning `null` from `createTrayIcon` (the backend's `-1`). That is a valid sentinel-based alternative and arguably better (no separate query to keep in sync), but it diverges from the suite's explicit-`is*Supported` pattern. | Optional: add `isTraySupported(): boolean` for suite symmetry, or document that `createTrayIcon` returning `null` is the supported-check. No rename needed. |

## Clean

- **Full, unabbreviated type word in most exports.** `createTrayIcon`, `destroyTrayIcon`, `createWebTrayBackend`, `getTrayBackend`, `setTrayBackend`, `onTrayEvent`, `setTrayIconTitle`, `setTrayIconTooltip` all carry `Tray`/`TrayIcon` in full; nothing abbreviated to `Tr*`/`*Tip`.
- **Globally unique root exports.** Every name is `*Tray*`-qualified, so no collisions with sibling capability barrels or the SDK barrel.
- **Command-capability triad is exact.** `createWebTrayBackend` / `getTrayBackend` / `setTrayBackend` match the documented command-capability shape and the `notification`/`clipboard`/`menu` siblings verbatim, including lazy default-backend creation in `getTrayBackend` (no top-level side effect).
- **`on*`-over-`subscribe*` event seam.** `onTrayEvent` is the exact name blessed in the codebase map (index.md L247) and correctly wraps the backend's `subscribe`, returning an unsubscribe `() => void`.
- **Allocation discipline.** The only allocators are `createTrayIcon` (returns a new `{ id }`) and `createWebTrayBackend` (returns a new backend) — both correct `create*`. No `out`-param hot-path functions exist, so alias-safety is N/A.
- **Sentinels for expected failure, never throws.** `createTrayIcon` maps the backend's `-1` to a `null` `TrayIcon`; the web backend's `create` returns `-1` and every mutator is a guarded no-op. No thrown errors for the expected web/no-tray case, no validation of unreachable invariants.
- **`get*` accessor, no boolean-getter misuse.** `getTrayBackend(): TrayBackend` returns a non-boolean; no `get*` returns a boolean and no boolean-returning function is mislabeled.
- **No teardown-verb drift.** Unsubscribe is a returned closure, not a mislabeled `dispose*`; `destroy*` is used for the native-handle teardown (see Findings) and `dispose*`/`acquire*`/`release*` are correctly absent.
- **Types sourced from `@flighthq/types`.** `MenuItemTemplate`, `TrayBackend`, `TrayEventType`, `TrayIcon`, `TrayIconOptions` are imported via a single `import type {}` line; nothing cross-package is defined inline. `setTrayContextMenu` reuses `MenuItemTemplate` consistent with `@flighthq/menu` and `@flighthq/app`'s `setAppDockMenu`.
- **Package manifest is clean.** Declares only `@flighthq/types`, `sideEffects: false`, single `.` export, no per-file subpaths.
- **Exports alphabetized** in `tray.ts`; `index.ts` is a thin `export *` barrel.
