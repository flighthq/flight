# API Alignment: @flighthq/effects

**Verdict:** Strongly conformant — a uniform set of `create*Effect` data factories that obey naming, allocation, `Readonly`, and types-layer rules; the only real issue is one factory missing the default-argument that its 42 siblings share, plus one structural asymmetry worth noting.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `createChannelMixerEffect` | Parameter-default asymmetry: every other `create*Effect` factory takes `options: Readonly<Omit<…, 'kind'>> = {}` (callable with no args), but this one omits the `= {}` default, so `createChannelMixerEffect()` is a type error while `createBloomEffect()` etc. all work. Breaks call-site symmetry across the package. If the intent is "channel-mixer has no usable default and the caller must supply a matrix", that is a deliberate exception that should be justified, not left as a silent inconsistency. | Add `= {}` to match siblings, or — if a matrix is genuinely required — make that explicit by typing the required field as non-optional (`Readonly<Omit<ChannelMixerEffect, 'kind'>>` where the matrix field is required in the type) so the asymmetry reads as intentional rather than accidental. |
| Low | `computeBloomBlurRadius` | Lone non-factory export in the package: it is the only effect with extracted "substrate-agnostic recipe math" (the file comment says the Gl and Wgpu bloom recipes derive identical parameters from it). The `compute*` verb and `Readonly<BloomEffect>` parameter are correct and it returns a plain `number` (no allocation, no `out` needed). The concern is consistency/completeness, not a rule violation: other effects with shared cross-backend math (blur radii, kernel sizes, grade coefficients) presumably duplicate that math in each backend instead of exposing a sibling `compute*`. | None required for conformance. Surface as a completeness question: should other multi-backend effects also expose their shared recipe math here so the agnostic package is the single source of derived parameters? This crosses into the backend packages, so raise it rather than acting on it. |

## Clean

- **Full, unabbreviated type words.** Every export spells the complete effect type name: `createBrightnessContrastEffect`, `createChromaticAberrationEffect`, `createScreenSpaceFogEffect`, `createLookupTableGradeEffect`, `createBokehDepthOfFieldEffect`. No abbreviations. Acronym effects (`createFxaaEffect`, `createSmaaEffect`, `createSsaoEffect`, `createSsrEffect`, `createTaaEffect`, `createCrtEffect`) carry the canonical industry term as the type word, which is the correct mature-naming choice.
- **Globally-unique, self-identifying names.** Each `create*Effect` is unique from the barrel; none collide with the backend siblings (`apply*EffectToCanvas` / `…ToGl` / `…ToWgpu` live in separate packages and are distinct).
- **Allocation discipline by verb.** `create*Effect` allocates a new descriptor — correct for the `create*` verb. `computeBloomBlurRadius` is pure math returning a primitive, so it needs no `out`/`target`; no allocation in a hot path.
- **`Readonly<T>` on inputs.** Every factory parameter is `Readonly<Omit<…, 'kind'>>`; `computeBloomBlurRadius` takes `Readonly<BloomEffect>`. Mutation is never the default.
- **Cross-package types from `@flighthq/types`.** No effect type is defined inline; each file imports its interface (`BloomEffect`, `GrayscaleEffect`, …) from `@flighthq/types`, which is the header layer. The `kind` discriminant model (`RenderEffect` open base contract, literal PascalCase `kind`) lives in types, and the factories emit the matching canonical literal (`{ kind: 'BloomEffect', ...options }`), so the string key, serialized form, and type name agree 1:1.
- **`import type {}` on its own line.** Every source file uses a dedicated `import type { … } from '@flighthq/types'` line — no mixed `import { type Foo, bar }` form anywhere.
- **No teardown/accessor/sentinel pitfalls.** The package has no `dispose*`/`destroy*`/`acquire*`/`release*` (those correctly live in the backend packages that own GPU/canvas resources), no `get*` returning a boolean, and no thrown errors for expected-missing cases — there are no failure paths to mis-handle; factories always succeed.
- **Consistent shape and parameter ordering.** All 43 factories follow one signature template (`options` first and only argument), making the package trivially predictable; `kind` is consistently `Omit`-ted from `options` and supplied by the factory.
- **Side-effect-free.** No top-level registration or mutable state; `"sideEffects": false` holds. Backend runner registration (`registerGlRenderEffect`, etc.) is correctly pushed to the backend packages, keeping this package a pure data/recipe layer.
