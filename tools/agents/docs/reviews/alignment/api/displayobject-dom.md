# API Alignment: @flighthq/displayobject-dom

**Verdict:** Mostly aligned — names carry the full `Dom` type word and out/allocation discipline is clean, but a misused `release*` teardown verb, an inconsistent `enable*`/`Support` family, a barrel that leaks ~12 module-only exports, and an inline cross-package overlay type are the real gaps.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `releaseDomRenderCache` | `release*` is reserved for pool `acquire*`/`release*` brackets. There is no `acquireDomRenderCache`; this drops a `WeakMap` reference so the target canvas becomes GC-eligible (the body comment literally says "dropping the reference frees it"). That is `dispose*` semantics, not a pool release. | Rename to `disposeDomRenderCache`. Keep `release*` only if a matching `acquire*` is introduced. |
| Medium | `defaultHtmlViewRenderer`, `drawDomHtmlView` | Naming is split: the renderer const drops the `Dom` token (`defaultHtmlViewRenderer`) while its draw fn keeps it (`drawDomHtmlView`), and the source file is `htmlView.ts` not `domHtmlView.ts`. Every sibling pairs `default*Dom*Renderer` + `drawDom*` (e.g. `defaultDomShapeRenderer`/`drawDomShape`). | Rename to `defaultDomHtmlViewRenderer` (and file `domHtmlView.ts`) so the HTML-view renderer matches the `Dom`-prefixed family. |
| Medium | `enableDomRenderCache` | Breaks the `enable*Support` naming the rest of the family uses (`enableDomClipSupport`, `enableDomCssFilterSupport`, `enableDomBlendModeSupport`) — no `Support` suffix — and takes the base `RenderState` while the others take `DomRenderState`. Inconsistent verb shape for the same "turn this DOM capability on" concept. | Align to `enableDomRenderCacheSupport`; if it genuinely operates on any `RenderState`, document why it is the one cross-tier member of the family. |
| Medium | `DomTextInputOverlay` (type) | Public, cross-file callback type defined inline in `domRichText.ts` and consumed by `domTextInput.ts` / `registerDomTextInputOverlay`. Every other DOM type in the package's public signatures (`DomRenderState`, `DomClipHooks`, `DomStageRectangle`, `DomClipEntry`, `RenderProxy2D`) is imported from `@flighthq/types`; this one is the lone inline exception. | Move `DomTextInputOverlay` to `@flighthq/types` so the header layer remains the full design surface. |
| Medium | barrel `index.ts` | The root barrel omits ~12 functions/consts that are `export`ed from their modules: `buildDomContourClipPath`, `pushDomClipContours`, `buildDomScale9Mapper`, `createDomScale9ShapeData`, `drawDomScale9Shape`, `defaultDomScale9ShapeRenderer`, `hasDomStructureChanged`, `processDomNode`, `reconcileDomContainer`, `swapDomOrderLists`, `getDomFontAscentCached`, `setDomFontAscentCached`. They are `export` (so cross-file reachable and test-visible) but not package-public, creating an ambiguous "exported but not really public" tier. | Decide each: if public, add to the barrel; if internal, drop `export` (use package-internal imports) so the public surface is exactly the barrel. |
| Low | `getDomFontAscentCached` | Returns `number \| undefined`; the convention's expected-failure sentinels are `null` / `-1`, not `undefined`. (`getDomRenderCacheTarget` and `getDomCssFilter` correctly use `null` / `undefined`-as-absent, but the numeric lookup should prefer `-1` or `null`.) | Return `-1` (or `null`) for a missing ascent to match the sentinel convention for numeric lookups. |
| Low | `_domFontAscentCache` (in `domFontSource.ts`) | Process-global mutable `Map<string, number>` at module scope, keyed by font string (not per-`RenderState`). The render-cache map is a `WeakMap` keyed per state; this one is genuinely shared top-level mutable state. Lazily populated so import stays side-effect-free, but it is the kind of shared global the ground rules discourage. | Consider keying the ascent cache off the `DomRenderState` runtime slot (as the render-cache targets are) rather than a module global. |
| Low | `setDomTransform` vs `applyDomBlendMode` | Both write a single CSS property onto an `HTMLElement`, yet one uses `set*` and the other `apply*`. The package otherwise splits `apply*` (write derived state to an element) from `set*` (store on state), but these two same-shaped element writers disagree. | Pick one verb for "write a CSS property to an element" (likely `setDomTransform` → keep `set*`, or move blend to `setDomBlendMode`) so element writers are uniform. |
| Low | `applyDomBlendMode(element, value)` | Parameter named `value: BlendMode \| null` is generic where the type word would self-identify it. | Name it `blendMode` to match the type-word-in-identifier preference seen elsewhere (`filter`, `transform`, `rect`). |

## Clean

- **Full, unabbreviated type words throughout.** Every exported name spells `Dom` and the full subject — `renderDomDisplayObject`, `getDomRenderStateRuntime`, `applyDomClipRectangles`, `setDomTransformWithOffset`. No `getDOBounds`-style abbreviation anywhere.
- **Globally distinctive names.** The `Dom`-prefixed surface (e.g. `drawDomShape`, `drawDomRichText`, `enableDomClipSupport`) reads unambiguously against the canvas/gl/wgpu siblings.
- **Allocation discipline by verb is correct.** `create*` allocates (`createDomRenderState`, `createDomRenderStateRuntime`, `createDomStageRectangle`); `ensureDomRenderCacheTarget` allocates-or-reuses and is honestly named `ensure*`; the `draw*`/`apply*`/`set*` writers don't allocate value types.
- **`createDomStageRectangle` is alias-safe** — it reads every `rect`/`transform` field into locals (the 8 corner products) before constructing the result; `setDomTransformWithOffset` likewise destructures the matrix first.
- **`Readonly<T>` applied where mutation isn't intended** — `Readonly<RectangleLike>`, `Readonly<MatrixLike>`, `Readonly<Matrix>`, and `readonly DomClipEntry[]` on the clip/transform/rectangle inputs.
- **No thrown errors for expected-missing cases** — lookups return sentinels (`getDomRenderCacheTarget` → `null`, `getDomCssFilter` → `undefined`); the package throws nowhere.
- **Accessor/boolean prefixes correct** — `get*` getters (`getDomCssFilter`, `getDomRenderStateRuntime`, `getDomRenderCacheTarget`) return values, and the boolean `hasDomStructureChanged` uses `has*`.
- **`import type { }` hygiene is clean** — every type import is on its own `import type` line; no `import { type Foo, bar }` mixing. Cross-package types (except `DomTextInputOverlay`) come from `@flighthq/types`.
- **Opt-in, side-effect-free registration** — no `registerRenderer` at module top level; renderers are exported `default*Renderer` consts plus `enable*` functions the caller invokes, consistent with the SDK's explicit-registration philosophy.
