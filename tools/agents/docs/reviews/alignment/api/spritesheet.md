# API Alignment: @flighthq/spritesheet

**Verdict:** Naming and sentinel discipline are clean, but the package leaks cross-package `*Data` types defined inline (consumed by `spritesheet-formats`), eagerly allocates signals with no opt-in or `dispose*`, and is inconsistent about `Readonly<>` on read-only parameters.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `SpritesheetData`, `SpritesheetAnimationData`, `SpritesheetFrameData`, `SpritesheetAnimationDirection` (in `spritesheetData.ts`) | Cross-package types defined **inline** in the package, not in `@flighthq/types`. They cross the boundary: `@flighthq/spritesheet-formats` imports them (`import type { SpritesheetAnimationData, SpritesheetData, SpritesheetFrameData } from '@flighthq/spritesheet'` in `asepriteParse.ts`, and similarly in the texturepacker/starling parse+serialize files). The CLAUDE rule: shared types that cross package boundaries belong in `@flighthq/types` (the header layer), not inline in a package, and consumers should import them from there. | Move the four type declarations to `@flighthq/types` (`SpritesheetData.ts` etc.), re-export from its index, and have both `spritesheet` and `spritesheet-formats` import them from `@flighthq/types`. The `create*Data` constructors stay in this package. |
| Medium | `createSpritesheetPlayer` | Eagerly allocates two signals unconditionally (`onComplete: ... ?? createSignal()`, `onLoop: ... ?? createSignal()`). The CLAUDE signals rule is that signal groups are opt-in via `enable*` functions "which is when the associated cost is assumed" — every player here pays for two signal objects whether or not anyone listens, and the signal machinery cannot tree-shake out. | Make the signals lazy/opt-in (e.g. an `enableSpritesheetPlayerSignals(player)` that creates the signals, with `updateSpritesheetPlayer` guarding `if (player.onComplete) emitSignal(...)`), mirroring `enableDisplayObjectSignals`. The `SpritesheetPlayer` type's `onComplete`/`onLoop` would become nullable. |
| Medium | `SpritesheetPlayer` (no `disposeSpritesheetPlayer`) | The player owns two signals but the package exports no teardown verb. An entity that holds signals should expose `dispose*` to detach listeners and release them to GC (the `dispose*` = detach/release-to-GC convention). Long-lived players that swap animations accumulate signal listeners with no documented release path. | Add `disposeSpritesheetPlayer(player)` that clears the signal listeners (and `queue`). Pairs naturally with the opt-in signal change above. |
| Low | `getSpritesheetAnimation(spritesheet: Spritesheet, label)` | Parameter `spritesheet` is read-only in the body but not marked `Readonly<Spritesheet>`, while sibling readers (`getSpritesheetPlayerFrame`, `createSpritesheetTimelineSource`) do use `Readonly<>`. Inconsistent application of the "Readonly by default" rule. | Change signature to `getSpritesheetAnimation(spritesheet: Readonly<Spritesheet>, label: string)`. |
| Low | `playSpritesheetAnimation`, `queueSpritesheetAnimation` | Named after the `Animation` they accept as the second argument, but they operate on (mutate) the `SpritesheetPlayer` passed first — unlike the player-named siblings `updateSpritesheetPlayer` / `getSpritesheetPlayerFrame`. The "function name includes the full type it operates on" rule points at `Player`. The current names read well as verbs, so this is a deliberate-tension call, but the asymmetry is worth a decision. | Either accept and document the verb-reads-better exception, or rename toward the operated-on type (e.g. `playSpritesheetPlayerAnimation` is clumsy — more likely keep as-is, but record the intent). |
| Low | `createSpritesheetAnimation` vs `createSpritesheetAnimationData` | Diverging defaults for the same conceptual fields: animation defaults `frameDuration: 0`, `loop: false`; the data form defaults `frameDuration: 100`, `loop: true`. Not a naming-convention break, but a surprising inconsistency between the runtime entity constructor and its data twin. | Align the defaults (or document why the runtime entity intentionally starts inert at `frameDuration: 0`). |

## Clean

- **Full, unabbreviated type words** in every export — `createSpritesheet`, `createSpritesheetAnimationData`, `getSpritesheetPlayerFrame`, `createSpritesheetFromTileset`. No abbreviation of `Spritesheet`/`Animation`/`Frame`/`Player`.
- **Globally unique names** — every export is `Spritesheet`-prefixed; no collision risk from the root barrel.
- **Allocation by verb** — `create*` allocate; lookups (`getSpritesheetAnimation`, `getSpritesheetPlayerFrame`) and mutators (`updateSpritesheetPlayer`, `playSpritesheetAnimation`, `queueSpritesheetAnimation`) do not allocate new entities.
- **Sentinels for expected failure** — `getSpritesheetAnimation` and `getSpritesheetPlayerFrame` return `null` on miss; nothing throws for expected-missing cases.
- **`get*` accessors** are genuine getters returning entities; no boolean-returning `get*`. `updateSpritesheetPlayer`'s boolean is an advanced/changed result on an update verb, not a mis-prefixed query.
- **`import type {}` on its own line** in every file; value and type imports are not mixed inline.
- **Entity constructors over literals** — `createSpritesheetFromTileset` builds via `createSpritesheet`/`createSpritesheetFrame` rather than object literals.
- **`createSpritesheetTimelineSource`** is exemplary: `Readonly<>` on both inputs, a per-target `WeakMap` so the source is shareable, and a thorough ownership/coordinate comment explaining the lazy-bitmap and the cross-package `TimelineSource` contract that keeps `timeline` and `spritesheet` decoupled through `@flighthq/types`.
