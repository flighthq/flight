# API Alignment: @flighthq/types

**Verdict:** Strong adherence overall — this is a disciplined header layer (flat `export *` barrel, no inline cross-package types, kinds as PascalCase strings, sentinel-not-throw backend contracts, `out`-params, `Readonly<>` on inputs) — but it has one clear convention break (`AppearanceFlags` ships merged-namespace methods that read as OOP-bound, non-self-identifying functions living in the header) plus several smaller consistency drifts (TS `enum` vs `as const` for the enum-like concept, acronym casing, filename↔type-name mismatch).

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `AppearanceFlags` (`namespace` merge: `.any`, `.has`, `.add`, `.remove`, `.clear`) | The `enum AppearanceFlags` is merged with a `namespace AppearanceFlags` carrying live functions. This breaks several rules at once: (1) functions in the **header layer** — `@flighthq/types` is meant to be type/kind declarations, and these are the only real runtime functions in the package, hidden from `npm run api` because they are namespace-scoped; (2) **not globally self-identifying** — `AppearanceFlags.has`/`.add`/`.remove` carry no type word, the opposite of the "function names contain the full unabbreviated type word, globally unique" rule; (3) **method/OOP binding shape** the map argues against (free functions over methods, C/C++ portability). | Move the logic to the owning package (`@flighthq/node` / appearance), expose as free functions `hasAppearanceFlags(flags, test)`, `addAppearanceFlags(flags, add)`, `removeAppearanceFlags(flags, remove)`, `anyAppearanceFlags(flags, test)`. Keep only the flag values in `@flighthq/types`. |
| High | `AppearanceFlags.clear()` | Named as a teardown/mutation verb but takes no arguments and just returns `AppearanceFlags.None` — it is a constant, not an operation. Misleading verb. | Drop it; callers use `AppearanceFlags.None` directly. |
| Medium | `BlendMode` (`enum`) | User-facing value type modeled as a TS `enum`, while the sibling user-facing enum-like value types (`PathCommand`, `ImageChannel`, `MouseButton`, `KeyCode`, `KeyModifier`) use the `const {…} as const` + `type` pattern. Two patterns for one concept; `enum` also resists tree-shaking and has no string-serialized form, unlike the `as const` peers. | Convert `BlendMode` to the `const BlendMode = {…} as const` + `export type BlendMode` pattern to match the user-facing enum-like value types. (Internal bitflag/format enums `BatchFormat`, `BatchBarrier`, `AppearanceFlags` are a defensible separate case; the inconsistency to resolve is the user-facing one.) |
| Low | `ClipboardBackend.readRTF` / `writeRTF` | Acronym casing diverges from its own neighbors: the same interface uses `readHtml` / `writeHtml` (PascalCase `Html`) but `readRTF` / `writeRTF` (all-caps `RTF`). Inconsistent acronym treatment within one interface. | Pick one acronym convention and apply it package-wide; given `Html`/`Url`/`Dom`/`Gl`/`Wgpu` PascalCase elsewhere, prefer `readRtf` / `writeRtf`. |
| Low | `DOMRenderOptions.ts` → `DomRenderOptions`; `DOMStageRectangle.ts` → `DomStageRectangle` | Filename uses all-caps `DOM` while the exported type inside is PascalCase `Dom` (and `DomRenderState.ts` already uses `Dom` in both). File↔type-name drift; the exported identifiers are consistent, the filenames are not. | Rename files to `DomRenderOptions.ts` / `DomStageRectangle.ts` to match the exported type and `DomRenderState.ts`. (Filename-dimension item; noted here because it desyncs from the exported name.) |
| Low | `FileSystemBackend.fileExists` | Boolean-returning method named `*Exists` rather than the `has*`/`is*` convention used elsewhere (`hasText`, `isSupported`, `isRegistered`). Minor: `fileExists` is a widely-understood standard spelling, so this is borderline. | Optional: `hasFile` for strict convention adherence, or leave as a deliberate standard-API spelling and note it. |
| Low | `StorageBackend.getItem` / `setItem` / `removeItem` / `keys` | Generic `Item` carries no `Storage` type word (`getItem` not `getStorageItem`). These are backend-interface methods (namespaced by `StorageBackend`, not exported free functions), and they intentionally mirror the Web Storage `localStorage` API for familiarity — so this is acceptable, but worth recording as a deliberate exception rather than silent drift. | Keep as-is (deliberate Web Storage mirror); ensure the corresponding `@flighthq/storage` **free** functions do carry the type word (e.g. `getStorageItem`). |

## Clean

- **Barrel shape.** `index.ts` is a flat alphabetized `export *` over every file with a single `.` package entry and `"sideEffects": false` — exactly the thin re-export / single-root-entry rule. No per-file `exports` subpaths.
- **Kind identity.** Every `*Kind` is a PascalCase **string** constant (`BitmapKind = 'Bitmap'`, `SpriteKind`, `MeshKind`, all the material/light/force kinds), never a `Symbol()` — matches the string-kind identity model. The remaining `unique symbol` exports (`EntityRuntimeKey`, `NullScene`, `DisplayObjectTraitsKey`, `SceneNodeTraitsKey`, `MeshGeometry` `__*Data` brands) are exactly the permitted internal runtime-slot / brand / sentinel uses.
- **Cross-package types centralized.** This _is_ the header layer; no inline cross-package type definitions leaking elsewhere, and no `import { type Foo, bar }` inline-mixed type imports (all type imports are on their own `import type {}` lines).
- **Sentinels over throws.** Backend seams (`FileSystemBackend`, `ClipboardBackend`, `StorageBackend`, `TrayBackend`) document and return `null` / `'' ` / `false` / `-1` / no-op for expected-failure cases (denied access, missing entry) rather than throwing — the sentinel rule applied consistently across the platform suite.
- **Teardown verbs correct.** `Renderer.destroyData` and `TrayBackend.destroy` free non-GC resources (GPU data / native handles) and are correctly `destroy*`, not `dispose*`; `disposeImageResource` (referenced from `ImageResource`) correctly uses `dispose*` for release-to-GC. No synonym drift.
- **`out`-param + `Readonly<>` discipline.** Getter-with-output methods consistently take an `out`/`target` last param (`getInfo(out)`, `getBounds(win, out)`, `getStatus(out)`, `getSafeAreaInsets(out)`, `getScreens(out)`), and input objects are wrapped in `Readonly<>` (`Readonly<TrayIconOptions>`, `Readonly<Uint8Array>`, `Readonly<GeolocationRequestOptions>`, `Readonly<FileWatchEvent>`).
- **Accessor / boolean verbs.** `get*` methods all return values (none return boolean); booleans use `has*`/`is*` (`hasText`, `hasImage`, `hasSingleInstanceLock`, `isSupported`, `isRegistered`). Only `fileExists` sits outside this (see Findings).
- **Type words preserved in exported names.** Backend/interface and type names carry the full unabbreviated type word (`DisplayObjectRenderer`, `TextShaperBackend`, `ApplicationWindow`, `SurfaceFingerprint`); no abbreviated type words in exported identifiers (uppercase `GL`/`DOM`/`HTML` occurrences are in comments/standard-API references, not Flight-owned names).
