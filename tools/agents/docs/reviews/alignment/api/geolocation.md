# API Alignment: @flighthq/geolocation

**Verdict:** Structurally clean command-capability seam, but the package is split between two domain prefixes (`Geo*` vs `Geolocation*`) for the same concept — pick one and apply it consistently.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `clearGeoWatch`, `watchGeoPosition` (vs `getGeolocationBackend`, `setGeolocationBackend`, `requestGeolocationPermission`, `createWebGeolocationBackend`) | Two prefixes for one domain. The backend/permission/factory functions spell out the full domain word `Geolocation`, while the position/watch functions abbreviate it to `Geo`. The map rule is that a function's type/domain word must be the full unabbreviated word and globally self-identifying; `Geo` is an abbreviation of the `Geolocation` domain. A reader scanning the barrel sees an inconsistent vocabulary. | Choose one domain prefix for the whole package. Aligning on the full word gives `clearGeolocationWatch` and `watchGeolocationPosition`, matching the other five exports and the package/type names. |
| Medium | `createGeoPosition`, `getCurrentGeoPosition`, `watchGeoPosition` (the `GeoPosition` type) | The position type itself is named `GeoPosition` (a deliberate rename to dodge the lib.dom `GeolocationPosition` collision, documented in `packages/types/src/Geolocation.ts`). So `createGeoPosition` _does_ carry its full type word — but that makes `Geo` simultaneously a legitimate type word (in `GeoPosition`) and an abbreviation of the domain (in `clearGeoWatch`). This is the root of the inconsistency above: the type rename leaks a second vocabulary into the function names. | Resolve at the type layer if possible: keeping the entity `GeoPosition` but the domain `Geolocation` is what forces the split. If `GeoPosition` stays, treat `Geo` strictly as the position type word (`createGeoPosition` ok) and use `Geolocation` everywhere the domain is meant (`watchGeolocationPosition` watches the _domain_, not a `GeoPosition` value). Make the boundary deliberate, not incidental. |
| Low | `clearGeoWatch` / `watchGeoPosition` | Verb-pair asymmetry on top of the prefix split: the start function is `watch*Position` (named after what it returns/observes) but the stop function is `clear*Watch` (named after the handle). The pairing is correct (Web API parlance: `watchPosition`/`clearWatch`), but the noun flips between the two halves. | Acceptable since it mirrors the platform vocabulary, but if renamed for the prefix fix, prefer a consistent noun, e.g. `watchGeolocationPosition` / `clearGeolocationWatch` (handle-named stop is the established idiom, so keep `Watch` on the clear side). |

## Clean

- **Command-capability shape is textbook.** `getGeolocationBackend` / `setGeolocationBackend` / `createWebGeolocationBackend` exactly match the platform-suite seam pattern (cf. `clipboard`, `shell`, `haptics`). Backend default is lazily created inside `getGeolocationBackend`, never registered at module top level — `"sideEffects": false` is honored and the module is side-effect-free.
- **Sentinels over throws, correctly.** `getCurrentGeoPosition` resolves `null`, `watchGeoPosition` returns `-1`, `requestGeolocationPermission` resolves `false`, and `clearGeoWatch` is a no-op on unknown/absent backends. No expected-failure path throws — matches the rule precisely, and the web backend guards every `navigator.geolocation` call.
- **`createGeoPosition` allocation verb is right** — it allocates a zeroed value and is named `create*`, consistent with the allocation-discipline rule.
- **`Readonly<>` applied where mutation isn't intended:** `options?: Readonly<GeolocationRequestOptions>` on both public reads, and the `handler`/`listener` payload is `Readonly<GeoPosition>`. The `id: number` and primitive params are correctly exempt.
- **Accessor prefixes correct:** `getGeolocationBackend`/`getCurrentGeoPosition` are real getters; no `get*` returns a boolean. `requestGeolocationPermission` returning `Promise<boolean>` is an action (a permission request), correctly not named `get*`/`is*`.
- **Cross-package types sourced from `@flighthq/types`** (`GeolocationBackend`, `GeolocationRequestOptions`, `GeoPosition`), imported on a dedicated `import type {}` line in both source and test. Nothing defined inline. The lib.dom collision is handled with a local `GlobalGeolocationPosition` alias so source never references the colliding global name directly.
- **Exports are alphabetized** within `geolocation.ts`; loose module state (`_backend`, `_emptyOptions`) and private helpers sit below the public API as the source-style rule prescribes.
- **`setGeolocationBackend(null)` fall-back-to-web** is the documented, consistent seam idiom across the platform suite.
