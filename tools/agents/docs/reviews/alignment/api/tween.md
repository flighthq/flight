# API Alignment: @flighthq/tween

**Verdict:** Mostly aligned on naming and `Readonly<>` discipline, but carries a top-level mutable-singleton side effect (`defaultManager`), an inline cross-package type (`ColorComponents`) leaking into a public return, a stale `0xRRGGBB` color convention, and `Tween<any>` leakage; a few verb/symmetry rough edges remain.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `defaultManager` (tweenManager.ts) | Exported module-top-level singleton built by calling `createTweenManager()` at import time. This is shared mutable state plus a top-level side effect (allocates a `Map`) in a `"sideEffects": false` package — exactly what the Ground Rules forbid ("Do not … mutate shared state at module top level"). It also silently backs the no-manager `createTween` overload, hiding where tween state lives. | Remove the eager singleton and the implicit-default `createTween` overload; require an explicit `TweenManager`. If a convenience default is wanted, expose it lazily via a `getDefaultTweenManager()` accessor that constructs on first call, not a top-level `const`. |
| High | `ColorComponents` (colorTween.ts) | Cross-package type defined inline and leaked into the public API as the return type `Tween<ColorComponents>`. Violates "cross-package types come from `@flighthq/types`" and exposes an undocumented anonymous shape (`{ b; g; r }`) to callers. | Define the color-component shape in `@flighthq/types` (e.g. `ColorTweenComponents`) and import it, or have `createColorTween` return `Tween<object>` so the internal component bag is not part of the contract. |
| Medium | `createColorTween` | JSDoc states "packed `0xRRGGBB` color" and interpolates only r/g/b — but the Flight convention is packed **`0xRRGGBBAA`** (RGBA) with one consistent convention SDK-wide. Alpha is silently dropped. | Operate on `0xRRGGBBAA` and interpolate all four channels (mask `& 0xff` per byte, including alpha), matching the documented color convention; update the doc comment. |
| Medium | `pauseTween`, `resumeTween`, `stopTween`, `stopAllTweens`, `stopTweens` | Parameters typed `Tween<any>` / `NumericProps<any>`. `any` leaks into the public surface and defeats the generic-`T` discipline used elsewhere in the package. | Use `Tween<object>` (and `NumericProps<object>` / `Readonly<NumericProps<object>>`) so the public signatures are `unknown`-safe rather than `any`-typed. |
| Medium | `completeTween` vs `stopTween(tween, { complete: true })` | Two public entry points produce overlapping "snap to end + fire onComplete" behavior, with the snap/round logic duplicated verbatim across `completeTween`, `stopTween`, and `updateTween`. Verb drift (`complete*` vs `stop*` with a `complete` flag) and copy-paste divergence risk. | Pick one canonical verb. Keep `completeTween` as the single "finish now" function and have `stopTween`'s `complete` path delegate to it; factor the shared snap loop into one internal helper (e.g. `writeTweenFinalValues`). |
| Medium | `applyTween` | Name does not describe the operation: it does **not** create or run a tween — it instantly assigns `propertyMap` values to `target` (after stopping overlapping tweens). A reader expects "apply a tween," gets an immediate set. Names should be self-identifying. | Rename to convey instant assignment, e.g. `setTweenTargetValues` or `snapTweenTarget`, or fold it into `stopTweens` with a "jump to values" option. |
| Low | `resetAllTweens` | Verb `reset*` is unique in the package and just calls `manager.tweens.clear()` — i.e. it discards all tweens without finishing them, unlike `stopAllTweens`. "Reset" is ambiguous against `stop`. | Rename to `clearAllTweens` (it clears) or `removeAllTweens` to distinguish from the `stop*` family that snaps/fires events. |
| Low | `createTween` (overloaded) | The manager-optional overload exists only to support the `defaultManager` singleton. Beyond the side-effect concern above, the dual signature makes the first positional argument polymorphic (`TweenManager \| T`), which is harder to read and to port to Rust (no overloads). | Drop the overload once the implicit default is removed; keep a single explicit-manager signature. |
| Low | `Tween<T>` fields (types/Tween.ts) | The entity exposes all fields mutable (`complete`, `paused`, `elapsed`, `reverse`, …) and `pauseTween`/`resumeTween` mutate them directly rather than through a guarded API. This is a `@flighthq/types` shape decision, not strictly this package's export, but it is what the package's public functions act on. | Out of scope for this package alone; flag to the types owner whether tween runtime state should move to a runtime/opaque tier per the entity/runtime split. |

## Clean

- **Full type words in names.** Every exported function carries the unabbreviated `Tween` type word: `createTween`, `createTweenManager`, `createTweenTimer`, `createColorTween`, `updateTweens`, `pauseTween`/`pauseTweens`/`pauseAllTweens`, etc. No abbreviation drift.
- **Allocation by verb.** `create*` functions (`createTween`, `createTweenManager`, `createTweenTimer`, `createColorTween`) are the only allocators; `updateTweens`, `pause*`, `resume*`, `stop*` mutate in place and do not allocate.
- **`Readonly<>` on options.** All option parameters are `Readonly<TweenOptions>` / `Readonly<TweenManagerOptions>` / `Readonly<StopTweenOptions>` / `Readonly<NumericProps<T>>` — mutation-not-intended inputs are correctly marked.
- **Sentinels over throws.** Expected-missing cases (`manager.tweens.get(target) === undefined`) return early rather than throwing; nothing throws for expected failure.
- **`import type {}` on its own lines.** `tween.ts`, `tweenManager.ts`, `updateTweens.ts`, `colorTween.ts`, `timer.ts`, `internal.ts` all keep `import type { … }` separate from value imports.
- **Types sourced from `@flighthq/types`.** `Tween`, `TweenManager`, `TweenOptions`, `TweenManagerOptions`, `StopTweenOptions`, `NumericProps`, `TweenPropertyDetail`, `EasingFunction` are all imported from the header layer (the lone exception is the inline `ColorComponents` flagged above).
- **Globally distinguishable names.** Exports do not collide with sibling packages; the `Tween`-suffixed/-infixed names are self-identifying from the barrel.
- **Consistent parameter ordering.** The `(manager, target, …)` order is uniform across `applyTween`, `pauseTweens`, `resumeTweens`, `stopTweens`, and the explicit-manager `createTween`.
