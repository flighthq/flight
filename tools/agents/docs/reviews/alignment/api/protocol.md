# API Alignment: @flighthq/protocol

**Verdict:** Strongly aligned — a textbook command+event platform capability that matches its `network`/`power`/`lifecycle` siblings; the only real friction is the `Url`-vs-`URL` acronym casing drift between the code (`onOpenUrl`) and the architecture map (`onOpenURL`).

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Info | `ProtocolHandler.onOpenUrl` | The architecture map (`tools/agents/docs/index.md`, `@flighthq/protocol` and `@flighthq/app` entries) names this signal `onOpenURL` (uppercase acronym), but the package — and the codebase at large (`openExternalUrl`, `loadImageResourceFromUrl`, `setFeedUrl`, `dataUrl`) — uses `Url`. The code is internally consistent and on the majority convention; the **map prose** is the outlier. | Leave the code as `onOpenUrl`; fix the two `onOpenURL` mentions in the map to `onOpenUrl` so doc and code agree. No package change. |
| Info | `attachProtocolHandler(handler)` / `detachProtocolHandler` / `disposeProtocolHandler` | Sibling event capabilities suffix the verb with the entity word and pass that entity (`attachNetwork(net)`, `attachPower(power)`, `attachAppLifecycle(app)`). Protocol names the type fully (`...ProtocolHandler`) — arguably _more_ correct per "full unabbreviated type word," but it is a parameter/naming-shape asymmetry vs the `attach<Entity>(<entity>)` pattern the other event caps share. | No change recommended — `attachProtocolHandler` is the more self-identifying form and the deviation is the better choice. Noted only for the `_consistency.md` synthesis to consider whether siblings should converge upward (`attachNetwork` → keep) vs. protocol's fuller form. |
| Info | `getProtocolBackend()` | Lazy-creates and caches the web default backend on first call (writes module state `_backend`). This is the documented seam pattern (every platform cap does it) and is import-side-effect-free, so it is compliant — flagged only because a `get*` accessor that mutates on first read can surprise. | None; matches `getNetworkBackend`/`getPowerBackend`. Behavior is documented in the doc-comment. |

No violations found for: type-word abbreviation, global uniqueness, allocation-by-verb, alias-safety, teardown-verb misuse, `Readonly<T>`, sentinel-vs-throw, or `import type` placement (see Clean).

## Clean

- **Full, unabbreviated type words everywhere.** Every export carries `Protocol` and, where applicable, the noun it acts on: `registerProtocolScheme`, `unregisterProtocolScheme`, `isProtocolSchemeRegistered`, `setProtocolSchemeAsDefault`, `setProtocolBackend`, `createWebProtocolBackend`, `createProtocolHandler`. No `Proto`/`Scheme`-only/`Url`-abbreviation drift.
- **Globally unique root names.** All eleven exports are `Protocol`-qualified; `npm run api` shows no collision with the only other `ProtocolBackend` producer (`host-electron`'s `createElectronProtocolBackend`).
- **Allocation by verb.** `create*` allocate (`createProtocolHandler` returns `{ onOpenUrl: createSignal() }`, `createWebProtocolBackend` returns a fresh backend object); no `create`/`get` allocates where it should not, and there are no math/out-param helpers to misuse. No `out`-param functions exist, so alias-safety is N/A and correctly avoided.
- **Teardown verbs used correctly and distinctly.** `detachProtocolHandler` tears down the backend subscription (the bracketed unsubscribe); `disposeProtocolHandler` detaches so the GC-managed signal becomes reclaimable. No `destroy*` is used — correct, because the handler owns no non-GC resource. `attach`/`detach` are paired brackets and `attachProtocolHandler` is explicitly idempotent (detaches a prior subscription first).
- **Sentinels, never throws, for expected failure.** Every backend op returns `boolean`; the web backend returns `false` (or a no-op `() => {}` unsubscribe for the inert `subscribe`) when the platform API is absent or unsupported, and `register` swallows the thrown `navigator.registerProtocolHandler` exception into `false`. No expected-missing case throws; no unreachable-invariant validation.
- **Accessors and booleans.** `getProtocolBackend` is the only `get*` and returns the backend (a real getter). All boolean queries/commands use `is*` (`isProtocolSchemeRegistered`) or an action verb returning success (`register*`/`setProtocolSchemeAsDefault`) — no boolean `get*`.
- **Verb consistency with siblings.** `create*`/`createWeb*Backend`/`get*Backend`/`set*Backend`/`attach*`/`detach*`/`dispose*` exactly mirror the `network`, `power`, and `lifecycle` event-capability shape, and `register`/`unregister`/`isRegistered`/`setAsDefault` form a coherent command set.
- **Types and imports.** `ProtocolBackend` and `ProtocolHandler` are defined in `@flighthq/types` (`packages/types/src/Protocol.ts`), not inline; `import type { ProtocolBackend, ProtocolHandler }` is on its own dedicated `import type` line, separate from the value import of `@flighthq/signals`.
- **Side-effect-free module.** Module state (`_backend`, `_subscriptions` WeakMap) and constants live at the bottom of the file after the alphabetized exports; nothing registers or runs at import time. `"sideEffects": false` is honored.
