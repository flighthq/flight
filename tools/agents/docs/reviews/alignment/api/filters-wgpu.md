# API Alignment: @flighthq/filters-wgpu

**Verdict:** Strong, internally consistent backend API that mirrors `filters-gl` closely; the only real issues are a barrel that exports the manual-lifetime gradient-ramp constructor while withholding the cached form its own docs prefer, and a missing `WgpuTripleSourcePipeline` type alias.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `createWgpuGradientRampTexture` / `getWgpuGradientRampTexture` | The barrel exports only `createWgpuGradientRampTexture` (caller owns the texture, must call `texture.destroy()`), but its own doc comment says to _prefer_ the cached, state-owned `getWgpuGradientRampTexture` — which is **not** exported from `index.ts`. The package publishes the form it tells users not to use and hides the recommended one. A user applying a gradient ramp outside the built-in filters has no access to the safe path. | Export `getWgpuGradientRampTexture` from `index.ts` (it already has a colocated test). Keep `createWgpuGradientRampTexture` too, but the cached accessor should be the surfaced default. |
| Low | `WgpuDualSourcePipeline` vs (missing) `WgpuTripleSourcePipeline` | `createWgpuDualSourcePipeline` returns a named alias `WgpuDualSourcePipeline` (= `WgpuFilterPipeline`), but `createWgpuTripleSourcePipeline` returns the bare `WgpuFilterPipeline`. Asymmetric: a reader expects a parallel `WgpuTripleSourcePipeline` alias, or no alias on either. The dual alias also adds zero type information (pure `= WgpuFilterPipeline`). | Either add `WgpuTripleSourcePipeline = WgpuFilterPipeline` for symmetry, or drop `WgpuDualSourcePipeline` and have all three return `WgpuFilterPipeline`. Prefer dropping the empty alias unless the dual/triple distinction is meant to become structural. |
| Low | `makeFilterState` / `makeRenderTarget` / `makeScratch` (in `wgpuTestHelper.ts`) | Use the `make*` verb where the SDK reserves `create*` for allocation. Not in the barrel (test-only), so not public surface, but the verb drift mirrors the same drift in `filters-gl` and is easy to fix while small. | Rename to `createWgpu*` (`createWgpuFilterState`, `createWgpuRenderTarget`, `createWgpuScratch`) to match the SDK allocation verb; keep them out of the barrel. |
| Low | `getWgpuFilterState` (in `wgpuFilterPass.ts`) | Exported from the module (consumed by the gradient filters) but correctly **not** in the barrel. It returns a large ad-hoc anonymous object bundling `uniformBG`/layouts/sampler plus `acquireSlot`/`writeSlot`/`beginPass` closures — an internal escape hatch, not a typed public seam. Acceptable as internal, but flagged so it is not promoted to the barrel as-is. | Leave out of the barrel. If gradient filters are the only consumer, consider folding the helper into `wgpuFilterPass` internals rather than a cross-file export, or give the return a named internal type. |

## Clean

- **Full, unabbreviated type words throughout.** `applyDisplacementMapFilterToWgpu`, `applyColorMatrixFilterToWgpu`, `createWgpuGradientRampTexture`, `getWgpuInvertTintShader` — no abbreviations; every filter name carries its full filter type.
- **Two consistent naming families, matching the `filters-gl` sibling 1:1.** Filter applicators use `apply<Filter>ToWgpu`; shader/pass infrastructure uses a `Wgpu`-infix (`applyWgpuTintPass`, `createWgpuFilterPipeline`, `drawWgpuTripleSourcePass`, `getWgpuBlitShader`). The backend token placement differs between the two families but does so identically in `filters-gl`, so it is a deliberate cross-backend convention, not drift.
- **Accessor verbs correct.** `get*` is used only for cached-object accessors (`getWgpuBlitShader`, `getWgpuTintShader`, all returning a `WgpuFilterPipeline`/`WgpuDualSourcePipeline`); none return booleans or perform work beyond lazy build-and-cache.
- **Allocation discipline by verb.** `create*` functions (`createWgpuFilterPipeline`, `createWgpuGradientRampTexture`) are the allocating ones; `apply*`/`draw*`/`clear*` write into GPU targets and return `void`.
- **`throw` is used only for genuine programmer error.** The five throws (`ColorMatrixFilter requires 20 values`, convolution dimension/size checks, `No active command encoder`) are all precondition violations that correct code cannot reach — exactly the sanctioned use. No thrown errors for expected-missing cases.
- **`Readonly<T>` on filter descriptor params.** Every `apply*Filter*` takes `Readonly<Omit<XFilter, 'kind'>>`; blur/box options are `Readonly<{...}>`. The mutable `scratch: WgpuRenderTarget[]` and `dest` params are deliberately mutable (GPU write targets / scratch ring), which is the correct opt-out.
- **`import type {}` on its own line.** All cross-package type imports (`WgpuRenderState`, `WgpuRenderTarget`, filter descriptors) come from `@flighthq/types` via dedicated `import type` lines; no mixed `import { type Foo, bar }`. No cross-package types defined inline.
- **Parameter ordering is symmetric.** Every entry point leads with `state`, then `source`, then `dest`/`map`, then `scratch`, then `filter`/`options`; the multi-source draws extend it cleanly (`source0`, `source1`, `source2`, `dest`, `pipeline`, `setUniforms`).
- **Exports alphabetized; one colocated `*.test.ts` per source file.**
