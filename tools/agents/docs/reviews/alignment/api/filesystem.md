# API Alignment: @flighthq/filesystem

**Verdict:** Partially aligned — the backend seam, sentinels, and `Readonly<>` discipline are correct, but the core file operations break the suite-wide "full type word in the exported name" rule (e.g. `readTextFile` vs the sibling pattern `readClipboardText`), and the package mixes `make`/`copy`/`rename`/`watch` verbs against the `create*` / `*FileSystem*` conventions used elsewhere.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `readTextFile`, `writeTextFile`, `readBinaryFile`, `writeBinaryFile`, `appendTextFile`, `fileExists`, `removeFile`, `makeDirectory`, `readDirectory`, `statFile`, `copyFile`, `renameFile`, `watchPath` | None of the 13 core operations carry the package's domain type word, breaking the suite convention that public free functions are globally self-identifying by embedding the full type word. Every sibling command capability does this without exception: clipboard → `readClipboardText`/`writeClipboardHtml`/`clearClipboard`; storage → `getStorageItem`/`setStorageItem`/`removeStorageItem`. Within this package the convention is even applied inconsistently — `getFileSystemPath` and `setFileSystemBackend` carry `FileSystem`, but `readTextFile`/`makeDirectory`/`copyFile` do not. Names like `removeFile`, `statFile`, `copyFile`, `makeDirectory`, `fileExists`, `watchPath` are generic enough to plausibly collide or read ambiguously from the SDK barrel. | Embed the domain word the way the siblings do. Cleanest is a `FileSystem` infix matching `getFileSystemPath`: `readFileSystemTextFile` is awkward, so the better golden-path choice is a `*File` / `*Directory` object-word scheme that is unambiguous on its own — keep `readTextFile`/`writeTextFile`/`readBinaryFile`/`writeBinaryFile`/`appendTextFile`/`statFile`/`copyFile`/`renameFile`/`removeFile` (these already end in the `File` object word) but rename the directory/path/watch ops that lack any type word: `makeDirectory` → `createFileSystemDirectory` (or `createDirectory`), `readDirectory` → `readFileSystemDirectory`, `fileExists` → `fileSystemFileExists` or `hasFile`, `watchPath` → `watchFileSystemPath`. At minimum align on one rule and apply it to all 13, not a subset. |
| Medium | `makeDirectory` | Uses the `make*` verb for a creation operation, against the SDK's dominant `create*` verb (the public API has ~200 `create*` functions; `make*` appears almost exclusively in internal render-state / test helpers like `makeGlState`, `makeScratch`, `makeFakeGl`). `make` is idiomatic to POSIX `mkdir`, but the SDK rule is to choose the word the codebase already reached for. | Rename to `createDirectory` / `createFileSystemDirectory` to match `create*` everywhere else. Update the `FileSystemBackend.makeDirectory` method name in `@flighthq/types` to track it. |
| Medium | `copyFile` / `renameFile` vs backend `copy` / `rename`; `watchPath` vs backend `watch`; `getFileSystemPath` vs backend `getPath` | The public function name and the `FileSystemBackend` method name diverge inconsistently. Some public functions match the backend method 1:1 (`fileExists`, `removeFile`, `makeDirectory`, `readTextFile`), while others are renamed (`copy`→`copyFile`, `rename`→`renameFile`, `watch`→`watchPath`, `getPath`→`getFileSystemPath`). This is defensible (bare `copy`/`rename`/`watch` are too generic for the barrel) but the rename rule is not applied uniformly — `readDirectory`/`fileExists` are left bare on both sides while `copy`/`rename` are decorated only on the public side. | Decide one rule: either every public op is decorated with the domain/object word (preferred, see High finding) or none are. The backend interface may keep terse method names; the divergence itself is fine, but it should be consistent across all members. |
| Low | `renameFile` | The doc comment says "Renames or moves a file" and the backend implements it as a move (copy-then-remove on OPFS), but the exported name only says "rename". Move semantics are part of the contract. | Acceptable as-is (`rename` is the canonical filesystem verb for move), but consider documenting move semantics in the name or keeping the comment as the single source — no rename needed if the comment stays. |

## Clean

- **Backend seam matches the sibling platform packages.** The `createWebFileSystemBackend` / `getFileSystemBackend` / `setFileSystemBackend` trio mirrors `clipboard`, `shell`, `storage`, `device`, and `platform` one-for-one, including the lazy web (OPFS) default (`getFileSystemBackend` allocates on first call) and the `set*(null)` reset-to-web semantics. These three names correctly carry the full `FileSystem` type word.
- **Sentinels for expected failure, never throwing.** Reads resolve to `null` (`readTextFile`, `readBinaryFile`, `statFile`) or `[]` (`readDirectory`); writes and mutations resolve to `false` when access is denied or the entry is missing (`writeTextFile`, `removeFile`, `copyFile`, `makeDirectory`, …). The web backend wraps every OPFS call in try/catch and degrades to the sentinel when `navigator.storage.getDirectory` is absent (jsdom / non-secure context). No thrown errors for expected-missing cases.
- **`Readonly<T>` applied where it matters.** `writeBinaryFile(path, data: Readonly<Uint8Array>)` and the `FileWatchEvent` listener (`(event: Readonly<FileWatchEvent>) => void`) are correctly read-only; the web backend honors it by copying with `data.slice()` before passing to `createWritable().write()` so the caller's buffer is never mutated. Other params are primitives (`string`) and exempt.
- **No misused teardown verbs.** No `dispose*` / `destroy*` / `acquire*` / `release*` present. `removeFile` is a domain delete operation, not a teardown verb, and is named appropriately. The watch unsubscribe is returned as a plain `() => void`, matching the event-seam pattern.
- **Cross-package types live in `@flighthq/types`.** `FileSystemBackend`, `FileSystemPathKind`, `FileEntry`, `FileStat`, `FileWatchEvent`, and `FileWatchEventType` are all defined in `packages/types/src/FileSystem.ts`; nothing is redefined inline.
- **`import type {}` on its own line.** `import type { FileEntry, FileStat, FileSystemBackend, FileSystemPathKind, FileWatchEvent } from '@flighthq/types';` — no inline `type` mixed with value imports.
- **No top-level side effects.** The only module state is `let _backend = null`, populated lazily through `getFileSystemBackend`. No registration, listeners, or globals at import time; `"sideEffects": false` holds.
- **Out-param alias-safety N/A.** All functions are async value-returning; there are no `out`/`target` parameters to alias.
- **Exports alphabetized** within `filesystem.ts` (order:check clean).
