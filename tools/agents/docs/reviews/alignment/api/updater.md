# API Alignment: @flighthq/updater

**Verdict:** Strong adherence to the event-capability seam pattern; the one real issue is naming asymmetry among the four lifecycle command functions, which inconsistently carry the `Updater` domain word and use mixed `Update`/`Updates` forms.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `checkForUpdates`, `downloadUpdate`, `quitAndInstallUpdate` | These three lifecycle commands do not carry the `Updater` domain word, while the sibling commands `setUpdaterFeedUrl`, `getUpdaterBackend`, `setUpdaterBackend` do. The sibling `power` package is uniform — every non-entity command carries the domain word (`setPowerKeepAwake`, `getPowerStatus`). From the SDK barrel, `checkForUpdates`/`downloadUpdate` read as generic verbs with no self-identifying domain anchor (the Design Constraint: "A function should be globally self-identifying without context"). | Rename to carry the domain word consistently: `checkForUpdaterUpdates` is awkward; prefer anchoring on the entity/domain noun, e.g. `requestUpdaterCheck`/`checkUpdater`, `downloadUpdaterUpdate`→`downloadUpdate` is still ambiguous. At minimum align all four under one domain prefix (`Updater*`) the way `power` does. |
| Medium | `checkForUpdates` (plural) vs `downloadUpdate` / `quitAndInstallUpdate` (singular) | Inconsistent pluralization of the same `Update` noun within one package: one command says "Updates", the others "Update". | Pick one form. The lifecycle deals with a single available update, so singular `Update` is the consistent choice; reword `checkForUpdates` accordingly. |
| Low | `quitAndInstallUpdate` (entity fn) vs `quitAndInstall` (backend method) | The public function appends `Update` but the `UpdaterBackend.quitAndInstall` method it delegates to does not. The thin wrappers otherwise pass straight through (`checkForUpdates`→`checkForUpdates`, `downloadUpdate`→`downloadUpdate`), so this one wrapper renames its backend method, breaking the 1:1 readability between public command and backend method. | Either keep the public name aligned with the backend method, or rename the backend method to match — keep wrapper↔method names identical across all four commands. |
| Low | `setUpdaterFeedUrl` / backend `setFeedUrl` | Convention asymmetry vs `network`/`power`, where the backend surface is `getStatus`/`subscribe` and the public function adds the domain word once (`getNetworkStatus`). Here the public `setUpdaterFeedUrl` is correct, but it is the only _config_ command — worth confirming `FeedUrl` is the canonical term (electron-updater uses "feed URL"; Sparkle uses "appcast URL"). Not a violation, just verify the term transfers instantly. | Keep `setUpdaterFeedUrl`; confirm `FeedUrl` over `Appcast`/`UpdateUrl` is the intended canonical noun for the cross-backend seam. |

## Clean

- **Entity-event seam shape is exemplary** and matches `network`/`power` 1:1: `createAppUpdater` (allocates inert signals) → `attachAppUpdater` / `detachAppUpdater` (idempotent, tears down prior subscription first) → `disposeAppUpdater` (delegates to detach, releases to GC). This is the canonical event-capability quartet from the platform-suite docs.
- **Teardown verbs are correct.** `disposeAppUpdater` is `dispose*` (detach listeners → release to GC, nothing non-GC to free) — exactly the documented meaning; no misuse of `destroy*` for a GC-only entity.
- **Backend seam is the documented command-capability shape:** `getUpdaterBackend` (lazy web default, "there is always a backend"), `setUpdaterBackend(backend | null)` (null falls back to web default), `createWebUpdaterBackend` (`create*` allocates). Names carry the full `Updater`/`UpdaterBackend` type word.
- **Full unabbreviated type words** in entity functions: `attachAppUpdater`, `createAppUpdater`, etc. all spell `AppUpdater`; backend functions spell `UpdaterBackend`. No abbreviations.
- **`get*` is a true accessor** returning `UpdaterBackend`; no boolean-from-`get*` or non-getter `get*`. No boolean-returning functions exist, so no `has*`/`is*` obligation is missed.
- **Sentinels over throws:** the web backend no-ops every command and returns inert unsubscribes rather than throwing when no native updater exists — matches the platform-suite rule (return `null`/`false`/`-1`/no-op when a capability is unavailable). No precondition throws, no internal-invariant validation.
- **`import type {}` on its own line:** `import type { AppUpdater, UpdaterBackend } from '@flighthq/types';` is a dedicated type-only import, not mixed inline with value imports.
- **Cross-package types from `@flighthq/types`:** `AppUpdater`, `UpdaterBackend`, `UpdateInfo` all live in `packages/types/src/Updater.ts`; nothing is redefined inline. `UpdateInfo` payloads are `Readonly<UpdateInfo>` in both the signal and backend signatures.
- **No allocation in hot/command paths** beyond the documented `create*` allocators; no `out`-param functions to alias-check (this capability has no value-returning snapshot like `getNetworkStatus(out)`, which is appropriate — updates are push-only events, not a pollable status).
- **Module-private state at file bottom:** `_backend` and `_subscriptions` (a `WeakMap` keyed by entity, the documented per-entity subscription store) sit after the exported functions; package is `"sideEffects": false` with no top-level registration.
- **Exports alphabetized** within `updater.ts`; barrel is a thin `export * from './updater'`.
