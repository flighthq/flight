# API Alignment: @flighthq/camera

**Verdict:** Strong — full unabbreviated type words, disciplined `out`/alias-safe helpers, correct sentinel and `Readonly<>` usage throughout; only minor verb/parameter-naming consistency nits.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Low | `createCamera`, `createOrthographicProjection`, `createPerspectiveProjection` | Options parameter is named `opts`; the dominant codebase convention is `options` (59 vs 24 across `create*` functions), and the immediate sibling `@flighthq/effects` `createCameraMotionBlurEffect(options)` uses `options`. The matching `CameraOptions` / `OrthographicProjectionOptions` / `PerspectiveProjectionOptions` interfaces already spell the word out. | Rename the parameter to `options` for cross-package symmetry. |
| Low | `getCameraInverseViewProjectionMatrix4` | A `get*` accessor that returns `boolean` as an invertibility sentinel. The `get*` prefix normally signals a value accessor, and a boolean-returning function normally takes `is*`/`has*`; here the boolean is a success flag for an `out`-writing computation, which is a legitimate but slightly mixed signal. (The verb is defensible because the primary effect is writing the matrix into `out`; flagged only for awareness, not a required change.) | Keep `get*` (the `out` matrix is the real product), or document the boolean-as-sentinel convention so it reads consistently against sibling math packages. |
| Info | `createCamera` | Stores `options.projection` (a `Readonly<Projection>` via the `Readonly<CameraOptions>` parameter) directly into the mutable `Camera.projection` field, so the new camera aliases the caller's projection descriptor. Not an alias-safety violation per the checklist, but the shared reference means later mutation of the descriptor mutates the camera. | Acceptable for a value-style descriptor; if independence is wanted, shallow-copy the projection on construction. Otherwise note the shared-reference ownership in the doc comment. |

## Clean

- **Full, unabbreviated type words.** Every export names its operand in full: `Camera`, `Projection`, `OrthographicProjection`, `PerspectiveProjection`, and the dimensioned `Matrix4` / `Vector3` suffixes (`getCameraViewProjectionMatrix4`, `setCameraViewMatrix4FromLookAt`, `setCameraViewMatrix4FromMatrix4`). No abbreviations.
- **Globally unique, self-identifying names.** All eleven exports are unambiguous from the root barrel; `getCameraViewProjectionMatrix4` vs `getCameraInverseViewProjectionMatrix4` and the `...FromLookAt` / `...FromMatrix4` pair read cleanly without context. No collisions with sibling `camera`-named exports (`createCameraMotionBlurEffect`, `setGlMeshCameraPosition`).
- **Allocation by verb.** `create*` functions allocate (`createCamera` via `createEntity`/`createMatrix4`, the two projection builders return descriptors). All math/transform helpers (`getCamera*Matrix4`, `setProjectionMatrix4`, `setCamera*`) write into an `out`/`camera` target and allocate nothing on the hot path — scratch matrices (`__scratchProjection`, `__scratchViewProjection`) are module-level singletons, parked at file bottom per source style.
- **Alias safety, explicitly documented.** `getCameraViewProjectionMatrix4`, `getCameraInverseViewProjectionMatrix4`, `setProjectionMatrix4`, and `setCameraViewMatrix4FromLookAt` all read inputs into scratch/locals before writing `out`, and each carries a comment stating the aliasing guarantee. Correct and verified against the read-before-write rule.
- **Sentinels, no spurious throws.** `getCameraInverseViewProjectionMatrix4` returns `false` (leaving `out` untouched) for the expected non-invertible case rather than throwing. No function throws for expected-missing/failure conditions, and there is no validation of unreachable internal invariants.
- **`Readonly<T>` discipline.** Read-only inputs are marked everywhere mutation isn't intended: `Readonly<CameraOptions>`, `Readonly<Camera>` on the getters, `Readonly<Vector3Like>` / `Readonly<Matrix4Like>` / `Readonly<Projection>`. The in-place setters correctly take a mutable `camera: Camera` because mutation is the deliberate effect.
- **Boolean predicates use `is*`.** `isOrthographicProjection` / `isPerspectiveProjection` are correctly prefixed and serve as discriminated-union narrowers (`projection is …`).
- **Type sourcing and imports.** All cross-package types (`Camera`, `Projection`, `Matrix4Like`, `Vector3Like`, the projection variants) come from `@flighthq/types`; none are defined inline. Every `import type {}` is on its own line — projection.ts splits value and type imports cleanly.
- **`*Options` / `*Like` shapes.** Structural input interfaces are named `*Options` and exported alongside their builders; `*Like` types are used for the structural matrix/vector inputs, matching the entity-vs-structural rule.
