# API Alignment: @flighthq/render

**Verdict:** Naming spells out full type words and the `register*`/`prepare*`/`update*`/`dispose*` verb families are correct, but the package has two structural convention breaks — module-level mutable singletons (`installRenderAdaptHook` + `_installed` flag in `renderProxyAdapter`) that violate the no-shared-top-level-state rule, and a redundant duplicate export (`updateDisplayObjectRenderTransform`) — plus several lower-severity issues: `compute*` allocators that should be `create*`, an exported empty no-op, a non-descriptive `obj` parameter, and a `noopRendererData` name that does not lead with a verb.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `installRenderAdaptHook` (+ module-level `_adaptHook` in `renderProxy.ts`, `_installed` in `renderProxyAdapter.ts`) | The render-adapter feature is wired through process-global mutable singletons, not per-`RenderState` runtime state. `installRenderAdaptHook(fn)` writes a module-level `let _adaptHook`, and `setRenderProxyAdapter` flips a module-level `let _installed` to lazily install `applyRenderProxyAdapter` once for the whole process. This is exactly the "shared top-level mutable state" / "register at module level" the map forbids: two `RenderState`s share one hook, install order is global, and there is no uninstall. It also makes `walkNode`'s `_adaptHook?.()` call depend on whether any state ever called `setRenderProxyAdapter`. | Move the hook onto `RenderStateRuntime` (a nullable `adaptHook` slot, the runtime-slot pattern the map prescribes) and have `setRenderProxyAdapter` set it on `getRenderStateRuntime(state)` instead of a module `let`. Drop `installRenderAdaptHook` from the public API (or make it `state`-scoped). Removes the `_installed` global entirely. |
| High | `updateDisplayObjectRenderTransform` | Exact duplicate export of `updateRenderProxy2DTransform` — its whole body is `return updateRenderProxy2DTransform(state, data, parentData)`. Two public names for one operation breaks "globally unique exported function names" and the map's verb-consistency rule; a caller cannot tell which is canonical. Sprites and display objects now share one `RenderProxy2D` (per the type-collapse decision recorded in the codebase map), so the DisplayObject-named alias is vestigial. | Delete `updateDisplayObjectRenderTransform` and keep the trait-named `updateRenderProxy2DTransform` (consistent with `updateRenderProxy2D`, `updateRenderProxyAppearance`, `updateRenderProxyMaterial`). Update any callers/examples. |
| Medium | `computeRenderTargetSize` | Named `compute*` (the package's write-into-`out` verb — see `computeDisplayObjectRenderTargetTransform`, `computeRenderCacheTransform`, which both write an `out` matrix) but it **allocates** a fresh `{ width, height }` object on every call. `compute*` siblings here are explicitly allocation-free out-writers; this one silently allocates on what can be a per-cache hot path. | Either rename to `createRenderTargetSize` (it allocates, so a `create*` verb is honest) or give it an `out: { width: number; height: number }` first parameter and write into it like the other two `compute*` functions. Prefer the out-param form for symmetry with the sibling transform computers. |
| Medium | `computeTextFormatFontString` | Same verb/allocation mismatch: `compute*` but returns a freshly built `string`. (Primitive return, so the cost is smaller than the object case, but the verb still signals "writes out, no alloc" while it constructs a value.) | Rename to `getTextFormatFontString` (it derives a value from the format with no out-param) so `compute*` stays reserved for the out-writing transform/size functions. `get*` is the accessor verb the map prescribes for "derive and return a value." |
| Medium | `createRenderState(obj?: Partial<RenderState>)` | Parameter named `obj` — non-descriptive, against "choose names whose meaning transfers instantly." Every other constructor in the package names its operand for what it is. `obj` also reads as the entity being created rather than seed overrides. | Rename to `init` or `overrides: Partial<RenderState>` to convey it is a partial seed, not the resulting state. |
| Low | `beginRenderProxyUpdate(_source, _data)` | Exported function whose entire body is `{}` — a no-op with two unused (underscore-prefixed) parameters. It is dead public surface: it does nothing, allocates nothing, and is not paired with an `end*`. Exporting an empty stub couples the published API to a placeholder. | Remove it from the public API until it has behavior; if it is a reserved seam, keep it internal (un-exported) rather than shipping an exported no-op. |
| Low | `noopRendererData` | Does not lead with a verb and is not an entity constructor/accessor — it is a default `Renderer.createData` implementation that always returns `null`. The name reads as a noun ("noop renderer data"), unlike the verb-first exports around it (`createRenderProxy`, `registerRenderer`, `getRenderProxyCache`). | Rename to `createNoopRendererData` (matches the `Renderer.createData` slot it fills and the `create*`-returns-or-allocates convention), or fold it into `renderer.ts` as an internal default rather than a public export. |
| Low | `getRenderProxyAdapter` / `setRenderProxyAdapter` (`adapter` param) | `setRenderProxyAdapter` takes `adapter: RenderProxyAdapter \| null` non-`Readonly`. It only stores the reference into a `WeakMap` (no mutation), so under the Readonly-by-default rule the stored reference should be `Readonly<RenderProxyAdapter>`. (Note `useRenderCache` then mutates `existing.cache`, so the cache-adapter path is intentionally mutable — but the generic adapter setter does not mutate and should be marked const.) | Type the parameter `Readonly<RenderProxyAdapter> \| null` (and the `getRenderProxyAdapter` return likewise) unless a backend genuinely needs to mutate the stored adapter through these functions. |
| Low | `applyRenderProxyAdapter` (`data` param) | Second parameter typed `RenderProxy2D & { traverseChildren: boolean }` — an inline structural intersection that re-states a field already on `RenderProxy2D`. Redundant (the `& { traverseChildren: boolean }` adds nothing `RenderProxy2D` lacks) and pushes an ad-hoc shape into the public signature instead of the shared `@flighthq/types` type. | Drop the intersection and type it `data: RenderProxy2D`. If the intent is to require `traverseChildren`, that belongs on the `RenderProxy2D` type in `@flighthq/types`, not inlined at the callsite. |
| Low | `prepareSceneRender` doc vs behavior | Not an API-naming defect, but the export's contract comment says `aspect` is "taken from the render state's pixel-space target (width / height)"; the implementation always passes the constant `DEFAULT_VIEWPORT_ASPECT = 1` (RenderState carries no canonical viewport size, as a later comment admits). The two comments contradict each other in one public function's doc. | Reconcile the doc: state plainly that a perspective camera must carry its own projection aspect and the function uses a neutral `1` fallback (the second comment is the accurate one). |

## Clean

- **Full, unabbreviated type words.** Every export names its operand in full — `prepareDisplayObjectRender`, `updateRenderProxyAppearance`, `computeDisplayObjectRenderTargetTransform`, `getRenderStateRuntime`, `registerRenderCacheRenderer`, `packSceneLightBlock`. No `RenderProxy`→`RP`, `DisplayObject`→`DO`, or `Transform`→`Xform` shortening; `RenderProxy2D` keeps the `2D` qualifier distinct from the base `RenderProxy`.
- **`register*` opt-in, no module-level registration.** `registerRenderer` / `registerRenderCacheRenderer` are explicit functions taking `(state, kind, renderer)`; nothing registers a renderer at module top level, matching the renderer-registration rule. (The one top-level-state violation is the _adapter hook_, flagged above — the renderer registry itself is correctly per-state.)
- **Teardown verbs used correctly.** `disposeRenderProxy` / `disposeDisplayObjectRender` are `dispose*` — they detach the proxy from the `WeakMap` (release-to-GC) and _cascade_ to the renderer's `destroyData?.()` to free the non-GC GPU resource. The dispose-vs-destroy split is observed precisely: the framework proxy is GC-managed (dispose), the renderer's GPU data is freed now (destroy), and the comments state exactly that. No `acquire*`/`release*` leaks into this package's public surface (pools live in the backend packages).
- **Allocation discipline on the transform/light path.** `computeDisplayObjectRenderTargetTransform` and `computeRenderCacheTransform` write into an `out` matrix using module-scope scratch (`_tempInvLocal`, `_tempTranslation`) and allocate nothing; `packSceneLightBlock` writes into `out: SceneLightBlock` and reuses `scratchColor`/`scratchProjection`; `prepareSceneRender` returns a per-`RenderState` reused `SceneRenderList` (WeakMap-keyed scratch), not a fresh allocation per frame. (The two stray `compute*` allocators are the medium nits above.)
- **Alias-safe out-params.** `computeDisplayObjectRenderTargetTransform` resolves `inverseMatrix` into a scratch and builds the translation in scratch before the final `multiplyMatrix(out, …)`, so `out` aliasing an input is safe; `setSceneViewProjectionMatrix4` reads the camera into `scratchProjection` first (its comment calls out the alias-safety); `setRenderStateBackgroundColor` reads `color` into `uint` before writing any `_state` field.
- **`get*` accessors and `is*` predicates.** Accessors are `get*` returning values (`getRenderProxy2D`, `getRenderProxyAdapter`, `getRenderProxyCache`, `getRenderStateRuntime`); boolean type-guards lead with `is*` (`isRenderCache`, `isRenderCacheAdapter`, `isRenderProxyDirty`, `isRenderProxyVisible`). No `get*` returns a boolean and no boolean predicate is mis-prefixed.
- **`enable*` for opt-in signal cost.** `enableRenderCacheAdapterSignals` follows the `enable*Signals` convention for opting an entity into its signal group, lazily creating the signal only when called.
- **`import type` hygiene + types from `@flighthq/types`.** Type-only imports are on their own `import type { … }` lines throughout (`renderProxy.ts`, `renderProxyAdapter.ts`, `sceneRender.ts`); cross-package types (`RenderProxy2D`, `RenderState`, `SceneLightBlock`, `Renderer`) come from `@flighthq/types`, with no cross-package type defined inline. `RenderProxyVisitor` / `AdaptHook` are local function-type aliases scoped to the walk, which is acceptable.
- **`Readonly<>` on the read-heavy 3D/transform inputs.** `packSceneLightBlock(out, lights: Readonly<SceneLights>)`, `prepareSceneRender(…, scene: Readonly<SceneNode>, camera: Readonly<Camera>, lights: Readonly<SceneLights>)`, and the `bounds: Readonly<RectangleLike>` parameters on the target-transform/size functions correctly mark non-mutated object inputs const. The deliberately-mutable returns (`createRenderStateRuntime`, `getRenderStateRuntime`) document why they opt out.
