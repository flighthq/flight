# API Alignment: @flighthq/textinput

**Verdict:** Strongly aligned — full type words, `Readonly<>` discipline, `out`-first parameters, and `enable*`/`get*`/`has*` opt-in conventions are all observed; the only real flag is one expected-failure case that throws instead of returning a sentinel, plus a few minor verb/symmetry notes.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `getInputState` (internal) → reached via `appendTextInput`, `applyTextInputRestriction`, `deleteTextInputBackward`/`Forward`, `getTextInputCaretIndex`, `getTextInputDisplayText`, `getTextInputSelectionBeginIndex`/`EndIndex`/`Text` | These public `get*`/edit functions throw `Error('text input is not enabled…')` when input was never enabled. The map reserves `throw` for programmer error and wants sentinels for expected-missing state. Querying selection/caret on a not-yet-enabled `RichText` is a plausible expected-missing case (the package even ships `hasTextInput`/`getTextInputState` returning a `null` sentinel for exactly this state), so the readers are inconsistent with their own probe. The file comment argues it is misuse, which is defensible for the _mutating_ editors but weak for the pure `get*` readers. | Either (a) treat all of these as misuse-only and document the precondition uniformly, or (b) for the `get*` readers return a sentinel (`-1` for index getters, `''` for text getters) when `getTextInputState` is `null`, matching `hasTextInput`. Pick one policy and apply it consistently across the editing readers. |
| Low | `handleTextInputKeyboard` | Declared `: boolean`, but the trailing `switch (command)` has no `default` and no post-switch `return`; if `command` ever falls outside the handled cases the function returns `undefined` for a `boolean` signature. Currently masked only by the exhaustive `KeyboardCommand` union. | Add a final `return false;` after the switch (or a `default: return false`) so the boolean contract holds without relying on union exhaustiveness. |
| Low | `applyTextInputRestriction` | The `apply*` verb usually implies mutating the target in place; here it is a pure function returning a _new_ filtered string and never touches `source`. The verb reads as a side-effecting op but it is value-in/value-out. | Consider `getRestrictedTextInput` / `filterTextInput` (returns a value) to signal purity, or keep `apply*` but ensure the package documents it as non-mutating. Minor — name is not misleading once read. |
| Low | `SelectableRichText*` family vs `TextInput*` family | Verb symmetry is good (`create*`/`focus*`/`blur*`/`dispatch*` parallel across both), but the families are asymmetric in teardown wiring: `TextInput` has `connectInputToTextInput` returning a disconnect closure, while the `SelectableRichText` family exposes no equivalent `connect*`/teardown for its input signals. Not a naming violation, but a public-surface asymmetry between two sibling manager families in the same package. | If selectable rich text is meant to be input-driven too, add a parallel `connectInputToSelectableRichText` returning a disconnect closure; otherwise document why the selectable family is wiring-free. Surface as a design question, not an autonomous change. |

## Clean

- **Full, unabbreviated type words everywhere.** Every export spells out `TextInput`, `SelectableRichText`, `Caret`, `Selection`, `Rectangle`, `Character`, `Index` — no abbreviations (`getTextInputCharacterIndexAtPoint`, `getTextInputSelectionBeginIndex`, `selectWordAtTextInputIndex`). Names are globally self-identifying and the root barrel names are globally unique.
- **`out`-first parameter convention.** `getTextInputCaretRectangle(out, source, layout)` and `getTextInputSelectionRectangles(out, source, layout)` both place `out` first and are symmetric with each other and with the codebase norm (`copyMatrix`, `getDeviceInfo`, `unionAabb`). Both write into `out` and do not allocate.
- **`Readonly<>` discipline.** Read-only paths take `Readonly<RichText>`, `Readonly<TextLayoutResult>`, `Readonly<InputKeyboardData>`, `Readonly<TextInputOptions>`, `Readonly<ReplaceTextInputOptions>`, `Readonly<HandleTextInputKeyboardOptions>`; mutating editors correctly take a plain mutable `RichText`. Opt-out is deliberate and matches what each function does.
- **Opt-in `enable*` seam.** `enableTextInput`/`disableTextInput`/`getTextInputState`/`hasTextInput` follow the map's `enable*` subsystem pattern exactly: side-effect-free import, lazy slot allocation on the `RichText` runtime, idempotent enable, `null` sentinel from `getTextInputState`.
- **Accessor/boolean prefixes correct.** `get*` functions all return values (never booleans); `hasTextInput` is the only boolean accessor and uses `has*` correctly.
- **Verb consistency with siblings.** `connectInputToTextInput` mirrors `@flighthq/input`'s `connectInputToInteraction` (same `connectInputTo*` shape, same disconnect-closure return). `create*` managers, `dispatch*` event entry points, and `focus*`/`blur*` pairs are internally consistent across both manager families.
- **Sentinels for the genuinely-expected cases.** Manager dispatchers return `false`/early-return when there is no focus target or empty text (`dispatchTextInput`, `dispatchTextInputKeyDown`, `dispatchSelectableRichTextKeyDown`); `getTextInputState` returns `null`; `getSelectableRichTextSelectionText` returns `''`.
- **`import type {}` hygiene + types from `@flighthq/types`.** Every cross-package type (`RichText`, `TextInputState`, `TextInputManager`, `SelectableRichTextManager`, `TextSelectionRectangle`, `ReplaceTextInputOptions`, etc.) is imported `import type` on its own line from `@flighthq/types`; no cross-package types are defined inline. The only local type, `KeyboardCommand`, is a file-private union and correctly not exported.
- **Exports alphabetized** within each source file and re-exported in barrel order; `npm run order` is satisfied (not re-reported here).
