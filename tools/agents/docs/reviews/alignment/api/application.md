# API Alignment: @flighthq/application

**Verdict:** Strong, highly consistent command/event surface; the only real problems are two dead alias files leaking abbreviated names into source and tests, two fullscreen helpers that throw on expected failure, and a `Readonly` cast hole.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `webWindow.ts` (`createAppWindow`, `createWebWindow`, `AppWindow`) | Dead alias file: not exported from the `index.ts` barrel, referenced only by its own `webWindow.test.ts`. It also defines names that violate conventions — `createAppWindow` / `AppWindow` abbreviate the type word `ApplicationWindow` (Design Constraints: "Never abbreviate type names"), and `createWebWindow` is a redundant `web`-prefixed alias for the canonical `createApplicationWindow`. Stale `dist/html5Window.d.ts` shows a prior `createHTML5Window` rename, confirming this is churn, not API. | Delete `webWindow.ts` and `webWindow.test.ts`. The canonical `createApplicationWindow` is already barreled. |
| High | `webApplication.ts` (`createWebApplication`) | Same pattern: dead file, not in the barrel, referenced only by `webApplication.test.ts`. It re-exports `createApplication` under a redundant `web`-prefixed name. The `web` qualifier is wrong as public API anyway — `createApplication` is backend-agnostic; the web specificity lives in `createWebWindowBackend`. | Delete `webApplication.ts` and `webApplication.test.ts`. |
| Medium | `requestApplicationFullscreen`, `exitApplicationFullscreen` | Both return the raw browser promise (`element.requestFullscreen()` / `document.exitFullscreen()`), which **rejects** when fullscreen is denied (no user gesture) or unsupported — an expected failure, not programmer error, so it should not throw (Design Constraints: "Throw only for programmer errors"). They also dereference `document`/`element` with no `typeof document === 'undefined'` guard, so they throw in non-DOM hosts — inconsistent with `createWebWindowBackend.setFullscreen`, which guards and swallows the rejection. | Guard for DOM, swallow the rejection (return `Promise<boolean>` / `Promise<void>` that resolves to a sentinel), mirroring `setFullscreen`. Better: route fullscreen through the `WindowBackend` seam like every other window command rather than two bare DOM functions. |
| Low | `getWindowBounds` | Declares `win: Readonly<ApplicationWindow>` then immediately casts it away: `getBounds(win as ApplicationWindow, out)`. The cast defeats the `Readonly<>` contract the signature advertises and is a latent mutation hole. | Widen the `WindowBackend.getBounds` backend signature to accept `Readonly<ApplicationWindow>` (it only reads `win`), removing the cast. |
| Low | `createWebWindowBackend` vs `getWindowBounds` out-param | `WindowBackend.getBounds`/`getWindowBounds` take `out` last and return it (good, matches geometry), but the other `out`-style entry `computeWindowDeviceTransform(win, out)` is the only fill-and-return in `window.ts` and is fine. No asymmetry within the package — noting only that `getBounds` on the backend is the one place the `Readonly` discipline isn't carried through (see above). | Covered by the `getWindowBounds` fix. |

## Clean

- **Full, unabbreviated type words everywhere in the barrel.** Every exported function spells out `Window` / `ApplicationWindow` / `Application` (`setWindowAlwaysOnTop`, `attachWindowRenderState`, `computeWindowDeviceTransform`, `disposeApplicationWindow`). No abbreviation in any _barreled_ export — the abbreviation problems are confined to the two dead alias files.
- **Verb consistency is excellent and symmetric.** `attach*`/`detach*` pair across all nine window subsystems and the app `Exit`; `set*` for every command mutator; `get*` only for genuine accessors (`getWindowBackend`, `getWindowBounds`); `is`/`has` not misused; `request*` reserved for veto/permission-gated actions (`requestWindowClose`, `requestWindowAttention`, `requestApplicationFullscreen`); `start*`/`stop*` bracket the loop.
- **Teardown verbs used correctly.** `dispose*` (`disposeApplication`, `disposeApplicationWindow`) detaches listeners/clears the observer side-table → GC; no `destroy*` misuse (the package owns no GPU/native handle). `detach*` is the per-subsystem inverse of `attach*`, distinct from `dispose*` (which tears down all). No synonym drift.
- **Out-param discipline.** `computeWindowDeviceTransform` reads `win.devicePixelRatio` into a local before writing `out` fields, so `out === win`-style aliasing is safe, and its doc comment explicitly states "Reads win before writing out, so out may alias an input." `Readonly<ApplicationWindow>` on the input.
- **Sentinels for expected failure.** `closeWindow` / `requestWindowClose` / `openWindow` return `boolean` (veto / host-opened-a-window) instead of throwing; backend no-ops return cleanly. (The two fullscreen helpers are the lone exception — see Findings.)
- **`Readonly<T>` on read-only object params:** `computeWindowDeviceTransform`, `getWindowBounds` (`Readonly<ApplicationWindow>`), `openWindow` (`Readonly<WindowOptions>`). Command mutators correctly take a mutable `ApplicationWindow` because they update entity state.
- **No top-level side effects.** Backend is lazily created via `getWindowBackend`; observer registries are `WeakMap` side tables initialized on first use; no `register*` at module top level. Matches `"sideEffects": false`.
- **Type hygiene.** All cross-package types (`ApplicationWindow`, `Application`, `Matrix`, `RenderState`, `WindowBackend`, `WindowBounds`, `WindowOptions`) come from `@flighthq/types`; nothing defined inline. `import type { … }` is on its own dedicated line in every file, never mixed with value imports.
- **Dependencies minimal and correct:** only `@flighthq/signals` + `@flighthq/types`; does not import `@flighthq/sdk`.
