# API Alignment: @flighthq/textlayout

**Verdict:** Strong overall — naming uses full type words, `out`/`create`/`get` verbs are disciplined and sentinels are used well — but two systemic gaps stand out: missing `Readonly<>` on `TextFormat` parameters across the whole package, and a `layout` parameter that drifts position between sibling functions.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `getTextFormatAscent`, `getTextFormatDescent`, `getTextFormatHeight`, `getTextFormatLeading`, `mergeTextFormat`, `createTextFormatRange`, `createTextLayoutGroup` | Take `format: TextFormat` (mutable) where they never mutate it. The map mandates `Readonly<T>` on every parameter where mutation is not intended (the C++ `const` rule). `mergeTextFormat(base, override)` reads both and returns a fresh object — neither input is mutated. `createTextFormatRange`/`createTextLayoutGroup` _store the reference_ without cloning, so they especially want `Readonly` to signal the caller's object is captured as-is. | Change these parameters to `Readonly<TextFormat>`. For `mergeTextFormat`, both `base` and `override` become `Readonly<TextFormat>`. |
| Medium | `getRichTextSelectionRectangles(out, beginIndex, endIndex, layout)` | Parameter-order asymmetry: `layout` is the package's dominant trailing-context argument and sits in **second** position almost everywhere (`getRichTextLineMetrics(layout, lineIndex)`, `getRichTextLineCount(layout)`, all of `richTextMetrics`, `getTextMetrics(out, layout)`). Here `layout` is dead last, after two index scalars. | Reorder to `getRichTextSelectionRectangles(out, layout, beginIndex, endIndex)` so `out` then `layout` (the read context) precede the scalar range, matching `getTextMetrics(out, layout)` and `computeTextBoundsRectangle(out, spec, layout)`. |
| Low | `getRichTextCharBoundaries(out, text, layout, charIndex)`, `getRichTextCharIndexAtPoint(text, layout, x, y)`, `getRichTextLineText(text, layout, lineIndex)` | `text` precedes `layout` here, but the rest of the family leads with `layout`. The `(text, layout, …)` vs `(layout, …)` split is a second axis of the same ordering inconsistency. | Settle one rule: `layout` is the primary context, so lead with it where present (`getRichTextLineText(layout, text, lineIndex)` etc.), keeping `out` first when present. Apply package-wide. |
| Low | `getRichTextScrollYOffset(lineHeights, firstVisibleLine)` | Axis vocabulary drift: every sibling scroll function uses the Flash `V`/`H` axis words (`getRichTextMaxScrollV`, `getRichTextMaxScrollH`, `getRichTextBottomScrollV`), but this one switches to `Y`. A reader cannot tell from the name that it is the vertical pixel offset for a `scrollV`. | Rename to `getRichTextScrollVOffset` (or `getRichTextScrollVPixelOffset`) to keep one axis vocabulary across the scroll API. |
| Low | `getRichTextCharIndexAtPoint`, `getRichTextLineIndexAtPoint` | Return `0` for both the empty-layout / not-found case and a genuine hit on the first char, conflating "no result" with "index 0". Sibling lookups use a real sentinel (`getRichTextLineMetrics` → `null`, `getRichTextLinkAtPoint` → `null`, `getTextLineBreakIndex` → `-1`). | If callers need to distinguish "no hit", these are conceptually caret positions where `0` is a defensible clamp — keep `0` but document it as a clamp, not a sentinel; otherwise return `-1` on empty layout to match `getTextLineBreakIndex`. |
| Low | `TextBoundsSpec` (exported from `textBounds.ts`) | A cross-package shape contract (its own comment: "so both TextLabel and RichText satisfy it structurally") is defined inline in a package file rather than in `@flighthq/types`, the header layer. It is not currently imported by name elsewhere (consumers satisfy it structurally via `RichTextData`/`TextLabelData`), so this is a soft flag, but the map says the full API shape should be navigable from `@flighthq/types` alone. | Move `TextBoundsSpec` to `@flighthq/types` and re-export, so the text-bounds contract is discoverable from the header. |
| Low | `computeTextBoundsHeight`/`computeTextBoundsWidth`/`computeTextBoundsOffsetX` vs `getRichTextTextHeight`/`getRichTextTextWidth` | Two verbs for closely-related derived scalars: `compute*` (TextBounds family, returns a number) and `get*` (RichText family, also returns a ceil'd derived number). The `compute*` scalars are pure reads of `spec`+`layout` with no `out`, blurring the line with `get*`. | Acceptable if `compute*` is reserved for "derives a value from policy + layout" and `get*` for "reads a layout/result field"; if so, note the rule in a comment. Otherwise consider `get*` for the scalar bounds accessors and reserve `compute*` for the out-writing `computeTextBoundsRectangle`. |
| Info | `computeTextBoundsRectangle` | The doc comment warns "`out` may not alias `spec`", but `out: RectangleLike` and `spec: TextBoundsSpec` are different types and cannot alias — the alias-safety note is moot (and `spec` is `Readonly`). Harmless, but the warning misleads a reader into thinking there is a real aliasing hazard. | Drop or reword the aliasing caveat; the real out-param alias case to test is `out === a Rectangle input`, which does not arise here. |

## Clean

- **Full, unabbreviated type words throughout.** `getDisplayObject`-style discipline holds: `getRichTextSelectionRectangles`, `computeTextBoundsRectangle`, `getTextFormatAscent`, `createTextLayoutGroup` — no `getRTBounds`/`getFmt` abbreviations anywhere.
- **Verb discipline on allocation.** `create*` functions (`createRichTextContent`, `createTextLayoutResult`, `createTextMetrics`, `createTextLayoutGroup`, `createTextFormatRange`) all allocate and return; the hot-path / read functions (`computeTextLayout`, `getTextMetrics`, `getTextLineBreaks`, `computeTextBoundsRectangle`, `getRichTextSelectionRectangles`) write into an `out` and return `void`, with reusable module scratch (`_lineBreaks`, `_charAdvances`) kept at file bottom. This matches the explicit-allocation rule precisely.
- **`out` is consistently first** on the write-into functions (`computeRichTextContent`, `computeTextLayout`, `computeTextBoundsRectangle`, `getTextMetrics`, `getRichTextCharBoundaries`, `getRichTextSelectionRectangles`, `getTextLineBreaks`).
- **Sentinels for expected failure.** `getRichTextLineMetrics` → `TextLineMetrics | null`, `getRichTextLinkAtPoint` → `string | null`, `getTextLineBreakIndex` → `-1`, `getTextLayoutMeasureProvider` → `null`. No thrown errors for missing/empty cases.
- **Accessor prefixes correct.** `get*` for value accessors; `getRichTextCharBoundaries` correctly returns `boolean` to signal hit/miss while filling `out` (it is a fill-or-fail, not a pure accessor, so the `boolean` return is appropriate rather than a `has*` rename).
- **`Readonly<TextLayoutResult>` / `Readonly<RichTextData>` applied consistently** on the read-only layout and data inputs across `richTextMetrics`, `richTextQuery`, `textBounds`, and `textMetrics` — the gap is specifically `TextFormat`, not the larger structures.
- **`import type {}` on its own line in every file**; all cross-package entity types (`TextFormat`, `TextLayoutResult`, `RichTextData`, `Rectangle`, `TextSelectionRectangle`, …) are imported from `@flighthq/types`, with the lone inline exception (`TextBoundsSpec`) flagged above.
- **Clean measure-provider seam.** `getTextLayoutMeasureProvider`/`setTextLayoutMeasureProvider` keep the module singleton at file bottom, default to `null`, prefer the `@flighthq/textshaper` backend, and document the explicit-injection escape hatch — a side-effect-free, opt-in provider rather than top-level registration.
