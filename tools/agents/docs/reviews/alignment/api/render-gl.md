# API Alignment: @flighthq/render-gl

**Verdict:** Mostly aligned — the `Gl` backend token, `create*`/`destroy*`/`acquire*`/`release*` verbs, `Readonly<>` discipline, and full type words are applied consistently; the real problems are a test helper (`makeGlState`/`makeGL`) leaking into the public barrel, a dead unexported `registerGlBitmapShader`, and a sentinel/verb inconsistency cluster.

## Findings

## Clean

- **Backend token is consistent.** Every export is prefixed `Gl` (`createGlRenderState`, `glRenderTarget`, `drawGlFullscreenPass`), matching the established `gl` backend token for `*-gl` packages. `Gl` is the accepted token here (mirrors the `GlRenderState`/`GlBitmapShader` types in `@flighthq/types`), not a forbidden abbreviation.
- **Full type words in names.** Functions carry the complete operand word: `createGlRenderTarget`, `getGlRenderStateRuntime`, `resolveGlMaterialRenderer`, `acquireGlRenderTarget`. No `RenderTarget` → `RT` style elision.
- **Allocation verbs are disciplined.** `create*`/`acquire*` allocate; `drawGl*`, `setGl*`, `applyGlBlendMode`, `renderGlBackground` write into pre-allocated runtime buffers (`quadVertexData`, `matrixArray`) with no per-call allocation on the hot path. The one hot-path allocation in `beginGlRenderTarget` (`createMatrix()`) is a documented state-save, and `drawGlRenderTargetResult` correctly uses `acquireMatrix`/`releaseMatrix` as paired brackets.
- **Teardown verbs are correct and distinct.** `destroyGl*` (`destroyGlRenderState`, `destroyGlRenderTarget`, `destroyGlRenderTargetPool`) free GPU resources immediately via `deleteProgram`/`deleteBuffer`/`deleteFramebuffer`/`deleteTexture` — the exact `destroy*` semantics. `acquireGlRenderTarget`/`releaseGlRenderTarget` are honest pool brackets (release returns to the free list, keeping GPU storage). No `dispose*`/`destroy*` synonym drift.
- **Pool bracket symmetry.** `acquireGlRenderTarget` ↔ `releaseGlRenderTarget`, `createGlRenderTargetPool` ↔ `destroyGlRenderTargetPool` — matched pairs, documented as brackets in `glRenderTargetPool.ts`.
- **`Readonly<>` applied where mutation is not intended.** `Readonly<RenderTargetDescriptor>`, `Readonly<Matrix>`, `ReadonlyArray<WebGLTexture>`, `Readonly<GlRenderTarget> | null`, `Readonly<GlFullscreenProgram>` on parameters that must not be mutated; the intentionally-mutable `GlRenderStateRuntime` return is explicitly documented as such.
- **Sentinels over throws for expected-missing.** `getGlMaterialRenderer`/`getGlMaterialShader`/`resolveGlMaterialRenderer` return `null` for unregistered kinds; throws are reserved for genuine programmer/environment errors (no WebGL2 context, shader compile/link failure).
- **Accessor prefixes correct.** `getGl*` for accessors; no boolean-returning `get*`. Setters use `setGl*`, registrations use `registerGl*`, resolution uses `resolveGl*` — verb roles are consistent (aside from the `registerGlBitmapShader` setter-misuse flagged above).
- **`import type {}` hygiene.** Type imports are on dedicated `import type { }` lines throughout; cross-package types are sourced from `@flighthq/types`, and `glShaderTypes.ts` is a pure re-export of the header types rather than inline redefinitions (the only inline-type lapse is the structural `Matrix`/viewport shapes flagged above).
- **Exports alphabetized within files** (order:check clean).
