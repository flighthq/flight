# API Alignment: @flighthq/render-wgpu

**Verdict:** Largely well-aligned — full unabbreviated type words, correct allocation verbs, and a clean `acquire`/`release` target pool — but two real leaks (test-only exports in the public barrel, an `acquire*` verb used for a non-pool lazy accessor), plus pervasive inline structural transform types that should be `Readonly<MatrixLike>` from `@flighthq/types`.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `createWgpuRenderStateForTest`, `installWgpuMock` | Test scaffolding (from `wgpuTestHelper.ts`) is re-exported from the public barrel `index.ts` (line 11), so it ships in the published API surface. Both are pure JSDOM test fixtures. | Drop them from `index.ts`; import directly from `./wgpuTestHelper` inside tests. If a test entry is genuinely needed, it should not sit on the package root. |
| High | `acquireWgpuFrameCaptureTexture` | `acquire*` is reserved for pool brackets with a matching `release*`. This is a lazy get-or-create accessor (creates/resizes the capture texture on demand, freed later by `destroyWgpuRenderState`), with no `release*` partner — verb misuse that implies a pooling contract that does not exist. | Rename to `getWgpuFrameCaptureTexture` (or `ensureWgpuFrameCaptureTexture`); reserve `acquire*` for the real pool in `wgpuRenderTargetPool.ts`. |
| Medium | `drawWgpuQuadWithTransform`, `drawWgpuRenderTargetResult`, `setWgpuMatrixFromTransform`, `writeWgpuMatrixOnlyUniforms`, `writeWgpuQuadUniforms` | The transform argument is an inline structural literal `{ a; b; c; d; tx; ty }` redefined at five call sites instead of the shared cross-package type. `@flighthq/types` already exports `MatrixLike` (the exact `EntityWithoutRuntime<Matrix>` shape). CLAUDE rule: cross-package types come from `@flighthq/types`, and `*Like` is the sanctioned structural-input form. | Replace each inline `{ a, b, c, d, tx, ty }` param with `Readonly<MatrixLike>`. Same for the inline `renderProxy` shapes (`{ alpha; transform2D; ... }`) where a `RenderProxy`/`RenderProxy2D` already covers them. |
| Medium | `drawWgpuQuad`, `drawWgpuQuadWithTransform` | Object params not mutated but lack `Readonly<>`: `renderProxy: RenderProxy2D` / `RenderProxy` and `textureEntry: WgpuTextureEntry`. Sibling helper `getWgpuRenderProxyColorTransform` correctly takes `Readonly<RenderProxy>`, so the package is inconsistent with itself. | Mark read-only inputs `Readonly<...>`; keep `state: WgpuRenderState` mutable (legitimately written each frame). |
| Low | `getWgpuShader` | Returns `WgpuBitmapShader \| undefined`, while the package's other lookups (`resolveWgpuShader`, `getWgpuMaterialRenderer`, `resolveWgpuMaterialRenderer`) return `... \| null`. Sentinel-type drift for the same "not found" case. | Return `... \| null` for consistency with the package's null sentinel convention. |
| Low | `createWgpuTextureEntry` / `updateWgpuTextureEntry` / `bindWgpuTexture` | Cross-sibling asymmetry: render-gl exposes `createGlTexture` / `updateGlTexture` / `bindGlTexture`. The wgpu trio carries an `Entry` suffix on two of three (and `bindWgpuTexture` returns a `WgpuTextureEntry` but the create/update pair operate on it), making the texture-API names diverge from the GL renderer for the same concept. | Align naming with the GL sibling (`createWgpuTexture` / `updateWgpuTexture`) unless `WgpuTextureEntry` is a deliberately distinct entity; if so, make all three consistently `*TextureEntry`. |
| Low | `enableWgpuBlendModeSupport` | `enable*` is described in CLAUDE as the verb for opting into signal groups. This is an install-a-hook function (`state.applyBlendMode = applyWgpuBlendMode`). Not flagged as a hard violation because the render-gl sibling uses the identical `enableGlBlendModeSupport` — the renderers have established `enable*BlendModeSupport` as a shared idiom. | Leave as-is for sibling consistency; if the signal-group reservation is to be strict, both renderers should move to `installWgpuBlendModeSupport` together (cross-package decision, surface to user). |

## Clean

- **Full, unabbreviated type words throughout.** Every export spells the operand: `getWgpuRenderProxyColorTransform`, `createWgpuRenderTargetPool`, `resolveWgpuMaterialRenderer`, `setWgpuMatrixFromTransform`. No abbreviated type words.
- **Globally unique names** via the consistent `Wgpu` infix; no collision risk from the barrel.
- **Allocation discipline by verb is correct.** `create*` (`createWgpuRenderState`, `createWgpuRenderTarget`, `createWgpuTextureEntry`) allocate; the hot-path uniform writers (`writeWgpuQuadUniforms`, `writeWgpuMatrixOnlyUniforms`, `setWgpuMatrixFromTransform`) write into pre-allocated runtime arrays and allocate nothing.
- **`setWgpuMatrixFromTransform` is alias-safe** — it only writes `matrixArray` from a separate `t`/`viewport`, no read-after-write hazard.
- **Pool brackets are correct:** `acquireWgpuRenderTarget` ↔ `releaseWgpuRenderTarget` are genuine paired brackets over `WgpuRenderTargetPool`, with `createWgpuRenderTargetPool`/`destroyWgpuRenderTargetPool` for lifecycle.
- **Teardown verbs are right:** `destroyWgpuRenderState`, `destroyWgpuRenderTarget`, `destroyWgpuRenderTargetPool` all free non-GC GPU resources (textures/buffers) immediately — `destroy*`, not `dispose*`, is the correct choice, and the doc comments explicitly justify what is and isn't freed.
- **Sentinel-vs-throw is mostly correct:** lookups return `null` for expected-missing; `createSurfaceFromWgpuRenderState` and `createWgpuRenderState` `throw` only on genuine programmer error / environment precondition (capture not enabled, no WebGPU adapter) — the right place to throw.
- **Accessor prefixes correct:** `get*` for accessors, `isWgpuSupported` uses `is*` for its boolean return.
- **`import type {}` hygiene:** every file keeps `import type { ... }` on its own line, separate from value imports (e.g. `wgpuDraw.ts`, `wgpuRenderState.ts`).
- **Readonly used where it matters** on the capture/material/color-transform read paths (`Readonly<WgpuRenderState>`, `Readonly<RenderProxy>`, `Readonly<WgpuRenderTarget>`, `Readonly<Matrix>`).
- **Barrel and within-file ordering** are alphabetized; `order:check` has nothing additional to report here.
