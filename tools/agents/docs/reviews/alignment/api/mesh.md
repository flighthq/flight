# API Alignment: @flighthq/mesh

**Verdict:** Strong — full unabbreviated type words, correct `out`-param/alias-safety with documented guarantees, correct `destroy*` vs `dispose*` choice, and sentinel-over-throw discipline; the one substantive nit is that the `compute*` helpers allocate scratch buffers despite their no-alloc `out`-param shape.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `computeMeshGeometryNormals`, `computeMeshGeometryTangents` | Both take an `out` parameter (the signal for a no-alloc, hot-loop-safe helper per the allocation-by-verb rule), yet each allocates a `Float64Array` scratch buffer per call — `accum` in normals, `tan` + `bitan` in tangents. `compute*` is not one of the documented allocating verbs (`create*`/`clone*`/`acquire*`), so the `out`-param shape promises hot-path safety the implementation does not deliver. This is a real per-call allocation in what reads as a recompute primitive. | Either accept a caller-provided scratch/`out` for the accumulator (true no-alloc), pool the `Float64Array` via `acquire*`/`release*`, or — if these are explicitly one-shot finalize helpers — document the per-call allocation in the doc comment so the `out`-param shape isn't read as a hot-path guarantee. `computeMeshGeometryBounds` is the clean reference: it is genuinely allocation-free. |
| Low | `MeshGeometryOptions` | Exported from the root barrel (public surface) but defined inline in `meshGeometry.ts` rather than in `@flighthq/types`. It is the public input contract for `createMeshGeometry`; per the header-layer rule, cross-package/public type shapes belong in `@flighthq/types`. (It is currently only consumed within `mesh`, so this is low, not blocking.) | Move `MeshGeometryOptions` into `@flighthq/types` alongside `MeshGeometry`/`MeshSubset`/`VertexAttributeLayout` so the full input shape is navigable from the header layer. |
| Info | `destroyMeshGeometryGlData`, `destroyMeshGeometryWgpuData` | Both take `Readonly<MeshGeometry>` and then mutate the attached runtime (`runtime.webglData = null`) through an `EntityRuntimeKey` cast. This is the established entity/runtime pattern (the entity is logically unchanged; only its private runtime slot is cleared), and the `Readonly<>` on the entity is defensible — flagged only so the read-only-param-that-mutates-runtime pattern is a conscious choice, not drift. | No change required; the `destroy*` verb and the doc comments correctly justify freeing the GPU slot. |

## Clean

- **Full, unabbreviated type words.** Every export names its operand in full — `createBoxMeshGeometry`, `createConeMeshGeometry`, `createCylinderMeshGeometry`, `createPlaneMeshGeometry`, `createQuadMeshGeometry`, `createSphereMeshGeometry`, `createTorusMeshGeometry`, `createMeshGeometry`, `cloneMeshGeometry`, `computeMeshGeometry{Bounds,Normals,Tangents}`, `getMeshGeometry{Index,Vertex}Count`. `Gl`/`Wgpu` in the `destroy*` pair are backend tokens (the codebase's accepted convention), not abbreviations of the `MeshGeometry` type word.
- **Globally unique, self-identifying names.** All exports carry the `MeshGeometry` stem and are unambiguous from the root barrel. No collision with `@flighthq/scene`'s `createMesh` (that builds a `Mesh` scene node, a distinct type) or with the `*MeshGeometry` consumers in `scene-gl`/`scene-wgpu`.
- **Allocation by verb.** `create*`/`clone*` allocate as expected (fresh typed arrays + entity/runtime). `cloneMeshGeometry` deep-copies vertices, indices, subsets, and bounds, and resets `version` to 0 with its own runtime — no shared GPU slots. `getMeshGeometry{Index,Vertex}Count` are pure reads. `computeMeshGeometryBounds` is allocation-free into `out`.
- **Alias safety, explicitly documented.** `computeMeshGeometryBounds` reads every position into locals before writing the six corner fields, so `out === geometry.bounds` is safe. `computeMeshGeometryNormals`/`Tangents` accumulate into scratch and only write the normal/tangent slots of `out.vertices` (reading the disjoint position/normal/uv slots of `geometry.vertices`), so the in-place `out === geometry` case the builders rely on is safe. Each function carries a comment stating the aliasing guarantee, and the file header restates it.
- **Teardown verbs correct.** `destroyMeshGeometryGlData`/`destroyMeshGeometryWgpuData` correctly use `destroy*` (free a non-GC GPU upload slot now), not `dispose*`, with comments that explicitly justify the choice against the dispose/destroy distinction. No `dispose*`/`acquire*`/`release*` misuse.
- **Sentinels, no spurious throws.** Nothing throws. Degenerate inputs return empty/zero results — empty vertex stream yields an empty AABB (`min = +Infinity`, `max = -Infinity`), zero/negative stride yields a 0 vertex count, degenerate UVs fall back to an arbitrary perpendicular tangent. No validation of unreachable internal invariants.
- **`Readonly<T>` discipline.** Read-only inputs are marked throughout: `Readonly<MeshGeometry>` on clone/getters/compute-source/destroy, `Readonly<MeshGeometryOptions>` on the constructor, `Readonly<VertexAttributeLayout>` on the private layout helper, and `Readonly<...Array<ArrayBuffer>>` on index inputs. The `out`/`out.vertices` targets and the deliberate `vertices` ownership-handoff into `createMeshGeometry` are correctly left mutable.
- **Accessors.** `getMeshGeometryIndexCount`/`getMeshGeometryVertexCount` use `get*` and return values; no boolean-returning `get*` and no count predicate that should be `has*`/`is*`.
- **Parameter-order symmetry.** All three `compute*` functions share the `(out|target, geometry)` order; the `create*Geometry` builders share an ascending dimensions→segments→flags ordering (`createCylinderMeshGeometry(topRadius, bottomRadius, height, radialSegments, capped)`, `createTorusMeshGeometry(radius, tube, radialSegments, tubularSegments)`), and `createConeMeshGeometry` cleanly delegates to the cylinder builder.
- **Imports & types.** Every `import type {}` is on its own line in all three source files; entity/scene-graph types (`MeshGeometry`, `MeshGeometryRuntime`, `MeshSubset`, `PrimitiveTopology`, `VertexAttributeLayout`, `AabbLike`) all come from `@flighthq/types`. `MeshGeometryOptions` is the lone inline type (see Findings).
- **Single construction point.** All builders funnel through `buildCanonicalMeshGeometry`, and both that path and `cloneMeshGeometry`/`createMeshGeometry` route through one private `createMeshGeometryRuntime`, so every `MeshGeometry` shares one runtime shape — module-level constants (`CANONICAL_VERTEX_LAYOUT`, offset constants, `UINT16_INDEX_CEILING`) sit at file bottom per source style.
