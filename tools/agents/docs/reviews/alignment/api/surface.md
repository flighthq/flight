# API Alignment: @flighthq/surface

**Verdict:** Strong overall — full type words, alias-safe out-params, sentinel/throw split all correct; the one real wart is a split naming scheme (filters use `operationSurface` while the same "apply a filter" concept also appears as `applySurface*`) plus a `dest`/`out` synonym drift for the write target.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `bevelSurface`, `boxBlurSurface`, `gaussianBlurSurface`, `glowSurface`, `innerGlowSurface`, `dropShadowSurface`, `innerShadowSurface`, `gradientBevelSurface`, `gradientGlowSurface`, `colorMatrixSurface`, `convolveSurface`, `displaceSurface`, `dilateSurface`, `erodeSurface`, `medianSurface`, `pixelateSurface`, `sharpenSurface` vs `applySurfaceColorTransform`, `applySurfacePaletteMap`, `applySurfaceThreshold` | Two naming schemes for the same concept (apply an image-processing operation to a surface). The filter/effect family is `operationSurface` (object-last, no leading verb), while three sibling operations use the explicit `applySurface*` verb-first form. A reader scanning the root barrel cannot predict which scheme any given filter uses. | Pick one scheme for the whole filter/effect family. Either move the filters to verb-first `applySurface<Operation>` (e.g. `applySurfaceGaussianBlur`), or — if `operationSurface` is the intended house style for filters — rename the three outliers to match (`thresholdSurface`, `colorTransformSurface`, `paletteMapSurface`). The first is more consistent with the dominant `verbSurface*` style used everywhere else in the package. |
| Low | `dest: Readonly<SurfaceRegion>` writers (e.g. `fillSurfaceRectangle`, `copySurfacePixels`, `compositeSurfaceRegion`, `flipSurfaceHorizontal`, `applySurface*`) vs `out: Surface` writers (`setSurfacePixel`, `setSurfacePixelRgb`, `floodFillSurface`, `scrollSurface`) vs `out: Uint8ClampedArray` filters | The mutated destination parameter is named three ways for one role: `dest` (region targets), `out` (Surface / typed-array targets). The map says mutable outputs are "usually named `out` or `target`"; `dest` is a third synonym. The split tracks the parameter _type_, not a semantic difference. | Standardize the write-target name. Reserve `out` for the written destination across all three families (the region writers would become `out: Readonly<SurfaceRegion>`), or document the convention "`dest` = region, `out` = surface/buffer" explicitly. Consistency beats the type-driven split. |
| Low | `mergeSurface(dest, source, redMultiplier, greenMultiplier, blueMultiplier, alphaMultiplier)` | Takes four positional channel multipliers while every other configurable operation in the package (`bevelSurface`, `convolveSurface`, `displaceSurface`, blurs, shadows, glows, `resizeSurface`) takes a `Readonly<Surface*Options>` object. Four same-typed `number` args in a row are easy to transpose. | Consider a `Readonly<SurfaceMergeOptions>` (or accept the four-arg form is fine for a fixed, well-known RGBA quartet — OpenFL's `BitmapData.merge` is positional too; low priority). |
| Info | `convertSurfacePixelOrder`, `premultiplySurfacePixels`, `unpremultiplySurfacePixels`, `computeGaussianKernel` | These operate on raw buffers / kernels rather than a `Surface`/`SurfaceRegion`, so the "Surface" word in the name describes the domain, not an operated-on type. Not a violation (the full domain word is present and they are genuinely surface-pixel helpers), but worth noting these are the buffer-level layer of the package, distinct from the region API. | No change; flagged only so the dual layering (raw-buffer helpers vs region API) is visible. |

## Clean

- **Full, unabbreviated type words everywhere.** Every export leads or contains `Surface` spelled out; no `getSurfPixel`/`SurfRegion`-style abbreviations. `Rgb` and `32` suffixes (`getSurfacePixelRgb`, `extractSurfacePixels32`) are domain/encoding terms, not abbreviated type names.
- **Allocation discipline by verb is correct.** `createSurface`, `cloneSurface`, `createSurfaceRegion`, `createSurfaceFingerprint`, `createSurfaceFrom*`, `createImageResourceFromSurface` allocate; `setSurfaceRegion` mutates in place and returns `out` for chaining; all hot-path math/filter/copy helpers write into a caller-provided `out`/`dest` and allocate nothing.
- **Out-params are alias-safe and documented.** `flipSurfaceHorizontal`/`flipSurfaceVertical` explicitly detect the same-region case (`isSameRegion`) and swap in pairs, with doc comments stating the in-place contract; `setSurfaceRegion` reads inputs into params before writing. No read-after-write clobbering found.
- **Sentinel vs throw split is correct.** Expected-failure paths return sentinels (`compareSurface` → `Surface | null` when identical, `getSurfaceColorBoundsRectangle` → `RectangleLike | null`, `parseSurfaceFingerprint` → `… | null`); `throw` is used only for genuine programmer error (dimension mismatch in `compareSurface`/`getSurfaceMismatch`/`compareSurfaceFingerprints`, malformed color/convolution matrices, `gridSize < 1`) — precondition violations, not internal-invariant checks.
- **Accessors and getters.** All read-only queries use `get*` (`getSurfacePixel`, `getSurfaceHistogram`, `getSurfaceCoverage`, `getSurfaceMismatch`); none returns a boolean, so no `get*`-boolean mismatch and no missing `has*`/`is*` (the only boolean predicates, `isSameRegion`/`inBounds`, are private helpers correctly named).
- **`Readonly<T>` is applied thoroughly.** Read-only inputs are `Readonly<Surface>` / `Readonly<SurfaceRegion>` / `Readonly<Uint8ClampedArray>` / `ReadonlyArray<number>`; mutability is opted into only on the written target. The `dest: Readonly<SurfaceRegion>` writers are correctly `Readonly` on the _struct_ (its fields are not mutated — only `dest.surface.data` is written), which is the right modeling.
- **No teardown-verb misuse.** The package owns no GC-roots or GPU/native handles, so it correctly exposes no `dispose*`/`destroy*`/`acquire*`/`release*`; nothing is mislabeled.
- **Type imports and cross-package types are clean.** All shared types (`Surface`, `SurfaceRegion`, `SurfaceFingerprint`, `ImageChannel`, `PixelOrder`, `ThresholdOperation`, etc.) come from `@flighthq/types`; `import type {}` is always on its own line (no inline `import { type Foo, bar }`); the barrel is a thin re-export with a single `.` entry and `"sideEffects": false`.
- **Output-first parameter ordering is consistent** across all write families (`out`/`dest` first, then `source`, then options), and the scratch-buffer filters uniformly use `(out, scratch, source, options)`.

_(Note: `initSurfaceRs` appears in `npm run api surface` output but belongs to the separate `@flighthq/surface-rs` package, not this one; out of scope here.)_
