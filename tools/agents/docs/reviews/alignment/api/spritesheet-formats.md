# API Alignment: @flighthq/spritesheet-formats

**Verdict:** Well aligned — full `Spritesheet` type word everywhere, a clean symmetric `parse*`/`parse*Document`/`serialize*` trio across all three formats; the notable gaps are mutable `existing`/`document` round-trip types that should be `Readonly`, and parse functions that `throw` (via `JSON.parse`) on malformed input where the convention prefers a sentinel.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `parseAsepriteSpritesheet`, `parseAsepriteSpritesheetDocument`, `parseTexturePackerSpritesheet`, `parseTexturePackerSpritesheetDocument` | These call `JSON.parse(json)` unguarded, so malformed file content throws a `SyntaxError`. Parsing an externally-supplied file is an **expected** failure case (a user feeds a bad/empty/wrong-format file), not programmer misuse — the convention says return a sentinel (`null`) for expected failure and throw only on misuse. The Starling parsers do not throw (regex-based, return an empty doc), so the package is internally inconsistent: bad XML yields an empty `SpritesheetData` but bad JSON throws. | Wrap `JSON.parse` and return `null` on parse failure (changing the return type to `SpritesheetData \| null` / `AsepriteParsed \| null`), or make Starling equally strict. Either way, make the three formats agree on the failure contract. |
| Low | `serializeAsepriteSpritesheet`, `serializeStarlingSpritesheet`, `serializeTexturePackerSpritesheet` (the `existing?` param) | `existing` is a read-only input the function never mutates, but its type is bare `Partial<AsepriteDocument>` / `Partial<StarlingDocument>` / `Partial<TexturePackerDocument>`. The map's `Readonly<T>` rule applies to "function parameters … where mutation is not intended". `data` is correctly `Readonly<SpritesheetData>`, so the asymmetry is visible within each signature. | Wrap as `Readonly<Partial<…>>` to match the adjacent `Readonly<SpritesheetData>` and signal that `existing` is not written. |
| Low | `AsepriteParsed.document`, `StarlingParsed.document`, `TexturePackerParsed.document` | The `document` returned for round-tripping is a mutable `*Document`. It is a freshly allocated value (acceptable to hand back mutable), but the intended contract is "preserve and feed back into `serialize*`", i.e. treat as opaque. A mutable field invites in-place edits that the serialize path does not expect. | Consider typing the returned `document` as `Readonly<…Document>` (and the `existing` param accepts `Readonly<Partial<…>>` per above) so the round-trip object reads as immutable carry-through. Low because the value is caller-owned and newly allocated. |
| Info | `AsepriteDocument`, `TexturePackerDocument`, `StarlingDocument`, `AsepriteSerializeOptions`, `StarlingParseOptions`, `TexturePackerParseOptions`, `TexturePackerSerializeOptions`, `AsepriteParsed`/`StarlingParsed`/`TexturePackerParsed` | These exported types are defined inline in the package, not in `@flighthq/types`. The map says cross-package types belong in the header layer. These are format-schema and per-format option/result types consumed only by this package and its direct users (round-trip), so they read as legitimately package-local public API rather than cross-package contracts. No action recommended; recorded so a future cross-package consumer (e.g. a `resources` loader that wants `AsepriteDocument`) is a signal to promote them. |
| Info | `// ─── Internal helpers ───` / `// ─── Public API ───` dividers (all six parse/serialize files) | Source Style says "Avoid structural divider comments such as `// ---- setup ----`." Every parse/serialize file uses box-drawing section dividers. Not an exported-API defect (the public surface is unaffected), but flagged as a Source Style drift to clean up when these files are next touched. | Remove the dividers; the existing `// Public API` boundary is already conveyed by the exported `function` keyword and the helpers sitting above it. |

## Clean

- **Full, unabbreviated type word everywhere.** Every exported function carries the complete `Spritesheet` type word — `parseAsepriteSpritesheet`, `serializeTexturePackerSpritesheet`, `parseStarlingSpritesheetDocument`. No `parseAsepriteSheet` / `parseTPSpritesheet` style abbreviation, and the format name (`Aseprite`/`Starling`/`TexturePacker`) is spelled in full.
- **Symmetric, globally-unique trio across all three formats.** Each format exposes exactly `parse<Format>Spritesheet` (→ `SpritesheetData`), `parse<Format>SpritesheetDocument` (→ `<Format>Parsed` for round-trip), and `serialize<Format>Spritesheet`. The `parse`/`serialize` verb pairing and the `…Document` suffix for the round-trip variant are consistent format-to-format; names are unique within the barrel and across the SDK.
- **Consistent parameter ordering.** All parsers take `(input: string, options?)`; all serializers take `(data, existing?, options?)`. `data` first, optional carry-through second, options last — symmetric across formats.
- **`create*` constructors used, no object literals for entity types.** Parsers build results via `createSpritesheetData` / `createSpritesheetFrameData` / `createSpritesheetAnimationData` from `@flighthq/spritesheet`, per the "constructors over literals for SDK entity types" rule.
- **Allocation verbs correct.** `parse*` / `serialize*` are allocating transforms with no `out`/`target` claim, so the no-allocation `out`-param rule does not apply. No `create*` that fails to allocate, no hot-path helper that does.
- **No teardown-verb misuse.** No `dispose*` / `destroy*` / `acquire*` / `release*` present; none is warranted for a stateless format codec.
- **`import type {}` on its own line.** Every file separates `import type { … }` from value imports — e.g. `asepriteParse.ts` imports the types on a dedicated `import type` line and `createSpritesheet*` on a separate value line. No inline `import { type Foo, bar }`.
- **`data` parameter is `Readonly<SpritesheetData>`** in all three serializers, correctly marking the primary input immutable.
- **Sentinel-friendly internals where present.** Helpers degrade gracefully on missing data (`durationMap.get(n) ?? 100`, `findIndex` `>= 0 ? : 0`, Starling returning empty docs) rather than throwing — the right instinct; the Medium finding is only that the JSON parsers do not extend it to top-level parse failure.
- **No `get*`/`has*`/`is*` naming hazards.** The package exports no accessor or boolean-returning functions, so there is nothing to mis-prefix.
- **Exports alphabetized** within each file (`order:check` clean) and tests colocated as `<source>.test.ts` for each parser.
