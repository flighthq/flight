# API Alignment: @flighthq/signals

**Verdict:** Mostly aligned — the connect/disconnect/emit verb vocabulary is clean and the entity/runtime split, `Readonly<>` discipline, and cross-package type sourcing are correct; the one real defect is `disconnectAllSignals`, whose plural pluralizes the wrong noun (it clears all _slots_ from one _signal_).

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `disconnectAllSignals` | The name claims to operate on multiple **signals**, but the function takes a single `signal: Signal<T>` and removes every connected **slot** from that one signal. The noun in the name is wrong (it should describe slots, not signals) and the plural misleads the reader into thinking it disconnects across many signals. It also breaks the symmetry with its sibling `disconnectSignal(signal, slot)` — both act on one signal, yet one is singular and one is plural. This is the full-unabbreviated-correct-type-word rule plus verb/parameter symmetry. | Rename to `disconnectAllSlots(signal)` (or `disconnectSignalSlots(signal)`). Both read as "from this one signal, disconnect all its slots," restoring symmetry with `disconnectSignal` / `isSlotConnected` (which already center the slot noun). |
| Medium | `connectSignalAtRate` vs `connectSignal` | The two "connect" functions have **inconsistent return contracts**: `connectSignal` returns `void` (caller must keep the slot reference and call `disconnectSignal` to undo), while `connectSignalAtRate` returns a disconnect closure `() => void` and hides its wrapping `handler` so the caller _cannot_ `disconnectSignal` it manually. Same verb family, two different teardown idioms — a caller cannot predict from the name which one applies. | Pick one teardown model for the package. Either have `connectSignal` also return a disconnect closure (most ergonomic, but allocates), or have `connectSignalAtRate` return the wrapper `handler` (typed as the slot) so it teardowns through the same `disconnectSignal` path as everything else. Document the chosen convention in the package map. |
| Low | `connectSignalAtRate` | `AtRate` is a vague rate qualifier — it does not name what is being rated. The function specifically throttles a per-frame `(deltaTime: number) => void` tick signal to a target `fps`. Compare the sibling throttle helper naming elsewhere in the SDK; "rate" alone is less precise than "fps"/"frame rate." Minor, since the `fps` parameter disambiguates at the call site. | Consider `connectSignalAtFrameRate` (matches the `fps` parameter and the `deltaTime` payload) or `connectSignalThrottled`, so the qualifier names the thing being limited rather than the abstract notion of a rate. |
| Low | `connectSignalAtRate` | Lives in `throttle.ts` but is named `connect*`, so the file name (`throttle`) and the only exported symbol do not share a stem. Filenames in this package otherwise track their exported concept (`signal.ts`→`createSignal`, `slot.ts`→connect/disconnect/`isSlotConnected`, `emitter.ts`→`emitSignal`/`cancelSignal`). | If the symbol is renamed (e.g. `connectSignalThrottled`), rename the file to match the symbol stem, or move it into `slot.ts` alongside the other connect/disconnect functions. |

## Clean

- **Verb vocabulary.** `createSignal` (allocates — correct verb), `connectSignal`/`disconnectSignal` (paired connect/disconnect), `emitSignal`, `cancelSignal`, `isSlotConnected` form a coherent, greppable set. No `make`/`new`/`add` drift for creation, no synonym churn for connect.
- **`is*` boolean.** `isSlotConnected` returns `boolean` and uses the `is*` prefix correctly; no `get*` returning a boolean anywhere.
- **`Readonly<>` discipline matches mutation intent.** `isSlotConnected` takes `Readonly<Signal<T>>` (read-only — correct) and `connectSignal`'s `options` is `Readonly<SignalConnectOptions>`; the mutating functions (`connectSignal`, `disconnectSignal`, `disconnectAllSignals`, `emitSignal`, `cancelSignal`) deliberately take a **mutable** `Signal<T>` because they splice its `data` or flip `cancelled`. The const-by-default rule is applied with the right opt-outs.
- **Cross-package types from the header layer.** `Signal`, `SignalData`, `SignalConnectOptions` are all imported from `@flighthq/types` and re-exported, not defined inline. The package's only runtime dependency is `@flighthq/types`, consistent with the map's "fundamental infrastructure, few dependencies" note for signals.
- **`import type {}` discipline.** Every type import is on a dedicated `import type { ... } from '@flighthq/types'` line; no mixed `import { type Foo, bar }` form. Type re-exports (`export type { Signal }`, `export type { SignalConnectOptions }`) are also kept on their own `export type` lines.
- **No teardown-verb misuse.** The package correctly avoids `dispose*`/`destroy*` for slot removal: a signal holds only GC-managed slot arrays with no native resource, so "disconnect" (not `dispose`/`destroy`) is the right verb. `release*` is correctly reserved (none here).
- **Sentinels vs throw.** No expected-failure path throws: `disconnectSignal`/`isSlotConnected` no-op or return `false` when `data === null` (signal never connected) rather than throwing. No internal invariants that correct usage cannot reach are validated.
- **Module-private state placement.** `nullSignalEmit` is isolated in `internal.ts` with a doc comment explaining the empty-signal no-op invariant, and is documented as _not_ barrel-exported; the private `initSignal`/`makeDispatch` helpers sit after the exported functions in `slot.ts`, per the source-style rule.
- **Globally unique names.** Every export is prefixed/suffixed with `Signal` (`createSignal`, `emitSignal`, `connectSignal`, …) or the slot noun (`isSlotConnected`), so there is no collision risk from the SDK barrel.
- **Side-effect-free.** `"sideEffects": false` and no top-level registration/listener/mutation; signal state is allocated lazily on first connect via `initSignal`, not at import.
