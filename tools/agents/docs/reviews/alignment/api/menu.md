# API Alignment: @flighthq/menu

**Verdict:** Strongly aligned — the exported surface follows the command-capability seam pattern and naming conventions almost exactly; the only notable issues are a public/backend verb mismatch (`showContextMenu` → `popupContextMenu`) and a missing teardown path for the `onMenuSelect` listener.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- | --- |
| Low | `showContextMenu` | Public verb `show*` (correct, matches `showMessageDialog` / `showNotification` / `showItemInFolder`) delegates to a backend method named `popupContextMenu`. The two names for one operation introduce a second vocabulary word (`popup`) on the seam. Internal-only, but the `MenuBackend` interface is exported from `@flighthq/types`, so the mismatch is publicly visible to anyone implementing a backend. | Rename the `MenuBackend.popupContextMenu` method to `showContextMenu` so the seam uses one verb. (Tray uses `setTrayContextMenu`; the `Context Menu` noun is already shared — align the verb too.) |
| Low | `createMenuItemTemplate` | Allocates a new template but shallow-spreads `template`, so the returned object shares the same `submenu` array reference as the input. A `create*` is expected to yield an independently-owned value; mutating the result's `submenu` mutates the caller's input. | Either clone `submenu` (and nested templates) when present, or document explicitly that `submenu` is shared by reference. Given the entity-construction intent, a shallow clone of `submenu` is the safer default. |
| Low | `onMenuSelect` | Returns an unsubscribe closure (good), but there is no package-level teardown and the listener is wired through the live backend. If `setMenuBackend` swaps the backend after a listener is registered, the returned unsubscribe targets the old backend and the new backend has no listener. No `dispose*` or re-attach path documents this lifecycle. | Document that listeners must be re-registered after `setMenuBackend`, or route subscriptions through a package-level registry that survives backend swaps (mirror however `onNotificationClick` / `onTrayEvent` resolve the same concern, for cross-package consistency). |
| Info | `MenuItemType`, `MenuItemRole` | Both are closed string-literal unions in `@flighthq/types`. The map prefers open contracts over closed unions for kind-like vocabularies that hosts may extend (a native host could expose roles beyond the listed set). Not a hard violation since these are descriptor enums, not `*Kind` registries. | Consider widening `role`/`type` to `MenuItemRole | (string & {})` (or a documented open string) so native hosts can pass through host-specific roles without a types change. Leave as-is if the closed set is deliberate for the MVP. |

## Clean

- **Full unabbreviated type words.** Every export spells the operated-on type in full: `createMenuItemTemplate`, `getMenuBackend`, `setMenuBackend`, `createWebMenuBackend`, `setApplicationMenu`, `showContextMenu`, `onMenuSelect`. No abbreviations.
- **Globally unique root exports.** None of the seven exports collide with sibling packages; `setApplicationMenu` / `showContextMenu` are menu-specific and unambiguous. (`MenuItemTemplate` is correctly shared via `@flighthq/types`, consumed identically by `app`, `tray`, `application`.)
- **Command-capability seam shape.** Matches the documented pattern exactly: flat free functions + `getMenuBackend` / `setMenuBackend` / `createWebMenuBackend`, lazily-created web default, `setMenuBackend(null)` falls back to the web backend. Identical in shape to `clipboard`, `dialog`, `shell`, `notification`, `tray`.
- **Sentinels for expected failure, no throws.** Web backend returns `false` from `setApplicationMenu`, `null` from the context-menu popup, and a no-op unsubscribe from `subscribeSelect` — never throws for the unsupported-on-web case. `getMenuBackend` always returns a backend (no null sentinel needed). No validation of unreachable invariants.
- **Allocation discipline.** Only `create*` functions allocate (`createMenuItemTemplate`, `createWebMenuBackend`); no hot-path/out-param functions exist here, so the no-allocate rule does not apply. Verbs are correct.
- **`Readonly<>` discipline.** `createMenuItemTemplate(template?: Readonly<Partial<MenuItemTemplate>>)` and all `readonly MenuItemTemplate[]` array params are immutable. `setMenuBackend(backend: MenuBackend | null)` correctly leaves the backend mutable (it is stored, not read-only), matching `setClipboardBackend` / `setTrayBackend`.
- **Accessor / boolean prefixes.** `getMenuBackend` is a true getter. `setApplicationMenu` returns a `boolean` success flag — acceptable as a command result rather than a misnamed getter (it performs an action, not a query), consistent with `writeClipboardText` returning `Promise<boolean>`.
- **Event hook naming.** `onMenuSelect(listener): () => void` matches the documented `on*(listener): () => void` convention and sibling `onNotificationClick` / `onTrayEvent`.
- **Imports & types layout.** `import type { MenuBackend, MenuItemTemplate } from '@flighthq/types'` is on its own `import type` line; no inline `{ type X }` mixing; no cross-package types defined inline (all live in `packages/types/src/Menu.ts`). Module-private `_backend` is at the bottom of the file per source style.
- **Alphabetization.** Exports are in alphabetical order (`createMenuItemTemplate`, `createWebMenuBackend`, `getMenuBackend`, `onMenuSelect`, `setApplicationMenu`, `setMenuBackend`, `showContextMenu`).
- **Package manifest.** `"sideEffects": false`, single `.` export, `@flighthq/types` as the only runtime dependency — no top-level registration; the web backend is created lazily inside `getMenuBackend`.
