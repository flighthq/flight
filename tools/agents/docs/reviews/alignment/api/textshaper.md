# API Alignment: @flighthq/textshaper

**Verdict:** Strong alignment — a textbook command-capability backend seam; the only real issue is that `shapeText` over-promises (it is named for shaping but returns only an advance, diverging from the backend's own `measureText` verb), with one minor `Readonly<>` nit shared suite-wide.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `shapeText(text, format): number` | The verb `shape` over-promises. Shaping canonically produces glyphs/clusters/advances/offsets; this function returns a single `number` (horizontal advance width) and delegates to `TextShaperBackend.measureText`. The package's only operating verb therefore disagrees with the backend method it calls (`measureText`) and with the value it returns. The source comment itself concedes "Advances are all current shaping produces (canvas measureText)." A reader sees `shapeText(...) : number` and cannot tell whether `number` is a width, a glyph count, or a status. | Rename to match the actual semantics and the backend verb, e.g. `measureTextWidth` / `getTextShaperAdvance`. Reserve a future `shapeText` for when the seam returns real shaped output (glyph ids/clusters/offsets). If `shapeText` is kept deliberately as the seam's forward-looking name, document on the function that it currently returns advance width only and will widen to a shaped-glyph result. |
| Low | `shapeText(text, format)` | Name does not carry the seam's type word. Siblings `getTextShaperBackend`/`setTextShaperBackend` are globally self-identifying (the `TextShaper` infix leads straight back to this package); `shapeText` is generic enough that it does not lead back to the textshaper domain from a bare reference. The map's "globally self-identifying without context" rule applies to the package root barrel. | Fold into the rename above — any `TextShaper`-infixed name (`getTextShaperAdvance`, `measureTextShaperWidth`) restores self-identification and barrel uniqueness. |
| Low | `setTextShaperBackend(backend: TextShaperBackend \| null)` | The stored backend is only ever read (its `measureText` is invoked, never mutated), but the parameter and the `_backend` module reference are not `Readonly<>`. The map's "default to `Readonly<>`, opt out only when mutation is deliberate" applies to stored references. | Type as `setTextShaperBackend(backend: Readonly<TextShaperBackend> \| null)` and store `_backend: Readonly<TextShaperBackend> \| null`. This is suite-wide (`setClipboardBackend`, `setStorageBackend`, `setPlatformBackend` share the non-`Readonly` form), so fix as a coordinated seam change, not in `textshaper` alone. |
| Info | `getTextShaperBackend(): TextShaperBackend \| null` | Returns nullable, unlike sibling `getClipboardBackend`/`getStorageBackend`/`getPlatformBackend` which lazily create a web default and return non-null. This is an **intentional, documented** divergence: the canvas backend needs DOM + font-string computation, so it lives in `@flighthq/textshaper-canvas` and is installed via `setTextShaperBackend` — there is no light in-package web default to lazily create. The leading comment explains this and ties it to text-layout's historical null-until-registered measure provider. No action — recorded so the divergence from the suite's non-null getter is on the record as justified. | — |

## Clean

- **Sentinel over throw.** `shapeText` returns `-1` when no backend is registered (expected pre-setup state), and the comment explicitly distinguishes the `-1` sentinel from a real zero-width advance. No expected-missing case throws; no unreachable internal invariant is validated. Exactly the map's sentinel rule.
- **No teardown-verb misuse.** No `dispose*`/`destroy*`/`acquire*`/`release*` present; the seam holds no non-GC resource and `setTextShaperBackend(null)` is the correct clear path, so there is no synonym drift to flag.
- **`get*`/`set*` accessor naming.** `getTextShaperBackend` returns a value (not a boolean); `setTextShaperBackend` is a plain setter. No `get*` returns a boolean; no boolean getter is missing `has*`/`is*`.
- **Backend-seam verb consistency (registration).** `get*Backend` / `set*Backend(backend: … | null)` exactly matches `clipboard`, `storage`, and `platform`. The companion `createCanvasTextShaperBackend()` in `@flighthq/textshaper-canvas` follows the suite's `create<Host>*Backend(): *Backend` shape. The registration half of the seam is a clean instance of the documented command-capability pattern.
- **Type hygiene.** `import type { TextFormat, TextShaperBackend } from '@flighthq/types'` is on its own dedicated `import type` line; both cross-package types live in `@flighthq/types` (`TextShaper.ts`, `TextLayout.ts`), none redefined inline. `Readonly<TextFormat>` is correctly applied to the `format` parameter of `shapeText`.
- **Import side-effect-free / opt-in registration.** No top-level registration; `_backend` initializes to `null` and is only ever populated through the explicit `setTextShaperBackend` opt-in. `"sideEffects": false` holds and the only dependency is `@flighthq/types`.
- **Source style.** Module state (`let _backend`) sits below the exported surface per source-style rules; exports are alphabetized (`get` < `set` < `shape`), so `order:check` is clean. One source file (`textShaper.ts`) with a colocated `textShaper.test.ts` covering all three exports including the alias/last-write-wins and `-1` sentinel paths.
