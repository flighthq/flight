# API Alignment: @flighthq/filters-gl

**Verdict:** Strong, internally consistent backend API — naming, allocation discipline, and `state`/`source`/`dest` ordering are exemplary; the one real gap is two shader-location types that escape through exported `get*` return positions without being exported or living in `@flighthq/types`.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| High | `getGlBlitOffsetShader`, `getGlInvertTintShader`, `getGlTintShader` | Return the inline `BlitOffsetShaderLocations` / `TintShaderLocations` types (defined in `glBlitShader.ts` / `glTintShader.ts`), but neither type is exported from the barrel. The return value is unnameable by callers, and the cross-package type is defined inline instead of in `@flighthq/types`. The wgpu twin handles this correctly — `filters-wgpu` does `export type { WgpuDualSourcePipeline, WgpuFilterPipeline }`. | Move `BlitOffsetShaderLocations` and `TintShaderLocations` (the shader-program-locations contract) into `@flighthq/types` alongside the existing `GlShaderLocations`, then `export type { ... }` them from `filters-gl`'s `index.ts`, mirroring the wgpu sibling. |
| Low | `createGlGradientRampTexture` | Sole exported function that does not take `state: GlRenderState` first — it takes a raw `gl: WebGL2RenderingContext`. Every other export is `(state, ...)`. Asymmetric entry shape for one function. | Acceptable if intentional (it is a pure GPU-resource constructor with no render-state dependency), but consider `(state, colors, alphas, ratios)` for surface symmetry, deriving `gl` from `state.gl` as the rest of the package does. Document the deviation if kept. |
| Low | `applyBevelFilterToGl`, `applyDropShadowFilterToGl`, `applyGradientGlowFilterToGl`, `applyGradientBevelFilterToGl`, `applyInnerGlowFilterToGl`, `applyInnerShadowFilterToGl`, `applyOuterGlowFilterToGl`, `applySharpenFilterToGl` | `scratch: GlRenderTarget[]` is a mutable array; the functions only read its slots (they mutate target GPU contents, never reassign array elements). Per the `Readonly<>`-by-default rule this should be `ReadonlyArray<GlRenderTarget>`. Cross-package, not local: the wgpu sibling uses the same mutable `WgpuRenderTarget[]`, so this is a shared convention to settle, not a one-package slip. | If applied, change in both `filters-gl` and `filters-wgpu` together (`ReadonlyArray<...>`) so the scratch-buffer convention stays symmetric across backends. |
| Low | `glOuterGlowFilter.ts:4`, `glDropShadowFilter.ts:4` | Two separate `import type` lines from `@flighthq/types` (one for the filter type, one for `GlRenderState`) where a single `import type { DropShadowFilter, GlRenderState }` would do — every other file in the package already collapses these. Source-style cleanliness, not API. | Merge into one `import type { ... } from '@flighthq/types'` line. |

## Clean

- **Full, unabbreviated type words everywhere.** `applyColorMatrixFilterToGl`, `applyDisplacementMapFilterToGl`, `applyGradientBevelFilterToGl`, `createGlGradientRampTexture` — no `DOBevel`/`ColorMat`-style abbreviation anywhere. The `*ToGl` suffix and `Gl*` prefix make every name globally self-identifying and unique against the canvas/css/surface/wgpu siblings.
- **Allocation discipline by verb is exact.** Only `createGlGradientRampTexture` allocates (and says so in its doc with explicit caller-owns/`gl.deleteTexture` ownership). Every `apply*` hot-path function writes into caller-provided `dest`/`temp`/`scratch` targets and allocates nothing — confirmed in the doc comments and bodies (e.g. `applyDropShadowFilterToGl` "The filter allocates nothing itself").
- **`Readonly<Omit<XFilter, 'kind'>>`** on every filter descriptor parameter — consistent, correct immutability on the data input, and correctly drops the `kind` discriminant the backend does not need.
- **Cross-package types come from `@flighthq/types`** (`DropShadowFilter`, `ColorMatrixFilter`, `GlRenderState`, `GlFullscreenProgram`, etc.) and `import type` is always on its own line — no `import { type Foo, bar }` mixing anywhere.
- **Parameter-ordering symmetry** is excellent: every `apply*` is `(state, source, dest, ...)`, with the displacement-map's extra input slotted as `(state, source, map, dest, filter)` — matching `applyDisplacementMapFilterToWgpu` exactly. Blit/tint passes follow `(state, source, dest, ...)` identically.
- **Verb consistency with the wgpu twin.** `applyGlBlitPass`/`applyGlBlitOffsetPass`/`getGlBlitShader` and `applyGlTintPass`/`applyGlInvertTintPass`/`getGl*TintShader` mirror `filters-wgpu`'s `applyWgpu*`/`getWgpu*` set name-for-name, and the `apply*Pass` vocabulary for sub-passes is used consistently.
- **`get*` accessors return values, not booleans**; no boolean `get*` and no boolean-returning function that should be `has*`/`is*`. No `dispose*`/`destroy*`/`acquire*`/`release*` misuse — the package owns no teardown surface (resource ownership is pushed to the caller, who deletes the ramp texture).
- **Throws are reserved for programmer error.** The only `throw`s (`glColorMatrixFilter` 20-value check, `glConvolutionFilter` positive-dimensions / 7×7-max / matrix-length checks) are precondition violations on malformed filter descriptors — correct use of throw-on-misuse rather than sentinels.
