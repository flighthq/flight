# API Alignment: @flighthq/filters

**Verdict:** Strongly aligned — a clean, data-descriptor package whose 14 `create*Filter` factories and 2 `computeBoxBlur*` math helpers all obey the naming, allocation, type-import, and sentinel rules; the only real tension is `createColorMatrixFilter`'s positional `matrix` argument breaking the otherwise-uniform `options`-object factory shape.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `createColorMatrixFilter(matrix: ReadonlyArray<number>)` | Sole factory that takes a **bare positional argument** instead of an `Omit<…, 'kind'>` options object. Every other factory — including `createConvolutionFilter`, whose type also has one required array field (`matrix`) among several — uses the uniform `options` object. This breaks parameter-shape symmetry across the package: a caller learns the convention from 13 functions then hits one exception. `ColorMatrixFilter` has exactly one field beyond `kind` (`matrix`), but so do `MedianFilter`/`PixelateFilter`/`SharpenFilter` etc., which still go through `options?`. | Change to `createColorMatrixFilter(options: Omit<ColorMatrixFilter, 'kind'>)` (required, since `matrix` is required) for symmetry with `createConvolutionFilter`/`createGradientBevelFilter`/`createGradientGlowFilter`. If the positional form is kept for ergonomics, record it as a deliberate exception in the package map so the divergence is documented rather than silent. |
| Low | `createConvolutionFilter` (and `createColorMatrixFilter`, `createGradientBevelFilter`, `createGradientGlowFilter`) | The returned descriptor's `matrix` (and `colors`/`alphas`/`ratios`) is **stored by reference** from the caller-supplied array via spread/assignment. The type marks them `ReadonlyArray<number>`, so the descriptor cannot mutate them, but a caller who later mutates the original array silently mutates the "immutable" filter. This is consistent with the package's value-descriptor philosophy (no defensive copy), so it is a note, not a defect. | Optional: document in the type or factory that array fields are retained by reference and the caller must not mutate them after construction, or clone in the factory if defensive immutability is desired. Leaving as-is is acceptable for a plain-data package. |

## Clean

- **Full, unabbreviated type words in every name.** All 14 factories spell out the complete filter type — `createDisplacementMapFilter`, `createGradientBevelFilter`, `createColorMatrixFilter` — none abbreviated. The two math helpers, `computeBoxBlurRadius` and `computeBoxBlurPassRadius`, name the full concept (box blur, pass radius).
- **Allocation by verb is correct.** The allocating factories use `create*` (they return a freshly built descriptor object); the two hot-path math helpers `computeBoxBlurRadius` / `computeBoxBlurPassRadius` use `compute*` and **allocate nothing** — they take only `number` primitives and return a `number`. No `out`/`target` parameter is needed because there is no object output, so the no-allocation-math rule is satisfied without one.
- **`out`-param / alias-safety not applicable, correctly.** No function takes an `out`/`target`; the math helpers return scalars, so there is no aliasing surface to get wrong.
- **Globally unique names.** The `create*Filter` suffix and `computeBoxBlur*` prefix namespace the entire surface; no collision risk from the SDK barrel, and the near-twins `computeBoxBlurRadius` vs `computeBoxBlurPassRadius` are clearly distinguished by the `Pass` token.
- **Sentinels vs throw.** No expected-failure path is forced through an exception, and no unreachable internal invariants are validated. `computeBoxBlurRadius`/`computeBoxBlurPassRadius` return `0` for `sigma <= 0` (a clean in-band sentinel) rather than throwing.
- **`Readonly<T>` discipline.** Array inputs are typed `ReadonlyArray<number>` (`createColorMatrixFilter`); the `Omit<…, 'kind'>` option objects inherit the all-`readonly` fields from the `@flighthq/types` interfaces, so option props are immutable. Scalar parameters (`sigma`, `passes`, `pass`) are primitives and exempt.
- **Cross-package types from `@flighthq/types`.** Every filter interface (`BevelFilter`, `BlurFilter`, …, `DisplacementMapMode`) and `BitmapFilter` is imported from the header layer and re-exported from the barrel — none redefined inline. The package's only runtime dependency is `@flighthq/types`.
- **`import type {}` discipline.** Every source file uses a dedicated `import type { … } from '@flighthq/types'` line; no mixed `import { type Foo, bar }` form. The barrel re-exports types in a single `export type { … }` block separate from the value `export`s.
- **Kind-string convention.** Each factory stamps a canonical PascalCase string kind (`kind: 'BevelFilter'`) matching the type's `readonly kind` literal — strings, not `Symbol()`, per the kind-identity model.
- **Required vs optional `options` is principled.** Filters with required fields take a required `options` (`createConvolutionFilter`, `createGradientBevelFilter`, `createGradientGlowFilter` — all have required `matrix`/`colors`/`alphas`/`ratios`); all-optional filters take `options?` (`createBlurFilter`, `createMedianFilter`, etc.). The split tracks the type definitions exactly.
- **No teardown / accessor / boolean surface.** No `get*`/`has*`/`is*`, `dispose*`/`destroy*`, or `acquire*`/`release*` functions exist — appropriate for a stateless data-descriptor + math package, so none can be misused.
- **Source style.** One source file per filter domain, named for the object (`bevelFilter.ts`, `displacementMapFilter.ts`) with colocated `*.test.ts`; `blurMath.ts` keeps its two module-private helpers (`computeBoxBlurLowerWidth`, `computeBoxBlurLowerPassCount`) below the exported functions, and the load-bearing variance math is documented in comments rather than restating obvious code.
