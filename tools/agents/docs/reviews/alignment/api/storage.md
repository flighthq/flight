# API Alignment: @flighthq/storage

**Verdict:** Strongly aligned — a textbook command-style platform-integration cell; only one minor `Readonly<>` nit and a small naming-asymmetry note, no real defects.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Low | `setStorageBackend(backend: StorageBackend \| null)` | The stored backend reference is not marked `Readonly<>`. Per the Design Constraints, object references that are not mutated should default to `Readonly<T>` (mirroring C++ `const`). The function only stores and later invokes methods on it; it never mutates the object's own fields. | Type the parameter `backend: Readonly<StorageBackend> \| null` (and the `_backend` slot / `getStorageBackend` return as `Readonly<StorageBackend>`), consistent with the const-by-default rule. Verify the sibling command packages (`clipboard`, `shell`) so the whole suite moves together rather than diverging — this is a suite-wide convention call, not storage-specific. |
| Info | `getStorageItem` / `setStorageItem` / `removeStorageItem` vs `clearStorage` / `getStorageKeys` | Mild lexical asymmetry: the single-key operations carry the noun `Item` (`getStorageItem`), while the whole-store operations drop it (`clearStorage`, `getStorageKeys`). Not a violation — the split is meaningful (per-item vs whole-store) and tracks the `localStorage` vocabulary the docs explicitly endorse ("the right word is the one a reader would have reached for independently"). | No change required. Flagged only so it is a recorded, deliberate choice rather than drift. The current spelling reads naturally and is preferable to forcing `clearStorageItems`. |

## Clean

- **Full, unabbreviated type word everywhere.** Every export names `Storage` in full — `clearStorage`, `createWebStorageBackend`, `getStorageBackend`, `getStorageItem`, `getStorageKeys`, `removeStorageItem`, `setStorageBackend`, `setStorageItem`. No abbreviations, no `getStoreItem` / `getKVItem` shortcuts.
- **Globally unique names.** All seven exports are package-prefixed (`*Storage*`); none collide with the generic `getItem`/`keys`/`clear` verbs, which correctly stay confined to the `StorageBackend` seam interface in `@flighthq/types`, not the package root.
- **Sentinels, never throws for expected failure.** Writes/removes/clear return `boolean` (false on denial), reads return `string | null`, `getStorageKeys` returns `[]`. The web backend wraps every `localStorage` call in try/catch and converts the throw into the documented sentinel — exactly the "expected-failure surface" rule. No precondition-validation throws.
- **Command-capability shape matches the suite.** `createWebStorageBackend` / `getStorageBackend` / `setStorageBackend(... | null)` mirror `@flighthq/clipboard` (`createWebClipboardBackend` / `getClipboardBackend` / `setClipboardBackend`) one-for-one — same verbs, same null-to-reset semantics, same lazy web default. The `create*` verb is correctly used (it allocates a backend object); `get*`/`set*` used correctly for the accessor/installer pair.
- **Accessor conventions.** `get*` prefix used for the three readers (`getStorageBackend`, `getStorageItem`, `getStorageKeys`); none return a bare boolean dressed as a getter. The boolean-returning operations (`setStorageItem`, `removeStorageItem`, `clearStorage`) are imperative commands, correctly not forced into `has*`/`is*`.
- **No allocation-verb misuse, no teardown-verb drift.** No `dispose*`/`destroy*`/`acquire*`/`release*` appear — none are needed (no GC-reachable listeners and no non-GC resource owned), so their absence is correct rather than a gap. `clearStorage` is data clearing, not entity teardown, and is correctly not named `disposeStorage`.
- **Synchronous-by-design, documented.** The `StorageBackend` seam returns values directly (not Promises) with an inline rationale tying it to `localStorage` being synchronous — matches the platform-suite "web backend guards every API and returns sentinels" contract.
- **Types sourced from `@flighthq/types`.** `StorageBackend` is defined once in `packages/types/src/Storage.ts` and imported via `import type { StorageBackend } from '@flighthq/types';` on its own dedicated `import type` line — no inline cross-package type, correct import-type style.
- **Side-effect-free / opt-in.** `"sideEffects": false`; the default web backend is created lazily inside `getStorageBackend`, not registered at module top level — honoring the import-side-effect-free rule.
- **Source style.** Exports alphabetized (`clearStorage` → `setStorageItem`); module-private state (`_backend`, `getWebStorage`) placed at the bottom after the public surface; comments carry the ownership/failure semantics that names cannot.
- **TS↔Rust.** Crate `flighthq-storage` exists (`crates/flighthq-storage/src/storage.rs`) and the browser-bound verbs are an explicitly recorded conformance item in `tools/agents/docs/rust/conformance.md` (the `host-web` seam list) — no silent divergence.
