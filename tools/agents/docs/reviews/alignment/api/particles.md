# API Alignment: @flighthq/particles

**Verdict:** Strong, convention-disciplined surface — full type words, no inline cross-package types, no thrown errors for expected cases; the only real issues are a verb inconsistency (`build*` vs `create*`) and shared module-level scratch buffers that break re-entrancy / C-port intent.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- | --- |
| Medium | `buildParticleColorCurve`, `buildParticleCurve` | These allocate and return a new LUT (`number[]`), but use the verb `build*` while every other allocating constructor in the package uses `create*` (`createParticleEmitterConfig`, `createParticleEmitterState`, `createParticleObjectsState`). `build*` appears nowhere else in the package and breaks the allocation-verb convention (`create*`/`clone*`/`acquire*` are the sanctioned allocators). | Rename to `createParticleColorCurve` / `createParticleCurve` (they take a generator fn + sample count and return a fresh array — exactly a `create*`). Keep `*FromKeyframes` / `*ToKeyframes` as the conversion pair. |
| Medium | `applyParticleForces`, `applyParticleCollisions` (+ object variants) | Use file-module-level shared scratch (`const accel`, `const s`) mutated inside the per-particle loop. This makes the functions non-re-entrant: nested/concurrent calls (e.g. a force pass that triggers a sub-emitter force pass, or future worker-parallel emitters) clobber each other's scratch. It also fights the C/C++ portability goal (hidden file-static mutable state). | Move the scratch to a stack local (`const accel: [number, number] = [0, 0]`) inside each function — the allocation is once per call, not per particle, so the hot loop stays allocation-free. |
| Low | `sampleParticleColorCurve(lut, t, out, offset)` | The `out` sink is typed as an anonymous structural `{ [index: number]: number }` rather than a named/`Readonly`-aware type. It is the only `out`-param in the package and the only one not using a `@flighthq/types` shape, so the intent (a writable RGB triple sink at `offset`) is carried by the param name alone. | Type `out` as `Float32Array | number[]`(the actual call sites pass`data.colors`), or a named `MutableNumberArray`-style alias from `@flighthq/types`, so the writable contract is explicit. |
| Low | `sampleParticleColorCurve` | Out-param alias-safety is not documented. It reads `lut[a]`/`lut[b]` then writes `out[offset..]`; when `out === lut` and `offset` overlaps the read window the result is fine only because reads precede the writes for that index — but this is incidental, not stated. | Add a one-line comment noting the read-before-write ordering and that `out` may alias `lut` only at a non-overlapping `offset`, or assert the contract. |
| Low | `validateParticleEmitterConfig` / `normalizeParticleEmitterConfig` co-located in `validateParticleEmitterConfig.ts` | `normalizeParticleEmitterConfig` is a public allocator exported from a file named after a sibling function; its home file does not name it. (Filename-dimension concern, noted for cross-reference; not an exported-name defect.) | Out of scope for the API dimension — flag to the `filenames` pass: consider a `particleEmitterConfig`-domain file or a `normalizeParticleEmitterConfig.ts`. |

## Clean

- **Full, unabbreviated type words everywhere.** Every export carries `Particle`, `ParticleEmitter`, `ParticleObjects`, `ParticleColorCurve`, or `ParticleCurve` in full — no `Emitter`/`Obj`/`Cfg` shortenings. `ensureParticleEmitterStateCapacity` vs `ensureParticleObjectsStateCapacity` keep the two state families fully distinguished.
- **Globally unique root exports.** No bare/generic names (no `update`, `apply`, `sample`); all are prefixed by their subject, safe in the SDK barrel.
- **Sentinels over throws.** Zero `throw` in the package. `emitParticleBurst` returns `0` when capped, `validateParticleEmitterConfig` returns `[]` for a clean config, curve samplers return `0` for empty LUTs — expected-failure paths all use sentinels; no validation of unreachable internal invariants.
- **Allocation discipline (mostly).** `create*`/`*FromKeyframes`/`*ToKeyframes` allocate and say so; `ensure*Capacity` grow in place; the hot passes (`applyParticle*`, `updateParticle*`, `sampleParticle*`) are allocation-free in the loop. Only the `build*` naming (above) deviates.
- **`Readonly<T>` on read-only object params.** `config: Readonly<ParticleEmitterConfig>`, `state: Readonly<ParticleEmitterState>` (on the `is*` queries), `worldTransform?: Readonly<WorldTransform2D>`, and `ReadonlyArray<ParticleForce | ParticleCollider | …Keyframe>` are used consistently; mutable params (`emitter`, the simulated `state`, `objects`) are deliberately writable.
- **`is*` for booleans.** `isParticleEmitterComplete` / `isParticleObjectsComplete` correctly use the `is*` prefix and return `boolean`; no `get*` returns a boolean.
- **No inline cross-package types.** Every interface/type alias (`ParticleEmitterConfig`, `ParticleEmitterState`, `ParticleConfigIssue`, `ColorKeyframe`, all force/collider kinds, `WorldTransform2D`, `RandomSource`) is imported from `@flighthq/types` and only re-exported (`export type { … }`) for convenience — none defined locally.
- **`import type {}` hygiene.** Every type import is on its own `import type { }` line; no mixed `import { type Foo, bar }`.
- **No teardown-verb misuse.** The package owns no GC-detach or non-GC resources, and correctly exposes no `dispose*`/`destroy*` (state is plain GC-managed typed arrays); `acquire*`/`release*` are correctly absent (it leans on `@flighthq/geometry` `reserveFloat32Array` for growth).
- **Opt-in, side-effect-free passes.** `applyParticleForces`/`applyParticleCollisions` are documented as fully opt-in with no top-level registration — matches the tree-shaking and "no hidden update pass" rules.
