# API Alignment: @flighthq/effects-gl

**Verdict:** Strongly conformant — names carry full type words, `Readonly<>`/sentinels/teardown verbs are used correctly, and the registry/runner seam is idiomatic; the only real issues are signature asymmetry across the `apply*ToGl` family (per-effect extra params) and a couple of runners that silently drop the depth G-buffer they document needing.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `applyBloomEffectToGl` vs the rest of the `apply*EffectToGl` family | Signature asymmetry. Most effects are `(state, source, dest, effect)`; bloom inserts `pool` at arg 4 (`state, source, dest, pool, effect`); the depth/velocity effects insert a texture at arg 4 (`...dest, depthTexture, effect)` / `...dest, velocityTexture, effect`). `effect` is sometimes the 4th and sometimes the 5th positional arg, so the family has no stable tail. | Keep `effect` last is already honored, but the inserted slot breaks scanability. Consider standardizing the optional G-buffer/pool inputs into a single trailing options position, or at minimum keep the inserted resource immediately after `dest` consistently (bloom's `pool` and the others' `depthTexture`/`velocityTexture` already are) and document the family shape so the variance is intentional, not accidental. |
| Medium | `applySsaoEffectToGl`, `applySsrEffectToGl` (+ their `default*Runner`s) | Both are `[DEPTH]`-class screen-space effects but their `apply*` signatures take no depth texture and their runners ignore `ctx.sceneDepthTexture` — unlike `applyBokehDepthOfFieldEffectToGl` / `applyScreenSpaceFogEffectToGl`, which thread `ctx.sceneDepthTexture` through a `depthTexture` param. SSAO/SSR cannot read the G-buffer they conceptually require, so they are silent color-only placeholders with no seam to upgrade. | For consistency with the other depth effects, give `applySsaoEffectToGl`/`applySsrEffectToGl` a `depthTexture: WebGLTexture \| null` param (after `dest`) and have the runners pass `ctx.sceneDepthTexture`, matching bokeh/fog. (Correctness/completeness, but it manifests as an API asymmetry within the depth-effect subset.) |
| Low | `getGlEffectProgram` | `get*` accessor that lazily compiles + caches a program on a miss (`compileGlFullscreenProgram`), i.e. it allocates/creates a GPU resource, which the verb grammar reserves for `create*`/`acquire*`. It reads as a pure accessor but has a one-time create side effect. | Acceptable as a memoized getter, but consider `acquireGlEffectProgram` (it is effectively a cache acquire) or a doc note that first call compiles. Not blocking; the get-or-create memo pattern is used elsewhere. |
| Low | `createGlRenderEffectPipeline(_state, options)` | First param `_state` is declared but unused (underscore-prefixed). It exists only for call-site symmetry with the other pipeline functions. A required-but-ignored param is a small API smell. | Keep for symmetry (reasonable) but confirm it is load-bearing for a future GL-context-bound allocation; if not, dropping it would tighten the surface. Document why it is retained. |
| Low | `GlRenderEffectRunner` doc (in `@flighthq/types`, consumed here) | The type's comment says runners are "registered against an effect `type`," but the discriminant field is `kind` and the code correctly uses `kind` (`registerGlRenderEffect(state, kind, runner)`, `RenderEffect.kind`). Doc/term drift adjacent to this package's public seam. | Fix the comment in `packages/types/src/GlRenderEffectPipeline.ts` to say `kind`. (Out of this package, but surfaced because it describes effects-gl's registration contract.) |
| Info | `applyTaaEffectToGl(state, source, dest, _effect)` | The `_effect` underscore correctly signals an unused param for a documented passthrough placeholder. Flagged only so it is understood as intentional, not an oversight. | No change. The header comment explains the placeholder; this is the right way to keep the stage shape. |

## Clean

- **Full, unabbreviated type words.** Every `apply*EffectToGl` / `default*EffectRunner` name embeds the exact effect type word from `@flighthq/types` (`applyBrightnessContrastEffectToGl` ↔ `BrightnessContrastEffect`, `applyLiftGammaGainEffectToGl` ↔ `LiftGammaGainEffect`). Acronym names (`Fxaa`, `Crt`, `Smaa`, `Ssao`, `Ssr`, `Taa`) match their canonical type names and are the established term for the effect, not abbreviations of a spelled-out word.
- **Globally-unique, self-identifying names.** The `…EffectToGl` suffix and `Gl…` prefix make every export unambiguous from the barrel; no collision risk with sibling `effects-canvas`/`effects-wgpu` families.
- **`Readonly<>` discipline.** All `source`/`dest`/`effect`/`options` params are `Readonly<…>`; mutable args are exactly the intended outputs (`state`, `pipeline`, the GL target the pass writes). `GlRenderEffectContext` and `RenderEffectPipelineOptions` fields are `readonly` where appropriate.
- **Teardown verbs.** `destroyGlRenderEffectPipeline` correctly uses `destroy*` — it frees GPU render targets and the target pool now. The pipeline borrows intermediates via `acquireGlRenderTarget`/`releaseGlRenderTarget` as matched brackets inside `endGlRenderEffectPipeline`. No `dispose`/`destroy` synonym drift.
- **Sentinels over throws.** `getGlRenderEffectRunner` returns `null` for an unregistered kind; `endGlRenderEffectPipeline` skips unknown runners (`if (runner === null) continue`) and early-returns on a null scene target. No thrown errors for expected-missing cases anywhere in the package.
- **Registry / runner seam.** `registerGlRenderEffect` + `getGlRenderEffectRunner` over a per-state `WeakMap` registry, last-write-wins, with `default*EffectRunner` constants importable individually — the opt-in, tree-shakable material-renderer pattern the map prescribes. No top-level registration side effects (`"sideEffects": false` honored).
- **`import type {}` hygiene.** Every file puts type-only imports on dedicated `import type { … }` lines; no mixed `import { type Foo, bar }`. All cross-package types come from `@flighthq/types` (`GlRenderState`, `GlRenderTarget`, the per-effect `*Effect` interfaces, `GlRenderEffectRunner`), none defined inline.
- **Out-target safety.** `apply*` functions read `source.texture` (input) and write to `dest` (output); source and dest are distinct ping-pong targets by construction in the pipeline, and inputs are read before the draw writes the output — no read-after-write aliasing hazard.
- **`begin`/`end` bracket pair.** `beginGlRenderEffectPipeline` / `endGlRenderEffectPipeline` form a clear scoped pair around scene rendering, consistent with the render-gl target begin/end convention they wrap.
