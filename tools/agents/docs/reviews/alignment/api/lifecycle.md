# API Alignment: @flighthq/lifecycle

**Verdict:** Well aligned — full type words, correct event-capability seam, proper `dispose` semantics; the only real issue is a split root word (`AppLifecycle` entity vs `Lifecycle` backend) that breaks symmetry with sibling event packages.

## Findings

| Severity | Symbol | Issue | Suggested fix |
| --- | --- | --- | --- |
| Medium | `createAppLifecycle`, `attachAppLifecycle`, `detachAppLifecycle`, `disposeAppLifecycle`, `getAppLifecycleState` vs `getLifecycleBackend`, `setLifecycleBackend`, `createWebLifecycleBackend` | The package uses **two** root words for one concept: the entity and its operations carry `AppLifecycle`, but the backend seam and its accessors carry plain `Lifecycle` (`LifecycleBackend`, `getLifecycleBackend`). Sibling event packages use a single root word throughout — `network` is `Network` + `NetworkBackend` + `getNetworkBackend`; `power` is `Power` + `PowerBackend` + `getPowerBackend`. The `App` prefix appears only on the entity half, so the barrel reads as two loosely related families instead of one. | Pick one root word and apply it package-wide. Either drop the `App` prefix (entity `Lifecycle`, `createLifecycle`/`attachLifecycle`/`getLifecycleState`, matching `network`/`power` exactly), or carry `App` through the backend (`AppLifecycleBackend`, `getAppLifecycleBackend`, `createWebAppLifecycleBackend`). The `network`/`power` precedent favors the unprefixed `Lifecycle`. Note this touches `@flighthq/types` (`AppLifecycle`/`AppLifecycleState`) too. |
| Info | `createAppLifecycle` | Builds the entity from an inline object literal of `createSignal()` calls rather than a constructor/helper. This matches the sibling event packages (`createNetwork`, `createPower` do the same) and the entity is a flat bag of signals with no runtime/binding identity, so the "prefer constructors over literals" rule does not bite here. Recorded for transparency, not a defect. | No change. |
| Info | `getLifecycleBackend` | A `get*` accessor that lazily allocates the web default on first call (`_backend = createWebLifecycleBackend()`), slightly outside the "allocation only in `create*`/`clone*`/`acquire*`" verb rule. | No change — this is the deliberate, cross-package backend-seam convention (identical lazy-init in `getNetworkBackend`, `getPowerBackend`, `getClipboardBackend`, `getStorageBackend`). |

## Clean

- **Full, unabbreviated type word.** Every function spells out its type word — `AppLifecycle` on the entity functions, `Lifecycle`/`LifecycleBackend` on the seam, `AppLifecycleState` as the return of `getAppLifecycleState`. No abbreviation of a type name. (The defect above is _which_ word, not a truncation.)
- **Event-capability seam matches the documented pattern.** The map specifies event capabilities expose `create*`/`attach*`/`detach*`/`dispose*` plus a `get*Backend`/`set*Backend`/`createWeb*Backend` triad; lifecycle implements all of it, mirroring `network` and `power` one-for-one (entity create, attach/detach/dispose, three backend functions, state reader).
- **Teardown verbs used correctly.** `disposeAppLifecycle` detaches the backend subscription and releases the entity to GC (the signals are plain GC memory afterward) — exactly the `dispose*` meaning. No `destroy*` is present, which is correct: the entity owns no non-GC resource. `detachAppLifecycle` is a domain "stop delivery" operation, distinct from teardown, and is named as such.
- **Idempotent / safe re-entry.** `attachAppLifecycle` tears down any prior subscription first; `detachAppLifecycle` is safe when not attached; `disposeAppLifecycle` delegates to `detach`. No precondition throws.
- **Sentinels, no throwing for expected cases.** `createWebLifecycleBackend` degrades to state `'active'` and a no-op unsubscribe when `document`/`window` are absent rather than throwing; `setLifecycleBackend(null)` resets to the lazy web default. No thrown errors for expected-missing host APIs.
- **Boolean/accessor prefixes correct.** `getAppLifecycleState` and `getLifecycleBackend` are genuine getters returning values; no `get*` returns a boolean, and there are no boolean-returning exports that should be `is*`/`has*`.
- **`import type {}` on its own line.** `import type { AppLifecycle, AppLifecycleState, LifecycleBackend } from '@flighthq/types';` is a dedicated type-only import, separate from the value import of `@flighthq/signals`.
- **Cross-package types live in `@flighthq/types`.** `AppLifecycle`, `AppLifecycleState`, and `LifecycleBackend` are all defined in `packages/types/src/Lifecycle.ts`, not inline.
- **No top-level side effects.** Module-level state (`_backend`, `_subscriptions`) is inert until a function is called; no registration, listener, or timer runs at import. `"sideEffects": false` is honest.
- **Out-param alias-safety N/A.** No `out`/`target` parameters — all functions either return a fresh value or act on the singleton backend / a passed entity. No hot-path math to allocate in.
- **`Readonly<T>` consistent with the package class.** The entity params (`app: AppLifecycle`) are unmarked, matching `attachNetwork`/`disposePower`; these functions read the entity's signal fields and pass them to `emitSignal`. The omission is uniform across the event-capability class, so it is not a lifecycle-specific defect.
- **Exports alphabetized** within `lifecycle.ts` (order:check clean), and the test `describe` blocks mirror the exported names in the same order.
