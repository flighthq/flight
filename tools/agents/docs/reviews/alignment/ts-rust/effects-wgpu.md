### Suggested divergence-map additions

1. **wgpu effect-program-cache surface is larger than TS by design.** The existing line-61 entry ("`effects-wgpu` still pending ‚Ä¶ requires a separate `WgpuFilterState`, a Rust-only API split") already names the root cause; extend it to record the resulting extra cache symbols (`ensure_wgpu_effect_pipeline`, `get_wgpu_dual_source_effect_pipeline`, `build_wgpu_effect_module_wgsl`, `clear_wgpu_effect_pipeline_cache`, `draw_wgpu_effect_filter_pass`, `draw_wgpu_dual_source_effect_pass`, `draw_wgpu_effect_gaussian_blur`) so they read as expected, not as drift. Once bloom adopts the filters-wgpu Gaussian and the `WgpuFilterState` plumbing is folded in, revisit whether these locals collapse.
2. **wgpu effect types live in the backend crate, not the Rust types header.** Same finding as effects-gl: TS keeps `WgpuRenderEffectContext` / `WgpuRenderEffectRunner` / `WgpuRenderEffectPipeline` in `@flighthq/types`; Rust defines them in `flighthq-effects-wgpu`. Record once as the intended port-wide GPU-backend-type pattern (covering effects-gl + effects-wgpu + likely filters-gl/filters-wgpu) rather than per-crate.
3. **Minor extra registry helpers** `clear_wgpu_render_effect_registry` / `wgpu_render_effect_type` have no TS counterpart (test-support / accessor). A blanket note that GPU effect-registry crates carry a clear+type accessor pair would cover both effects-gl and effects-wgpu.
