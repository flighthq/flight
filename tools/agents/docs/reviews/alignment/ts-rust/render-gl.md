# TS↔Rust Alignment: @flighthq/render-gl

**Verdict:** Strong name/filename alignment (40 of 45 TS exports map 1:1); the substantive issues are three undocumented function-level divergences — a missing `clearGlRenderTarget`, a reshaped fullscreen pass that drops `compileGlFullscreenProgram` + `GlFullscreenProgram`, and a missing `setGlQuadMatrixFromOffset` — plus a Rust-only `composite_gl_cached_texture` public primitive; none are in the divergence map and none are visible to the conformance script (render-gl is GPU-name-match-excluded).

`render-gl` is a **GPU backend core**, listed in the name-match exclusion set in both `tools/agents/docs/rust/conformance.md` (line 115, "their conformance is visual — the parity matrix at the `gl`/`wgpu` cells") and `scripts/rust-conformance.ts` (`GPU_CRATES`, line ~165). The summary row `| render-gl | 45 | 3 | 19 | 42 |` is therefore **not a hard-gate failure**: every gapped function is classified `gpu-backend` and validated by the functional render+fingerprint gate, not by name matching. The script also only checks for _TS functions lacking a Rust test_ — it never inspects Rust-side names — so the findings below (renames, omissions, Rust-only additions) are exactly the drift it cannot see. Crate name is identity (`flighthq-render-gl` ↔ `@flighthq/render-gl`); the former `-webgl`→`-gl` divergence is gone (conformance.md line 30).

## Name map findings

| TS symbol / file | Rust symbol / file | Issue |
| --- | --- | --- |
| `clearGlRenderTarget` (`glFullscreenPass.ts`) | — | **Missing.** TS clears a render target to transparent (bind FBO, viewport, `clearColor(0,0,0,0)`, `clear`). No Rust counterpart; `begin_gl_render_target` (`render_target.rs`) binds + sets viewport but does **not** clear. Genuine GPU-core function gap, not in the map. |
| `compileGlFullscreenProgram` (`glFullscreenPass.ts`) + `GlFullscreenProgram` (type) | — | **Missing.** Rust has no dedicated fullscreen program / compile path; `draw_gl_fullscreen_pass` reuses the bitmap shader instead (see below). The `GlFullscreenProgram` type is absent. Undocumented API divergence. |
| `drawGlFullscreenPass(state, program: GlFullscreenProgram, inputs: WebGLTexture[], dest: GlRenderTarget\|null, setUniforms)` | `draw_gl_fullscreen_pass(state, shader: &GlBitmapShader, texture)` (`fullscreen_pass.rs`) | **Reshaped signature, name preserved.** TS is a multi-input post-process pass with an explicit `dest` target and a `setUniforms` callback; Rust takes a single texture, reuses the bitmap shader, and has no dest/uniform-callback. Behavior is narrower than upstream. Should be recorded or brought to parity. |
| `setGlQuadMatrixFromOffset` (`glDraw.ts`) | — | **Missing.** TS builds the quad matrix with an `(dx,dy)` pre-offset and uploads it. No Rust export; the offset-quad path is not factored out. |

The two `set_gl_matrix_*` fns are `pub` in `shader.rs` but **not re-exported from `lib.rs`** (module-public only); their TS counterparts are package exports. |
| `setGlAttributes` / `setGlBaseUniforms` (`glShader.ts`) | `set_gl_attributes` / `set_gl_base_uniforms` (`shader.rs`) | In sync **by name**, but module-public only (not in the `lib.rs` `pub use` barrel) and `set_gl_base_uniforms(gl, loc, alpha: f32)` takes a raw alpha vs TS `(gl, loc, renderProxy: RenderProxy)`. Minor signature reshape, acceptable for a visual core. |
| `createGlCanvasElement` (`glElement.ts`) | — | Expected substrate divergence: no `HTMLCanvasElement` in the box. Native hosts own surface creation. Worth a one-line map note like the `render-wgpu` `createWgpuCanvasElement` entry. |
| `makeGlState` (`glTestHelper.ts`) | — | TS test-only helper (jsdom/webgl mock). No Rust analogue expected — Rust tests use native/headless glow. Not drift. |

In TS this compositing lives in the Canvas-substrate renderers; Rust consolidates it here. Genuine extra public surface that should be recorded. |

Idiomatic Rust, defensible; an "options-vs-descriptor / id-vs-object" map note would close the loop. |

## In sync

- **Crate name + filenames track cleanly:** `glBackground.ts`↔`background.rs`, `glDraw.ts`↔`draw.rs`, `glFullscreenPass.ts`↔`fullscreen_pass.rs`, `glMaterialRegistry.ts`↔`material_registry.rs`, `glRenderState.ts`↔`render_state.rs`, `glRenderTarget.ts`↔`render_target.rs`, `glRenderTargetPool.ts`↔`render_target_pool.rs`, `glShader.ts`↔`shader.rs`, `glShaderBinding.ts`↔`shader_binding.rs`. Every TS `gl*`-prefixed source basename maps to its de-prefixed snake_case Rust file. (TS `glElement.ts`/`glTestHelper.ts`/`glShaderTypes.ts`/`internal.ts` have no Rust file — substrate/test-only/types, expected.)
- **Teardown/pool verbs preserved:** `destroyGl*`→`destroy_gl_*`, `acquireGlRenderTarget`/`releaseGlRenderTarget`→`acquire_*`/`release_*`. Verb semantics carry across exactly. (`destroy_*`, not `dispose_*`, is correct here — these free GPU framebuffers/textures.)
- **Sentinel conventions:** `getGlMaterialRenderer: … | null` → `Option<&GlMaterialRenderer>`, `getGlMaterialShader`/`getGlShader`/`getGlRenderTarget` → `Option`. `Readonly<>` params → `&` borrows; mutating paths → `&mut`. Correct.
- **Material registry, blend, shader-binding, background, draw quad, render-target lifecycle, texture upload** map 1:1 with full type words: `registerGlMaterialRenderer`→`register_gl_material_renderer`, `applyGlBlendMode`/`enableGlBlendModeSupport` identity, `renderGlBackground`→`render_gl_background`, `drawGlQuad`→`draw_gl_quad`, `begin/end/resize/resolve/drawGlRenderTargetResult` identity, `bind/create/updateGlTexture` identity, `compileDefaultGlProgram`/`compileGlBitmapProgram`/`createDefaultGlBitmapShader`/`createGlBitmapShader` identity, `resolveGlShader`/`setGlShader`/`getGlShader` identity, `useGlProgram`→`use_gl_program`.
- **Manifest:** `Cargo.toml` depends only on `flighthq-types`, `flighthq-render`, `glow` — consistent with the `render-gl->displayobject` map note (line 123: types via header, leaves live in `displayobject-gl`).

### Suggested divergence-map additions

1. **Fullscreen pass reshape** — `compileGlFullscreenProgram` + `GlFullscreenProgram` dropped; `drawGlFullscreenPass` narrowed to a single-texture bitmap-shader pass (no `dest`, no `setUniforms`, no multi-input). Either bring Rust to parity or record the simplification with rationale. Currently silent.
2. **`clearGlRenderTarget` missing** — record whether the clear is intentionally folded into `begin_gl_render_target` (it currently is **not**) or is a genuine TODO. This is real GPU-core behavior, not visual-only plumbing.
3. **`setGlQuadMatrixFromOffset` missing** — record the offset-quad path's Rust home or restore the export.
4. **`composite_gl_cached_texture`** — Rust-only public primitive; one line under the canvas-texture-compositing consolidation rationale (mirrors `render-wgpu`'s `composite_wgpu_cached_texture`, which is equally unrecorded).
5. **`createGlCanvasElement`** — one line under the substrate-absence rationale (no `HTMLCanvasElement` in the box), as suggested for `render-wgpu`.
