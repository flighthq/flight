# TS↔Rust Alignment: @flighthq/displayobject

**Verdict:** All 41 TS exports are ported 1:1 (correct snake_case, full type words, clean file-name tracking), but the crate adds **61 extra display-object-specific functions with no `@flighthq/displayobject` counterpart** — concrete hierarchy/transform/accessor/`prepare` implementations that in TS live as generic `*Node*` functions in `@flighthq/node` (and `prepareDisplayObjectRender` in `@flighthq/render`) — and none of this is recorded in the divergence map.

## In sync

- **Every TS exported function is ported, name-exact.** All 41 `@flighthq/displayobject` barrel exports map 1:1 to a Rust `pub fn` with correct camelCase→snake_case and the full type word preserved. Zero missing ports.
- **Out-param / sentinel conventions carry across:** `compute*LocalBoundsRectangle(out, source)` → `compute_*_local_bounds_rectangle` over `&mut`; `isDisplayObject` → `is_display_object` (bool); `getDisplayObjectStage` → `get_display_object_stage` (Option-returning lookup).
- **`createDisplayObjectGeneric<R>`** → `create_display_object_generic` (generic preserved); the full `create*`/`create*Runtime`/`create*Data`/`get*Runtime` quartet ports cleanly for bitmap, video, stage, htmlView, renderView, displayContainer, displayObject.
- **Stage signals:** `createStageSignals`/`getStageSignals`/`enableStageSignals` → `create_stage_signals`/`get_stage_signals`/`enable_stage_signals`. Verb `enable*` preserved.
- **File-name tracking is clean:** `bitmap.ts↔bitmap.rs`, `displayContainer.ts↔display_container.rs`, `displayObject.ts↔display_object.rs`, `htmlView.ts↔html_view.rs`, `renderView.ts↔render_view.rs`, `stage.ts↔stage.rs`, `video.ts↔video.rs`, `index.ts↔lib.rs`. Every Rust basename tracks its TS counterpart (camelCase→snake_case).

## Should be added to the divergence map

The 61 extra functions are not arbitrary — they are the visible surface of a real, defensible port decision (the **slotmap arena**, `tools/agents/docs/rust/index.md` line 41: "free functions over `(&mut NodeArena<T>, NodeId)`"). But that decision is documented only at the _mechanism_ level and authorizes generic `*_node_*` free functions — which `flighthq-node` already provides correctly (`add_node_child`, `get_node_alpha`, …). It does **not** explain why `flighthq-displayobject` also carries a parallel `*_display_object_*` re-spelling of the node hierarchy/transform/bounds API plus de-novo `get_/set_` accessors for bitmap/video/stage/view fields. Two things are missing from `tools/agents/docs/rust/conformance.md`:

1. **A `displayobject` entry recording the typed-arena accessor layer.** Decide and document: are `*_display_object_*` hierarchy/transform accessors the canonical Rust seam (typed `DisplayObjectArena` instead of generic `NodeArena<T>`), or should display-object code call the generic `flighthq-node` `*_node_*` functions (matching TS, where `@flighthq/displayobject` exports none of these)? Right now both exist and the displayobject-named set is silent drift. This parallels the open `timeline`/`create_movie_clip` decision already noted in the map.
2. **`prepare_display_object_render` crate placement.** The map's `render->displayobject` edge note covers the dependency, but should explicitly state that the `prepare_display_object_render` _function_ lives in `flighthq-displayobject` (vs TS `@flighthq/render`).

The conformance script (`scripts/rust-conformance.ts`, line ~502) intentionally does **not** flag Rust-only "extra" functions, so this entire 61-function surface is invisible to the gate — exactly the drift this human review is meant to catch. Note also: `npm run rust:conformance` currently aborts in this worktree (`npx tsx ./scripts/api.ts --json` fails — looks like a build/env issue, not specific to displayobject), so the gate is not running here at all.
