# TS↔Rust Alignment: @flighthq/effects-gl

**Verdict:** Strongly aligned — all 43 `apply*EffectToGl` recipes, both registry functions, and the 5 pipeline functions map 1:1 with correct snake_case/full-type-word names; the only gaps are layering relocations (effects-gl re-implements render-gl fullscreen primitives, and hosts its own GL effect types instead of the `types` header) that are not recorded in the divergence map.

`effects-gl ↔ flighthq-effects-gl` is the documented identity map (conformance.md line 30: "`effects-gl`/`effects-wgpu` map by identical name"). The crate is in the GPU-shader exclusion set (`scripts/rust-conformance.ts` line 169), so its conformance is visual (parity matrix at the `gl` cell), not name-match — this review covers the name/file/layering alignment the script does not score. The structural gate could not be run to completion here: `scripts/rust-conformance.ts` spawns `api.ts --json`, which is OOM-killed (status 137) in this environment; findings below are from direct symbol comparison plus the committed divergence map.

## In sync

- **All 43 effect recipes** (`apply_*_effect_to_gl`) present, named correctly, full type words preserved, themed-module grouping aside.
- **All 43 default runner constants** present (`DEFAULT_GL_*_EFFECT_RUNNER`), tracking TS `defaultGl*EffectRunner`.
- **Registry** (`get_gl_render_effect_runner`, `register_gl_render_effect`) — names and `render_effect_registry.rs` basename match; sentinel/`&mut` conventions carried.
- **Pipeline** (`begin_/create_/destroy_/end_gl_render_effect_pipeline`, `set_gl_render_effect_velocity_texture`) — all 5 present; `destroy_*` teardown verb correctly preserved (frees GPU targets); `render_effect_pipeline.rs` basename matches `glRenderEffectPipeline.ts`.
- **Program cache** `get_gl_effect_program` ↔ `getGlEffectProgram`, file basename matches.
- **Dependencies** — Rust deps (`effects`, `filters-gl`, `render-gl`, `types`) match TS real value-import edges. The TS `@flighthq/filters` package.json entry is declared-but-unused (no value import) and correctly absent from Rust; `@flighthq/geometry` (`createMatrix`) is a `FOLDABLE_DEPS` mechanical translation and correctly not a Rust crate dep.
- **Recorded divergence honored:** the bloom-Gaussian-blur fix (conformance.md line 61) — Rust now depends on `flighthq-filters-gl` and calls `apply_gaussian_blur_filter_to_gl` rather than an inlined kernel.

### Suggested divergence-map additions

1. **effects-gl re-implements render-gl fullscreen primitives.** Record that `compileGlFullscreenProgram` / `clearGlRenderTarget` / `GlFullscreenProgram` were not ported into `flighthq-render-gl`, so `flighthq-effects-gl` defines local `compile_gl_effect_program` / `clear_gl_effect_target` / `GlEffectProgram` / `draw_gl_effect_fullscreen_pass` / `EFFECT_VERTEX_SRC`. Tie this to the planned render-gl split (line 75): once render-gl exposes the fullscreen-program primitives, effects-gl should consume them and these locals should be removed.
2. **GL effect types live in the backend crate, not the Rust types header.** TS keeps `GlRenderEffectContext` / `GlRenderEffectRunner` / `GlRenderEffectPipeline` in `@flighthq/types`; Rust defines them in `flighthq-effects-gl`. If this is the intended port-wide pattern for GPU backend types, record it as a blanket exception rather than silent per-crate drift.
