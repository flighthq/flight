# Depth Review: @flighthq/render-wgpu

**Domain**

WebGPU renderer **core** for the Flight SDK: the subject-agnostic GPU plumbing that sits under the per-subject leaf renderers. Per the render-layering decision (docs `conformance.md#the-render-layering`), `render-wgpu` owns "render state, targets, shaders, draw, fullscreen/surface passes" — device/adapter/context bring-up, the bitmap quad pipeline, blend/stencil pipeline state, a per-frame uniform ring buffer, texture upload/caching, offscreen render targets + pool, a readback/capture path to a CPU `Surface`, and the registration seams (material renderer, shader binding) that the leaf renderers (`displayobject-wgpu`, `scene-wgpu`, `filters-wgpu`, `effects-wgpu`) build on. The actual per-primitive drawing (shapes, text, sprite batches, particles, 3D scene) lives in those sibling packages, NOT here — so this review judges the depth of the _backend core_, not of a full WebGPU 2D renderer.

**Verdict: solid** — completeness 72/100.

For its true domain (a WebGPU backend core that other packages draw through), this is a real, working, thoughtfully-built foundation: it boots a device, manages frame command encoding, runs a premultiplied bitmap pipeline with a dynamic-offset uniform ring buffer, supports stencil masking, offscreen targets with a pool, HDR-format targets, and a genuinely careful headless capture path. It is well past "stub" and past "partial." It falls short of "authoritative" because several pieces a mature GPU core is expected to provide are stubbed or absent: blend modes are mostly unimplemented (11 of 15 fall back to Normal), there is no MSAA, no mipmaps/anisotropy, no scissor application despite the runtime carrying scissor state, no GPU-error/device-loss handling, and several runtime capabilities (particles, sprite-batch, clip contours) are allocated here but have no exported entry points in this package.

## Present capabilities

- **Device / context bring-up** — `createWgpuRenderState` (async adapter→device→context, `powerPreference`, preferred-format selection, `premultiplied` alpha, `RENDER_ATTACHMENT | COPY_SRC` usage), `isWgpuSupported`, `createWgpuCanvasElement` (pixel-ratio sizing), `destroyWgpuRenderState` (destroys uniform/particle/depth/sprite-batch buffers, correctly leaves the shared device, pipelines, bind groups, samplers to GC), runtime accessors (`createWgpuRenderStateRuntime`, `getWgpuRenderStateRuntime`).
- **Frame lifecycle** — `renderWgpuBackground` (open command encoder, clear color from background, depth-stencil ensure, capture-texture redirect), `submitWgpuRenderPass` (upload used ring slice via `writeBuffer`, encode capture copy, submit, free retired buffers). Clean encoder/pass state machine.
- **Bitmap pipeline + shaders** — a complete WGSL bitmap shader (quad expanded in the vertex stage from corner uniforms, premultiplied output, unpremultiply→colorTransform→repremultiply in the fragment stage, alpha discard) plus a stencil-write mask shader. Pipeline caching keyed on `blendMode-stencilMode-format`, pipeline warming (`warmWgpuPipelines`), bind-group/pipeline-layout builders, `getWgpuPipeline` / `getActiveWgpuPipeline`.
- **Uniform ring buffer** — 4096-slot, device-alignment-aware dynamic-offset ring (`writeWgpuQuadUniforms`, `writeWgpuMatrixOnlyUniforms`), with a documented 128-byte slot layout and `setWgpuMatrixFromTransform` (2D affine → clip-space mat3x3 with the WGSL vec3 column padding handled).
- **Draw primitives** — `drawWgpuQuad`, `drawWgpuQuadWithTransform`, `drawWgpuRenderTargetResult` (composes node + cache-offset transform, Y-flips the target). These are the low-level quad emitters the leaf renderers call.
- **Texture handling** — `bindWgpuTexture` (type-aware sizing for Canvas/Image/Video/ImageBitmap/OffscreenCanvas, premultiplied upload, WeakMap cache), `createWgpuTextureEntry`, `updateWgpuTextureEntry`, linear + nearest samplers selected by `allowSmoothing`.
- **Offscreen render targets** — `createWgpuRenderTarget` (optional `rgba16float` HDR format, own depth-stencil), `begin/endWgpuRenderTarget` (a proper push/pop stack saving view, viewport, transform, color format, mask state), `resizeWgpuRenderTarget`, `destroyWgpuRenderTarget`, and an `acquire`/`release` pool (`wgpuRenderTargetPool`) with format/size matching.
- **Stencil masking** — three stencil modes (normal/masked/maskwrite) wired into pipeline state and `setStencilReference` on draw; mask depth tracked on the runtime.
- **Blend mode seam** — `applyWgpuBlendMode`, `enableWgpuBlendModeSupport` (installs the hook tree-shakably), Add + Normal/Layer implemented as real `GPUBlendState`s.
- **Headless capture / readback** — `enableWgpuFrameCapture`, `acquireWgpuFrameCaptureTexture`, `encodeWgpuFrameCapture`, `createSurfaceFromWgpuRenderState`. This is the most polished part: it correctly handles software-adapter swapchain-readback-returns-zero, same-frame copy timing, 256-byte row alignment, and BGRA↔RGBA normalization, with a well-reasoned premultiplied-stays-premultiplied decision documented inline.
- **Registration seams** — `registerWgpuMaterialRenderer` / `getWgpuMaterialRenderer` / `resolveWgpuMaterialRenderer` (default-material fallback, unresolved = no-op), and `setWgpuShader` / `resolveWgpuShader` / `getWgpuShader` (per-node shader binding via a lazily-installed resolver). Both are nullable-hook style and tree-shake until used.
- **Test mocks** — `installWgpuMock`, `createWgpuRenderStateForTest`, and a colocated `.test.ts` per source file.

## Gaps vs an authoritative WebGPU-renderer-core library

- **Blend modes are mostly stubbed.** Of 15 `BlendMode` entries only `Add`, `Normal`, and `Layer` produce a real `GPUBlendState`; `Multiply`, `Screen`, `Darken`, `Lighten`, `Overlay`, `Hardlight`, `Difference`, `Invert`, `Subtract`, `Erase`, `Shader` are all `null` and silently fall back to Normal. `Subtract` (`reverse-subtract`/`subtract` op) and `Erase`/`Multiply`/`Screen` (`zero`/`one-minus-src`/`dst` factors) are expressible with fixed-function blend alone and are simply not done. The separable Photoshop modes (Overlay/Hardlight/Difference/etc.) need a fragment read-back or dual-source path — an authoritative core would at least document that and implement the fixed-function-expressible ones. This is missing-by-omission, not by-design.
- **No MSAA.** Explicitly deferred (the pool comment calls it "intentionally out of scope" pending multisample pipeline variants). For an authoritative 2D GPU core, antialiased edges on rotated/scaled geometry are table stakes; the single bitmap-quad pipeline is `sampleCount: 1` only.
- **No scissor / clip-rect application.** The runtime carries `scissorStack` and `currentScissorRect` and they are reset everywhere, but nothing in this package ever calls `setScissorRect`. Rect clipping (a standard renderer-core capability, and the cheap alternative to stencil) is absent at the API level here.
- **No mipmaps or anisotropic filtering.** Textures upload a single mip level; samplers are linear/nearest with no mipmap filter or `maxAnisotropy`. Minified/downscaled bitmaps will shimmer/alias — a quality gap an authoritative core addresses.
- **No GPU error / device-loss handling.** No `pushErrorScope`/`popErrorScope`, no `device.lost` handling, no out-of-memory guard on texture/target creation. `requestAdapter`/`requestDevice`/context failures `throw`, which is reasonable for misuse, but a runtime device loss mid-session is unhandled.
- **No required-features / limits negotiation.** `requestDevice()` is called with no `requiredFeatures`/`requiredLimits`. HDR (`rgba16float`) render targets are created without confirming `float32-filterable`/blendable support; large-canvas paths don't check `maxTextureDimension2D`. An authoritative core negotiates and degrades.
- **Capabilities allocated but not exported here.** The runtime reserves `particleInstanceBuffer`, the whole `spriteBatch*` pool, `clipContourPipelines`/`clipContourStack`/`clipForms`, and `shapeMeshPipelines`, but this package exports no functions to drive them — they belong to `displayobject-wgpu`. That is the correct layering, but it means the _core's own_ API surface for instanced/batched draw is empty; the only draw primitives it ships are single 6-vertex quads. A "draw, batched" capability that an authoritative core would own is pushed entirely to the leaf.
- **No depth ordering.** Depth writes are disabled (`depthCompare: 'always'`), so the core is painter's-order only — fine for 2D, but there is no z-based path even though depth-stencil textures are allocated.
- **`drawWgpuQuad` vs `drawWgpuQuadWithTransform` duplicate the full 32-float uniform write** inline rather than both routing through `writeWgpu*Uniforms`; the transform variant re-implements `setWgpuMatrixFromTransform` and the color-transform packing by hand. A robust core would have one uniform-writing path.

## Naming / API-shape notes

- Naming is strong and consistent: `Wgpu` infix on every export, full unabbreviated type words (`createWgpuRenderTarget`, `acquireWgpuFrameCaptureTexture`), correct teardown verbs (`destroy*` for GPU resources, `dispose*` absent because nothing here is detach-to-GC), and proper `acquire*`/`release*` pool brackets. Allocation verbs (`create*`, `acquire*`) vs in-place (`resize*`, `update*`, `write*`) follow the house rules.
- `enableWgpuBlendModeSupport` / `enableWgpuFrameCapture` / `setWgpuShader` are good tree-shakable opt-in hooks rather than always-on behavior — matches the "gate optional subsystems with nullable hooks" rule.
- Several public signatures leak inline structural transform literals (`{ a; b; c; d; tx; ty }`) instead of `Readonly<Transform2DLike>` — e.g. `drawWgpuQuadWithTransform`, `writeWgpuMatrixOnlyUniforms`, `setWgpuMatrixFromTransform`, `writeWgpuQuadUniforms`. These should use a named `*Like` type from `@flighthq/types` for grepability and symmetry with the rest of the SDK.
- `drawWgpuRenderTargetResult` takes a `renderProxy as never` cast internally and an ad-hoc `{ alpha; material; transform2D }` literal — a sign the render-proxy contract for the target-result path isn't fully typed.
- `index.ts` re-exports `WgpuRenderTarget` / `WgpuRenderTargetPool` types from `@flighthq/types`, keeping the header layer authoritative — good.

## Recommendation

Treat this as a **solid core that is one focused pass away from authoritative for its layer**. Prioritize, in order: (1) implement every fixed-function-expressible `BlendMode` (`Subtract`, `Erase`, `Multiply`, `Screen`, `Darken`, `Lighten`) and explicitly document/route the separable modes that need a read-back path, so the blend table stops silently degrading to Normal; (2) wire `setScissorRect` to the existing `scissorStack`/`currentScissorRect` so rect clipping actually works; (3) add device-loss + `pushErrorScope` handling and `requestDevice` feature/limit negotiation (especially gating `rgba16float` targets); (4) add MSAA pipeline variants and mipmap/anisotropy sampler options for quality parity with a mature GPU core. Collapse the two quad-draw paths onto the single `writeWgpu*Uniforms` writer to kill the duplicated uniform-packing. Replace the inline `{a,b,c,d,tx,ty}` literals with a named `Transform2DLike`. None of this changes the package's correct scope — it is the backend core, and the per-subject draw work rightly lives in `displayobject-wgpu`/`scene-wgpu`; closing these gaps makes the core itself exhaustive rather than merely functional.
