# Depth Review: @flighthq/render

**Domain**: Backend-agnostic render core — renderer registration, render state, the scene-graph→render-node update/prepare pipeline, the render-cache seam, render-target geometry math, and the 3D scene-prepare (cull/light-pack) pass. The concrete per-backend draw loops (Canvas/DOM/GL/WGPU) and batching live in sibling packages by design; this package owns the contracts and the CPU-side preparation they all consume.

**Verdict**: solid — 74/100

This is not a "renderer" library in the conventional sense (it draws nothing). Judged for what it actually is — the shared render-core/abstraction layer that every backend plugs into — it is a deliberate, well-factored, complete-feeling core for the 2D path, with a thinner-but-coherent 3D prepare pass. It falls short of "authoritative" mainly because two pieces a render-core layer is normally expected to own (a backend-agnostic draw/submit driver and a frame-graph/render-pass abstraction) are absent or only hinted at.

## Present capabilities

The exported surface is small but the responsibilities are real:

- **Renderer registry (`renderer.ts`)** — `registerRenderer(state, kind, renderer)` over a `Map<Kind, Renderer>`, last-write-wins (matches the documented override convention), with a `rendererMapId` generation counter so proxies lazily rebind their renderer/data when registrations change. `copyRenderersFromRenderState` / `copyAllRenderersFromRenderState` for cloning a registry plus clip hooks across states. `noopRendererData` as the canonical empty `createData`. The `Renderer` contract (`createData` / `destroyData` / `submit` / optional `format: BatchFormat`) cleanly separates per-node allocation, GPU teardown (`destroy*` semantics), and batch-flush hints.
- **Render state (`renderState.ts`)** — `createRenderState` / `createRenderStateRuntime` / `getRenderStateRuntime`. The entity/runtime split is honest: the public `RenderState` is a thin data record (smoothing, background color triple, pixel ratio, clip depth, `renderAlpha`/`renderBlendMode`/`renderTransform2D` roots, `roundPixels`, `sceneGraphSyncPolicy`); the runtime carries the frame counter, proxy WeakMaps, registry, and a reusable `tempStack`. Backend runtimes extend the base runtime, so the frame id / proxy maps / registry are genuinely shared across all four backends.
- **The 2D prepare pipeline (`renderProxy.ts`)** — the heart of the package. `prepareDisplayObjectRender` drives a single dirty-checked, stack-based pre-order `walkNode`; one `RenderProxy2D` allocator serves both display objects and sprites (the documented type collapse); per-node visitor `updateRenderProxy2D` composes appearance + transform + material + clip-depth and runs the adapt hook. Correct dirty propagation via revision ids (`lastLocalTransformId` / `lastAppearanceId` / `lastLocalContentId`) and per-frame `transformFrameId`/`appearanceFrameId` parent-dirty checks; `sceneGraphSyncPolicy` lets a caller force full refresh. Visibility culling (`isRenderProxyVisible` — alpha, visible flag, zeroed-scale). Teardown counterpart `disposeDisplayObjectRender` / `disposeRenderProxy` cascading to `destroyData`.
- **Update steps as free functions** — `updateRenderProxyAppearance` (alpha/visible/blend inheritance from parent or state roots, with early-outs), `updateRenderProxy2DTransform` (parent×local matrix compose, alias-safe, dirty-gated), `updateRenderProxyMaterial` (non-inheriting material + materialData resolve), `updateNodeClip` (clip nesting depth). Each is independently testable and tree-shakable.
- **Render-proxy adapter seam (`renderProxyAdapter.ts`)** — a general per-source `adapt` hook that can swap a node's renderer/kind and control child traversal during prepare; lazily installs itself into the walk. This is the extension point that powers caching and could power LOD/filter substitution.
- **Render cache seam (`renderCache.ts`)** — `createRenderCache` (backend-agnostic handle owning no resource), `createRenderCacheAdapter`/`useRenderCache` (substitutes the cache renderer + folds in placement transform during prepare), `registerRenderCacheRenderer`, `getRenderProxyCache`, optional `onPrepare` signal for lazy refresh. This is the cacheAsBitmap analog done as data + a seam rather than a stateful flag.
- **Render-target geometry math (`renderTarget.ts`)** — `computeDisplayObjectRenderTargetTransform`, `computeRenderCacheTransform`, `computeRenderTargetSize` (ceil + padding + min clamp). Pure, alias-safe scratch-based transform helpers for offscreen capture.
- **3D scene prepare (`sceneRender.ts`)** — `prepareSceneRender` walks a `SceneNode` graph, composes view-projection (perspective/orthographic), builds a frustum, frustum-culls meshes against world-space AABBs, and packs directional+ambient lights into an std140 `SceneLightBlock` (sRGB→linear premultiplied). Returns a reused per-state `SceneRenderList`. A real, backend-agnostic 3D prepare pass — modest in feature count but correct.
- **Misc resolved-value helpers** — `setRenderStateBackgroundColor` (packs RGBA int → number/float[]/hex-string triple), `computeTextFormatFontString` (CSS font string), `RenderProxyVisitor`/`walkNode` exposed for custom passes.
- **Testing** — 134 `it` cases across 12 colocated test files (`renderProxy.test.ts` alone has 45), including dirty-tracking, alias cases, and adapter/cache behavior. Coverage is genuinely thorough for the logic present.

## Gaps vs an authoritative render-core library

A mature, backend-agnostic render-core/abstraction layer is normally expected to provide more than preparation:

- **No backend-agnostic draw/submit driver.** The `Renderer` contract has `submit(state, node)` and a `format?: BatchFormat` flush hint, and the doc map says the package owns the "render queue" and "update pipeline" — but there is no exported function here that walks prepared proxies and calls `submit` / flushes batches on format change. The actual frame draw loop lives in each backend (`renderCanvasDisplayObject`, `renderGlDisplayObject`, etc.). A render core would normally own a `submitRenderProxy`/`drawRenderList`/`flushRenderBatch` driver so the per-backend code is only the leaf draw, not the whole walk. This is the single biggest depth gap. (Partly missing-by-design — the immediate-mode backends each drive their own walk — but the `submit`/`format` contract implies a shared driver that does not exist.)
- **No explicit render-queue / sort order.** `RenderStateRuntime` has no draw-order list, no z/depth sort, no opaque-vs-transparent partition for the 2D path. Order is implicit in scene-graph traversal. An authoritative core usually exposes a queue abstraction (sortable, layerable). The `tempStack` is scratch, not a retained queue.
- **No render-pass / frame-graph abstraction.** There is render-target _math_ but no `RenderPass`/`RenderGraph`/multi-target dependency description in the core. Each backend reinvents begin/end target (`beginCanvasRenderTarget`, `beginGlRenderTarget`, `beginWgpuRenderTarget`). A shared, backend-agnostic pass/attachment descriptor would be the canonical home for that. Reasonably missing-by-design given the immediate-mode backends, but it is a recognized render-core feature.
- **Blend mode is a single field, not a stack.** `renderBlendMode` is a root override; clip has a depth counter, but blend/alpha inheritance has no save/restore stack in the core (left to backends). Fine, but worth naming.
- **2D culling is visibility-only.** No viewport/scissor frustum cull for the 2D path (the 3D path has frustum culling). Bounds-based offscreen rejection of 2D nodes is absent from the core.
- **3D prepare is narrow.** Exactly one directional + one ambient light (`LIGHT_BLOCK_FLOATS = 12`), no point/spot lights, no shadow-caster collection, no draw-order/material sort, no instancing batching, no LOD. Adequate as a seam; far from an authoritative 3D render core. The doc map flags `scene` as deliberately under-built, so this is mostly missing-by-design.
- **No stats/diagnostics.** No draw-call/triangle/batch counters, no per-frame timing hooks, no debug capture seam in the core. Mature render layers usually expose these.
- **`computeTextFormatFontString` is an odd resident.** A CSS font-string builder reads like it belongs in `text` or a backend, not the render core. Minor scope leak.

## Naming / API-shape notes

- Naming is consistent and self-identifying: `prepareDisplayObjectRender` / `prepareSceneRender`, `updateRenderProxy*`, `computeRenderTarget*`, the `create*`/`get*`/`dispose*`/`destroyData` verb discipline is correct (dispose detaches/GC; destroyData frees GPU now). Full type words are spelled out.
- `RenderProxy` is the chosen name for the per-node render record (formerly "render node"); comments and the `RenderProxyVisitor`/`RenderProxyAdapter` family keep it coherent.
- `updateDisplayObjectRenderTransform` is a thin alias that just calls `updateRenderProxy2DTransform` — two public names for one behavior. One could go.
- `beginRenderProxyUpdate` is an exported empty no-op (`{}`) — a reserved hook with no body. Either give it a purpose or drop it from the public surface; an exported no-op is API debt.
- `installRenderAdaptHook` plus the module-level `_adaptHook` is a single global hook slot mutated at runtime; functional but slightly at odds with the "no shared top-level mutable state" guidance, and it means only one adapt hook can be installed process-wide. Worth noting even though it is internal-ish.
- Several backend-facing draw structs are typed inline (`{ a; b; c; d; tx; ty }`) in the backend packages rather than via a shared `RenderProxyTransform` — not this package's surface, but the seam originates here.

## Recommendation

Treat this as a **solid render-core that is one major piece short of authoritative**: it owns preparation, registration, the proxy/update pipeline, caching, and target math cleanly and with strong tests, but it does not own the backend-agnostic _draw driver_ its own `Renderer.submit` / `BatchFormat` contract implies. The highest-value depth additions, in order:

1. Add a shared submit/draw driver (`drawRenderProxy`/`submitRenderList`/`flushRenderBatch`) that walks prepared proxies and dispatches `submit` with automatic batch-flush on `format` change, so backends contribute leaf draws rather than re-implementing the walk. This is what would move it toward authoritative and would deduplicate the four backends.
2. Introduce a backend-agnostic render-pass / target-attachment descriptor to host the begin/end-target pattern the backends each reinvent (pairs naturally with the existing `renderTarget.ts` math).
3. Add a retained, sortable render-queue concept (even a thin layer/order list) for the 2D path.
4. Housekeeping: remove the `beginRenderProxyUpdate` no-op and the `updateDisplayObjectRenderTransform` alias, and relocate `computeTextFormatFontString`.
5. Leave the 3D prepare as-is for now (intentionally minimal per the `scene` roadmap), but note point/spot lights and a draw sort as the eventual completeness bar.
