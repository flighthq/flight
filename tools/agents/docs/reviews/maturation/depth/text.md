# Maturation Roadmap: @flighthq/text

**Current verdict:** solid — 72/100. An architecturally-correct text _display-object_ layer (`TextLabel`/`RichText`/`NativeText` entities) that is let down only by an incomplete mutator/accessor surface; the hard text-engine work is correctly delegated to `@flighthq/textlayout`, `@flighthq/textinput`, and `@flighthq/textshaper`.

Scope note carried from the depth review: this package owns the _entities and their field-level get/set surface_, not the text engine. Every tier below stays inside that charter. Engine-side maturation (shaping, bidi, line-break tables) belongs in the sibling packages and is only referenced here where a `text` API depends on it. The single biggest lever is closing the setter/getter asymmetry the review flagged: string/format-range/scroll/autoSize have first-class mutators while ~14 `RichTextData` fields have none.

## Bronze

Minimum viable: make the field surface internally consistent so a user can drive every stored field through a named function with invalidation, and read back what they wrote. This is the 20% (the missing `set*`/`get*` pairs) that delivers 80% of the "authoritative field" value, all field-local, no cross-package work.

- **Complete the `RichText` field mutators** (each diffs, bumps `invalidateNodeLocalContent`, and `invalidateNodeLocalBounds` only when it can change the box): `setRichTextBackground`, `setRichTextBackgroundColor`, `setRichTextBorder`, `setRichTextBorderColor`, `setRichTextTextColor`, `setRichTextWordWrap` (bounds), `setRichTextMultiline` (bounds), `setRichTextSelectable`, `setRichTextMaxChars`, `setRichTextMouseWheelEnabled`, `setRichTextCondenseWhite`, `setRichTextDefaultTextFormat`, `setRichTextStyleSheet`, `setRichTextHtml` (sets `htmlText`, clears/regenerates content), `setRichTextWidth`, `setRichTextHeight`.
- **`setTextLabelWidth`** — present in source but confirm it is exported and tested alongside the others (the review lists it among the established setters; verify it is in the barrel).
- **Read accessors restoring OpenFL symmetry** (sentinel-returning, no allocation where avoidable): `getTextLabelFormat`, `getRichTextHtml`, `getRichTextDefaultTextFormat`, `getRichTextLength`, `getNativeTextString`/`getTextLabelString`/`getRichTextString` convenience readers (or document `source.data.text` as canonical — pick one and be consistent).
- **Effective-format-at-index reader:** `getRichTextFormatRangeAt(out, source, index)` returning the merged effective `TextFormat` at a character index (composes `defaultTextFormat` + overlapping `textFormatRanges` via `mergeTextFormat`, already in `textlayout`). This closes the OpenFL `getTextFormat(beginIndex, endIndex)` read half.
- **Retire the `internal.ts` cast.** Replace the `RichTextDataInternal` `scrollH`/`scrollV` write-cast with the runtime-slot convention the map prefers (store scroll on `RichTextRuntime`, or make the fields plainly mutable on `RichTextData` and drop the `readonly`). The current `internal.ts` is the legacy pattern the codebase map explicitly says not to extend.
- **Colocated tests** for every new `set*`/`get*` (the `exports:check` gate requires one test per export); each setter test asserts the diff-skip path and the invalidation bump, mirroring the existing `richText.test.ts` style.

## Silver

Competitive and solid: matches what a well-regarded `TextField`-class library offers — the convenience operations developers actually reach for, a clean signals story, and entity-level conveniences that wrap the `textlayout` queries so a `text` user never has to import the engine for everyday tasks.

- **Append/insert/replace text operations** (non-editing, field-level — editing variants stay in `textinput`): `appendRichTextString`, `appendRichTextHtml`, `insertRichTextString(source, index, value)`, `replaceRichTextString(source, beginIndex, endIndex, value)`, `appendTextLabelString`. These re-index `textFormatRanges` correctly on insert/delete (the load-bearing detail — splice ranges, do not just concatenate text).
- **`setRichTextFormatRange` symmetry helpers:** `removeRichTextFormatRangesIn(source, begin, end)`, `getRichTextFormatRangeCount`, `getRichTextFormatRangeByIndex(out, source, i)` so the range list is fully introspectable and editable, not just push/clear.
- **Entity-level metric conveniences** (thin wrappers over `textlayout`, ensuring layout first, so a `text`-only import covers the common reads): `getRichTextLineCount(source)`, `getRichTextTextWidth(source)`/`getRichTextTextHeight(source)`, `getRichTextLineMetrics(out, source, lineIndex)`, `getRichTextCharIndexAtPoint(source, x, y)`, `getRichTextMaxScrollH(source)`/`getRichTextMaxScrollV(source)`/`getRichTextBottomScrollV(source)`. Each `ensureTextLayout` then delegates to the existing `textlayout` free function.
- **Signals group, opt-in via `enable*`:** `enableTextFieldSignals(source)` allocating a signal group on the runtime with `onTextFieldChange`, `onTextFieldScroll`, and `onTextFieldLink` (link-click, paired with `getRichTextLinkAtPoint`). Define the payload structs (`TextFieldChangeEvent`, `TextFieldScrollEvent`, `TextFieldLinkEvent`) in `@flighthq/types` first; mutators emit only when the group is enabled (zero cost otherwise), matching `enableDisplayObjectSignals`.
- **`NativeText` parity with the entity model:** `getNativeTextStyle`, `getNativeTextMeasuredWidth`/`getNativeTextMeasuredHeight` readers, and a `setNativeTextStyle` partial-merge variant (`patchNativeTextStyle`) so common single-property style changes do not require rebuilding the whole `NativeTextStyle`.
- **HTML round-trip seam via a `-formats` neighbor package:** introduce `@flighthq/text-formats` for HTML/`TextField`-HTML parse and serialize (`parseRichTextHtml` → `{ text, textFormatRanges }`, `serializeRichTextHtml`). Keeps the regex/parse weight out of the core `text` bundle (the map's `-formats` importer/parser pattern). `setRichTextHtml` in core calls the registered parser through a seam so the dependency stays optional and tree-shakable.
- **`condenseWhite` / `styleSheet` actually applied**, not just stored: wire `condenseWhite` and `RichTextStyleSheet` into the content-build path (`computeRichTextContent` already takes the data; ensure the parser/seam honors them). Today these are stored fields with no behavioral effect.

## Gold

Authoritative / AAA: nothing a `TextField` expert finds missing, full edge-case and error discipline, and 1:1 Rust-port parity. The frontier here is breadth of OpenFL `TextField` semantics plus the cross-backend and Rust conformance guarantees.

- **Full OpenFL `TextField` semantic coverage at the entity layer:** `restrict` (allowed-character set, stored on data and enforced through the `textinput` seam), `displayAsPassword` promotion path on a static field, `embedFonts`/`antiAliasType`/`sharpness`/`thickness`/`gridFitType` as stored descriptor fields (consumed by renderers/shaper), `useRichTextClipboard`/`alwaysShowSelection` flags, and `replaceSelectedText` (delegating to `textinput` when editable).
- **Image/inline-object support in HTML** (`<img>` tags → inline placeholder boxes in the layout stream): requires a `text`-side `RichTextInlineObject` type in `@flighthq/types` and a layout-side inline-box hook. Surface to the user as a design-decision item (it touches `textlayout`).
- **Tab-stop and list/bullet rendering completeness** verified end-to-end (the `TextFormat` fields exist; ensure `tabStops`, `bullet`, `indent`, `blockIndent`, `leftMargin`/`rightMargin` produce correct layout across all raster backends — gated by `test:functional:parity`).
- **Performance:** content-build and range-splice operations are O(ranges) not O(text); large-document scroll (`scrollV` on 10k+ lines) uses the cached line-height prefix sums rather than re-summing per call; `appendRichTextString` does not re-layout the whole field when not wrapped. Add a benchmark scene.
- **Exhaustive edge-case + error handling:** empty string, all-whitespace, surrogate-pair and grapheme-cluster indices (char-index APIs must not split a surrogate pair), zero/negative width under `wordWrap`, `maxChars` truncation, `scrollV`/`scrollH` clamping at exact boundaries, format ranges that overlap or exceed text length (clamp, do not throw — throw only on genuine misuse like a non-finite index). Tests for each.
- **Functional + parity test coverage:** functional scenes for multi-format `RichText`, autoSize anchors (`left`/`right`/`center`), word-wrap reflow, scroll, links, and `NativeText`, each with a screenshot baseline and cross-backend parity assertion (`functional-test` skill).
- **Full Rust-port mirror `flighthq-text`:** every `create_*`/`set_*`/`get_*`/`compute_*_local_bounds_rectangle`/`ensure_text_layout` ported (snake_case, `&mut`/out-params, `Option` sentinels), conforming to the TS package and recorded in the conformance map. Pairs with `flighthq-textlayout` + the `textshaper` (rustybuzz) stack so non-Canvas backends render `RichText` identically. Assertion-ported unit tests plus parity-matrix cells (`rust:skia ~ ts:canvas`).
- **API + docs:** `npm run api text` shows a complete, symmetric `create/get/set/compute/ensure` surface per entity; a package-level usage doc covering the static-vs-editable split, the autoSize anchor model, and the HTML/styleSheet seam.

## Sequencing & effort

Recommended order, with dependencies and items to surface:

1. **Bronze first, in one pass (small, fully in-package, ~1–2 days).** No type changes are needed for the setters — every field already exists on `RichTextData`/`NativeTextData` in `@flighthq/types`. Do the `internal.ts` retirement _with_ the scroll work so the `readonly scrollH/scrollV` decision (runtime slot vs. mutable data field) is made once. This is the highest value-per-effort and unblocks clean accessors. Run `npm run exports:check` and `npm run order:fix` after — every new export needs a colocated test and alphabetized placement.

2. **Silver append/insert/range helpers (in-package, but design-sensitive, ~2–3 days).** The load-bearing risk is `textFormatRange` re-indexing on insert/delete; get that right and tested before the signals/metrics conveniences. The metric conveniences are trivial wrappers once the pattern is set.

3. **Silver signals + types (touches `@flighthq/types`, ~1 day).** Define `TextFieldChangeEvent`/`TextFieldScrollEvent`/`TextFieldLinkEvent` in the header layer first, then `enableTextFieldSignals` in this package. Coordinate the event-name vocabulary with the `enableDisplayObjectSignals` precedent.

4. **Silver `@flighthq/text-formats` neighbor package (cross-package, design decision — surface to user).** Spinning up a new package and the HTML parse/serialize seam is the biggest Silver item and the right home for HTML weight. **Decision to surface:** confirm the `-formats` split (vs. folding HTML into `textlayout`) and whether `styleSheet`/`condenseWhite` semantics live in the parser or the content-build path. Depends on `RichTextStyleSheet` (already typed) and `computeRichTextContent` (in `textlayout`).

5. **Gold OpenFL-semantic fields + inline objects (cross-package, ~1–2 weeks).** `restrict`/`maxChars`/`replaceSelectedText` couple to `@flighthq/textinput`; inline `<img>`/tab/bullet rendering couples to `@flighthq/textlayout` and the renderer packages. **Surface as design decisions:** inline-object layout model and embed-font/anti-alias descriptor fields both cross package boundaries and should be agreed before implementation.

6. **Gold performance, edge-case hardening, functional/parity tests, then the Rust mirror (large, last).** The Rust port should follow only once the TS surface is final, since it conforms to TS; doing it earlier means re-porting. Rust depends on `flighthq-textlayout` + `flighthq-textshaper` being ready for any non-Canvas rendering.

**Cross-package / decision items to raise up front:** (a) the `internal.ts` scroll-field convention (runtime slot vs. drop `readonly`); (b) creation of `@flighthq/text-formats` and where HTML/styleSheet semantics live; (c) the inline-object (`<img>`) layout model spanning `text`/`textlayout`/renderers; (d) embed-font / anti-alias descriptor fields that only renderers and the shaper consume; (e) signal event-name vocabulary alignment with the display-object signals group.
