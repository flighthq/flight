# Maturation Roadmap: @flighthq/effects

**Current verdict:** solid — completeness 72/100 ("solid breadth, thin depth": the effect _catalog_ is near-authoritative, but only 1 of 43 descriptors carries real substrate-agnostic recipe math, and there is no introspection, no shared color-science primitives, and no composition layer).

This package is deliberately a thin descriptor layer: it owns the _catalog of effect intents_ (`RenderEffect` plain-data descriptors) plus the substrate-agnostic _recipe math_ both GPU backends derive parameters from. Per-backend recipes (`gl*`/`wgpu*`/`canvas*` runners), pipeline/dispatch, and registration correctly live in `@flighthq/effects-gl|wgpu|canvas` and `@flighthq/render` — that boundary is right and these tiers do **not** move it. Maturation here means (1) building out the recipe-math half of the charter that `computeBloomBlurRadius` is currently the lone instance of, (2) deepening under-specified descriptors, and (3) adding data-driven introspection/metadata so the pipeline can reason about an intent list without backend knowledge. All shared types land in `@flighthq/types` first; the Rust `flighthq-effects` crate (already at 1:1 parity with the current thin shape) mirrors each addition.

## Bronze

The minimum that makes this a _library_ rather than a catalog of factories: shared math the backends can stop duplicating, a uniform base contract, and the introspection the pipeline needs to validate an intent list.

- **Uniform base-contract fields.** Promote the ad-hoc, per-effect blend controls to `RenderEffect` in `@flighthq/types`: add optional `intensity?: number` (effect strength / dry-wet mix, default 1) and `enabled?: boolean` (default true; a disabled intent is skipped by the pipeline without removing it from the list). Today some effects have `intensity`, most do not — make it canonical and let per-effect fields override only where the semantics differ.
- **Required-render-input as data, not comments.** Add `RenderEffectInput` string-kind union to `@flighthq/types` (`'Hdr' | 'Depth' | 'Motion' | 'Temporal'`) and a `getRenderEffectInputs(effect: Readonly<RenderEffect>): readonly RenderEffectInput[]` free function in this package backed by a static per-`kind` table. This promotes the `[HDR]`/`[DEPTH]`/`[MOTION]`/`[TEMPORAL]` comment tags into queryable data so the pipeline can validate (does the scene target carry depth/velocity/history this list needs?) without backend knowledge.
- **Gaussian / blur recipe math** (the highest-value math gap). Add free functions, following the `computeBloomBlurRadius` "one source of truth" pattern (already consumed by `gl`/`wgpu`/`canvas`):
  - `computeGaussianSigmaFromRadius(radius): number` and `computeGaussianRadiusFromSigma(sigma): number`.
  - `createGaussianKernelWeights(radius, sigma, out: Float32Array): number` (writes normalized 1D weights, returns count; out-param, alias-safe).
  - `computeSeparableBlurPassCount(effect): number` for the blur family (motion/directional/radial/camera-motion), replacing per-recipe guesses at `samples`.
- **Color-temperature math.** Add `computeColorTemperatureRgb(kelvin, out: [number, number, number])` (Planckian-locus / Tanner-Helland approximation) and `computeWhiteBalanceMultipliers(temperature, tint, out)` so `WhiteBalance`/`ColorGrade` temperature handling is one shared implementation, not re-derived per backend.
- **Effect-kind enumeration.** Add `RenderEffectKind` string-union (or exported `const` array) and `getRenderEffectKinds(): readonly Kind[]` so tools/editors can enumerate the catalog. Pure data, tree-shakable.
- **Tests + Rust mirror.** Colocated `*.test.ts` for every new function (kernel normalization sums to 1, sigma↔radius round-trips, temperature endpoints); mirror each new descriptor field, the input-table, and each math function into `flighthq-effects` (`compute_gaussian_*`, `get_render_effect_inputs`, etc.) so conformance stays 1:1.

## Silver

Competitive with a well-regarded post-process stack (Unity URP/HDRP volume components, Godot environment/glow, PostProcessing v2): every common descriptor exposes the parameters a colorist/TD expects, and the shared color-science math is real.

- **Multi-mip bloom.** Extend `BloomEffect` with `mipCount?`, `thresholdKnee?` (soft-knee bright-pass), and `mipWeights?: readonly number[]` (per-mip scatter). Add `computeBloomMipCount(width, height, effect): number`, `computeBloomThresholdKnee(effect, out: [number, number, number, number])` (the `(threshold, threshold-knee, 2*knee, 0.25/knee)` curve constants), and `computeBloomMipWeights(mipCount, effect, out): number`. This is the single biggest per-effect depth gap the review names.
- **Tone-mapping operator math.** `ToneMapEffect` already has the `operator`/`exposure`/`white` fields; add the shared evaluators so all backends share one curve: `computeReinhardToneMap`, `computeAcesToneMap` (the Narkowicz fit + the full ACES RRT/ODT matrices), `computeFilmicToneMap`, `computeAgxToneMap`, `computeUncharted2ToneMap`, plus `getAcesInputMatrix(out)` / `getAcesOutputMatrix(out)`. Add `computeExposureScale(exposure): number` (EV → linear multiplier) shared by `Exposure`/`ToneMap`/`ColorGrade`.
- **ASC CDL color-science primitives.** Add a shared slope/offset/power form: `computeCdlFromLiftGammaGain(effect, out: CdlValues)` and `applyCdl`-style param derivation so `ColorGrade` and `LiftGammaGain` share one color pipeline instead of being independent descriptors. Add shadows/midtones/highlights split parameters to `ColorGradeEffect` (`shadows?`, `midtones?`, `highlights?` as packed RGBA wheels) — the standard 3-way grade.
- **LUT-coordinate math.** Add `computeLookupTableCoord(rgb, lutSize, out)` and `getLookupTableTileLayout(lutSize, out)` for the common 2D-strip-encoded 3D LUT used by `LookupTableGrade`, so each backend samples the LUT identically.
- **Quality/sample fields across the catalog.** Audit and add `samples?`/`quality?` where missing: `RadialBlur`, `DirectionalBlur`, `CameraMotionBlur`, `BokehDepthOfField` (`apertureBlades?`, `maxBlurRadius?`), `Ssr` (`maxSteps?`, `thickness?`, `maxDistance?`), `ChromaticAberration` (`samples?`). Each new field gets the matching `compute*` derivation where it drives a pass count or kernel.
- **Vignette / depth-of-field / SSAO depth math.** `computeDepthOfFieldCoc(depth, focusDistance, aperture, focalLength): number` (circle-of-confusion), `computeSsaoSampleKernel(samples, out: Float32Array): number` (hemisphere kernel), `computeLinearDepthFromNonlinear(depth, near, far): number` shared by fog/DoF/SSAO/SSR.
- **Effect-stack validation helper.** `validateRenderEffectList(effects, available: readonly RenderEffectInput[]): RenderEffectInput | null` — returns the first required input the scene target does not provide, or `null` if the list is satisfiable. Sentinel-style (no throw); composition/ordering still belongs to the render layer, this is pure data validation.
- **Defaulting / normalization.** `getRenderEffectDefaults(kind): Readonly<RenderEffect> | null` and `normalizeRenderEffect(effect, out): void` so a sparse intent can be filled to a fully-specified one for serialization, editor display, and deterministic backend behavior.
- **Rust parity.** Every Silver descriptor field, math function, and validation helper mirrored in `flighthq-effects`; the math functions are the _ideal_ conformance targets (value-in/value-out, deterministic, no GPU) — add ported assertion tests so the matrix differ can compare TS↔Rust math directly.

## Gold

Authoritative reference for a 2D/3D post stack: nothing a domain expert finds missing in the _intent + recipe-math_ layer, full edge-case handling, performance-aware math, and exhaustive cross-backend / cross-language conformance.

- **Complete color-science suite.** Full working-space awareness: `getRec709LuminanceWeights(out)`, `getRec2020LuminanceWeights(out)`, sRGB↔linear transfer functions (`computeSrgbToLinear`/`computeLinearToSrgb`), `computeRgbToHsl`/`computeHslToRgb`, `computeRgbToOklab`/`computeOklabToRgb` (perceptual saturation/hue for `HueSaturation`/`ColorGrade`), and the gamut matrices the ACES path needs. This makes every color effect colorimetrically defensible rather than approximate.
- **Catalog completeness pass.** Fill the remaining recognized post families a AAA stack ships and that are absent or stubbed: `AutoExposureEffect`/`EyeAdaptationEffect` (histogram-based, `[HDR]`), `MotionBlurEffect` camera-vs-object distinction with shutter-angle, `ContactShadowsEffect` (`[DEPTH]`), `VolumetricLightEffect` (beyond `GodRays`), `BarrelDistortionEffect`/`PanniniProjectionEffect`, `FilmEmulationEffect` (grain + halation + gate weave), `ColorBlindSimulationEffect` (accessibility), and a `CustomShaderEffect` escape hatch carrying a backend-keyed source reference (still plain data — a registry key, not a runtime object).
- **Per-effect recipe math for the whole catalog.** Bring every descriptor with non-trivial parameter derivation up to the `computeBloomBlurRadius` standard: halftone cell/angle math, dither matrices (Bayer/blue-noise table generation, `createBayerMatrix(order, out)`), CRT mask/curvature constants, glitch block math, Kuwahara sector weights, outline edge-threshold derivation, scanline phase. The end state: no backend recipe contains its own parameter math that a sibling backend re-derives.
- **Animation-friendly interpolation.** `lerpRenderEffect(a, b, t, out): void` per-kind (numeric fields lerp, packed-RGBA fields lerp in the right space, enums snap) so tween/timeline can animate an effect intent. Plus `canLerpRenderEffects(a, b): boolean` sentinel guard for mismatched kinds.
- **Serialization & versioning.** Stable round-trip: `serializeRenderEffect`/`deserializeRenderEffect` (or confirm structural-clone suffices) with a versioned migration hook consistent with the SDK's scene-migration model, so saved effect stacks survive descriptor evolution.
- **Performance & numerical rigor.** All `create*GaussianKernel`/matrix helpers write into caller-provided typed arrays (zero per-frame allocation), documented alias-safety on every out-param function, and SIMD-friendly memory layout that the Rust crate can match. Hot math benchmarked.
- **Exhaustive tests + docs.** Property tests for every transfer function and matrix (round-trip, identity, known reference values from the ACES/Rec.709 specs), table-driven `getRenderEffectInputs` coverage for all kinds, and golden-value tests pinning each tone-map operator at reference exposures. A package-level doc enumerating every effect, its inputs, its parameters, and the backend recipes that consume it.
- **1:1 Rust conformance.** `flighthq-effects` mirrors the complete catalog, all recipe math, interpolation, and serialization; the divergence map records any intentional TS↔Rust differences (expected: none for the value-typed math). Effects math joins the "mixable leaf" conformance set — deterministic, GPU-free, headlessly fingerprintable — and is wired into the parity matrix differ so every operator and kernel is checked TS↔Rust per commit.

## Sequencing & effort

Recommended order, with dependencies and items to surface:

1. **Bronze base-contract + input-data first** (small, unblocks everything). Adding `intensity`/`enabled` to `RenderEffect` and the `RenderEffectInput` data table is a `@flighthq/types`-first change that the pipeline (`@flighthq/render`) and editors immediately benefit from. **Cross-package surface item:** the render-effect pipeline in `@flighthq/render` / `effects-gl|wgpu|canvas` should be updated to _honor_ `enabled` (skip) and `getRenderEffectInputs` (validate) — flag this to the owners of those packages rather than editing across the boundary autonomously.
2. **Bronze gaussian + temperature math** (medium, highest-value-per-line). Pure additive functions; the backends already import `computeBloomBlurRadius`, so they can migrate their duplicated blur/temperature math onto these incrementally. Coordinate the migration with `effects-gl`/`effects-wgpu`/`effects-canvas` owners — the math lands here, the recipe edits land there.
3. **Silver multi-mip bloom + tone-map operators** (medium-large). These are the two descriptor-depth gaps the review calls out by name; the tone-map field shape already exists, only the shared evaluators are missing. ACES matrices are a one-time, well-specified addition.
4. **Silver CDL / 3-way grade + LUT math** (large). **Design-decision item to surface:** whether `ColorGrade` and `LiftGammaGain` should _unify_ onto a shared CDL primitive or stay separate descriptors that both _consume_ it. The review flags their lack of shared color-science; the cleaner end-state is a shared math core with two descriptor faces — confirm the API shape (do users want both kinds, or is `ColorGrade` the canonical one?) before committing.
5. **Silver validation + defaulting helpers** (small-medium). `validateRenderEffectList`, `getRenderEffectDefaults`, `normalizeRenderEffect` round out the introspection story.
6. **Gold catalog/color-science/interpolation/serialization** (large, ongoing). The catalog-completeness pass and per-effect math are the genuine frontier; `lerpRenderEffect` depends on tween/timeline wanting to animate intents (**cross-package:** confirm `@flighthq/tween`/`@flighthq/timeline` will consume it); serialization should reuse the SDK's existing scene-migration mechanism rather than inventing a new one (**design-decision:** align with `conventions/types-layout.md` versioned-migration model).

**Rust mirror runs alongside every tier**, not as a final step — the `flighthq-effects` crate already tracks the thin shape 1:1, and the math functions are the easiest, most valuable conformance targets (deterministic, GPU-free). Add the ported assertion tests and wire the math into the parity differ as each function lands, so TS↔Rust never drifts.

**Boundary that does not move:** recipes, registration (`register*RenderEffect`), and pipeline dispatch stay in `@flighthq/render` and the `effects-<backend>` packages. This roadmap deepens the _intent + recipe-math_ charter only; it never pulls backend rendering into this package.
