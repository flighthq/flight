# Maturation Roadmap: @flighthq/device

**Current verdict**: partial — 45/100. A clean, correctly-shaped backend seam with a 7-field `DeviceInfo` snapshot plus `SafeAreaInsets`, but under-populated for its domain — missing a stable device id, display/DPI metrics, CPU/GPU capability, total-vs-available memory, and form-factor granularity that an authoritative device-identity library exposes.

The package today is a single `DeviceInfo` value (`model`, `manufacturer`, `osName`, `osVersion`, `platform`, `isVirtual`, `memory`) + `SafeAreaInsets`, read through `getDeviceBackend` / `setDeviceBackend` / `createWebDeviceBackend` with a web UA classifier and documented sentinels. Maturation is mostly about widening the value snapshot and resolving two real conformance hazards (the `platform` overload and the `memory` unit ambiguity) before any native host conforms to the seam. The seam shape, sentinel discipline, out-param/allocator discipline, and entity-less flat-value design are already right and should be preserved.

All shared type changes land in `@flighthq/types/src/Device.ts` first (the header layer), then the implementation in `@flighthq/device` follows. The Rust mirror is `flighthq-device` (a `native`-feature default backend over std/sysinfo; value-typed, so it is a mixable leaf).

## Bronze

The minimum genuinely-useful version: fix the two conformance hazards and add the single most-requested missing field (a stable id). All of this is flat-value additions to `DeviceInfo` plus one new read — no new sub-systems.

- **Resolve the `platform` overload.** In `@flighthq/types` rename the architecture concept to `arch: string` (CPU architecture, e.g. `'arm64'`, `'x64'`) and stop overloading it with the UA string. If the raw platform/UA string is still wanted, add a separate `platformString: string`. Update the web backend to set `arch` from `navigator.userAgentData?.platform`/UA inference (sentinel `''` when unknown) instead of dumping the whole `userAgent` into it. Pre-release, this is a clean rename, not a migration.
- **Resolve the `memory` unit ambiguity.** Replace the single coarse `memory` with `totalMemory: number` (bytes, `-1` sentinel) and document the unit explicitly in the type comment. The web backend converts `navigator.deviceMemory` (GiB-rounded) to bytes (`deviceMemory * 1024**3`) so web and native agree on units. (`availableMemory` is deferred to Silver — it is live-ish and coarser on web.)
- **Add a stable device/install id.** New `getDeviceId(): string` free function (the single most-used field in `react-native-device-info`/Capacitor `getId`) backed by `DeviceBackend.getId(): string`. Web default: `crypto.randomUUID()` persisted via a storage seam — but to keep `@flighthq/device` dependency-light, the web backend reads/writes through `@flighthq/storage` if present and otherwise returns `''`. Sentinel `''` when no stable id can be formed. Document that this is an _install_ id (resettable by clearing storage), not a hardware serial.
- **Add `cpuCores: number`** to `DeviceInfo` (web: `navigator.hardwareConcurrency`, sentinel `-1`). Cheap, universally available, high value.
- **Add `createDeviceId`-free convenience is not needed** — keep `getDeviceId` as a direct string read; do not introduce a runtime object.
- **Doc cross-references for the deliberate splits.** Add comments in `Device.ts` and the Package Map entry pointing at `@flighthq/platform` (locale, OS/desktop/mobile kind, touch), `@flighthq/power` (battery/thermal), `@flighthq/app` (app version/build/bundle id), and `@flighthq/screen` (display metrics) so omissions read as design, not gaps.
- Tests: extend `device.test.ts` for `arch`, `totalMemory` unit conversion, `getDeviceId` (registered-backend, web-with-storage, web-without-storage `''`, and stability across two reads), `cpuCores`.

## Silver

Competitive and solid — matches a good device library (Capacitor `Device`, `react-native-device-info`, `expo-device`): fuller identity, form-factor, display device data, and a real safe-area path.

- **Memory split + low-end hint.** Add `availableMemory: number` (bytes, `-1`) and `isLowEndDevice: boolean` (web heuristic over `deviceMemory`/`hardwareConcurrency`; native: real thresholds). Keep `totalMemory` from Bronze.
- **Form-factor / device class.** Add a `DeviceFormFactor` _Kind_-style string union home in `@flighthq/types` (`'Phone' | 'Tablet' | 'Desktop' | 'TV' | 'Watch' | 'Car' | 'Unknown'`) exposed as `formFactor` on `DeviceInfo`, plus the granular predicates that have no home in `@flighthq/platform`: `isTablet`. (`isMobile`/`isDesktop` stay delegated to `@flighthq/platform` with a doc note; only the tablet/TV/watch granularity device class lives here.) Web default infers from UA + viewport + touch; native reports real class.
- **Display device metrics** (device-class data a device library is still expected to surface, distinct from `@flighthq/screen`'s live multi-display enumeration). Add `DeviceDisplayMetrics` to `@flighthq/types` (`pixelRatio`, `densityDpi`, `colorDepth`, `physicalWidth`, `physicalHeight`, `logicalWidth`, `logicalHeight`) and `getDeviceDisplayMetrics(out): DeviceDisplayMetrics` + `createDeviceDisplayMetrics()` allocator + `DeviceBackend.getDisplayMetrics`. Web: `devicePixelRatio`, `screen.colorDepth`, `screen.width/height`. Doc-cross-reference `@flighthq/screen` for live work-area/scale.
- **OS detail.** Add `osBuild: string` (build/kernel/API-level string, `''` sentinel) to `DeviceInfo`; populate the web `osVersion` from UA where reliably parseable (currently always `''`).
- **GPU identity.** Add `gpuVendor: string` and `gpuRenderer: string` to `DeviceInfo` (web: `WEBGL_debug_renderer_info` over a transient context, `''` when blocked/unavailable). Document that browsers may mask these.
- **Real safe-area path on web.** Replace the zero-stub `getSafeAreaInsets` with a CSS-var bridge: `enableWebSafeAreaInsets()` (or a backend helper) that mounts a probe element using `env(safe-area-inset-*)` and reads computed values, so web PWAs on notched devices get real insets instead of `0`. Keep zero as the fallback when the probe yields nothing.
- **Security / integrity flags.** Add `isJailbroken: boolean` and `isRooted: boolean` (web: `false`; native: real checks) alongside the existing `isVirtual`.
- **Backend completeness check.** Add an `exports:check`-covered test asserting every `DeviceBackend` method has a web default and a sentinel-return path; document the full backend contract in `Device.ts`.

## Gold

Authoritative / AAA — the canonical device-identity reference, with exhaustive coverage, a parser neighbor package, async live-ish reads where needed, full Rust parity, and conformance tests.

- **`@flighthq/device-formats` neighbor package.** Move the UA-string parsing (`detectWebOsName` and any UA→OS/version/device-class inference) into a `-formats` sibling as pure, tree-shakable parser free functions (`parseUserAgentOsName`, `parseUserAgentOsVersion`, `parseUserAgentFormFactor`, `parseUserAgentArch`). This keeps `@flighthq/device` a thin seam and makes the (large, churny) UA table independently testable and updatable. The web backend imports it; native hosts do not pull it in.
- **Async storage-derived id done right.** If `getDeviceId`'s storage seam is async on some hosts (OPFS, native keystore), add `getDeviceIdAsync(): Promise<string>` over `DeviceBackend.getIdAsync`, keeping the sync `getDeviceId` as a best-effort cached read. Native: hardware-backed stable id (keystore/`ANDROID_ID`/IDFV) where the platform permits; sentinel `''` where policy forbids.
- **Exhaustive identity surface** to match the union of competitors: `productName`, `marketingName`, `boardName`, `hardwareModel` (raw vs human), `installerSource`/`installSource` (delegated to `@flighthq/app` if it lands there — surface a design-decision item), `webViewVersion: string`, `supportedAbis: ReadonlyArray<string>`, `fontScale: number` (accessibility text scale), `isHdr: boolean`, `colorGamut: string`.
- **Capability flags block.** A `DeviceCapabilities` value (`hasTouch` — doc-delegated to `@flighthq/platform`; `hasKeyboard`, `hasMouse`, `hasStylus`, `hasVibration` — doc-delegated to `@flighthq/haptics`; `hasGeolocation`, `hasCamera`) read via `getDeviceCapabilities(out)` — but only for the dimensions with no other package home; the rest are explicit doc cross-references to avoid duplication.
- **Performance + hot-path guarantees.** Document and test that all `get*(out)` reads are allocation-free in steady state (snapshot cached in the backend, re-filled into `out`); add a `refreshDeviceInfo()` to invalidate the cache for the rare fields that change (orientation-affected insets, available memory).
- **Full edge-case + error handling.** Every backend method returns a documented sentinel on every failure mode (no `navigator`, blocked GPU info, denied storage, SSR/no-DOM); a test matrix exercises each. No throws anywhere except genuine API misuse.
- **1:1 Rust parity.** `flighthq-device` mirrors the final type surface as value types: `DeviceInfo`, `SafeAreaInsets`, `DeviceDisplayMetrics`, `DeviceFormFactor`, `DeviceCapabilities`; `DeviceBackend` trait + `set_device_backend`; a `native`-feature default backend over `sysinfo`/`os_info`/`raw-window-handle`; `get_device_info(&mut out)`, `get_device_id`, `get_device_display_metrics(&mut out)`. As a value-typed leaf, register it in the conformance map as **mixable** (a future `device-rs` wasm drop-in) and add a parity cell so Rust↔TS device reads are fingerprint-compared.
- **Docs**: a package README/doc enumerating every field, its unit, its sentinel, its web vs native source, and the cross-package delegation table — the canonical reference the verdict asks for.

## Sequencing & effort

Recommended order (each tier is cumulative; do not skip the Bronze hazard fixes — native hosts cannot conform to the seam until `platform`/`memory` are resolved):

1. **Bronze, hazard fixes first (small, ~half day).** `arch` rename + `totalMemory` bytes are pure type+web-backend renames in `@flighthq/types`/`@flighthq/device`; do them before anything depends on the current ambiguous shape. Then `cpuCores` (trivial) and `getDeviceId` (the one item with an external dependency — see below). Then the doc cross-references. Land, run `npm run check` + `npm run api device` + `npm run exports:check`.
2. **Silver (medium, ~2–3 days).** Form-factor and display metrics are the highest-value; do those before GPU/OS-detail. The web safe-area CSS-var bridge is the only browser-fiddly item — budget extra. `isJailbroken`/`isRooted` are web-trivial (always `false`) but only meaningful once a native host exists.
3. **Gold (large, ongoing).** The `-formats` split is the natural first Gold step (it unblocks clean UA-table maintenance and shrinks the seam). Async id, the exhaustive field set, and Rust parity follow. Rust parity should track the _final_ TS shape — do it after the type surface stabilizes, not mid-flux.

**Cross-package / design-decision items to surface (do not decide autonomously):**

- **`getDeviceId` storage dependency.** Persisting the install id needs `@flighthq/storage`. Decide whether `@flighthq/device` takes a (web-only, lazily-used) dependency on it, or whether the host injects a storage function into the backend. Recommend the latter (keeps `device` dependency-light and tree-shakable) but it is a seam-shape decision worth confirming.
- **Display metrics vs `@flighthq/screen` boundary.** `getDeviceDisplayMetrics` (device-class static data) overlaps conceptually with `@flighthq/screen` (live, multi-display, work-area). Confirm the split: device = the built-in display's static characteristics; screen = enumeration + live geometry. Needs a one-line ruling in the Package Map.
- **`installerSource` / app-context fields.** Competitors bundle app version/build/bundle-id/installer into the device library; Flight scatters these into `@flighthq/app`. Confirm `installerSource` belongs in `@flighthq/app`, not here, before adding it anywhere.
- **`isTablet`/form-factor vs `@flighthq/platform`.** `@flighthq/platform` owns desktop/mobile/web kind and touch. Confirm that tablet/TV/watch granularity (`formFactor`, `isTablet`) lives in `@flighthq/device` and is not duplicated in `platform`.
- **Capabilities duplication.** Several Gold `DeviceCapabilities` flags (touch, vibration, geolocation, camera) have owning packages (`platform`, `haptics`, `geolocation`, `webcam`). Decide per-flag: surface here as a convenience read, or keep as a doc cross-reference only. Default to cross-reference to avoid drift.
