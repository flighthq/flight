# Maturation Roadmap: @flighthq/platform

**Current verdict**: solid — 72/100. The right shape and complete _as a seam_ (flat free functions over a swappable `PlatformBackend` with `get*`/`set*`/`createWeb*`, sentinel-returning web guards, out-params, value-typed `PlatformInfo` in `@flighthq/types`), but two of the six advertised `PlatformInfo` fields — `version` and `arch` — are dead on the only shipped backend, and "web" collapses every non-native host into one bucket.

This is a deliberately narrow root-identification seam, not a kitchen-sink platform library — clipboard/fs/dialog/etc. are sibling packages by design, and device/memory/battery are routed to `@flighthq/device`/`@flighthq/power`. So maturation here is _deepening one seam_, not absorbing capabilities. The work is: make the existing fields actually populate on web, add a runtime/engine signal so the host can be classified beyond "web", and (Gold) make this the canonical environment-identification reference with version-comparison helpers, a UA-parser neighbor package, and Rust parity.

All shared type changes land in `@flighthq/types/src/Platform.ts` first (the header layer), then `@flighthq/platform` implements against them. The Rust mirror is `flighthq-platform` (a `native`-feature default backend over std/`os_info`; value-typed, so it is a **mixable** leaf). Preserve the existing seam shape, sentinel discipline, out-param/no-alloc convenience reads, and entity-less flat-value design throughout.

## Bronze

The minimum genuinely-useful version: make the fields the type already advertises actually populate on web, fix the one correctness nit, and add the single most-requested missing classification (host runtime). No new sub-systems — flat-value fills plus one new field.

- **Fill `arch` on the web backend.** Read `navigator.userAgentData?.architecture` + `bitness` (low-entropy is `''`; arch is high-entropy — see the async item below for the proper resolve), with a UA-string fallback heuristic (`x86_64`/`Win64`/`WOW64` → `'x64'`, `arm`/`aarch64` → `'arm64'`). Sentinel `''` when undetectable. Normalize the value space to the same canonical tokens native will use (`'x64'`, `'arm64'`, `'x86'`, `'arm'`, `'wasm'`).
- **Fill `version` on the web backend.** Parse OS version from the UA string where reliably available (the `Windows NT 10.0` / `Mac OS X 10_15` / `Android 14` / `CPU OS 17_4` patterns), upgraded by `userAgentData.getHighEntropyValues(['platformVersion'])` when the async path lands. Sentinel `''` when unparseable. Keep the type comment's "never parsed for semantics here" boundary for the _string_; semantic comparison is a separate Silver helper.
- **Add a host-runtime signal.** New `PlatformRuntime` string union in `@flighthq/types` — `'web' | 'electron' | 'tauri' | 'capacitor' | 'native' | 'unknown'` — as `runtime` on `PlatformInfo`, plus `getPlatformRuntime(): PlatformRuntime`. This is the field that stops "web" from being the only non-native classification: `kind`/`name` describe the OS, `runtime` describes the host shell. Web default infers from globals (`window.process?.versions?.electron`, `window.__TAURI__`, `window.Capacitor`) and defaults to `'web'`; native hosts set it explicitly. This is the single highest-value addition.
- **Add `enginePlatformName` / browser-engine field.** `PlatformEngine` union (`'blink' | 'gecko' | 'webkit' | 'unknown'`) as `engine` on `PlatformInfo` with `getPlatformEngine(): PlatformEngine`. Cheap UA inference (`Chrome`/`Edg` → blink, `Firefox` → gecko, `Safari`-without-Chrome → webkit). High value for web-quirk branching; native sets `'unknown'`.
- **Fix the `"iOs"` doc-comment typo** on `isPlatformMobile` (→ `"iOS"`).
- **Document the `_scratch` aliasing for the port.** Add a one-line comment on the module-level `_scratch` noting it is a single-threaded JS no-alloc convenience and that the Rust/native mirror uses a per-call local or thread-local (the review flagged this).
- Tests: extend `platform.test.ts` for `arch` (UA-fallback tokens + `''` sentinel), `version` (each UA pattern + `''`), `getPlatformRuntime` (each global probe + registered-backend + `'web'` fallback), `getPlatformEngine`, and the iOS comment is purely cosmetic.

## Silver

Competitive and solid — matches what a good environment-detection library (`bowser`, `platform.js`, `ua-parser-js`, Capacitor `Device.getInfo`, Tauri `os` plugin) offers: an async resolve path so the rich web fields actually fill, version-comparison ergonomics, and complete cross-backend value normalization.

- **Async identity resolve path.** The web `userAgentData` high-entropy values (`platformVersion`, `architecture`, `bitness`, full browser version) are Promise-based; the current sync `getInfo` cannot fill them. Add `getPlatformInfoAsync(out): Promise<PlatformInfo>` over a new optional `PlatformBackend.getInfoAsync?(out): Promise<PlatformInfo>`, and a `refreshPlatformInfo(): Promise<void>` that resolves high-entropy values once and caches them so subsequent sync `getPlatformInfo` reads return the enriched snapshot. Native backends may omit `getInfoAsync` (sync is complete there). This is the seam the review identified as the blocker for `version`/`arch` on web — do it before relying on those fields being populated in browsers.
- **Version-comparison helpers.** The string `version` stays unparsed, but add free functions that operate _on_ it: `comparePlatformVersions(a, b): -1 | 0 | 1` (numeric-segment, dotted compare; non-numeric tail ignored) and `isPlatformVersionAtLeast(minimum): boolean` (reads the live `version`, returns `false` when `version === ''`). These give the "satisfies / at-least" ergonomics competitors provide without turning `version` into a parsed semantic field on the type. Sentinel: comparison of `''` sorts lowest; `isPlatformVersionAtLeast` returns `false` on unknown.
- **Canonical value-space documentation + normalization.** Pin and document the exact allowed tokens for every union (`PlatformName`, `PlatformKind`, `PlatformRuntime`, `PlatformEngine`, and the `arch` token set) in `Platform.ts` so web and native produce identical strings — this is the cross-backend consistency the conformance suite will fingerprint. Add a test asserting the web backend only ever emits canonical tokens.
- **`isPlatformNative()` predicate.** Convenience over `runtime !== 'web' && runtime !== 'unknown'` — the common "are we in a host shell?" branch, complementing `isPlatformWeb`. Also `isPlatformDesktop`/`isPlatformMobile` already exist; round out with the runtime-based questions hosts actually ask.
- **Browser/engine version field.** Add `engineVersion: string` (browser/engine version, distinct from OS `version`) to `PlatformInfo`, filled from UA / high-entropy `fullVersionList`. `''` on native. Many web-targeting libs surface this separately from OS version.
- **Backend-completeness check.** An `exports:check`-covered test asserting every `PlatformBackend` method has a web default and a documented sentinel-return path (no `navigator`, SSR/no-DOM, blocked high-entropy). Document the full backend contract inline in `Platform.ts`.

## Gold

Authoritative / AAA — the canonical environment-identification reference: exhaustive coverage, a UA-parser neighbor package, full edge-case/error handling, performance guarantees, docs, and 1:1 Rust parity.

- **`@flighthq/platform-formats` neighbor package.** Move all UA-string parsing (`detectWebPlatformName`, OS-version extraction, arch inference, engine/runtime inference) into a `-formats` sibling as pure, tree-shakable parser free functions: `parseUserAgentPlatformName`, `parseUserAgentPlatformVersion`, `parseUserAgentArch`, `parseUserAgentEngine`, `parseUserAgentRuntime`. This keeps `@flighthq/platform` a thin seam and makes the large, churny UA table independently testable and updatable; the web backend imports it, native hosts do not pull it in. This is the natural first Gold step.
- **Endianness / pointer-width signals** relevant to the C/C++ port and binary-format code: add `endianness: 'little' | 'big' | 'unknown'` and `pointerWidth: 32 | 64 | -1` to `PlatformInfo` (web: infer `64` from `bitness`/arch, `'little'` as overwhelming default with a documented caveat; native: real values). The review called these out as common in native-targeting identification layers.
- **Distribution / OS-detail fields** to match the union of competitors: `osBuild: string` (build/kernel string, `''`), and on Linux `distro: string` + `distroVersion: string` (native via `os-release`; web `''`). Document each as `''`-sentinel and web-vs-native source.
- **Capability-version block for graphics-relevant checks.** A small `PlatformGraphics` value read via `getPlatformGraphicsInfo(out)` covering only the dimensions with no other package home and high relevance to renderer selection: `hasWebgl2: boolean`, `hasWebgpu: boolean`, `prefersReducedMotion: boolean`. Anything device-hardware-shaped (GPU vendor/renderer) stays a doc cross-reference to `@flighthq/device`; this block is strictly host/runtime feature flags. Surface as a design-decision item (see below) — it may belong in a renderer-capabilities seam instead.
- **Performance + hot-path guarantees.** Document and test that all sync `get*(out)` / scalar reads are allocation-free in steady state (snapshot cached in the backend after the async resolve, re-filled into `out`/returned from the cached scratch). The async `refreshPlatformInfo` is the only allocation point.
- **Full edge-case + error handling matrix.** Every backend method returns a documented sentinel on every failure mode (no `navigator`, no `userAgentData`, denied/timed-out high-entropy resolve, SSR/no-DOM, exotic UA). A test matrix exercises each; no throws anywhere except genuine API misuse (the existing throw-only-on-misuse rule holds — there is currently no misuse surface, which is correct).
- **1:1 Rust parity.** `flighthq-platform` mirrors the final type surface as value types: `PlatformInfo`, `PlatformName`, `PlatformKind`, `PlatformRuntime`, `PlatformEngine`; `PlatformBackend` trait + `set_platform_backend`; a `native`-feature default backend over `os_info`/`std::env::consts` (`OS`, `ARCH`) / `target_endian`; `get_platform_info(&mut out)`, `get_platform_runtime`, `compare_platform_versions`, `is_platform_version_at_least`. As a value-typed leaf, register it in the conformance map as **mixable** (a future `platform-rs` wasm drop-in) and add a parity cell so Rust↔TS identity reads are fingerprint-compared. The async resolve path is web-only (`host-web`); the native default is fully sync — keep the authoritative seam native-clean per the Rust host-layer async/`Send` note.
- **Docs**: a package README/doc enumerating every field, its value space, its sentinel, its web vs native source, and the cross-package delegation table (`@flighthq/device`, `@flighthq/power`, `@flighthq/screen`, `@flighthq/app`) — the canonical reference the verdict asks for.

## Sequencing & effort

Each tier is cumulative. The dependency that gates everything web-rich is the **async resolve path** (Silver): without it, `arch`/`version` can only ever be UA-string best-effort. Bronze ships the UA-string version of those fields immediately (no async needed), so Bronze is genuinely shippable on its own.

1. **Bronze (small, ~half day).** All flat type+web-backend additions in `@flighthq/types`/`@flighthq/platform`, no external dependencies. Order: `runtime` field + `getPlatformRuntime` first (highest value, unblocks host classification), then `arch`/`version` UA-string fills, then `engine`, then the typo + `_scratch` comment. Land, run `npm run check` + `npm run api platform` + `npm run exports:check` + `npm run order`.
2. **Silver (medium, ~1–2 days).** The async resolve path is the keystone — do it first so the Bronze `arch`/`version` fields upgrade from "UA best-effort" to "high-entropy accurate" on web. Then the version-comparison helpers (pure, trivial), `isPlatformNative`, `engineVersion`, and the canonical-token normalization test. The async seam is the only browser-fiddly item — budget extra for the `getHighEntropyValues` permission/timeout edge cases.
3. **Gold (large, ongoing).** `-formats` split first (it shrinks the seam and unblocks clean UA-table maintenance), then endianness/pointer-width and OS-detail fields, then Rust parity. Rust parity should track the _final_ TS shape — do it after the type surface stabilizes, not mid-flux. The `PlatformGraphics` block is the most uncertain Gold item — confirm its home before building (see below).

**Cross-package / design-decision items to surface (do not decide autonomously):**

- **`runtime` vs `@flighthq/app`/`@flighthq/device` boundary.** Host-shell identity (`electron`/`tauri`/`capacitor`) could be argued to belong in `@flighthq/app` (application/process layer) rather than the OS-identification seam. Recommend `runtime` stays here as a low-level _classification_ (it pairs naturally with `name`/`kind`), while `@flighthq/app` owns app _identity_ (name/version/bundle id) — but confirm the split with a one-line Package Map ruling.
- **`PlatformGraphics` capability block home.** `hasWebgl2`/`hasWebgpu`/`prefersReducedMotion` may belong in a renderer-capabilities seam (e.g. alongside `@flighthq/render` backend selection) rather than the platform-identification package. Decide before building — default to _not_ adding it to `platform` unless no better home exists, to avoid scope creep into the renderer domain.
- **Async-seam shape consistency across the suite.** Adding `getInfoAsync`/`refreshPlatformInfo` sets a precedent for the whole platform-integration suite's web high-entropy reads (`@flighthq/device` has the same `userAgentData` problem). Coordinate the async-resolve pattern once so `platform` and `device` (and any future high-entropy reader) share one shape rather than diverging — surface as a suite-wide seam decision.
- **Canonical `arch` token set.** Pin the exact arch vocabulary (`'x64'` vs `'amd64'`, `'arm64'` vs `'aarch64'`) once, shared with `@flighthq/device`'s `arch` field (per the device roadmap's Bronze rename) so the two packages never disagree on the same concept. This is a shared-types decision, not a per-package one.
