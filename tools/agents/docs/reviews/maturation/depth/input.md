# Maturation Roadmap: @flighthq/input

**Current verdict:** solid — 70/100. Authoritative on normalization and SDL-style key-code mapping, but a thin normalization seam against the full breadth of a mature input system: no queryable state, no semantic gamepad layer, no timestamps, no backend seam, no actions/gestures.

The depth review's core call holds: this is a deliberately high-quality _normalization layer_, not yet an _input library_. The roadmap below moves it toward "input library" depth where that belongs in-package (state, gamepad semantics, payload richness, timestamps, portability) and explicitly hands off the two layers that should be bounded neighbor packages (action/binding maps, gesture recognition). Every payload addition lands in `@flighthq/types` first.

## Bronze

The minimum that turns the edge-only normalizer into something a game/app can actually poll and trust. Highest value-per-effort.

- **Event timestamps.** Add `timeStamp: number` (monotonic, ms) to `InputKeyboardData`, `InputPointerData`, `InputGamepadButtonData`, `InputGamepadAxisData` in `@flighthq/types`; populate from `event.timeStamp` (DOM) and `performance.now()` (gamepad poll). Unblocks repeat-timing, velocity, inertia, double-tap downstream. Touches every `set*Data` writer and every scratch singleton.
- **Queryable held-state snapshot.** A plain-data `InputState` entity (in `@flighthq/types`) plus `createInputState()`, `isInputKeyDown(state, keyCode): boolean`, `isInputPointerButtonDown(state, button): boolean`, `getInputGamepadAxis(state, gamepad, axis): number`, `getInputGamepadButtonValue(state, gamepad, button): number`. Fed by `connectInputStateToInputManager(state, manager): () => void` (subscribes to the manager's signals; returns a disposer). This is the single biggest gap the review names — give consumers "is W held?" without each one re-deriving it.
- **Pointer payload enrichment.** Add `pressure: number`, `tiltX: number`, `tiltY: number`, `twist: number`, `width: number`, `height: number` to `InputPointerData`; populate in `setInputPointerData` from the `PointerEvent` (default `0` for `WheelEvent`/relative). The `pen` pointer type already advertises support that the payload silently drops today.
- **`InputTextData` payload.** Stop overloading `TextSelectionRange` for inserted runs. Add `InputTextData { text: string; isComposing: boolean }` to `@flighthq/types`; retarget `onTextInput`/`onTextEdit`. Carry `isComposing` from the DOM event so consumers can distinguish committed text from in-flight composition. (Honest type, called out as a smell in the review.)
- **Options symmetry.** Accept `Readonly<AttachInputOptions>` on `attachRelativePointerInput`, `attachTextInput`, and `attachGamepadInput` so `preventDefault` (and future options) are uniform across all attach pairs.

## Silver

Competitive with a good input library (SDL input subsystem / Lime input / Unity's lower layer). Common professional use and the important edge cases, plus the portability seam.

- **`InputBackend` seam.** Define `InputBackend` in `@flighthq/types` and add `getInputBackend()` / `setInputBackend(backend)` / `createWebInputBackend()`. Web backend wraps the current DOM attach paths; native hosts (winit/SDL via the Rust port, Electron) feed normalized events through the _same_ `InputManager` API. Makes "portable" structural rather than DOM-implicit — the review's explicit portability recommendation, and a prerequisite for honest Rust parity.
- **Semantic gamepad mapping.** Add `GamepadButton` and `GamepadAxis` string-`*Kind` identifiers (A/B/X/Y, shoulders, triggers, dpad, stick-click, start/back/guide; left/right stick X/Y, trigger axes) in `@flighthq/types`, plus `getGamepadButtonName(mapping, index)` / `getGamepadAxisName(...)` honoring the W3C `"standard"` mapping vs `"raw"`. Surface `mapping: GamepadMappingKind` on connect data. Buttons/axes are currently raw indices only.
- **Dead-zone filtering.** `applyGamepadAxisDeadZone(value, deadZone): number` and `applyGamepadStickDeadZone(out, x, y, deadZone)` (radial), with a configurable per-manager default. Stick noise filtering is table stakes for any gamepad layer.
- **Gamepad rumble / vibration.** `hasGamepadVibration(gamepad): boolean`, `setGamepadVibration(gamepad, { strongMagnitude, weakMagnitude, duration }): boolean`, `stopGamepadVibration(gamepad)` over the `GamepadHapticActuator` / dual-rumble effect, returning sentinel `false` when unsupported. Routed through the `InputBackend` so native hosts implement it natively.
- **Coalesced & predicted pointer events.** `getCoalescedInputPointerData(out, event)` / iteration over `getCoalescedEvents()` and an opt-in `coalesced` flag on pointer attach, plus predicted events for stylus/high-frequency input. Matters for the `pen`/high-Hz paths already advertised.
- **Key-repeat synthesis config.** First-class `InputKeyRepeatOptions { delay, interval }` and a repeat timer so non-DOM sources (gamepad d-pad, native backends) get consistent auto-repeat, instead of relying on the DOM's `repeat` flag only.
- **Pointer-lock & capture helpers.** `requestInputPointerLock(element): Promise<boolean>`, `exitInputPointerLock()`, `hasInputPointerLock()`, and `setInputPointerCapture` / `releaseInputPointerCapture` wrappers — the relative-pointer path exists but the lock lifecycle it depends on is left to the caller.
- **Snapshot frame edges.** Extend `InputState` with per-frame edge queries: `wasInputKeyPressed(state, keyCode)` / `wasInputKeyReleased(...)` (down/up this frame) and `endInputStateFrame(state)` to roll edges. This is the polling idiom games expect on top of the held-state snapshot.

## Gold

Authoritative / AAA. Exhaustive coverage, the higher layers as bounded neighbors, performance, full edge handling, and 1:1 Rust parity.

- **`@flighthq/input-bindings` neighbor package.** Action/binding-map layer: `InputAction` + `InputActionSet` (plain descriptors), `createInputBindingMap()`, `bindInputAction(map, action, binding)`, `getInputActionValue(map, state, action)`, `isInputActionActive(...)`, runtime rebinding, and chord/combo recognition (`InputChord`). Serializable binding maps (string-kind actions, JSON round-trip) so apps can persist user rebinds. Large and orthogonal — a separate cell, as the review recommends, not swelling `input`.
- **`@flighthq/gestures` neighbor package.** Gesture recognizers over normalized multi-touch pointers: tap, double-tap, long-press, pan/swipe (with velocity), pinch-zoom, rotate. Recognizer entities with `enable*Gesture` signal groups, fed by the `InputManager`. Depends on Bronze timestamps and Silver coalesced events.
- **`@flighthq/gamepad-mappings` ("-formats" neighbor).** SDL `GameControllerDB`-style mapping database: `parseGamepadMapping(text)` / `getGamepadMappingForId(db, id)` so non-standard pads map correctly. Follows the `-formats`/importer neighbor pattern; keeps the parse table out of the core bundle.
- **Exhaustive key-code table parity.** Fill `keyCodesByCode` / `keyCodesByKey` to the full `KeyCode` enum (media/browser/system keys, F13–F24, international/IME keys, the numpad mem/hex/binary variants the enum already declares) so the mapping is exhaustive, not just the common subset present today.
- **Multi-device identity & hot-plug for keyboard/pointer.** Per-device `InputDeviceId` so a native host with multiple keyboards/mice can disambiguate; `onInputDeviceConnect`/`Disconnect` signals generalized beyond gamepads.
- **Sensor & extended sources** (where they belong here vs `@flighthq/sensors`): clarify the boundary and, if any raw normalization lands here, route it through `InputBackend`. Surface as a design question rather than assume.
- **Full test + conformance.** Property tests for the held-state snapshot vs signal replay, alias-safety tests for every new `out`-param writer, dead-zone numeric tables, and a functional/visual harness scene driving synthetic input across backends.
- **Rust port `flighthq-input` parity.** Mirror the normalized payloads, the `InputBackend` trait, the `InputState` snapshot, and gamepad semantics in the Rust crate; native backends (winit/SDL) implement the seam directly. Record any intentional TS↔Rust divergence (e.g. DOM-only fields) in the conformance map. The `InputBackend` seam (Silver) is the precondition that makes this 1:1 rather than a parallel reimplementation.

## Sequencing & effort

Recommended order, with dependencies and the items needing a decision before coding:

1. **Bronze, types-first, in one pass** (small–moderate). Add timestamps, pointer enrichment, and `InputTextData` to `@flighthq/types`, then update the `set*Data` writers, scratch singletons, and `createInputSignals` together — they touch the same lines. `InputState` + `connectInputStateToInputManager` is the largest Bronze item and depends only on the existing signals. Options symmetry is trivial cleanup. No cross-package coordination beyond `types`.
2. **`InputBackend` seam (Silver) early** (moderate). Do this before the gamepad-depth work and before the Rust port: it is the structural precondition for both native gamepad vibration and 1:1 Rust parity. Land `getInputBackend`/`setInputBackend`/`createWebInputBackend` and move the DOM attach bodies behind the web backend. **Design decision to surface:** whether the attach functions keep taking DOM targets directly (web-default ergonomics) or route entirely through the backend — affects the public signatures and the Rust mirror.
3. **Gamepad depth (Silver)** (moderate). Semantic mapping → dead-zones → vibration, in that order (vibration needs the backend from step 2). Mapping kinds and `GamepadHapticActuator` shapes go in `@flighthq/types` first.
4. **Remaining Silver** (moderate): coalesced/predicted events, key-repeat config, pointer-lock helpers, frame-edge snapshot queries. Frame-edge queries extend the Bronze `InputState`; pointer-lock is self-contained.
5. **Gold neighbor packages last** (large each). `@flighthq/input-bindings`, `@flighthq/gestures`, and `@flighthq/gamepad-mappings` are independent cells built on the matured core. **Surface as design questions before building:** (a) confirm these belong as separate packages vs in-package (the review leans separate — agreed); (b) name and scope each neighbor with the user before creating package shape; (c) `gestures` depends on Bronze timestamps + Silver coalesced events, so it cannot precede them.
6. **Rust `flighthq-input` parity** tracks the TS core: do it after the `InputBackend` seam and the `InputState` snapshot stabilize, recording divergences in the conformance map.

Cross-package / decision items to raise with the user up front: the three neighbor-package names and boundaries (steps 5), the attach-vs-backend signature question (step 2), and the `input` vs `@flighthq/sensors` boundary (Gold). Everything in Bronze and most of Silver is self-contained to `input` + `@flighthq/types` and can proceed without further design sign-off.
