# Maturation Roadmap: @flighthq/power

**Current verdict**: solid — 70/100. Covers the two pillars (battery/charging status with change events, screen keep-awake) over a correct swappable backend seam, but omits several power signals that `powerMonitor` and mobile power plugins treat as core (on-battery source, idle/lock, real low-power mode, thermal state, battery time/health).

This is an event-style platform-integration cell, not an algorithmic sub-library, so "authoritative" means _complete coverage of the canonical power-monitoring surface_ (Electron `powerMonitor` + `powerSaveBlocker`, W3C Battery Status, Screen Wake Lock, Capacitor/Cordova battery, Tauri battery/power) — not endless features. Every addition below is additive to the existing `PowerBackend` seam and `Power` entity; no restructuring is required. All shared types land in `@flighthq/types/Power.ts` first, then the package implements against them.

## Bronze

The 20% that closes the most glaring, zero-design-risk gaps — all of which the underlying web APIs _already_ expose and the current code discards or never reads. Shippable, basic, no new backend concepts.

- **Surface battery time fields.** Add `chargingTime: number` and `dischargingTime: number` (seconds; `-1` / `Infinity`-mapped-to-`-1` sentinel when unreported) to `PowerStatus` in `@flighthq/types`. The web `getBattery()` manager already exposes `chargingTime`/`dischargingTime` and the current `createWebPowerBackend` reads only `level`/`charging` and throws the rest away. Wire the two existing `timechange` event listeners (`chargingtimechange`, `dischargingtimechange`) into the existing `subscribe` path.
- **Add `isOnBattery` (power source).** Add `isOnBattery: boolean` to `PowerStatus`. `isCharging` is _not_ a power-source read (a full battery on AC is not charging). The Electron backend already has `powerMonitor.onBatteryPower`; the web backend derives it from `charging === false && level >= 0` as a degraded heuristic (sentinel `false` when unknown). This is the single most-requested missing desktop signal.
- **`PowerKeepAwakeMode` kind for the two `powerSaveBlocker` modes.** Define `PowerKeepAwakeMode` as a `*Kind` string union seam: `'PreventDisplaySleep'` (the current screen wake-lock behavior, default) and `'PreventAppSuspension'`. Change `setKeepAwake(enabled)` → `setKeepAwake(enabled, mode?)` on the backend, and `setPowerKeepAwake(enabled, mode?)` in the package (default `'PreventDisplaySleep'` preserves current behavior). Web honors only display-sleep and returns `false` for app-suspension; Electron maps both.
- **`hasPowerKeepAwake(): boolean`** — report whether a keep-awake lock is currently held. The web backend already tracks `_wakeLockSentinel`; expose it rather than making callers track state. Also re-acquire the wake lock on `visibilitychange` (browsers auto-release it when the tab is hidden) so the lock survives tab switches — a real correctness bug in the current implementation.
- **Tests** for each new `PowerStatus` field and the keep-awake mode/`has` path, including the aliased-`out` case for `getPowerStatus`.

## Silver

Competitive with a well-regarded power library: the signals that make apps reach for a power module in the first place, plus real OS-flag reads instead of heuristics, and resolving the suspend/lifecycle ownership question.

- **System idle.** Add `getPowerSystemIdleTime(): number` (seconds since last input, `-1` if unsupported) and `getPowerSystemIdleState(thresholdSeconds): PowerIdleState` where `PowerIdleState` is a `*Kind` union `'Active' | 'Idle' | 'Locked' | 'Unknown'`. These are first-class `powerMonitor` capabilities (`getSystemIdleTime`, `getSystemIdleState`) and a primary reason to use a power module. Web returns `-1`/`'Unknown'` (no native idle API; the Idle Detection API is permission-gated and could be an optional fill later); Electron backs them directly.
- **Lock-screen events.** Add `onLockScreen` and `onUnlockScreen` signals to `Power`, plus `subscribeLockScreen`/`subscribeUnlockScreen` on `PowerBackend`. Electron's `lock-screen`/`unlock-screen` events back these; web returns no-op unsubscribers. Wire them into `attachPower`'s subscription teardown set alongside the existing suspend/resume wiring.
- **Real OS low-power / battery-saver flag.** Split today's homegrown `isLowPower` (a `level <= 0.2 && !charging` charge heuristic) into two honest fields: keep `isLowPower` as the _OS Low Power Mode / battery-saver_ flag (iOS Low Power Mode, Windows battery saver, GNOME power-saver) and add `isBatteryLow: boolean` for the charge-threshold heuristic. Add `onLowPowerModeChange` signal. Web has no real OS flag → `isLowPower` reads `false` (sentinel), `isBatteryLow` keeps the heuristic; native backends report the genuine flag. Document the distinction.
- **Resolve suspend/resume ownership with `@flighthq/lifecycle`.** (Surface as a design decision — see Sequencing.) The web backend maps `subscribeSuspend`/`subscribeResume` to page `freeze`/`resume`, which are tab-lifecycle events, not machine sleep/wake. Decide whether OS sleep/wake stays in `power` (native-honest, web-degraded) while app active/inactive/background moves to / stays in `lifecycle`. Document the boundary in both Package Map entries; rename if the boundary shifts (pre-release, no compat cost).
- **Native default backend (Rust-aligned).** Per the platform-suite pattern, ensure the web backend remains the ambient default in TS; the design note that native hosts replace it via `setPowerBackend` is already satisfied by `host-electron`. Confirm `host-electron` is extended to back every new field (`isOnBattery` already present; add idle/lock/low-power-mode wiring).
- **Tests** covering lock/unlock signal edges, idle-state thresholds, the `isLowPower` vs `isBatteryLow` split, and backend-swap fallback (set a fake native backend, assert delivery; `setPowerBackend(null)` restores web).

## Gold

Authoritative for the domain: exhaustive coverage, thermal awareness, battery health, full edge-case/error handling, and 1:1 Rust-port parity. Nothing a power-aware app or game would find missing.

- **Thermal state.** Add `PowerThermalState` `*Kind` union (`'Nominal' | 'Fair' | 'Serious' | 'Critical' | 'Unknown'`, mirroring macOS `NSProcessInfo.thermalState` and Android `PowerManager` thermal status) to `PowerStatus`, plus `getPowerThermalState()` and an `onThermalStateChange` signal. Web returns `'Unknown'`; native (winit/SDL host or a future thermal API) reports real pressure. This is increasingly expected in games and compute-heavy apps for dynamic quality scaling.
- **Battery health detail.** Add an opt-in `PowerBatteryHealth` value type (`createPowerBatteryHealth()` + `getPowerBatteryHealth(out): PowerBatteryHealth | null`): `capacityWearLevel` (0..1, design-vs-current capacity), `cycleCount`, `temperatureCelsius`, `voltage`, `healthState` (`*Kind`: `'Good' | 'Fair' | 'Poor' | 'Unknown'`). Kept as a separate struct/reader (not bloating the hot `PowerStatus` snapshot) since it changes rarely and is native-only. Returns `null` where unsupported (web).
- **`enablePowerSignals` group + thermal/health change signals.** Audit whether the signal set should move behind an `enablePowerSignals(power)` group per the project's "users opt into signal groups via `enable*`" rule — the entity is small enough that always-on may be fine, but document the decision. Add `onIdleStateChange` so idle is observable, not just pollable.
- **Performance / allocation discipline.** Confirm the single internal `_scratch` reuse stays the only allocation in the change path; ensure the new `timechange`/lock/idle subscriptions all funnel through one debounced emit so a burst of OS events does not fan out N `onChange`s. Add a coalescing guard if the native backends prove chatty.
- **Full edge-case handling.** Wake-lock re-acquisition across `visibilitychange` _and_ fullscreen-exit; `dischargingTime === Infinity` → `-1` normalization; permission-denied paths for Idle Detection (if adopted) returning sentinels, never throwing; idempotent double-`attach`/`detach`/`dispose` (already correct — keep tested).
- **`flighthq-power` Rust crate (1:1 parity).** Mirror the matured `PowerBackend` trait + `set_power_backend` in `flighthq-types`/`flighthq-power`, native default in-crate behind a `native` cargo feature (battery via `battery` crate / OS APIs, thermal via OS, idle via OS), `!Send`/async seam kept native-clean per the Rust host-layer rule, web fill in `host-web`. As an event capability it is _not_ mixable (carries the `Power` entity); native is the production story. Conformance: the snapshot fields and signal-edge semantics must match TS, with any intentional web-vs-native divergence recorded in the conformance divergence map.
- **Docs + examples.** A short usage example (poll `getPowerStatus`, attach for change events, keep-awake toggle, thermal-driven quality scaling) and a doc note clarifying every web-degraded sentinel so the optimistic suspend/resume/idle naming is honest on web.

## Sequencing & effort

Recommended order, with dependencies and items to surface:

1. **Bronze first — low effort, high value, no design risk.** Battery time fields, `isOnBattery`, keep-awake mode kind + `hasPowerKeepAwake`, and the `visibilitychange` wake-lock re-acquire fix are all small, self-contained, and exercise APIs already in hand (web manager fields, Electron `onBatteryPower`/`powerSaveBlocker`). Land `@flighthq/types/Power.ts` changes, then `power.ts`, then `host-electron/electronPower.ts`, then tests. Half a day.
2. **Silver, idle + lock + low-power split (medium effort).** Idle and lock-screen are native-backed (`host-electron` does the real work; web is sentinel/no-op), so the package change is mostly seam-widening and signal wiring into `attachPower`'s teardown set. The `isLowPower`/`isBatteryLow` split touches the web heuristic and is a small but breaking-shape change to `PowerStatus` (fine pre-release). 1–2 days including `host-electron` wiring and tests.
3. **Cross-package design decision — surface to the user before Silver lands:** the **suspend/resume vs. `@flighthq/lifecycle` ownership overlap**. The Package Map assigns app active/inactive/background/resume/pause to `lifecycle`; `power` currently exposes suspend/resume backed by page-freeze on web. This needs a deliberate boundary ruling (OS machine-sleep in `power`, app-lifecycle in `lifecycle`) before more sleep-adjacent surface is added, to avoid building twice. It is the only item here that crosses package boundaries and requires a decision rather than autonomous work.
4. **Gold (higher effort, native-heavy).** Thermal and battery-health are gated on native backends having real data sources — the Electron backend can do some (battery health is limited on Electron), but full thermal/health is most meaningful in the Rust native hosts. Do the TS type/seam design and web-sentinel behavior first so the surface is stable, then fill natively.
5. **Rust `flighthq-power` crate last**, after the TS seam is matured and frozen — it must mirror the _final_ `PowerBackend` shape, not chase it. Coordinate with the Rust host layer (`host-winit`/`host-sdl` for native power/thermal/idle, `host-web` for the web fill). Record web-vs-native sentinel divergences in the conformance map.

Dependencies: all type changes block their package/`host-electron` implementations (header-first). No new neighbor `-formats` package is warranted (power has no importable file formats). No new third-party TS dependency is needed for Bronze/Silver; Gold battery-health/thermal on native may pull an OS-binding crate in the Rust port only.
