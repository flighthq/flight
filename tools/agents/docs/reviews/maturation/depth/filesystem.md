# Maturation Roadmap: @flighthq/filesystem

**Current verdict**: solid — 72/100. A clean, complete realization of the platform-suite command-capability pattern (flat free functions over a swappable `FileSystemBackend`, real OPFS web default, exemplary sentinel discipline), but the seam is the canonical CRUD + stat + watch + well-known-paths set only — missing recursive traversal, ranged/streaming I/O, an explicit directory-removal verb, path utilities, and symlink/permission ops, which is what separates it from an authoritative file-system API.

The package's shape is correct and on-spec; almost all of the work below is widening the `FileSystemBackend` _interface_ in `@flighthq/types` (so native hosts can implement the capability, not just the web default), then exposing matching free functions and OPFS implementations where feasible. The depth review's four priorities — recursive listing, ranged read, explicit `removeDirectory` + `directoryExists`, and path utilities — map directly onto Bronze.

## Bronze

The 20% that closes the most glaring depth gaps. Shippable, backend-uniform, mostly implementable on the OPFS web default, no new packages.

- **Explicit directory verbs (split the conflated `removeFile`).** Today `removeFile` recursively deletes directories too and `renameFile` moves directories despite the `File` suffix. Add to `FileSystemBackend` and export:
  - `removeDirectory(path: string, recursive?: boolean): Promise<boolean>` — explicit directory removal; non-recursive default fails (`false`) on a non-empty directory, matching POSIX `rmdir`. Keep `removeFile` for files; OPFS implements both over `removeEntry({ recursive })`.
  - `directoryExists(path: string): Promise<boolean>` — the missing companion to `fileExists` (which resolves only file handles). OPFS implements via `getDirectoryHandle` walk. The depth review flags this asymmetry directly.
- **Recursive directory traversal (the headline gap).** Add to the seam + free functions:
  - `readDirectoryRecursive(path: string): Promise<readonly FileEntry[]>` — depth-first walk returning all descendants with full relative paths, `[]` sentinel. OPFS implements by recursing the existing `asAsyncEntries` walk.
  - Optional Bronze stretch: a `FileWalkOptions` plain-data descriptor in `@flighthq/types` (`Readonly<{ maxDepth?: number; followSymlinks?: boolean }>`) threaded into the recursive call rather than a second function, keeping the surface small.
- **Ranged binary read.** Add `readBinaryFileRange(path: string, offset: number, length: number): Promise<Uint8Array | null>` to the seam + free functions. OPFS implements over `file.slice(offset, offset + length)` — feasible on the web backend today, per the depth review. `null` sentinel for missing/denied; `-1`-style empty handling for out-of-range returns an empty `Uint8Array`, not an error.
- **Public path utilities.** A small, allocation-explicit, pure-string module exported from the root (no backend; works everywhere, no `'/'`-handbuilding):
  - `joinFilePath(...segments: readonly string[]): string`, `getFileDirectoryName(path: string): string` (dirname), `getFileBaseName(path: string): string` (basename), `getFileExtensionName(path: string): string` (extname), `normalizeFilePath(path: string): string`, `isAbsoluteFilePath(path: string): boolean`. Names carry the full type word per the project rule (no `dirname`/`basename` abbreviations). This promotes the already-internal `splitWebPath`/`normalizeWebPath` logic to public API. **Decide**: ship inside `@flighthq/filesystem` (Bronze) vs. a sibling `@flighthq/path` package (Silver) — see Sequencing.
- **Tests**: backend-roundtrip + web-sentinel + jsdom no-op case for each new export, plus the OPFS-specific recursion and slice paths, matching the package's existing colocated test norm.

## Silver

Competitive with a good cross-platform fs library (Node `fs/promises`, Deno `fs`, Tauri `fs`). Adds streaming, atomic writes, the symlink write side, and cross-backend consistency where the seam currently only exposes read-only metadata.

- **Streaming / chunked I/O** (large-asset path; everything buffers today). Add to the seam as plain-data handle descriptors in `@flighthq/types`, not method objects:
  - `openFileReadStream(path: string): Promise<ReadableStream<Uint8Array> | null>` and `openFileWriteStream(path: string): Promise<WritableStream<Uint8Array> | null>`. OPFS implements over `file.stream()` and `createWritable()` (which is already a `WritableStream`). `null` sentinel when unavailable. Standard Web Streams keep the boundary plain and avoid a bespoke handle type.
  - Convenience `writeBinaryFileChunks(path, chunks: AsyncIterable<Readonly<Uint8Array>>): Promise<boolean>` for the common "write a large payload without holding it all" case.
- **Atomic write helper.** `writeFileAtomic(path: string, data: Readonly<Uint8Array> | string): Promise<boolean>` — write to a temp sibling then rename into place (the standard durability primitive). OPFS implements via temp file + the existing copy/remove rename; native hosts use real `rename`. Documented as best-effort on web (OPFS rename is copy+remove, not atomic).
- **Symlink write side** (the seam currently exposes `FileStat.isSymlink` with no operations to produce or follow it). Add to `FileSystemBackend` + free functions:
  - `createFileSymlink(target: string, linkPath: string): Promise<boolean>`, `readFileSymlink(path: string): Promise<string | null>` (the link target), `getFileRealPath(path: string): Promise<string | null>` (realpath, resolving symlinks). Web backend returns sentinels (OPFS has no symlinks) and documents them as native-only — but they belong on the _interface_ so native hosts are reachable, per the depth review's "omissions from the seam, not just the web backend" note.
- **Permissions / access mode.** Add `getFilePermissions(path): Promise<FilePermissions | null>` and `setFilePermissions(path, mode): Promise<boolean>` over a plain-data `FilePermissions` type in `@flighthq/types` (`Readonly<{ readable: boolean; writable: boolean; executable: boolean; mode?: number }>`), plus `canAccessFile(path, mode): Promise<boolean>` (POSIX `access`). Web returns sentinels; native (chmod/access) implements. Add `readonly` / mode fields to `FileStat` where a native host can populate them.
- **Glob / pattern matching.** `findFiles(rootPath: string, pattern: string): Promise<readonly FileEntry[]>` — glob (`**`, `*`, `?`) over the recursive walk. Pure pattern-matching can live in the path module (Bronze) and compose with `readDirectoryRecursive`, so this is mostly free-function glue over Silver's streaming-free pieces.
- **Disk / quota introspection.** `getFileSystemUsage(path?: string): Promise<FileSystemUsage | null>` over a `FileSystemUsage` plain type (`Readonly<{ usedBytes: number; quotaBytes: number }>`). OPFS implements over `navigator.storage.estimate()`; native over statvfs.
- **Web backend hardening**: respect the Permissions API and persistent-storage request (`navigator.storage.persist()`), and de-duplicate the many `getWebRoot()`/try-catch blocks behind shared internal helpers as the seam grows.

## Gold

Authoritative / AAA. Exhaustive coverage, performance, full edge-case + error handling, the file-watch _event_ capability done properly, the importer/parser neighbor, and 1:1 Rust-port parity.

- **File-watch as a first-class event capability.** Today `watchPath` is a bare command-style callback that no-ops on web. Promote to the suite's event-capability shape (mirror `@flighthq/network`): a `FileSystemWatch` entity of signals in `@flighthq/types` with `create*`/`attach*`/`detach*`/`dispose*`, recursive-watch support (`watchPathRecursive`), debounce/coalesce options, and rename-detection (delivering `moved` events, not delete+create). Gate behind an `enableFileSystemWatch`-style group so a non-watching import tree-shakes the signal code. Web stays a documented no-op; native delivers real events.
- **Locking & transactional writes.** `lockFile(path): Promise<FileLock | null>` / `releaseFileLock(lock)` (advisory locks, `acquire`/`release` bracket semantics), and OPFS `createSyncAccessHandle`-backed synchronous high-throughput access for worker contexts (`openFileSyncAccess(path)`), which is the OPFS-native fast path.
- **Watched-tree + bulk operations.** `copyDirectory(from, to, recursive)`, `moveDirectory`, `emptyDirectory`, `getDirectorySize(path)` (recursive byte sum), and a `walkFileTree(rootPath, visitor)` callback walk for streaming traversal of huge trees without materializing a `FileEntry[]`.
- **`@flighthq/filesystem-formats` neighbor** (the `-formats` pattern) for archive/container importers that are file-system-shaped but parser-heavy: zip/tar read+write as a virtual `FileSystemBackend` (mount an archive as a backend), so `setFileSystemBackend(createZipFileSystemBackend(bytes))` exposes an archive through the same free-function surface. Keeps the core package parser-free and tree-shakable.
- **Full edge-case + error sweep**: path normalization across platforms (Windows `\` vs `/`, drive letters, UNC), symlink-loop detection in recursive walks, large-file streaming back-pressure, partial-write recovery, encoding options on text I/O (`readTextFile(path, encoding)` beyond UTF-8), BOM handling, and a documented capability matrix (`getFileSystemCapabilities()` returning the supported op/feature set per active backend — streaming, symlinks, locks, watch, permissions).
- **Tests + docs**: roundtrip + sentinel + alias cases for every export, OPFS sync-access-handle tests in a worker harness, watch attach/detach/dispose lifecycle tests, glob/normalization unit tables, and a `tools/agents/docs` note on the per-backend capability matrix and the watch command→event migration.
- **Rust-port mirror — `flighthq-filesystem`** with the same seam: a `FileSystemBackend` trait in `flighthq-types`, `set_filesystem_backend`, and a **native-default backend gated behind the `native` cargo feature over `std::fs`** (the host-layer "native ambient default" rule explicitly names `filesystem` for std — so unlike the web-first TS default, Rust ships a real fs backend in-crate, no host required). Free functions `read_text_file` / `read_binary_file_range` / `read_directory_recursive` / `write_file_atomic` / `create_file_symlink` / `get_file_real_path`, snake_case with full type words, `Option`/`bool` sentinels, alias-safe out-params, `&[u8]` borrows for write inputs. Streaming maps to `Read`/`Write` traits; watch maps to a `notify`-backed native backend. Conformance-mapped 1:1 against the TS exports. The path module is a pure value-typed leaf and a clean mixing candidate.

## Sequencing & effort

Recommended order, with dependencies and items to surface:

1. **Bronze, path utilities first** (small, ~half day, pure strings, no backend). Everything recursive/glob downstream composes with these, and they have no `@flighthq/types` seam change — promote the existing internal `splitWebPath`/`normalizeWebPath` to a public module. **Design decision to surface**: ship in `@flighthq/filesystem` now vs. a sibling `@flighthq/path` package. A sibling is the more authoritative shape (path utilities are not fs-specific and other packages — resources, loader — likely want them), but adds a package; recommend Bronze-in-`filesystem`, Silver-extract-to-`@flighthq/path` if a second consumer appears.
2. **Bronze, directory verbs + `directoryExists`** (small). Pure additions to the seam + OPFS; no reshape of existing functions (keep `removeFile` for files, add `removeDirectory`). Touches `@flighthq/types` first.
3. **Bronze, recursive listing + ranged read** (small–medium). Both implementable on OPFS today; `FileWalkOptions` and the recursive walk in `@flighthq/types` first, then web backend, then tests.
4. **Silver, streaming I/O** (medium). Uses standard Web Streams to keep the boundary plain; OPFS `file.stream()` / `createWritable()` map directly. Do before atomic-write so `writeFileAtomic` can reuse the write path.
5. **Silver, atomic write + glob** (small once streaming + path utils land — both are glue over earlier tiers).
6. **Silver, symlink + permissions** (medium; mostly seam + native, web is sentinels). These are the depth review's "add to the _interface_ so native hosts are reachable" items — the value is unlocking native backends, so land the `@flighthq/types` additions even though the web default no-ops them. Add the `FilePermissions` / extended `FileStat` types in the same pass.
7. **Silver, usage/quota** (small; OPFS `estimate()` + native statvfs).
8. **Gold tiers** (large, ordered): watch event-capability reshape → locking/sync-access → bulk dir ops → `-formats` archive neighbor → capability introspection + edge sweep → Rust mirror. The Rust crate should track the TS seam only after Silver settles, to avoid re-porting a moving contract.

**Cross-package / design-decision items to surface (not autonomous):**

- **`@flighthq/path` extraction** (step 1) — whether path utilities live in `filesystem` or a shared sibling. Cross-package decision; check for other would-be consumers (`resources`, `loader`) before deciding.
- **`@flighthq/signals` dependency** for the Gold watch event-capability — acceptable per the codebase map ("effectively always present"), but it adds a dependency to a currently `@flighthq/types`-only package; gate the watch entity behind an `enable*` group so a read/write-only import still tree-shakes the signal code.
- **The `watchPath` command → `FileSystemWatch` event reshape** (Gold) demotes/replaces the current bare-callback `watchPath`; confirm before any native `host-*` backend commits to the old `watch(path, listener)` contract. The depth review notes watch is currently web-no-op, so the reshape has no live web consumers to break.
- **`removeFile` recursive-directory behavior**: Bronze adds explicit `removeDirectory`, but decide whether `removeFile` should _stop_ recursing into directories (the crisp POSIX split) or keep its current convenience behavior. A pre-release API reshape is cheap now and harder once native hosts exist.
- **The `-formats` archive-as-backend pattern** (Gold) is a larger design — mounting a zip/tar through `FileSystemBackend` is elegant but stretches the "host capability seam" framing toward a virtual-fs abstraction; surface as a proposal before building.
