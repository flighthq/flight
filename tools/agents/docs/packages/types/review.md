## Verdict

`solid — 88/100`. The single strongest expression of the codebase's design rules: ~503 one-concept-per-file modules covering the entire declared SDK surface, applied with rare uniformity (quartets, open contracts, string kinds, `*Like` strips, full unabbreviated names, a clean alphabetized barrel with zero runtime dependencies). The +6 over the prior depth review's 82 is for the Bronze tier landing in full and _verified against source_: the placeholder `missing.test.ts` is gone, eight genuine type-level assertion test files now exist (`expectTypeOf` / `@ts-expect-error`, typechecked by `tsc -b`), the `DOM`→`Dom` acronym drift is fixed (files renamed, barrel updated, old files absent), and `ParticleForce`/`ParticleCollider` are settled as closed-by-design with a non-provisional rationale citing the spec. It stays `solid` rather than `authoritative` because the header's three load-bearing promises are still prose, not mechanism: self-containment ("navigable from the header alone, no impl imports") is unenforced, the built-in kind vocabulary is not navigable from the header (`KindOf`/`KnownKinds` deferred), and the Rust `Signal<T>` conformance seam is knowingly divergent. The score is distance-to-authoritative, not a grade on the diff.

## Gaps (vs the authoritative header bar; charter silent, codebase-map standard applies)

- **Self-containment is unenforced.** The header's defining promise — "navigable from `@flighthq/types` alone, without importing any implementation package" — has no mechanism. Nothing asserts no `types` file transitively imports a `@flighthq/<impl>` package, nor that every cross-package type actually lives here. Both directions ("nothing extra leaked in" / "nothing missing") are prose. This is the largest remaining gap and is the difference between a header that _is_ closed and one that _happens to be_ closed today.
- **Built-in kind vocabulary is not navigable from the header.** `Kind = string` is correctly open, but there is no `KindOf<TEntity>` mapped type resolving an entity to its `*Kind` literal and no `KnownKinds` union assembled from the built-in constants — so the built-in vocabulary cannot be enumerated or machine-checked from the header (confirmed absent: no `KindOf`/`KnownKinds` in src). A reader must grep ~500 files to learn the built-in kinds.
- **`Signal<T>` is function-typed, diverging from the Rust port.** `Signal.ts` still parameterizes by the slot function type (`T extends (...args: any[]) => void`) and leans on `any` (file-level `eslint-disable no-explicit-any`). The locked Rust decision is `Signal<T>` by _payload_. The two header layers describe the same seam with different generic shapes — a conformance seam that is knowingly less than 1:1. There is no `Signal.test.ts`, and the per-subsystem signal-group payloads (`NodeSignals`, `StageSignals`, `InteractionSignals`, `MovieClipSignals`, `InputSignals`, …) have no assertion coverage.
- **No scene serialization / versioning contract.** The types-layout spec describes "scenes are versioned intent; migrate at load," but no `SceneDocument` / `SceneVersion` / `SceneMigration` types exist (confirmed absent). The round-trip seam the whole SDK would serialize through is a documented intent with no type.
- **No branded primitives for the value conventions.** The SDK relies on `0xRRGGBBAA` packed colors and a single angle convention, but there is no `PackedRgba` brand, no `Radians`/`Degrees` distinction. Misuse (passing an unpacked color, mixing angle units) is invisible to the type system at every callsite SDK-wide. High value, high blast radius — a deliberate gap, not an oversight.
- **Assertion coverage is invariant-bearing only, not exhaustive.** Eight tests cover the highest-value invariants; dozens of entity quartets' `*Like` strips, every open contract's narrowing, every `*Backend` option/return shape, and every math `*Like` runtime-strip are not individually asserted. For a header this is acceptable triage, but `authoritative` wants the invariant-bearing concepts covered and a check that prevents silent regression as types are added.
- **No 1:1 conformance lock with `flighthq-types` (Rust).** Nothing asserts every concept here has a named Rust counterpart or an explicit divergence-map entry. The `Signal` divergence above is the live example with nowhere committed to record it.

## Contract & docs fit

**Lives up to the contract — exemplary.** Full unabbreviated type words throughout (`StandardPbrMaterial`, `DomRenderState`, no abbreviations); filename = primary exported type in exact PascalCase casing, now including the fixed `DomRenderOptions.ts` / `DomStageRectangle.ts`; flat layout with no category folders; one concept per file with quartets never split; string-kind identity uniform (`export const BitmapKind = 'Bitmap'`) serving as registry key + serialized form + intent vocabulary; open base contracts where extension is a goal, closed unions only where a hot finite family justifies it (`ParticleForce`/`ParticleCollider`, both now documented as deliberate); single `.` export, `sideEffects: false`, zero runtime dependencies; `crate: flighthq-types` mirror named in the charter front matter. The internal `Symbol()` uses (`EntityRuntimeKey`, `*TraitsKey`) are correctly symbols (runtime privacy, never serialized) while all serialized kinds are strings — the spec's symbol-vs-string split is honored exactly.

**Defects / candidate revisions:**

- **`exports:check` vs. a type-only package is an unresolved tension.** The contract mandates one colocated test per source file and runs `exports:check`. This package has 8 test files for ~503 source files. The right answer is the one the worker took — pure declaration files (`BloomEffect`, `PointLight`, enum/option files) have no behavior to assert, so they are legitimately test-free — but nothing _records_ that exemption, so a future `exports:check` would either flag ~495 "missing" tests or has a `types`-specific carve-out that is undocumented. Flag for the tooling/contract owner: the header layer needs an explicit "invariant-bearing concepts carry tests; pure declarations do not" rule, not a per-file mandate. Candidate contract revision.
- **`Signal<T>` `no-explicit-any` is a real, file-level escape hatch.** `Signal.ts` disables `@typescript-eslint/no-explicit-any` wholesale. Defensible for a function-typed dispatcher, but it is the one place the header reaches for `any`, and the deferred payload-reshape would remove it. Recording the divergence (and the lint exception) in a conformance/divergence map is the missing step. Candidate revision for the conformance-map owner.
- **Status doc drift (informational, not a code defect).** The ingested status entry claims `ParticleForce` has 5 members and its test covers 5; the source HEAD has 7 (`CurlNoiseForce`, `PointGravityWellForce` added) and the test covers all 7 correctly. The worker report is a mid-session snapshot that drifted below the realized source — the _code is more complete than the report_. Likewise the report's "Silver partial" framing undersells how much landed. Verified: the source is correct; the status prose is stale. No action on the code; noted so a later agent trusts source over the report.
- **Package Map line is accurate** — "shared interfaces, kind symbols, and cross-package type contracts … the codebase's header layer" matches the code (modulo the wording "kind symbols", which post-dates the string-kind decision; kinds are strings, not symbols — a one-word Map-line staleness worth fixing). Candidate revision for the codebase-map owner.

## Candidate open directions (charter is a stub — the questions it should settle)

1. **North star.** What is the durable bar for the header? Likely: _the full SDK API shape is navigable and pinnable from `@flighthq/types` alone, with no impl import_ — and the corollary that this is _mechanically enforced_, not promised. Confirm so future work is judged against it.
2. **Self-containment: test or `packages:check` rule?** Both directions (header imports no impl; every cross-package type is declared here) should be enforced. Decide whether that lives as a `headerClosure.test.ts` or a graph-walk rule in `packages:check` (the worker leans toward the latter). This is the promise that defines the package.
3. **Test policy for a type-only package.** Bless the rule that invariant-bearing concepts carry a colocated type-level test while pure declaration/variant files do not, and reconcile it with `exports:check` so the carve-out is recorded, not incidental.
4. **`Signal<T>` shape and Rust conformance.** Reshape TS `Signal` to payload-parameterized to match the locked Rust `Signal<T>` (a cross-package change touching every `*Signals` group and `enable*` site), or formally record the divergence in the conformance map. Either way the seam should be explicitly 1:1-or-documented, not silently divergent.
5. **Kind vocabulary as first-class.** Add `KindOf<TEntity>` + a `KnownKinds` union (keeping `Kind = string` open) and a `VendorKind` brand for third-party authors, so the built-in vocabulary is navigable and checkable from the header.
6. **Scene serialization / versioning contract ownership.** Decide whether `SceneDocument` / `SceneVersion` / `SceneMigration` are declared here (and which package owns load/save/migration), turning the types-layout "versioned intent, migrate at load" prose into the canonical seam.
7. **Branded primitives.** Whether the SDK adopts `PackedRgba` (and `Radians`/`Degrees`) brands declared once here. High value, high blast radius — needs a deliberate go-ahead before any sweep.
8. **1:1 conformance lock with `flighthq-types` (Rust).** Whether a conformance manifest the Rust checker reads becomes a gate, so a TS type added without a Rust counterpart (or a divergence-map entry) fails CI.
