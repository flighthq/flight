## Recommended

Strictly sweep-safe: within `@flighthq/clip`, no cross-package coupling, no breaking change, no open design decision. Safe to bless as a set.

- **Correct the dishonest `intersectClipRegions` contour∩rect comment.** The comment "Contours ∩ rect: keep contours, clipped bounds computed above" (`clipRegion.ts:207-209`) implies the kept contours are clipped, but line 209 copies `aContours` verbatim (`aContours.map((c) => c.slice())`); only `out.rect` is intersected (198–201). So `out.contours` can describe geometry outside `out.rect` — an internal inconsistency the comment misrepresents. Reword to: bounds clipped, contours kept verbatim (conservative). **Do not change the algebra in this delta** — the conservative-bounds behavior is the parked boolean-algebra fork (Backlog); only the comment is in scope. Pure in-source doc fix, no signature change. (review.md MINOR — "leaves contours wider than the stated bounds rect".)
- **Reword the package `description` from product to library.** `package.json` still reads "ClipRegion: hard geometric clip product built from rectangles or paths" — a pre-expansion framing. Post-expansion the package is a clip _operations_ library (composition / queries / transform / pool); the description undersells it. The review confirms this is an `origin/main` carry-over (no `package.json` hunk in the delta), so it is **not** a merge blocker — flagged only as a post-merge metadata reword. Pure metadata, no code change. (review.md — overturned-as-merge-objection carry-over; charter Open direction #9.)

## Backlog

Parked: each waits on a cross-package coordination, a breaking types-layer change, or an Open direction the charter must settle first. Not eligible for blanket approval.

- **`clipRegionContainsRectangle` contour-form false positive (parked: needs a blessed correctness ruling — and it gates the merge).** On a concave/holed contour it returns `true` when the rectangle is inside the bounding box but outside the actual region (`clipRegion.ts:44-46` returns `enclosesRectangle(clip.rect, rectangle)`, ignoring `clip.contours`) — the _unsafe_ direction for a `contains` predicate (a culling / interaction consumer that trusts a `true` could skip a needed clip). Its sibling `clipRegionIntersectsRectangle` (50–52) over-claims in the _safe_ direction for a may-intersect query, so the two predicates have opposite safe directions and the `contains` form got it backwards. The mechanically-safe fix is small and in-cell — for the contour form return `false` (honest under-claim) unless an exact polygon-contains is implemented, and correct the line-43 comment to state the contour answer is a conservative _false_, not a conservative _true_; add a contour false-positive test (the existing test at `clipRegion.test.ts:104-114` only covers the rect form). But **whether `contains` must be exact, must under-claim, or may stay approximate is a correctness-contract decision** the review surfaces as charter Open direction #2. Routed there so the cleanup and the contract land together rather than baking in an unblessed semantics. **Note: the review treats this as a MUST-FIX merge gate** — see the dispatch brief — but the _direction_ of the fix is the charter's to bless before it lands on the approved floor.
- **`createClipRegionFromContours` ownership: clone vs documented borrow (parked: needs a blessed ownership ruling — and it gates the merge).** It captures the caller's array by reference — `return { contours, rect, version: 0, winding }` (`clipRegion.ts:106-110`), asserted by its own test `expect(clip.contours).toBe(contours)` (`clipRegion.test.ts:231`) — while every sibling producer clones (`createClipRegionFromRectangle` via `cloneRectangle` 134; `cloneClipRegion`/`copyClipRegion` `.map(c => c.slice())` 82,91; intersect/union deep-copy kept contours 209,218,413,420). The signature also takes mutable `number[][]`, not `Readonly<>`, advertising a borrow the contract forbids by default; a later caller mutation silently corrupts the region with no `invalidateClipRegion` bump. Two valid resolutions: clone for symmetry (`contours.map((c) => c.slice())`) and accept `Readonly<readonly number[][]>`, OR document the by-reference capture as a deliberate zero-copy fast path in a durable comment and keep the param mutable — either way the test must assert the chosen ownership. Because it is a fork between two acceptable shapes, it is the charter's Open direction #4 to bless, not an autonomous sweep. **Note: the review treats this as a MUST-FIX merge gate** — the silent asymmetry is not the final shape — so it must be resolved (in either direction) before the merge lands.
- **Exact boolean algebra — the defining gap (parked: cross-package design fork).** `intersectClipRegions`/`unionClipRegions` are bounds-plus-one-input conservative on any contour form; there is no `subtractClipRegions`, no `xorClipRegions`, no true contour intersection/union. This is the single largest thing between `solid` and `authoritative`, but it needs a polygon-clipping kernel (Vatti / Weiler–Atherton / Martinez–Rueda) that does not exist anywhere in the monorepo (`@flighthq/path` has only `flatten`/`tessellate`). Per **structural-fork A** (source-data vs. graph participation) and the **bedrock test**, the kernel almost certainly belongs in `@flighthq/path` or a new `@flighthq/path-boolean` neighbor, with `clip` composing the _region semantics_ over the _geometry kernel_ — not embedding a clipper in this leaf. Routed to charter Open direction #1: "where does exact boolean algebra live, and is it in scope for `clip` at all, or is conservative-bounds the permanent contract because the renderer's stencil-then-cover realizes the true geometry anyway?" Not a merge blocker — the conservative algebra stays in this delta. Do not act autonomously.
- **Contour storage `number[][]` → `Float32Array` (parked: breaking, cross-cutting types change).** The roadmap's Gold perf item — flat typed-array contours for cheap transform / GPU upload, removing the per-point `number[]` allocation in `transformClipRegion` (375–386) and the `.map(c => c.slice())` deep copies (82,91,209,…). This is a breaking change to `ClipRegion.contours` in `@flighthq/types` that coordinates with every backend clip module. A types-layer decision (charter Open direction #3) to settle _before_ the Rust port locks the seam, not in-cell work.
- **Winding helpers / normalization ownership (parked: cross-package ownership decision).** `getClipRegionWinding`, explicit-winding constructors, and even-odd↔non-zero conversion are absent; winding correctness lives entirely in backends today. Whether `clip` owns the conversion (vs. `path` or the backends) is charter Open direction #5 — a boundary decision, not a sweep item.
- **Functional / visual parity test (parked: cross-backend, scoped against the conservative-bounds contract).** No scene exercises nested `intersectClipRegions` across Canvas/DOM/WebGL to confirm the descriptor's bounds match what each backend actually clips — exactly the drift jsdom unit tests cannot catch. Parked rather than recommended because what it should assert depends on the boolean-algebra and containment rulings above (testing conservative bounds as the contract vs. exact geometry); author it once those are blessed so the baseline is not rewritten. (review.md Cross-package gaps; roadmap Gold tests-and-docs.)
- **Rust `flighthq-clip` crate (parked: separate worktree, sequenced after the TS seam stabilizes).** The charter front matter declares `crate: flighthq-clip`; it does not exist. `clip` is a value-typed wasm-mixable leaf (structural-fork D) and a strong early Rust↔TS conformance target, but the port should mirror a _stable_ surface — naturally after the storage (`Float32Array`) and boolean-kernel decisions land, since the crate inherits both. Track as TS-ahead-of-Rust in the register / conformance map.

## Notes for the charter's Open directions

Surfaced for an explicit direction conversation (do not edit the charter here). The two MAJOR merge-gate objections are not just gaps — they are the charter's existing Open directions #2 and #4, which is the signal to bless the ruling _before_ this lands on the approved floor, since the floor is what subsequent work conforms to:

1. **Open direction #2 — containment correctness.** Bless the rule for `clipRegionContainsRectangle` on contour forms: must it under-claim (return `false` when unsure — the safe direction), or is an exact polygon-contains required? The fix direction follows the ruling; the merge gate needs at minimum the honest under-claim.
2. **Open direction #4 — `createClipRegionFromContours` ownership.** Choose clone-for-symmetry (the conventional allocation-on-`create*` direction, `Readonly<>` param) or a documented deliberate zero-copy borrow. The merge gate needs the silent asymmetry resolved either way.
3. **Open direction #1 — where exact boolean algebra lives** (path-boolean kernel in `@flighthq/path` / a neighbor vs. conservative-bounds as the permanent contract). The largest fork; gates `subtract`/`xor` and the `solid → authoritative` step.
4. **Open direction #3 — contour storage** (`number[][]` vs `Float32Array`); a breaking `@flighthq/types` change to settle before the Rust seam locks.
5. **Open direction #5 — winding ownership** (does `clip` own conversion / explicit-winding constructors, or do backends?).
6. **Open direction #9 — package `description` reword** from "product" to clip _operations_ library once the North star is blessed (pure metadata, post-merge).
