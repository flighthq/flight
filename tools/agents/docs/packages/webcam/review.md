# webcam — Review

Evidence: `incoming/builder-67dc46d64/head/packages/webcam/` + the types under `incoming/builder-67dc46d64/head/packages/types/src/Webcam*.ts`. Findings reference `67dc46d64:<path>`. This survey supersedes `reviews/depth/webcam.md` (the prior depth review, which scored the package `partial — 45/100` _before_ the live-stream build-out); the bundle is the work that closed most of that review's headline gap.

## Verdict

`solid — 74/100`. The build-out the prior depth review and maturation roadmap asked for has largely landed: a full live-stream dimension (`startWebcamStream` / `destroyWebcamStream` / `grabWebcamFrame` into a `Surface`), device enumeration, the `MediaTrackCapabilities`/`applyConstraints` surface, a real 4-state permission query, an opt-in signal group, `MediaRecorder` recording, `ImageCapture` still capture, and honest dimension/duration decode on the file-input path. This is the most complete the package has ever been and it is genuinely Bronze-plus-most-of-Silver against the roadmap. The 74 (below the worker's self-estimated 78) reflects this review's stricter distance-to-authoritative bar: a clutch of copy-paste/stale doc errors in `@flighthq/types`, two internal symbols leaking through the package barrel, a `Readonly` entity field mutated by a cast, the known per-call `OffscreenCanvas` allocation in the frame-grab hot path, and a charter that is still a stub — so most of "what good means here" (chiefly: _is the file-input picker and the live stream one package or two?_) is assumed, not confirmed. The code is solid; the score measures the remaining road, not the diff.

## Present capabilities (verified against source)

**Backend seam** (`webcam.ts`). The textbook platform-suite command shape: `getWebcamBackend` (lazily creates the web default), `setWebcamBackend(backend | null)`, and `createWebWebcamBackend()` for granular use. The web backend is a single object literal implementing all 11 `WebcamBackend` methods. `WebcamBackend` and `WebcamBackendEvent` are homed in `@flighthq/types` (`Webcam.ts`).

**One-shot capture & pick** (`webcam.ts`). `takeWebcamPhoto`, `pickWebcamImage`, `recordWebcamVideo` over a transient `<input type="file">` (`capture="environment"` only when `source === 'camera'`), `FileReader` → data URL, `null` sentinel on absent-document / cancel / read-error. Photo dimensions are now decoded via an `Image` element and video duration via a transient `<video>` (`onloadedmetadata`) — the dishonest `0`s the depth review flagged are fixed (`67dc46d64:webcam.ts:86-93,134-143`). `requestWebcamPermission` (boolean) and `getWebcamPermissionState` (4-state `WebcamPermissionState`) over the Permissions API, both honestly documented as state-only (the web path cannot prompt).

**Live stream** (`webcam.ts` + `webcamStream.ts`). `startWebcamStream(options?)` opens `getUserMedia`, populates the `WebcamStream` entity from the negotiated `MediaTrackSettings`, attaches a muted auto-playing `<video>` to the runtime for frame grabbing, and wires the track `'ended'` event to flip `active`. `destroyWebcamStream` stops all tracks. `grabWebcamFrame(stream, out: Surface): boolean` draws the current video frame through an `OffscreenCanvas`/`<canvas>` and `out.data.set(...)`, returning `false` on inactive/not-ready/size-mismatch. `takeWebcamStreamPhoto` prefers the `ImageCapture` API with a canvas fallback. `getWebcamPhotoSurface(photo, out)` decodes a stored `dataUrl` into a caller-provided `Surface` (with a 500 ms jsdom timeout guard). The `WebcamStream` entity carries the entity/runtime split correctly — `createWebcamStreamEntity` allocates with an opaque runtime slot (`mediaStream` + `videoElement`).

**Devices, capabilities, constraints** (`webcamDevice.ts`, `webcam.ts`). `getWebcamDevices(): Promise<readonly WebcamDevice[]>` (video + audio inputs, `[]` when unavailable). `getWebcamStreamCapabilities(stream): WebcamCapabilities | null` maps `MediaTrackCapabilities` into the typed `WebcamCapabilities` (zoom/torch/focus/exposure/white-balance/frame ranges via `mapWebcamCapabilities`/`mapCapabilityRange`). `applyWebcamStreamConstraints(stream, constraints)` maps `WebcamConstraints` into a single `applyConstraints({ advanced: [...] })` call, `false` on out-of-range/unsupported.

**Recording** (`webcamRecording.ts`). `startWebcamRecording(stream): WebcamRecording | null` (a `MediaRecorder` over the live stream, `null` when MediaRecorder is absent — jsdom — or start fails) and `stopWebcamRecording(recording): Promise<WebcamVideo | null>` (decodes duration, reports `hasAudio` from the actual audio tracks, encodes to a data URL, revokes the object URL). `mimeType` selection negotiates vp9/vp8/webm/mp4. `WebcamRecording` is a proper entity with a runtime slot.

**Signals** (`webcamSignals.ts`). The event-capability lifecycle: `enableWebcamSignals()` allocates an inert `WebcamSignals` group (`onWebcamDeviceChange`/`onWebcamPermissionChange`/`onWebcamStreamEnded`), `attachWebcamSignals` subscribes to the backend `subscribe` seam and forwards by event kind, `detachWebcamSignals` (idempotent) and `disposeWebcamSignals` tear down. A module-level `WeakMap` holds the per-signals unsubscribe — correct opt-in-cost shape.

**Types & packaging.** Eleven `Webcam*` types in `@flighthq/types`, all exported from its barrel: `Webcam` (source/options/photo/video/backend/event), `WebcamStream`, `WebcamRecording`, `WebcamDevice`, `WebcamFacingMode`, `WebcamPermissionState`, `WebcamStreamOptions`, `WebcamConstraints`, `WebcamCapabilities`, `WebcamCapabilityRange`, `WebcamSignals`. `package.json` is `sideEffects: false`, single `.` export, deps limited to `entity` / `signals` / `types`. Notably the `Surface` out-param functions import only the `Surface` _type_ — there is **no runtime `@flighthq/surface` dependency**, so the bundle stays lean (the roadmap's bundle concern is avoided). Tests: 44 `it`s across 5 colocated `*.test.ts`, `describe` blocks mirror exported names 1:1 (status claims this lifecycle works against the jsdom sentinel paths — verified the files exist and exercise the sentinel branches).

## Gaps (vs the AAA camera-capture target; charter mostly silent, codebase-map standard applies)

- **Convenience setters absent.** No `setWebcamTorch` / `setWebcamZoom` / `setWebcamFocus` / `setWebcamExposure` / `setWebcamWhiteBalance`. The roadmap listed these as Silver; the worker deliberately omitted them because `applyWebcamStreamConstraints` covers the use case (see Charter contradictions — this is a defensible call, not a defect, but it is a named API-surface gap a Capacitor-class library would expose).
- **Frame-grab allocates per call.** `grabFrame` constructs a fresh `OffscreenCanvas`/`<canvas>` every call (`67dc46d64:webcam.ts:223-244`) — fine for a one-shot snapshot, a real regression for a per-frame loop. No `startWebcamFrameLoop` / `requestVideoFrameCallback` path, no reused scratch canvas, no pooled-surface bracket. Roadmap Gold.
- **No live `MediaTrackSettings` round-trip.** Capabilities (ranges) are exposed, but there is no `getWebcamStreamSettings(stream)` reporting the _actual_ negotiated zoom/torch/focus the way the spec distinguishes capabilities-vs-settings.
- **No burst / sequence capture** (`takeWebcamPhotoSequence`, `grabWebcamFrames(stream, count, out[])`). Roadmap Gold.
- **No still metadata beyond dimensions.** `WebcamPhoto` has `dataUrl`/`format`/`width`/`height` but no `orientation`/EXIF, `timestampMs`, byte `size`, or `pixelFormat`. The `webcam-formats` neighbor (EXIF/HEIC) is correctly deferred under the `-formats`/plurality guard.
- **Advanced ImageCapture axes unexposed** — `pan`, `tilt`, `pointsOfInterest`, `colorTemperature`, `iso`, `exposureCompensation` setters. Roadmap Gold.
- **Error/edge coverage is the happy-path-plus-sentinel set.** `OverconstrainedError`, device-in-use, hot-unplug mid-stream, permission-revoked-mid-stream are not surfaced as distinct events/sentinels beyond the generic `try/catch → null`. The `subscribe` seam only ever emits `DeviceChange` on the web backend; `PermissionChange` and `StreamEnded` are wired in the signal forwarder but the web backend never produces them (a native host would).
- **No native host backend and no Rust crate.** `host-electron` webcam and the `flighthq-webcam` crate are both out-of-worktree and correctly deferred. No functional-test scene yet (a grabbed frame rendered into a `Surface`).

## Charter contradictions

The charter's "What it is" is the only non-stub section, and it materially **disagrees with the code the bundle shipped.** The charter (seeded from the depth review) describes the package as "take a photo, record a video, and pick an existing image … in the mold of Capacitor `Camera` / Cordova `camera`" — i.e. the one-shot picker reading. The shipped package is much larger than that: it is now a full live-streaming webcam library (getUserMedia, frame grab, device enumeration, constraints, recording, signals). This is not a _violation_ of a blessed rule (North star / Boundaries / Decisions are all `TODO`), but the charter's stated identity now _under_-describes the implementation — the exact inverse of the `shape` case. The `package.json` description ("Webcam capture, photo picking, and live streaming over a swappable web/native backend") already reflects the larger reading; the charter has not caught up. The first job of a direction session here is to reconcile this: bless live streaming as in-scope (which the code, the roadmap, and the Rust-port `webcam` rename — explicitly "not photo capture, which is now `webcam`" — all assume), or split it.

The one design call the worker made unilaterally that touches identity — **dropping `'prompt'` from `WebcamSource`** — is correct and well-justified (no function used it, the file-input backend could never honor it) and is now documented in the type. Worth promoting to a Decision once the charter exists.

## Contract & docs fit

**Lives up to the contract:** full unabbreviated names (`getWebcamStreamCapabilities`, `applyWebcamStreamConstraints` — note `Stream` is kept in the name so it is grep-distinct from a hypothetical file-input constraint); `out`-param `Surface` functions (`grabWebcamFrame`, `getWebcamPhotoSurface`); sentinel returns throughout (`null`/`false`/`[]`, never throws on expected failure); types-first in `@flighthq/types`; single `.` export; `sideEffects: false`; opt-in signal cost via `enableWebcamSignals` (no top-level registration); correct teardown-verb choice — `destroyWebcamStream` frees the non-GC camera lock (`destroy*`), `disposeWebcamSignals` detaches-to-GC (`dispose*`); `crate: flighthq-webcam` mirror named in the charter front matter and the Rust map. The `Webcam*` prefix keeps every symbol grep-distinct from `@flighthq/camera` (the 3D camera). Good contract hygiene overall.

**Defects / candidate revisions:**

- **Two internal symbols leak through the package barrel.** `index.ts` does `export * from './webcamStream'`, which re-exports `createWebcamStreamEntity` and `getWebcamStreamRuntime` (and the `WebcamStreamRuntime` interface) from the package root. The status doc calls `webcamStream.ts` an "internal" file and these "internal accessor"s, but `export *` publishes them. A user can now construct a `WebcamStream` with a null placeholder `MediaStream` and reach into the runtime — neither is meant to be public API. Either drop the re-export from `index.ts` (keep the file's exports for the colocated test only) or make them non-`export`. This is the clearest surface-shape defect in the package.
- **`WebcamBackend.requestPermission` doc contradicts its signature.** The interface comment (`67dc46d64:types/src/Webcam.ts`) says "Requests camera permission (may prompt the user on some hosts). **Returns the new state.**" but the method returns `Promise<boolean>`, and the web implementation returns `status.state === 'granted'`. The "returns the new state" wording is stale (a `WebcamPermissionState` return was presumably considered, then `getPermissionState` was split out for that). Fix the comment to "Returns true when granted."
- **Copy-paste comment errors in `WebcamCapabilities`.** `whiteBalanceModes` is commented "ISO sensitivity range" and `zoom` is commented "Zoom range" but sits under a comment block that reads as drifted — the `whiteBalanceModes`/`zoom` doc lines are mismatched to their fields (`67dc46d64:types/src/WebcamCapabilities.ts`). Pure doc errors in the header layer; misleading to a reader navigating `@flighthq/types` as the design surface.
- **`Readonly` entity field mutated by cast.** Three sites cast away `readonly` to flip `active`: `(stream as { active: boolean }).active = false` in `destroyStream` and the track-`ended` handler, and `(recording as { active: boolean }).active = false` in `stopWebcamRecording`. The entity fields are declared `readonly` in the types. This is the established pattern for mutating entity state from the owning package (the field is read-only to _consumers_), but it is a `readonly`-bypass cast that the contract's "Readonly everywhere" rule makes worth a deliberate note/helper rather than an inline cast repeated three times. Candidate: a small internal `setWebcamStreamActive(stream, false)` / mutable-runtime mirror, or document the cast as the entity-mutation idiom.
- **`WebcamStreamRuntime.mediaStream` is `null as any as MediaStream`.** `createWebcamStreamEntity` initializes the non-nullable `mediaStream` slot with a null cast (commented: jsdom lacks `MediaStream`; always overwritten before use). Honest and documented, but it makes the slot type lie — a `MediaStream | null` slot with the null handled at read sites would be truthful. Minor.
- **Package Map line is stale (candidate revision).** `tools/agents/docs/index.md` still lists `@flighthq/webcam` as "take photo / pick image" — the pre-build-out one-shot reading. The package is now a live-streaming webcam library; the Map entry should be expanded (mirroring the `package.json` description) once the charter blesses the scope. Flag for the Map owner.

## Candidate open directions (charter is a stub — these are the questions it should settle)

1. **Scope: one package or two?** Is the file-input _picker_ (`takeWebcamPhoto`/`pickWebcamImage`/ `recordWebcamVideo`) and the live _stream_ (getUserMedia/frame-grab/recording) one cohesive `webcam`, or two subjects? The depth review framed this as the blocking design question; the bundle answered it _de facto_ (both shipped together) but the charter has not blessed it. The Rust port and the `package.json` description both assume "live streaming is in scope" — confirm it as the North star so future work is judged against it.
2. **North star.** Likely: a complete getUserMedia/Capacitor-class capture seam with the `Surface` bridge as the integration point to the SDK pixel layer, sentinels-not-throws, opt-in cost for signals. Confirm.
3. **Convenience setters (`setWebcamTorch`/`setWebcamZoom`/…): ship or not?** The worker deferred them as thin wrappers over `applyWebcamStreamConstraints`. Decide whether the named, discoverable torch/zoom API is part of the bar (most camera libraries expose it) or whether the single constraints function is the intended surface.
4. **Permission-state unification (cross-package).** `WebcamPermissionState` (`'denied'|'granted'|'prompt'|'unknown'`) duplicates `GeolocationPermissionState`. Should the platform suite settle on a shared `PermissionState` in `@flighthq/types`? A types-layer decision that touches every permission-bearing capability.
5. **`webcam-formats` neighbor (EXIF/HEIC).** Approve the deferred `-formats` split and its trigger (metadata weight, not feature count), per the triad/plurality guard.
6. **Frame-loop / zero-copy representation.** Whether `grabWebcamFrame` gains a pooled-scratch / `requestVideoFrameCallback` companion (`startWebcamFrameLoop`) — affects the Rust mirror's frame representation too.
7. **Internal-symbol policy.** Whether `createWebcamStreamEntity`/`getWebcamStreamRuntime` are meant to be public (then document them) or hidden (then stop barrel-exporting them) — settles the barrel leak above.
