## Verdict

`solid — 82/100`. A deliberately narrow, well-shaped velocity-field seam that grew from 8 to 21 exports (plus 2 in `affineVelocity.ts`) this pass: it now carries a complete `Velocity2D` value algebra, first-class angular velocity, `dt`/per-second normalization, and a genuine per-pixel affine reprojection primitive (`getVelocitySampleAt`). Contract hygiene is excellent — out-params, alias-safe by construction, sentinels-not-throws, single `.` export, `sideEffects: false`, types-first in `@flighthq/types`. The 82 (below the worker's self-estimated 91) reflects this review's stricter bar: one real naming/promise mismatch (`contributeAffineVelocity` does **not** produce affine velocity — it produces origin velocity identical to `contributeTransformVelocity`), a structural-type leak on `getVelocitySampleAt`'s matrix parameter, two near-duplicate subtree walkers, and a charter that is a stub so most of "what good means here" is assumed, not confirmed. The code is good and the value-helper surface is canonical; the score is distance-to-authoritative, not a grade on the diff.

## Present capabilities (verified against source)

**Value algebra** (`velocityField.ts`, 21 exports, alphabetized, `describe` blocks mirror them 1:1). The full `Velocity2D` algebra is present and out-param/alias-safe: `addVelocity`, `subtractVelocity`, `scaleVelocity`, `copyVelocity`, `zeroVelocity`, `lerpVelocity`, `normalizeVelocity` (zero-safe), `clampVelocity` (max blur length, zero-safe), `dampVelocity` (EMA across frames), `lengthOfVelocity`, `isVelocityZero(v, epsilon?)`. Each reads inputs into locals before writing `out`, so `out === input` is safe — and the tests assert the aliased case for every mutating helper (`67dc46d64:velocityField.test.ts`).

**Field accumulator** (`velocityField.ts`). `createVelocityField()` allocates `{ dt: 1, frameId: 0, samples: WeakMap }`; `beginVelocityFrame(field, dt?)` advances the frame and records `dt` (defaulting to 1, frame-locked); `contributeVelocity`/`suppressVelocity` set explicit linear velocity and mark `explicitFrameId` so the baseline leaves the source alone regardless of call order; `getVelocity` (out-param, zero for missing/stale), `hasVelocity` (stale-aware, non-zero-aware), and `ensureVelocitySample` (get-or-create) complete the read/write surface. The `WeakMap<object>` source-agnostic keying — credited by the depth review — is preserved.

**Angular velocity** (`velocityField.ts`). `contributeAngularVelocity(field, source, radians)` stores a radians/frame scalar in addition to the linear velocity and marks the source explicit this frame; `getAngularVelocity` reads it with a sentinel `0` for stale/missing. `VelocitySample.angularVelocity` is homed in `@flighthq/types` (`67dc46d64:types/src/Velocity.ts:21-27`).

**Time normalization** (`velocityField.ts`). `getVelocityPerSecond(field, source, out)` divides the per-frame delta by `field.dt`, with a clean `dt <= 0` sentinel-to-zero guard, returning the per-frame value when `dt === 1`. This resolves the depth review's "no dt / undocumented decision" gap and is a zero-breaking-change addition (existing no-`dt` callers are unaffected).

**Affine reprojection primitive** (`affineVelocity.ts`). `getVelocitySampleAt(sample, currentWorldTransform, pointX, pointY, out)` computes `current·p − previous·p` at an arbitrary local-space point — the genuine per-pixel velocity a fragment shader needs for rotation/scale — and returns zero when no previous transform is stored. This is the real payoff of the affine work and is correct (the rotation test at a non-origin point verifies `(-1, 1)` for a 90° turn, `67dc46d64:affineVelocity.test.ts:104-116`).

**Transform-delta contributors** (`transformVelocity.ts`, `affineVelocity.ts`). `contributeTransformVelocity(field, root)` walks a `Transform2DNode` subtree, derives each node's velocity from `world.tx/ty` delta vs. the previous frame, commits the new previous transform, and skips nodes an explicit contributor already set. `contributeAffineVelocity(field, root)` does the same walk and **the same origin-only velocity derivation**, but additionally retains `previousWorldTransform` (it already did) so `getVelocitySampleAt` can be called downstream.

**Types & packaging.** `Velocity2D`, `VelocitySample`, `VelocityField`, and the open `VelocityContributor` function-type contract all live in `@flighthq/types/Velocity.ts`, with the deliberate boundaries (per-frame not per-second; single previous frame; per-instance punted to the kind's velocity writer) documented as explicit missing-by-design notes. `package.json` is `sideEffects: false`, single `.` export, deps limited to geometry/node/types (displayobject is a dev-dep, used only by tests). **Test count verified: 61** (49 + 9 + 3 across the three `*.test.ts`), all `describe` blocks alphabetized and mirroring exports.

## Gaps (vs the AAA motion-vector / velocity-buffer target; charter mostly silent)

- **The affine contributor does not contribute affine velocity (the headline gap).** `contributeAffineVelocity` is the package's named, promised deliverable, but `visitAffineVelocity` (`67dc46d64:affineVelocity.ts:72-87`) derives `sample.velocity` from `world.tx − previous.tx` / `world.ty − previous.ty` — i.e. `current·p − previous·p` evaluated at the **origin** `p = (0,0)`, where the linear part `a,b,c,d` drops out. For a node whose anchor is the origin this is numerically **identical** to `contributeTransformVelocity`. The genuine affine reprojection lives only in `getVelocitySampleAt`, which the contributor never calls into `sample.velocity`. So the _stored_ per-node velocity is still translation-of-origin, not per-pixel-correct for rotation/scale — the exact gap the depth review flagged is closed for the _primitive_ but not for the _contributor_ that bears the "affine" name. The source comment and the status "Concerns" both acknowledge this honestly; it is a real promise/name mismatch, not a hidden bug.
- **No bulk iteration / `forEachVelocity`.** Still no way to enumerate every source moving this frame; a consumer must already hold each source object. The `WeakMap` keying (a credited strength) is what blocks this — the fix is a memory-model decision (hybrid live-list + `WeakMap` retain), correctly deferred in status as needing a design ruling.
- **No multi-frame history / trails.** Only one previous frame is kept; no `enableVelocityHistory` ring for TAA-style temporal reprojection. Reasonable to gate behind opt-in; named but unbuilt.
- **No acceleration / higher-order motion.** `VelocitySample.acceleration` + `contributeAcceleration` / `getAcceleration` (velocity delta across frames) absent. Named in the roadmap as Gold.
- **No buffer-write convention helper.** The screen→buffer scale + Y-axis convention each GL/Wgpu writer bakes in is still un-centralized (`createVelocityWriteParams` / a `@flighthq/types` descriptor). This is the only thing standing between the two backends + the Rust port agreeing on one convention — cross-package.
- **No opt-in signals.** `enableVelocityFieldSignals` (frame-begun, source-suppressed) absent; correctly deferred until a real consumer appears, per the `enable*` rule.
- **No functional/conformance scene.** No `tests/functional/velocity-*` exercising translation/rotation/ scale motion blur; blocked on the GL/Wgpu writers adopting `getVelocitySampleAt` first (cross-package).

## Charter contradictions

The charter is a stub — only "What it is" carries content (seeded from the depth review's domain line); North star, Boundaries, Decisions, and Open directions are all `TODO`. So there is no blessed rule for the code to contradict, and **no charter contradiction exists** in the strict sense. One soft tension is worth surfacing for when the charter is authored: the charter's "What it is" describes the package as the seam _contributed into_ and _read out of_, which the code honors well; it does not yet say whether the _contributors_ are obliged to store per-pixel-correct (affine) velocity or only anchor velocity — which is exactly the ambiguity the `contributeAffineVelocity` naming exposes (see Gaps). The charter should settle that, not the code.

## Contract & docs fit

**Lives up to the contract:** full unabbreviated names (`getVelocityPerSecond`, `lengthOfVelocity`, not abbreviated forms); `out`-param value helpers, every one alias-safe by construction _and_ alias-tested; sentinel returns (`getVelocity`/`getVelocityPerSecond` → zero for stale/missing, `getAngularVelocity` → `0`, `getVelocitySampleAt` → zero with no previous transform) with no throwing; types-first in `@flighthq/types`; single `.` export; `sideEffects: false`; opt-in nothing-registered-at-top-level (there is no registration here, which is correct). `crate: flighthq-velocity` mirror is named in the charter front matter. Contract hygiene is among the cleanest in the surveyed set.

**Defects / candidate revisions:**

- **`contributeAffineVelocity` is a misnomer for what it stores.** As above, the contributor's stored `sample.velocity` is origin/translation velocity, identical to `contributeTransformVelocity`'s — the "affine" adjective describes the _retained matrix that enables `getVelocitySampleAt`_, not the value the function writes. Either (a) make the contributor genuinely store per-pixel velocity at a chosen anchor (e.g. honor `pivotX/pivotY` via `getVelocitySampleAt`), or (b) rename/redescribe it so the name promises only "retains the world transform so per-pixel reprojection is available" — and document that the per-node _scalar_ velocity is anchor velocity. As written, an agent reading the export list would reasonably expect `contributeAffineVelocity` to yield different `getVelocity` results than `contributeTransformVelocity` for a rotating node; it does not. (Decision, not sweep — routed below.)
- **`getVelocitySampleAt`'s `currentWorldTransform` is an inline structural type, not `Readonly<Matrix>`.** `67dc46d64:affineVelocity.ts:35` types the parameter as `Readonly<{ a; b; c; d; tx; ty }>` rather than `Readonly<Matrix>` from `@flighthq/types`. This is a structural-literal leak that the types-layout convention discourages (entity-backed `Matrix` carries binding identity; a `*Like` is the sanctioned structural input). It also means the function silently accepts a non-`Matrix` literal — convenient, but it diverges from the "use `Matrix`/`*Like`, not ad-hoc structural shapes" rule and from how `getNodeWorldTransformMatrix` returns a real `Matrix`. A pure in-package signature tightening (use `Readonly<Matrix>`, or `Readonly<MatrixLike>` if structural input is intended) — sweep-safe.
- **Two near-duplicate subtree walkers.** `visitTransformVelocity` and `visitAffineVelocity` are byte-for-byte identical except that the latter has the same body (the affine branch computes the same `cx = world.tx` / `px = previous.tx` origin delta). Until the affine contributor genuinely differs, the two private walkers (and their two public entry points) duplicate the walk-and-retain logic. If the affine contributor is kept distinct only by name, this is dead duplication; if it is made genuinely affine, the duplication resolves. Flagged as a structural smell tied to the naming defect, not a separate sweep.
- **Package Map omission (admin-doc staleness — confirmed in both trees).** `@flighthq/velocity` is **absent** from the Package Map in `tools/agents/docs/index.md` — verified in the live tree _and_ in `incoming/builder-67dc46d64/head/tools/agents/docs/index.md`. For a package whose entire reason to exist is to be the shared seam between motion sources and the GPU motion-blur velocity writers, the missing entry hides its scope and its relationship to `displayobject-gl`/`-wgpu` `*Velocity` writers and `effects` motion blur. This is a candidate revision to a root-level admin doc — the user's gate, not the reviewer's, and the worker correctly deferred it as cross-package. (The depth review noted the same; it remains true.)
- **Contract doc reconciled (verified fixed).** The depth review's stale `contributeNodeVelocity` / `suppressNodeVelocity` references in `VelocityContributor`'s doc are gone — the comment now references `contributeVelocity` / `suppressVelocity` (`67dc46d64:types/src/Velocity.ts:43`). Good; this earlier finding is closed.

## Candidate open directions (charter is a stub — the questions it should settle)

1. **North star.** What is the durable bar for this seam? Likely: per-frame, alias-safe, allocation-free value algebra; a single source-agnostic accumulator; explicit-wins-over-baseline fencing; and tessellation/per-instance/buffer-write all delegated out. Confirm so future work is judged against it.
2. **Affine contributor semantics (the headline question).** Should `contributeAffineVelocity` store per-pixel-correct velocity (at the node anchor / honoring `pivot`), or is anchor velocity the right per-node scalar and per-pixel reprojection strictly a `getVelocitySampleAt` consumer concern? The name currently over-promises; the charter should rule on what the contributor is obliged to store — and whether it should be renamed or merged with `contributeTransformVelocity`.
3. **Bulk iteration vs. `WeakMap` keying.** Bless `WeakMap`-only (consumers hold sources) or add a live-this-frame iteration list (`forEachVelocity`), which changes the GC/ownership story. The depth review credits `WeakMap` keying as a strength; flipping it is a real memory-model decision.
4. **Buffer-write convention.** Centralize the screen→buffer scale + Y-axis convention (`createVelocityWriteParams` / a `@flighthq/types` descriptor) so the two backends and the Rust port agree — only pays off with cross-package adoption.
5. **History & acceleration scope.** Confirm `enableVelocityHistory` (N-frame ring, TAA) and `acceleration` as opt-in Gold features that stay off the allocation-light default path.
6. **Transform-trait hardening (cross-package).** Both contributors cast children `as unknown as Transform2DNode` because `HierarchyNode` lacks the transform trait. A `isTransform2DNode` guard here, or a typed child accessor in `@flighthq/node`, removes the cast — the cleaner fix is cross-package.
7. **`getVelocitySampleAt` matrix parameter.** `Readonly<Matrix>` (entity-backed) vs. an explicit `Readonly<MatrixLike>` (structural input intended). Decide whether structural input is a sanctioned convenience or a leak to tighten.
8. **Rust parity.** The 15 new TS functions are not yet mirrored in `flighthq-velocity`; the `WeakMap<object>` → `HashMap<u64>` keying divergence (and any iteration-semantics divergence) belongs in the conformance divergence map. Scoped to the `rust` worktree.
